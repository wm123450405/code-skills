# 代码评审清单(内置兜底)

> 由 `code-review` 技能引用。**项目级评审清单应在 `./assistants/rules/review-checklist.md` 中提供**,
> 本文件只放通用的兜底清单,供项目级不存在时使用。

## 应用方式
- 评审每个任务时,按本清单逐项检查
- 任何"不通过"项都形成"发现",记入评审笔记
- 严重程度:必须改 / 建议改 / 可选
- **不要跳过清单项**,即使时间紧

---

## 1. 正确性(P0)
- [ ] 实现了任务所声明的功能(对照 PLAN.md "关键变更")
- [ ] 对应的需求 FR/AC 被满足
- [ ] 边界条件处理:空、零、负、极值、超长、Unicode
- [ ] 异常路径处理:参数错误、依赖不可用、超时、并发
- [ ] 状态机迁移正确(若涉及):每个合法迁移都实现,非法迁移被拒绝
- [ ] 返回值/响应符合详细设计的入参/出参约定

## 2. 安全(P0)
- [ ] 输入校验:长度、类型、格式、范围
- [ ] 鉴权:谁能调用,如何验证
- [ ] 授权:谁能做什么,如何控制
- [ ] 敏感数据:密码/token/手机号/身份证 不在日志/错误信息/URL 中明文出现
- [ ] SQL/NoSQL 注入:参数化查询,无字符串拼接
- [ ] 命令注入:无 shell 拼接,使用安全 API
- [ ] 反序列化:不信任输入,使用安全反序列化器
- [ ] XSS:输出转义,无 dangerouslySetInnerHTML
- [ ] 依赖漏洞:无已知高危 CVE 的依赖
- [ ] 跨站请求伪造(CSRF):状态变更请求有 CSRF 保护
- [ ] 越权:水平/垂直越权检查

## 3. 规范(强制条款)(P0/P2)
> 具体条款从 `./assistants/rules/` 各文件中提取,本节只列通用维度
- [ ] 命名规范(变量/函数/类/常量/文件)符合规则
- [ ] 目录结构与模块边界符合规则
- [ ] 错误处理范式符合规则
- [ ] 日志/可观测性规范符合规则
- [ ] API 风格(REST/gRPC/事件)与错误码规范
- [ ] 数据建模(字段命名/类型/约束/索引)符合规则
- [ ] 测试覆盖率门槛达到

## 4. 详细设计符合度(P1)
- [ ] 函数/类签名与 `plan/RESULT.md` 一致
- [ ] 数据结构字段与 `plan/RESULT.md` 一致
- [ ] 接口入参/出参/错误码与 `plan/RESULT.md` 一致
- [ ] 状态机迁移与 `plan/RESULT.md` 一致
- [ ] 任何偏离都有显式授权(在 `code/RESULT.md` 的 deviations 中)

## 5. 性能(P2)
- [ ] 算法复杂度合理(无明显 O(n²) 或更差)
- [ ] 无 N+1 查询
- [ ] 无循环里的 IO
- [ ] 无同步阻塞关键路径
- [ ] 资源使用合理(连接/锁/内存)
- [ ] 缓存策略与失效策略正确
- [ ] 批量/异步使用得当

## 6. 可维护性(P2)
- [ ] 命名自解释,避免缩写
- [ ] 函数单一职责(不超过一屏)
- [ ] 入参 ≤ 3,超过用对象包装
- [ ] 早返回,避免深层嵌套
- [ ] 注释解释"为什么",不解释"是什么"
- [ ] 公共 API 有 JSDoc/docstring
- [ ] 无魔数/硬编码(应提取为常量)
- [ ] 无重复代码(DRY)
- [ ] 无过度抽象/过早优化

## 7. 测试质量(P1/P3)
- [ ] 正常路径有测试
- [ ] 边界/异常有测试
- [ ] 测试独立(无共享状态)
- [ ] 测试可读(命名表达意图)
- [ ] mock 外部依赖,不 mock 被测对象
- [ ] 不测实现细节,测行为
- [ ] 失败时输出能直接定位问题
- [ ] 关键路径有覆盖(无重大缺口)

## 8. 一致性(P3)
- [ ] 与项目既有代码风格一致
- [ ] 错误处理风格一致
- [ ] 日志格式一致
- [ ] 目录组织一致
- [ ] 命名风格一致

## 9. 接口与上下游一致性(P1)
- [ ] 不破坏其他任务的接口契约
- [ ] 不引入未声明的副作用
- [ ] 不修改不应修改的全局状态
- [ ] 公共 API 变更同步更新文档
- [ ] 错误码/异常类型与项目约定一致

## 10. 文档与代码同步(P3)
- [ ] 改 API 时同步更新 API 文档
- [ ] 改行为时同步更新 README/迁移说明
- [ ] 改部署时同步更新部署文档
- [ ] 必要的 TODO 注释带负责人与时间

---

## 评审结论判定
- **✅ 通过**:无必须改、无任何严重警告
- **⚠️ 条件通过**:无必须改,但有建议改(可由开发者接受/拒绝)
- **❌ 阻塞**:有必须改未解决,或发现需求/设计本身错误

## 评审产物去向
- 派生任务:`./assistants/<版本号>/review/<新任务编码>/RESULT.md`(给 code-it 消费)
- 整体报告:`./assistants/<版本号>/review/<需求编号>/REVIEW-REPORT.md`
- 未派生发现:`./assistants/<版本号>/review/<需求编号>/findings-no-task.md`
