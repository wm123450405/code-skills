# code-review — 代码评审(版本感知)

## 目标
在代码合并前**系统化发现缺陷**,把可执行、可追溯、可派工的评审意见转交给 `code-it` 处理,使代码质量与设计/需求/规范保持一致。

## 适用场景
- 任何进入主干/共享分支的 PR/MR
- 重大重构、关键路径变更、安全敏感改动
- 整个需求的代码全部就绪后做整体评审
- 单个任务的增量评审(可重跑,基于新增的 code/RESULT.md)

## 不适用
- 当前**没有激活的版本工作空间**(请先调 `code-version`)
- 需求尚未在 PLAN.md 中拆分任务
- 任务的开发状态全部为 `待开始` 或 `已取消`
- 单元测试结果尚未确定(测试状态不在 `已运行-通过` / `不适用` / `已运行-失败` 等已落地状态)
- 紧急线上修复(走 hotfix 流程,不在本流水线)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
```
./assistants/
├── rules/                                  # 项目级规范(跨版本,只读)
└── <版本号>/
    ├── RESULT.md                           # 版本看板(本技能追加区段)
    ├── require/<需求编号>/RESULT.md        # 上游需求(只读)
    ├── design/<需求编号>/RESULT.md         # 上游概要设计(只读)
    ├── plan/
    │   ├── <需求编号>/RESULT.md            # 上游详细设计(只读)
    │   └── <需求编号>/PLAN.md              # 上游任务计划(只读,但本技能会追加新任务)
    ├── code/
    │   └── <任务编码>/                     # code-it 产出(只读,作为评审上下文)
    │       └── RESULT.md
    ├── test/
    │   └── <任务编码>/                     # code-unit 产出(只读,作为评审上下文)
    │       └── RESULT.md
    └── review/
        ├── <需求编号>/                     # 本技能整体评审产物,可写
        │   ├── REVIEW-REPORT.md            # 整体评审报告(主产出)
        │   ├── work-log.md
        │   ├── review-checklist-applied.md
        │   └── findings-no-task.md
        └── <新任务编码>/                   # 每个"审查改修"任务一个目录,可写
            ├── RESULT.md                   # 改修要求(给 code-it 直接消费)
            ├── work-log.md
            └── ...
```

- 路径以**当前工作目录(CWD)**为基准
- **本技能不修改** `./assistants/rules/`、`<版本号>/require/`、`<版本号>/design/`、`<版本号>/plan/RESULT.md`、`<版本号>/code/`、`<版本号>/test/` 下的任何内容
- **本技能会修改** `<版本号>/plan/<需求编号>/PLAN.md`:**仅追加**新的"审查改修"任务(不改其他任务、不改结构)
- **本技能会创建** `<版本号>/review/<需求编号>/` 与 `<版本号>/review/<新任务编码>/` 下的文档

## 输入
- **需求编码**(必填)
- **上游需求**:`./assistants/<版本号>/require/<需求编号>/RESULT.md`(必须存在)
- **上游概要设计**:`./assistants/<版本号>/design/<需求编号>/RESULT.md`(必须存在,作为设计基准)
- **上游详细设计**:`./assistants/<版本号>/plan/<需求编号>/RESULT.md`(必须存在,作为详细基准)
- **上游任务计划**:`./assistants/<版本号>/plan/<需求编号>/PLAN.md`(必须存在,作为评审范围)
- **代码改修正文**:`./assistants/<版本号>/code/<任务编码>/RESULT.md`(每个任务,作为评审对象)
- **测试总结(可选)**:`./assistants/<版本号>/test/<任务编码>/RESULT.md`(每个任务,辅助评审)
- **项目级规范**:`./assistants/rules/` 下所有文件(建议存在)
- **当前项目代码**:CWD 下的源文件(实际评审对象)

## 输出
主产出物:
- `./assistants/<版本号>/review/<需求编号>/REVIEW-REPORT.md` —— 整体评审报告
- `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 中追加的"审查改修"类任务
- 每个新任务的 `./assistants/<版本号>/review/<新任务编码>/RESULT.md` —— 改修要求(给 `code-it` 直接消费)

辅助过程文档:
- `<版本号>/review/<需求编号>/work-log.md`
- `<版本号>/review/<需求编号>/review-checklist-applied.md`
- `<版本号>/review/<需求编号>/findings-no-task.md`
- 每个新任务的 `<版本号>/review/<新任务编码>/work-log.md`

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:
- 评审发现汇总 → "评审发现汇总"区段追加
- 派生新任务 → "派生任务记录"追加 + "任务清单"追加新行(由本技能在 PLAN.md 中已追加,本步骤同步到看板)
- 发现的严重缺陷 → "缺陷清单"追加
- 任意阶段 → "变更记录"追加

## 工具使用约定
- 读激活版本:`Read "./assistants/.current-version"`
- 读所有上游:`Read` 读取 `<版本号>/require/` / `design/` / `plan/` / `code/` / `test/` 下的相关文档
- 读规范:`Glob "./assistants/rules/**/*"` + `Read`
- **读源码**:`Glob` / `Grep` / `Read` 读 CWD 下的实际代码
- 写文档:`Write` / `Edit`
- 与用户澄清:优先 `AskUserQuestion`;只在需要判断优先级或冲突时用

## 工作流程

### 步骤 0 — 版本上下文检测(强制前置)
1. 读取 `./assistants/.current-version`
2. **文件不存在** → 立即停下,告知用户先调 `code-version`
3. 读取内容,记为 `<版本号>`,后续所有路径用 `assistants/<版本号>/...`

### 步骤 1 — 收集需求编码
- 若用户未提供,主动询问
- 校验 `./assistants/<版本号>/require/<需求编号>/RESULT.md` 与 `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 都存在,否则:
  > 错误:上游缺失。请先运行 `code-require` 与 `code-plan` 产出必要文档。

### 步骤 2 — 定位 / 创建工作目录
- 检查 `./assistants/<版本号>/review/<需求编号>/` 是否存在
- 不存在 → `mkdir -p` 创建
- 存在 → 复用(意味着重跑,基于上次的发现做增量评审)

### 步骤 3 — 读取项目级规范
- `Glob "./assistants/rules/**/*"`
- 不存在/为空 → `AskUserQuestion` 询问是否继续
- 存在 → 逐个 `Read`,归类与摘要写入 `work-log.md` 起始区
- 特别关注 `./assistants/rules/review-checklist.md`,本技能会优先采用

### 步骤 4 — 读取上游文档
依次:
1. `Read "./assistants/<版本号>/require/<需求编号>/RESULT.md"` —— 提取 FR / NFR / AC / 验收标准
2. `Read "./assistants/<版本号>/design/<需求编号>/RESULT.md"` —— 提取模块拆分 / 接口 / 数据结构 / 决策
3. `Read "./assistants/<版本号>/plan/<需求编号>/RESULT.md"` —— 提取详细设计 / 接口细节 / 异常处理 / 安全要求
4. `Read "./assistants/<版本号>/plan/<需求编号>/PLAN.md"` —— 提取所有任务、其**开发状态**与**测试状态**、任务编号、关联任务

写入 `work-log.md`。

### 步骤 5 — 列出待评审任务
从 `PLAN.md` 的"任务总览"中筛出**可评审**的任务:
- 开发状态 ∈ {`已完成`, `进行中`}(`进行中` 可评审但需注明)
- 测试状态 ∈ {`已运行-通过`, `已运行-失败`, `不适用`}(不能是 `未编写` / `已编写` / `阻塞`)
- 排除:开发状态 = `已取消`、任务类型 = `文档` 且测试状态 = `不适用` 的纯文档任务(可选评审)

把"待评审任务清单"写入 `REVIEW-REPORT.md` 的"评审范围"部分。

### 步骤 6 — 读取每个任务的代码改修正文
对每个待评审任务:
1. `Read "./assistants/<版本号>/code/<任务编码>/RESULT.md"` —— 实际改了什么、关键决策、已知问题
2. `Read "./assistants/<版本号>/test/<任务编码>/RESULT.md"`(若存在)—— 测试覆盖、覆盖率、发现的代码 bug
3. 根据 `code/RESULT.md` 的"详细改动"部分,`Read` CWD 下的实际源码文件

把"读了哪些文件"记入 `work-log.md`。

### 步骤 7 — 加载评审清单
- 优先:读取 `./assistants/rules/review-checklist.md`(项目级评审清单)
- 兜底:使用本技能内置的 `checklists/review-checklist.md`
- 写入 `review-checklist-applied.md`,本次评审实际应用了哪些检查项

### 步骤 8 — 逐任务评审
对每个待评审任务,按以下顺序检查:

**8.1 正确性** —— 代码是否真的实现了任务所声明的功能
- 对应需求中的 FR / AC
- 对应详细设计中的关键逻辑
- 边界条件(空、零、负、极值、超长、并发、超时)是否处理
- 异常路径是否覆盖
- **不通过** → 标"必须改"

**8.2 规范遵循** —— 是否符合 `./assistants/rules/`
- 命名规范(变量/函数/类/文件/常量)
- 目录结构与模块边界
- 错误处理范式
- 日志/可观测性规范
- API 风格(REST/gRPC/事件/错误码)
- 数据建模(字段命名/类型/约束/索引)
- 安全规范(鉴权/加密/防注入/敏感数据)
- 性能规范(缓存/批量/异步)
- 测试规范(覆盖率门槛)
- **不通过** → 视条款的强制程度标"必须改"或"建议改"

**8.3 详细设计符合度** —— 是否与 `plan/RESULT.md` 一致
- 函数/类签名是否一致
- 数据结构字段是否一致
- 接口入参/出参/错误码是否一致
- 状态机迁移是否正确
- 偏离了设计但有合理理由 → 记录为"用户授权的偏离"(若 code/RESULT.md 已有则接受,否则询问)
- **不一致且无授权** → 标"必须改"

**8.4 安全** —— 是否引入安全风险
- 输入校验是否充分
- 鉴权/授权是否到位
- 敏感数据处理(密码/token/手机号)
- SQL/NoSQL 注入、命令注入、反序列化漏洞、XSS
- 依赖已知 CVE
- **不通过** → 标"必须改"

**8.5 性能** —— 是否引入明显性能问题
- 算法复杂度是否合理
- 是否有 N+1 查询、循环里的 IO、同步阻塞
- 资源使用(内存/连接/锁)是否合理
- **不通过** → 视影响标"必须改"或"建议改"

**8.6 可维护性** —— 是否便于他人理解和扩展
- 命名是否自解释
- 函数是否单一职责
- 是否有不必要的复杂度、过度抽象
- 注释是否解释"为什么"
- 魔数/硬编码
- **不通过** → 标"建议改"

**8.7 测试质量** —— 测试是否真正起到守护作用
- 是否有正常路径测试
- 边界/异常是否有测试
- 测试是否独立、可读
- mock 是否滥用
- 是否有未覆盖的关键路径
- **不通过** → 视覆盖度标"必须改"或"建议改"

**8.8 一致性** —— 与项目既有代码风格是否一致
- 命名、目录、文件组织
- 错误处理风格
- 日志格式
- **不通过** → 标"建议改"

**8.9 与上下游任务的一致性**
- 是否破坏其他任务的接口契约
- 是否有未声明的副作用
- **不通过** → 标"必须改"

每条发现记入"评审笔记",包含:
- 位置(文件:行号 或 文件:函数名)
- 类别(正确性/规范/设计/安全/性能/可维护性/测试/一致性/接口)
- 严重程度(必须改/建议改/可选)
- 描述
- 建议的改修方案

### 步骤 9 — 分类发现,决定是否派生新任务
- **必须改** → 派生新的"审查改修"任务
- **建议改** → 询问用户:派生新任务 / 留作 follow-up 记录到 `findings-no-task.md`
- **可选** → 记录到 `findings-no-task.md`,不派生任务
- **发现的代码 bug / 设计问题** → 若不属于本任务范围(而是上游设计错误),记录到 `REVIEW-REPORT.md` 的"超出本次评审范围"区,建议回 `code-plan` / `code-design`

**派生新任务的合并原则**:
- 同一文件、相同类别的多个发现 → 合并到一个任务
- 跨文件但相关的发现 → 可合并,但任务描述中明确列出所有文件
- 超过 15 条独立发现 → 询问用户:是合并成少量大任务 / 还是拆为多个小任务

### 步骤 10 — 分配新任务编号并更新 PLAN.md
1. 读取 `PLAN.md`,找到当前最大任务序号 N
2. 为每个新"审查改修"任务分配 `N+1`, `N+2`, ...
3. 决定每个新任务的**类型**:
   - 大多数 → 类型=`修改`,触发/来源=`审查改修`
   - 涉及新增代码 → 类型=`新增`
   - 是 bug 修复 → 类型=`修复`
4. 决定每个新任务的**关联任务**:
   - "关联任务"字段 = 被其修正的原任务编号(可多个)
   - "触发任务"字段 = 同上(更明确的字段)
5. **追加到 `PLAN.md` 的"任务总览"表**:
   - 找到"任务总览"表,在最后一行后插入新行
   - 同时**追加对应的"### <任务编号>:[类型] 标题"任务详情节**到"任务详情"末尾
6. **在变更记录中追加**:
   ```
   YYYY-MM-DD HH:mm  增量更新(审查)  评审发现 N 个问题,新增任务 T-XXX, T-YYY, ...  <评审者>
   ```
7. **版本号大版本递增**(v2 → v3)
8. **不修改**任何已有任务,不修改 `PLAN.md` 任何其他字段

### 步骤 11 — 撰写每个新任务的 review/RESULT.md
对每个新"审查改修"任务,创建 `./assistants/<版本号>/review/<新任务编码>/` 目录并写 `RESULT.md`:

按 `templates/REVIEW-FIX.md` 的结构生成,内容:
1. 任务信息(任务编码、类型、触发/来源、触发时间、严重程度)
2. 触发的原任务(被审查发现问题的任务,可能多个)
3. 问题清单(每条问题带位置、类别、描述、改修建议)
4. 应当改修的文件(每个文件:现状、应改为、理由)
5. 验证手段
6. 关联依据(规范条款、设计章节、需求 AC)
7. 不需要做的(避免 code-it 越界)

> **`code-it` 消费这个 RESULT.md** 作为本任务的全部输入,**不读** `<版本号>/plan/RESULT.md`(因为本任务是修补,不是新设计)。

### 步骤 12 — 撰写整体 REVIEW-REPORT.md
按 `templates/REVIEW-REPORT.md` 的结构生成,内容:
1. 评审信息(需求编码、版本号、评审时间、评审范围)
2. 评审清单(实际应用的检查项)
3. 任务评审结果总览(每个任务一行:状态、通过/失败/警告计数、发现数)
4. 发现汇总(按类别/严重程度)
5. 派生的新任务列表(任务编码、标题、严重程度)
6. 未派生任务的发现(轻微/可选)
7. 超出本次评审范围的发现(回 code-plan / code-design)
8. 整体结论(是否可合并 / 阻塞项 / 风险)
9. 变更记录

### 步骤 13 — 同步版本看板(强制)
1. `Read "./assistants/<版本号>/RESULT.md"`
2. **更新"任务清单"区段**:本步骤在 PLAN.md 中追加的新任务行,同步追加到看板"任务清单"(字段对齐模板:任务编号 / 需求 / 类型 / 触发/来源=审查改修 / 标题 / 开发状态=待开始 / 测试状态=未编写 / 关联任务=原任务 / 涉及文件=留空)
3. **追加"评审发现汇总"区段**:
   - 每条评审发现一行:评审 ID(F-001, F-002, ...) / 需求 / 任务 / 维度 / 级别 / 描述 / 派生改修任务 / 状态
   - 严重程度统计(必须改 / 建议改 / 可选)
4. **追加"派生任务记录"区段**:
   - 每个派生的"审查改修"任务一行:派生任务 / 关联原任务 / 派生时间 / review 来源(指向本 REPORT) / 状态
5. **追加"缺陷清单"区段**(若有发现的严重缺陷):
   - 缺陷编号 / 严重度(P0/P1/P2) / 标题 / 状态=待修复 / 报告时间 / 关联任务 / 修复提交=留空
6. **追加"变更记录"**:
   ```
   YYYY-MM-DD HH:mm  评审发现  <需求编码> 评审完成(共 N 条发现,派生 M 个"审查改修"任务)  <需求编码>
   ```
7. 更新看板文档头"最近更新"

### 步骤 14 — 完善过程文档
- `work-log.md` 收尾
- `findings-no-task.md` 完整记录未派生任务的发现

## 衔接
- **下游**:
  - `code-it` 消费 `./assistants/<版本号>/review/<新任务编码>/RESULT.md` 执行每个"审查改修"任务
  - 评审通过后:合并 → 部署
- **上游**:`code-version`(必须);`code-it` / `code-unit` 的产出 + 需求/设计/计划文档
- **横向**:可能与 `code-require` / `code-design` 联动 —— 若评审发现需求/设计本身有错,记录到 REVIEW-REPORT.md 的"超出本次评审范围",建议用户回到对应上游

## 不要做的事
- 不要在没有 `./assistants/.current-version` 的情况下继续执行
- 不读上游文档就直接评审
- 评审时改 CWD 下的源码(那是 code-it 的工作)
- 评审时改 `<版本号>/plan/RESULT.md` / `<版本号>/code/RESULT.md` / `<版本号>/test/RESULT.md`
- 把发现的"建议改"全部强派生为任务(询问用户)
- 在 PLAN.md 中修改除"追加新任务"之外的任何内容
- 跳过 `./assistants/rules/review-checklist.md`(若存在)而只用内置清单
- 给出模糊批评(必须给出可执行建议)
- 把 review/RESULT.md 写成"建议改用更安全的写法"这种程度(必须具体到文件:行 + 改修方案)
- 修改 `./assistants/<版本号>/RESULT.md` 中非本技能负责的区段(本技能负责:任务清单 / 评审发现汇总 / 派生任务记录 / 缺陷清单 / 变更记录)
