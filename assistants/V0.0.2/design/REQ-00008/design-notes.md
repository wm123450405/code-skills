## 2. 候选方案 + 选定方案 + 否决理由

### DQ-1:SKILL.md 修改方式

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** 整体重写 | 抛弃既有 SKILL.md,写新 SKILL.md(同时支持 2 模式) | 章节骨架完全重设计,可读性高 | 违反 FR-7.AC-7.2(frontmatter 不变)+ FR-7.AC-7.4(模式 1 路径不重写);风险大 |
| **B** ✅ 增量追加 | 既有 SKILL.md 不动,在"步骤 1 解析参数"后插入"步骤 1.5 模式选择",新增"步骤 2 整版本模式"分支 | 模式 1 行为零变化;风险最小;frontmatter 不变 | 章节略长,但可读性可接受 |
| C | 新建独立技能 `code-review-all` | 物理隔离清晰 | 违反"增加整版本模式是 `code-review` 的扩展"语义;增加新技能目录需 marketplace.json 同步,违反 FR-8.AC-8.1 |

**选定**:B(增量追加)

**依据规范**:
- `skill-conventions §规则 1`:frontmatter 必含 name + description(既有已合规,本设计**不**改)
- FR-7.AC-7.2:`code-review` 现有 frontmatter **不**变
- FR-7.AC-7.4:本需求**仅**新增"无参入口"的处理逻辑,模式 1 路径**不**重写
- NFR-2:增量修改 SKILL.md,不重写模式 1

### DQ-2:整版本模式与单需求模式的"代码复用"边界

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** ✅ 完全复用 + 循环 | 整版本模式 = 步骤 1(过滤需求) + 步骤 2 ~ 步骤 15 循环(对每个被评审需求复用模式 1) | 0 重复代码;FR-6.AC-6.1/AC-6.2 强保证(`code-auto` 解析逻辑不变);NFR-2 友好 | 循环逻辑略复杂 |
| **B** 部分复用 | 整版本模式**不**循环,而是把所有被评审需求的发现合并为 1 份 REPORT | REPORT 字面量小(1 份而非 N 份) | 违反 FR-3.AC-3.1(每个被评审需求生成 1 份独立 REPORT)+ FR-6.AC-6.1(与模式 1 内容完全一致) |
| C | 完全独立 | 物理隔离清晰 | 100% 重复代码,违反 NFR-1 零重复 + NFR-2 增量修改 |

**选定**:A(完全复用 + 循环)

**依据规范**:
- FR-3.AC-3.1:每个被评审需求生成 1 份 `REVIEW-REPORT.md`,内容与"单需求模式"**完全一致**(复用模式 1 的输出)
- FR-6.AC-6.1:整版本模式生成的 `REVIEW-REPORT.md` 内容与模式 1 完全相同(无字段差异)
- FR-6.AC-6.2:`code-auto` 解析"必须改列表"的逻辑**不**变
- NFR-2:增量修改 SKILL.md

### DQ-3:聚合文件 REVIEW.md 的位置

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** ✅ 版本顶层(`./assistants/<版本号>/REVIEW.md`) | 与 `RESULT.md` 同级 | 看板模板约定一致(顶层);整版本模式"整版本"语义直观;聚合文件**不**进 `review/` 子目录(NFR-6 强约束) | 顶层文件数 +1(共 2 个顶层:.RESULT.md + REVIEW.md) |
| **B** | `review/ 子目录(review/ALL/REVIEW-REPORT.md)` | 与模式 1 路径同根 | 违反 NFR-6;`review/` 现有结构是"按需求/任务编号分子目录",插入 `ALL/` 不一致 |
| C | 二者并存(版本顶层 + review/ 子目录双写) | 2 份备份 | 违反 NFR-3 幂等(多次执行需 2 份同步);冗余 |

**选定**:A(版本顶层)

**依据规范**:
- 用户 Q-2 锁定:"聚合文件 = `./assistants/<版本号>/REVIEW.md`"
- NFR-6:聚合文件位置约束 — `REVIEW.md` 位于 `./assistants/<版本号>/REVIEW.md`(与 `RESULT.md` 同级);**不**在 `review/` 子目录

### DQ-4:多次执行整版本模式的幂等性

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** ✅ 覆盖(REVIEW.md + N 份 REVIEW-REPORT.md 都覆盖) | 多次执行结果**幂等** | 实现简单;NFR-3 强保证;与"前次比较"语义自洽 | 丢失前次历史(但需求分析阶段无"前次历史"需求) |
| **B** 追加 | 多次执行**不**覆盖,而是追加 | 保留历史 | 违反 NFR-3 幂等;聚合文件可能膨胀;`code-auto` 解析逻辑需识别"哪些是最新" |

**选定**:A(覆盖)

**依据规范**:
- NFR-3:整版本模式多次执行结果**幂等**(REPORT 覆盖)
- 用户 Q-2 锁定:"多次执行整版本模式更新修改这个文件"

### DQ-5:派生任务的写入策略

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** | 本设计**自己**生成新 TASK 编码(扫 `code-plan` 既有 TASK 编号取 max+1) | 整版本模式完全自包含 | 重复 `code-plan` 既有逻辑;违反 FR-5.AC-5.2(`code-review` 自身**不**生成编码,而是**追加**到 `PLAN.md` 任务总览的最末行) |
| **B** ✅ 沿用 `code-plan` 既有规则 | 本设计**不**生成 TASK 编码,而是**追加**到对应需求的 `PLAN.md` 任务总览的最末行(由 `code-plan` 既有逻辑决定编号) | 单一权威源;FR-5.AC-5.2 强保证 | 需要 `code-plan` 既有逻辑已支持"追加派生任务" — 已有(模式 1 已落地) |

**选定**:B(沿用 `code-plan` 既有规则)

**依据规范**:
- FR-5.AC-5.2:派生任务编码生成沿用 `encoding-conventions §规则 4`(`code-plan` 阶段生成);`code-review` 自身**不**生成编码,而是**追加**到 `PLAN.md` 任务总览的最末行
- FR-5.AC-5.3:`PLAN.md` 中"任务总览"区段被追加任务行(由 `code-review` 写,而不是 `code-plan`)
- `encoding-conventions §规则 4`:实施流程 — 任务 ID 由 `code-plan` 生成,`code-review` 不应重复

### DQ-6:整版本模式与 `code-auto` 评审循环的关系

| 候选 | 描述 | 优点 | 缺点 |
| --- | --- | --- | --- |
| **A** | `code-auto` 评审循环**改用**整版本模式(一次性评审所有已完成需求) | 性能更好(1 次评审 vs N 次) | 违反 NFR-4(`code-auto` 评审循环**继续**用模式 1);`code-auto` 派生任务逻辑针对单需求,改用整版本模式需重大改造;违反"模式 1 是 `code-auto` 内部消费的契约"(FR-6.AC-6.2) |
| **B** ✅ `code-auto` **继续**用模式 1;整版本模式由用户手动调(用于"全版本最后一次评审"场景) | 0 风险;NFR-4 强保证;`code-auto` 内部逻辑零变化 | 整版本模式不参与自动循环 |

**选定**:B(手动)

**依据规范**:
- 用户 Q-5 采纳默认:`code-auto` 继续用模式 1;整版本模式由用户手动调
- NFR-4:`code-auto` 评审循环**继续**用模式 1(本需求**不**触发 `code-auto` 升级)

## 5. 风险与缓解

| 风险 | 严重度 | 缓解 |
| --- | --- | --- |
| **R-1** SKILL.md 增量追加错位(插入位置不当影响既有行为) | 高 | Edit 工具严格按锚点"步骤 1 解析参数"插入"步骤 1.5 模式选择";插入后用 Grep 检查既有章节未变;不重写既有章节字面 |
| **R-2** 整版本模式循环时某需求评审失败导致整体中断 | 中 | 模式 1 既有 SKILL.md 已有"评审内部失败 → 跳过该需求 + 报告"逻辑(边界 E-2);整版本模式直接复用,无新增风险 |
| **R-3** 多次执行整版本模式产生重复派生任务 | 中 | NFR-8 实现去重键 `(需求编码, 发现描述前 50 字)`;同描述发现**不**重复追加任务行 |
| **R-4** `REVIEW.md` 聚合文件膨胀(发现多时) | 低 | NFR-7 实现跨需求去重;聚合文件字数与 N 个 `REVIEW-REPORT.md` 之和近似,但**不**重复 |
| **R-5** 派生任务追加到 PLAN.md 时位置错乱(插在中间而非末尾) | 低 | Edit 工具严格按"任务总览"表的最后一行后追加;不修改既有任务行;不修改其他字段 |
| **R-6** 看板同步不完整(整版本模式漏同步某个区段) | 低 | 模式 1 既有同步逻辑覆盖 5 区段(任务清单 / 评审发现汇总 / 派生任务记录 / 缺陷清单 / 变更记录);整版本模式不触发额外区段(NFR-7 + FR-7.AC-7.1) |
