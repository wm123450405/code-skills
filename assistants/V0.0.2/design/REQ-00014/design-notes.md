# 设计笔记 — REQ-00014

更新时间:2026-06-05 12:35
版本:V0.0.2
需求:REQ-00014(优化技能 `/code-plan` 的任务拆分维度)

## 核心目标

在 `plugins/code-skills/skills/code-plan/SKILL.md` §10A 的"任务拆分原则"段(当前 L195-199,4 行),**完全替换**为新原则:
- **核心原则:按"功能点"拆分**(1 个任务 = 1 个用户完整可用的能力 = 展示效果 + 功能逻辑 + 使用说明)
- **架构任务作为首个任务**(条件触发:较高扩展性/可维护性/可配置性 + 较高复杂度,或涉及未来不同实现方式,或对接不同三方组件)

## 关键设计决策

### 决策 D-1:§10A 4 行原则**完全替换**(Q-D1 用户锁定)

- **决策**:**完全替换**原 4 行原则
- **理由**:
  1. 用户在步骤 8A 澄清中明确选择"完全替换"
  2. 新原则与原"顺序友好"原则有冲突(按功能点拆 vs 按层拆)— 不能共存
  3. 既有 REQ-00007 偶然"功能点化"的成功案例证明新原则可行
- **影响范围**:L195-199 共 4 行
- **保留行**:L201-239(任务类型 / 编号 / 双状态 / 触发/来源)全部**不**修改
- **依据**:`skill-conventions §规则 1` 不约束工作流文字;`dashboard-conventions §规则 1` **不**触发(本优化不改字段)

### 决策 D-2:新原则 = 2 个子节

- **决策**:**2 个子节** = "核心原则:按功能点拆分" + "架构任务作为首个任务(条件性)"
- **理由**:
  1. 2 个子节对应需求 FR-1 + FR-2,语义清晰
  2. 架构任务作为"条件性"——简单需求可省略(避免过度工程)
  3. 既有"任务类型 / 编号 / 双状态 / 触发/来源"作为后续子节保留(沿用)
- **影响**:**0**(纯文字增改,不改字段)

### 决策 D-3:对既有 7 个 PLAN **不**追溯(Q-A3 锁定 A)

- **决策**:**不**追溯重拆既有 7 个 PLAN(REQ-00004/05/06/07 等)
- **理由**:
  1. 重拆会破坏历史状态(开发/测试/提交哈希)— 不可逆
  2. 既有 PLAN 已完成实施,无追溯必要
  3. 实施成本高
- **影响**:**0**(既有 PLAN 保持原样)
- **依据**:Q-A3 用户锁定 A
- **新规则生效范围**:**仅** V0.0.2 未来所有需求(REQ-00008+)

### 决策 D-4:不修改其他 12 个 `code-*` 技能 SKILL.md(FR-4 强约束)

- **决策**:**不**修改其他 12 个技能
- **理由**:
  1. 本优化**只**改 `code-plan` 的"拆分"行为,不改其他技能的"读"或"实施"行为
  2. 任务编号格式 / 双状态 / 触发/来源 **不**修改(沿用既有)
  3. 其他 8 个技能(`code-design` / `code-it` / `code-unit` / `code-review` 等)读 PLAN.md 的字段语义**不**变
- **影响**:**0**(无联动修改)

### 决策 D-5:不触发 `dashboard-conventions §规则 1` 三同步

- **决策**:**不**触发三同步
- **理由**:
  1. 本优化**不修改**看板字段定义(任务编号 / 双状态 / 触发/来源 **不**变)
  2. 本优化**不修改** `version-RESULT.md` 模板(看板样板)
  3. 本优化**不修改** `CLAUDE.md` 看板字段约定段
  4. 本优化**不修改** `dashboard-conventions.md` 自身
- **影响**:**0**(强约束"不触发三同步"达成)
- **依据**:`dashboard-conventions §规则 1` 的"强约束触发条件"是"字段扩展",本优化**不**满足

## 修改后 §10A 草稿(供 code-it 实施用)

```markdown
### 步骤 10A — 任务拆分

**任务拆分原则(REQ-00014 优化后,2026-06-05 生效)**:

#### 核心原则:按"功能点"拆分(必须)

- **1 个任务 = 1 个完整的功能点** = 1 个用户完整可用的能力
- **完整一套**:同一个功能点的**展示效果 + 功能逻辑 + 使用说明**必须在**同一个任务**中完整呈现
- **不拆分到多个任务**:例:"用户登录功能"的 UI / 鉴权逻辑 / README 入门说明 必须在 1 个任务中
- **可独立验证**:每个任务有清晰的"完成定义" — 既包含"产物是什么"也包含"产物如何被验证"
- **0.5–2 天内可完成**

#### 架构任务作为首个任务(条件性)

- **触发条件**(任一满足):
  - **触发 1**:**较高扩展性 / 较高可维护性 / 较高可配置性** + **较高复杂度**
  - **触发 2**:**涉及未来可能的不同实现方式**(为未来切换预留扩展点)
  - **触发 3**:**对接不同三方组件**(为未来替换/升级预留抽象层)
- **架构任务内容**:**架构设计 + 架构骨架** = 抽象层 + 接口契约 + 扩展点(**不**实现具体业务)
- **位置**:**首个任务**(TASK-XXX-00001),后续任务在此架构上填充功能点
- **可省略**:简单需求(单一功能点 + 低扩展性 + 无三方组件)可**不**插入架构任务

#### 生效范围

- **仅** V0.0.2 未来所有需求(REQ-00008+)应用本规则
- 既有 7 个 PLAN(REQ-00004/05/06/07 + 部分其他)**不**追溯重拆
- **不**修改其他 12 个 `code-*` 技能(任务编号/双状态/触发来源 **不**变)

---

**任务类型**:
- `新增`:全新代码/接口
- `修改`:对已有代码/接口的扩展或行为变更
- `重构`:不改外部行为的内部结构调整
- `修复`:针对已知 bug 的修复
- `测试`:独立提交的测试补齐
- `文档`:独立提交的文档更新

**任务编号**:
- 格式:`TASK-(REQ|BUG)-NNNNN-NNNNN`,如 `TASK-REQ-00001-00001`、`TASK-BUG-00001-00001`
- 任务序号自 00001 起递增,五位补零
- 首次拆分时,按"应被编码的先后顺序"分配序号(若首个任务是架构任务,则架构任务=00001)
- 任务类型作为前缀标签写在标题里(如 `[新增] 用户登录接口`),不参与编号

**任务双状态字段(每个任务)**:开发状态 与 测试状态
- **开发状态**(必填,初值 = `待开始`):
  - `待开始` / `进行中` / `已完成` / `已取消` / `阻塞` / `待重新评估`
  - 含义:代码本身是否被实现
- **测试状态**(必填,初值 = `未编写`):
  - `未编写` / `已编写` / `已运行-通过` / `已运行-失败` / `不适用` / `阻塞`
  - 含义:单元测试是否被编写/运行/通过

**任务双状态的初始化**:
- **代码类任务**(类型 = 新增/修改/重构/修复/测试):开发状态=`待开始`,测试状态=`未编写`
- **纯文档任务**(类型 = 文档):开发状态=`待开始`,测试状态=`不适用`(在"不适用理由"中注明"纯文档任务")
- 若任务范围明确不含可测代码(如纯配置变更),可在 `code-plan` 或 `code-it` 中讨论后设为 `不适用`

**双状态语义**:**任务"真正可发布" = 开发状态=已完成 ∧ 测试状态∈{已运行-通过, 不适用}**。只有两轴同时满足,任务才算真正完成。

**任务触发/来源字段(每个任务)**:回答"为什么创建本任务"
- 值:`需求新增` / `需求变更` / `需求撤回` / `设计变更` / `规范变更` / `代码变更` / `主动优化` / **`审查改修`** / `缺陷修复` / `性能优化` / `安全加固` / `测试补齐` / `其他`
- **关键差异**:`code-it` 执行时,根据 `触发/来源` 决定从哪里读输入
  - 大多数情况 → `./assistants/<版本号>/plan/<需求编号>/RESULT.md`(详细设计)
  - **`触发/来源=审查改修`** → `./assistants/<版本号>/review/<任务编码>/RESULT.md`(改修要求) ★
  - `触发/来源=需求撤回` → 任务标"已取消",无输入
- **填写规则**:
  - `code-plan` 首次拆分时:为每条新任务标注"是什么触发的"
  - `code-review` 派生新任务时:必须填 `审查改修`,并在"关联任务"指向被修正的原任务
  - 增量更新时:可改,但要写明理由(写到变更记录)
```
