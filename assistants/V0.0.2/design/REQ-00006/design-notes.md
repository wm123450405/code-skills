# 设计笔记 — REQ-00006

更新时间:2026-06-04 16:48

## 关键设计问题清单

本设计的关键决策点(只列"模型 + 选项 + 选定方案 + 理由");需求 v1 已锁定 4 项(Q-1/Q-2/Q-3/Q-4),设计阶段只需在"实现侧"做出剩余决策。

### D-1:`code-publish` 技能在 `code-*` 管道中的位置

- **候选**:
  - A. 独立第 9 个 `code-*` 技能,与 `code-version`/`code-init`/`code-rule` 同为"横向/支线"工具(不在主管道里)
  - B. 在主管道末尾,作为 `code-review` 之后的第 8 步(主管道延长)
  - C. 隐藏在 `code-review` 的"全通过"分支末尾(由 review 触发)
- **选定**:A(独立第 9 个 `code-*` 技能,横向/支线)
- **理由**:
  - 用户需求文档(REQ-00006 §1)直接将其定位为"覆盖开发完成 → 部署上线"的最后一步;独立技能符合用户口语化的"使用发布部署技能"语义
  - 主管道("code-version → code-require → ... → code-review")每条都是"按需求/任务"运转,**code-publish 是"按版本"运转**,粒度不一致,不宜插入主管道
  - 与 V0.0.2 已有的 `code-dashboard`(REQ-00004 引入,同样按版本运转)同位,2 个"版本级"技能与 7 个"需求/任务级"技能形成清晰的两层
  - 不依赖 review 通过 — 即使本版本没有任何需求(纯运维版本)也能调
- **依据规范**:`skill-conventions.md §规则 1`(独立技能要求独立 SKILL.md,name 与目录名一致)
- **被否决理由**:
  - B 项:主管道不应混合"按需求"与"按版本"粒度
  - C 项:用户语义是"主动调用一个技能",不是"review 副作用"

### D-2:前置检查的解析实现路径

- **候选**:
  - A. 用 Markdown 表格"按行"硬解析(每行 `|` split → 列字段 → 状态校验)
  - B. 引入第三方 Markdown 解析库(remark / mistletoe)
  - C. 转用 JSON/YAML 作为"状态镜像",看板写时同步写一份镜像,publish 读镜像
- **选定**:A(按行硬解析)
- **理由**:
  - NFR-1 锁定零新增依赖 → 排除 B
  - C 会强制改写 `code-version`/`code-require`/`code-plan`/`code-it`/`code-unit`/`code-fix`/`code-review` 7 个上游技能,触发 `dashboard-conventions.md §规则 1` 大规模联动 → 违反 NFR-2(纯只读) + FR-8.AC-8.2(不修改其他 8 技能)
  - V0.0.1 看板的表格结构已稳定 4 个月(REQ-00001 ~ REQ-00006 全部沿用),按行解析的健壮性已被实证
- **依据规范**:`dashboard-conventions.md §规则 1`(看板字段约定 3 处同步 — A 不触发,B/C 都触发)
- **实现要点**:
  - 锚点正则:`^## 需求清单$` / `^## 任务清单$` / `^## 缺陷清单$` → 到下一个 `^## ` 为该区段
  - 行解析:跳过 `|---|---|...|`(分隔行)和 `**统计**:...`(汇总行),只处理 `| .. | .. | .. |` 真数据行
  - 列识别:用首行 `| 字段A | 字段B | ... |` 提取列名 → 后续行按列名取值

### D-3:基线版本识别规则

- **候选**:
  - A. 规则 1:目录扫描 — 取 `./assistants/<all version dirs>` 中**字典序最小**的版本(假设 SemVer 排序 ≈ 字典序)
  - B. 规则 2:看板"里程碑"段含 `M0:工作空间就绪` 且全部里程碑为"初始" / 无下游里程碑
  - C. 规则 3:版本目录下无 `code/` / `test/` 子目录(说明从未编码)
- **选定**:A(目录扫描 + 字典序最小)
- **理由**:
  - NFR-7 显式锁定"采用规则 1(最简),规则 2/3 留作 v2"
  - 当前项目实际版本号为 `V0.0.0` / `V0.0.1` / `V0.0.2`,字典序最小确实是 V0.0.0(也是基线),规则简单可验证
  - 失败模式有限:若用户使用奇异版本号(如 `2026-Q2`),退化为"字典序对比",但**不致命**(只是 UPDATE.md 多写或少写一份,用户可以手动删)
- **依据规范**:无(本项目无"版本号格式"规范)
- **回退路径**:v2 升级为规则 2 或 3,或加 `--baseline` 参数显式声明

### D-4:`code-publish` 自身的输入 — 是否接受版本号参数

- **候选**:
  - A. 不接受 — 始终用 `.current-version`
  - B. 接受可选位置参数 `code-publish <版本号>`,缺省时取 `.current-version`
- **选定**:B(接受可选参数,缺省用 `.current-version`)
- **理由**:
  - 需求 §S-3 场景 `/code-publish V0.0.0` 明确含可选参数(用户希望按版本归档发布,不一定是当前激活版本)
  - 与 `code-dashboard` 设计一致(REQ-00004 是"双粒度":无参 / 给 REQ-NNNNN)
  - **不引入复杂度**:仅一个参数,缺省值清晰
- **依据规范**:无
- **实现要点**:
  - 解析:`<arg>` 非空且匹配版本号格式 → 用 `<arg>`;否则 → `Read .current-version`
  - 验证:`./assistants/<arg>/RESULT.md` 必须存在,否则报错退出(同步骤 0 失败)

### D-5:`--force` 强制发布的实现方式

- **候选**:
  - A. 接受位置参数 `code-publish --force` 或 `code-publish <版本号> --force`
  - B. 不在 v1 实现,需求 §S-6 是"未来扩展"
  - C. 通过 `AskUserQuestion` 在前置检查不通过时反向问"是否强制"
- **选定**:**B(v1 不实现 `--force`)**
- **理由**:
  - 需求 §S-6 虽然描述了 `code-publish --force`,但只是边界场景,**未在 FR/AC 中显式要求**(没有 `FR-X 强制发布` 条目)
  - 加 `--force` 增加参数解析复杂度;v1 优先简单可用
  - 用户实际需要"强制发布"时,可手动调 `code-publish` → 看到不通过报告 → 自行决定是否手动写 `publish/` 目录(逃逸阀)
- **依据规范**:无
- **回退路径**:v2 通过 `AskUserQuestion`(选项 C)在不通过时反向问,**比 `--force` 参数更安全**(用户看到问题清单后再决定)
- **降级说明**:本设计在 RESULT.md §11 标注"`--force` 留作 v2",并在过程文档 `clarifications.md` 记录

### D-6:`Q&A.md` 聚合 `qanda/` 的规则

- **候选**:
  - A. 全部聚合(qanda 里所有 .md 都收编进 Q&A.md)
  - B. 智能筛选(按"发布相关"关键词过滤)
  - C. 用户选择(列出 qanda/ 所有文件,让用户勾)
- **选定**:A(全部聚合)
- **理由**:
  - FR-5 §筛选规则 明确"v1 不实现智能筛选;**全部**聚合(用户自行组织)"
  - 智能筛选(B)涉及 NLP 判断,与"纯文档技能"特性不符
  - 用户选择(C)违反 NFR-3 隐式约定(不引入额外交互)
- **依据规范**:无
- **实现要点**:
  - 排除 `README.md`(README 是元说明,不是 Q&A 内容)
  - 每个聚合的 .md 在 Q&A.md 中独立为一个 `## <文件名(去后缀)>` 章节
  - 在每节首行加 `> 来源:qanda/<文件名>`(AC-5.4)
  - `qanda/` 不存在 或 仅有 README.md → 生成单段占位 + 提示(AC-5.1 / AC-5.2)

### D-7:文件覆盖策略(幂等性的具体语义)

- **候选**:
  - A. 始终覆盖(NFR-6 最朴素实现)
  - B. 仅当目标文件不存在或被识别为"由 code-publish 生成"时覆盖
  - C. 询问用户(每次重跑都弹 `AskUserQuestion`)
- **选定**:A(始终覆盖) + **在覆盖前向用户报告"将覆盖以下 N 个文件"**
- **理由**:
  - NFR-6 显式"再次生成同样的骨架" + "若已存在 → **覆盖**"
  - B 需要在文件首行加 `<!-- generated by code-publish -->` 标识 — 引入额外约定,且用户可能手动删了标识
  - C 违反"幂等"语义,且用户在 REQ-00006 §S-6 已表达"会手动审阅 → 手动 commit"模式,信息透明即可,不需要每次确认
- **依据规范**:无
- **实现要点**:
  - 写文件前 `ls -la publish/` 列出已有 → 在报告中显式说"将覆盖 N 个文件"
  - 用户审阅 dirty tree 时可通过 `git diff` 验证

### D-8:报告格式

- **候选**:
  - A. 多行纯文本(参考 needs §S-1 / §S-2 / §S-3 的样例输出)
  - B. 半结构化(JSON / YAML)
  - C. 增强 Markdown(含 ✓/✗ 图标 + 颜色编码)
- **选定**:A(多行纯文本)+ C 的"少量 ✓/✗/⚠ 图标"
- **理由**:
  - 需求 §S-1 / §S-2 / §S-3 / §S-4 的样例输出全部为"纯文本 + ✓/✗/⚠"
  - 报告是给用户看的(不是给下游技能消费),不需要结构化
  - **NFR-9 可读性优先**支持本选定
- **依据规范**:无
