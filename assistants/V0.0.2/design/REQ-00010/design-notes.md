## 关键设计问题清单

### Q-A:守卫插入位置
- **问题**:`code-it` 既有"步骤 0 版本上下文检测"+"步骤 7 处理任务前置依赖与状态",本需求新增"步骤 0a 前置任务守卫"应放哪?
- **候选**:
  - **A1**:**步骤 0 之前** — 命名为"步骤 0a"(沿用 REQ-00005 / REQ-00009 既有"步骤 0a"惯例)
  - **A2**:步骤 0 之后、步骤 1 之前 — 命名为"步骤 0b"
  - **A3**:在原"步骤 7 检查前置任务"处扩展 — 复用原"前置任务"概念
- **选定**:**A1**(沿用既有"步骤 0a 守卫"惯例)
- **理由**:
  1. 需求 FR-1.AC-1.1 明确要求"在'步骤 0'前显式列出'步骤 0a'"
  2. 与 REQ-00005 `code-require` 守卫、REQ-00009 `code-unit` 守卫同构(一致性)
  3. "无 `.current-version`"是步骤 0 的检查;**前置任务守卫是**"任务合法性"检查;后者不应在版本上下文缺失时被走
  4. "步骤 7 显式前置任务字段检查"是**另一种**机制(看"前置任务"字段),本需求是**新的**机制(看 PLAN.md 行序),不应合二为一(职责正交)
- **被否决方案**:
  - A2 步骤 0 之后:与既有"步骤 0a"惯例冲突
  - A3 步骤 7 扩展:混淆两种机制,且 NFR-3 零规范变更要求不破坏"前置任务"字段语义

### Q-B:"前置任务"信息源
- **问题**:"前置关系"从哪里取?
- **候选**:
  - **B1**:`PLAN.md` 任务总览区段**文件行序**(Q-1 锁定)
  - **B2**:新增"前置任务"字段(需求 FR-5 / NFR-3 明确禁止)
  - **B3**:看版"任务清单"区段(同上禁止)
  - **B4**:在 `code-plan` 阶段派生独立 `plan/REQ-NNNNN/deps.json` 文件
- **选定**:**B1**(零规范变更)
- **理由**:需求 FR-1.AC-1.2 明确要求"从 `plan/REQ-NNNNN/PLAN.md` 任务总览区段按文件行序解析所有任务";NFR-3 零规范变更

### Q-C:解析失败的容错
- **问题**:`PLAN.md` 任务总览区段格式错乱 / 文件不存在 / 任务未找到时,守卫应如何处理?
- **候选**:
  - **C1**:**通过守卫**(NFR-6 退化)
  - **C2**:**不通过守卫**(严格守卫,报"PLAN.md 解析失败")
- **选定**:**C1**(软失败)
- **理由**:需求 NFR-6 / 边界 E-3 / E-5 明确:`PLAN.md` 缺失或格式错乱时守卫**通过**(退化);同时任务合法性检查是**原 `code-it` 行为**,不重复

### Q-D:中止时屏幕输出格式
- **问题**:中止报告如何显示前置任务状态与"推荐执行命令"?
- **候选**:
  - **D1**:沿用 §6.1 需求模板(REQ-00010 FR-2)
  - **D2**:沿用 §"标题解析" REQ-00013 升级版(REQ-00010 §标题解析模板)
- **选定**:**D2**(升级版,与 `code-it` 现有 §"标题解析"段协同)
- **理由**:REQ-00010 §标题解析小节已为中止报告定义了升级模板(含"REQ-NNNNN · 标题" + "TASK-... · 标题"格式);既保持 `code-it` 屏幕输出一致性,又满足 FR-6.AC-6.2

### Q-E:中止时退出码
- **问题**:守卫不通过时退出码?
- **候选**:
  - **E1**:`exit 1`(通用错误码)
  - **E2**:新增 `exit 17`(与 `code-unit` 不通过守卫的退出码一致,假设有)
- **选定**:**E1**(不引入新退出码)
- **理由**:`code-auto` FR-7.AC-7.1 仅要求"退出码 ≠ 0",不区分具体值;NFR-1 零依赖;最小实现
- **回退路径**:v2 评估是否需要语义化退出码

### Q-F:与"原步骤 7 检查前置任务"的关系
- **问题**:需求 Q-1 锁定"零字段变更,按 PLAN.md 登记顺序判定",与原步骤 7 的"显式前置任务字段"如何共存?
- **候选**:
  - **F1**:**步骤 0a 守卫先于**步骤 7(本需求引入);步骤 7 仍按"显式前置任务"字段检查(原行为)
  - **F2**:删除步骤 7 的"显式前置任务"逻辑
- **选定**:**F1**(并存,职责正交)
- **理由**:
  1. 步骤 0a 守卫的依据 = PLAN.md 文件行序(零字段)
  2. 步骤 7 检查的依据 = PLAN.md 任务行的"前置任务"字段(显式字段)
  3. 两种机制覆盖不同场景:步骤 0a 兜底"PLAN.md 内所有任务按登记顺序串行";步骤 7 兜底"PLAN.md 内任务有显式依赖字段(如 BUG 任务的 '关联任务')"
  4. 选 F2 会破坏 NFR-3 零规范变更(原"前置任务"字段不被删除)
  5. 选 F2 也违反 FR-3.AC-3.1(不修改步骤 0 之后的原有流程)
- **协同**:`code-auto` 任务循环"按 PLAN.md 行序"调 `code-it` → 步骤 0a 守卫**总是通过**;用户手动乱序调 → 步骤 0a 守卫**触发**;两种调用模式下 `code-it` 行为一致

## 架构方案(选定)

### 整体方案
对 `code-it/SKILL.md` 进行**追加**式增量修改,新增"步骤 0a 前置任务守卫"小节(锚点 = "## 标题解析(REQ-00013 新增)" 段后 + "## 工作流程" 段前),完整结构包括:

1. **小节标题**:`## 步骤 0a — 前置任务守卫(REQ-00010 新增)`
2. **触发条件**:用户输入 = `TASK-...` 任务编码(`code-it` 步骤 1 已判定为任务分支)
3. **算法**:从 `plan/REQ-NNNNN/PLAN.md` 任务总览区段按文件行序解析所有任务 → 找到当前任务位置 → 前置任务 = 位置之前的所有任务 → 检查每个前置任务"开发状态"列
4. **通过条件**:所有前置任务开发状态 = `已完成`
5. **不通过处理**:打印"⛔ code-it 中止(存在未完成的前置任务)" + 前置任务状态清单 + "推荐执行"命令 + `exit 1`
6. **退化条件**:`PLAN.md` 不存在 / 任务总览格式错乱 → 视为"无法判定前置",**通过**守卫

## 关键算法(伪代码)

```ts
// 步骤 0a 前置任务守卫
function preTaskGuard(taskNum: string): void {
  // 1. 反推所属需求
  const reqNum = parseReqFromTask(taskNum)  // TASK-REQ-00001-00001 → REQ-00001
  if (!reqNum) {
    // 非任务分支(缺陷分支),守卫不触达
    return
  }
  // 2. 读 PLAN.md
  const planPath = `./assistants/${version}/plan/${reqNum}/PLAN.md`
  if (!fileExists(planPath)) {
    // NFR-6 退化
    log("⚠ PLAN.md 不存在,守卫通过(退化)")
    return
  }
  // 3. 解析任务总览区段
  const tasks = parsePlanOverview(planPath)  // 按文件行序
  // 4. 找到当前任务位置
  const idx = tasks.findIndex(t => t.num === taskNum)
  if (idx === -1) {
    // 任务不在 PLAN.md 中 — 由原 code-it 步骤 2 报错,守卫不重复
    return
  }
  // 5. 前置任务 = 当前任务之前的所有任务
  const preTasks = tasks.slice(0, idx)
  if (preTasks.length === 0) {
    log("✓ 前置任务守卫通过(无前置任务)")
    return
  }
  // 6. 判定每个前置任务
  const unfinished = preTasks.filter(t => t.devStatus !== '已完成')
  if (unfinished.length === 0) {
    log("✓ 前置任务守卫通过(全部已完成)")
    return
  }
  // 7. 中止
  logAbortReport(reqNum, taskNum, tasks, unfinished)
  exit(1)
}
```

## 边界与异常(沿用需求 + 本设计补充)

| ID | 场景 | 处理 | 依据 |
| --- | --- | --- | --- |
