## 1. 设计目标与范围

### 1.1 目标

将"更新版本看板"这一职责从 `/code-plan` 拆分出的派生任务中**移除**,下沉至 `/code-it` 自身完成。具体:

- **D-1** `/code-plan` 不再为"更新看板"单独拆出派生任务
- **D-2** `/code-it` 末尾兜底提交成功后,新增小步 P-1,自行把本任务的看板状态(开发状态:待开发→已完成)推进
- **D-3** 看板"任务清单"区段的"触发/来源"列保持 `详细设计`,**不再**出现 `更新看板` 来源
- **D-4** `/code-it` 推进看板小步沿用 `dashboard-conventions §规则 1` 的看板 3 区段解析锚点(`^## 任务总览$` / `^## 缺陷清单$` / `^## 评审发现汇总$`)
- **D-5** 看板"变更记录"区段追加"任务完成"事件行(沿用 V0.0.2 既有事件类型,**不新增**枚举值,因此不触发 `dashboard-conventions §规则 1` 3 处同步)

### 1.2 范围

| 范围 | 包含 |
| --- | --- |
| 改动 | `/code-plan` SKILL.md(步骤 4A 拆任务准则 + 步骤 9A 看板同步);`/code-it` SKILL.md(末尾兜底之后新增 P-1 小步) |
| **不**改 | `/code-unit` / `/code-review` / `/code-dashboard` / `/code-publish` / `/code-auto` 5 个 `code-*` 技能;`./assistants/rules/` 13 个文件;`plugins/code-skills/CLAUDE.md`;`plugins/code-skills/skills/code-version/templates/version-RESULT.md`(NFR-3 强约束) |

| 新增依赖 | 0 |
| 新增技能 | 0 |

### 1.3 不变量(强约束)

- **INV-1** `/code-plan` 拆出的任务数 == 真实任务数(无"更新看板"派生)
- **INV-2** `/code-it` 末尾兜底提交成功后,本任务看板"开发状态"列从"待开发"推进到"已完成"
- **INV-3** `/code-it` 推进看板失败时,**不重试**,**不阻断**(stderr 告警即可)
- **INV-4** 看板"任务清单"区段本任务行"触发/来源"列**始终**保持"详细设计"(不由 `/code-it` 改动)
- **INV-5** 看板"变更记录"区段在 `/code-it` 完成后追加 1 条"任务完成"事件(沿用既有事件类型,**不**触发 `dashboard-conventions §规则 1` 3 处同步)
- **INV-6** 看板 3 区段解析锚点(`^## 任务总览$` / `^## 缺陷清单$` / `^## 评审发现汇总$`)**字节级保留**(`/code-dashboard` 解析不受影响)

[/code-plan 步骤 4A 拆任务]
  输出 PLAN.md 任务总览(示意,以 REQ-00016 为例):

    | TASK-REQ-00016-00002 | 修改 code-plan/SKILL.md            | 触发/来源=详细设计 |

  共 4 个任务,无"更新看板"行 ✓  ← 但本需求关注的是:为什么有时会出现"更新看板"派生?

**REVIEW-REPORT 派生的"更新看板"任务**(`/code-review` 触发):
- 出现在历史 `code-review` 报告的"派生任务清单"区段,不是 `/code-plan` 拆的
- `code-auto` 步骤 6 评审循环会消费它,转成派生任务循环

但本需求描述中提到"为'更新看板'单独拆出一个派生任务,触发/来源=详细设计"——这其实是 `/code-plan` 在某些历史实现里**曾经**有过的问题(本仓库已修,但用户希望"修法明确写进 SKILL.md"以避免回归)。

  步骤 2: 暂存

## 3. 关键设计决策(8 项)

### D-1 `/code-plan` 步骤 4A 拆任务准则改为"一个任务 = 一个实际产出"

- **候选集**:`{代码改写, 测试编写, 文档改写, 数据迁移, 配置变更, 部署脚本}`
- **不在候选集**:`{看板更新, 文档评审, 任务分配}` 等纯协调/汇报类操作
- **理由**:与"任务粒度可执行 + 可独立验收"对齐;任务清单不再"为看而看"

### D-2 `/code-it` 末尾兜底后新增 P-1 小步"推进看板"

- **位置**:`/code-it` 末尾兜底"步骤 5 执行 commit 成功"之后,新增步骤 "P-1 推进看板"
- **职责**:仅推进**本任务**的看板状态,不波及他任务;不修改本任务"触发/来源"列(保持"详细设计")
- **不重试不阻断**:失败时 stderr 告警,流程继续;后续可由"补登任务"或下次 `/code-review` 派生任务补偿

### D-3 看板推进时只动"开发状态"列,"测试状态"留给 `/code-unit`

- **本任务无测试**:本任务完成后"开发状态"="已完成","测试状态"="不适用"(由 `/code-plan` 步骤 4A 写入时已设)
- **本任务带测试**:本任务完成后"开发状态"="已完成","测试状态"="待编写"(由 `/code-plan` 步骤 4A 写入时已设);`/code-unit` 推进"测试状态"列(本需求**不**改)
- **理由**:职责分离;`/code-it` 不应越界改"测试状态"列

### D-4 看板"变更记录"沿用既有"任务完成"事件类型,**不新增**枚举值

- **既有事件类型**:`需求新增 / 需求变更 / 概要设计完成 / 详细设计完成 / 任务完成 / 测试完成 / 审查完成 / 派生任务完成 / 缺陷修复 / 发布`(V0.0.2 既有 10 种)
- **本需求使用**:`任务完成`(既有,**不新增**)
- **结果**:**不**触发 `dashboard-conventions §规则 1` 3 处同步(规则 1 第 3 款"新增/删除/修改枚举值"才触发,本需求不新增)
- **验证**:V0.0.2 既有 11 个 `code-*` 实践历史中,`code-it` 完成时确实用"任务完成"事件(参考 `assistants/V0.0.2/RESULT.md` line 386 T-004 完成记录)

### D-5 看板推进失败处理:不重试,不阻断

- **失败场景**:`/code-it` P-1 步骤解析看板失败 / 写回失败 / 权限不足
- **行为**:`stderr` 打印 `⚠ 看板推进失败(<原因>),请人工核对`,**不重试**,**不阻断**;`/code-it` 整体仍视为完成
- **理由**:看板推进是"文档同步"职责,失败不应阻塞实际代码提交的完成
- **补偿路径**:用户可重跑 `/code-it`(幂等)或下次 `/code-review` 派生"看板补登"任务

### D-6 看板推进的幂等性

- **幂等条件**:本任务"开发状态"已是"已完成"时,P-1 步骤**不重复写**,只跳过
- **场景**:同一任务被多次执行(修正 bug / 重跑)
- **验证**:`code-auto` 步骤 6 评审派生任务循环可能重跑 `/code-it`;此时看板不会重复追加"任务完成"行(只在状态变化时追加)

### D-7 看板解析沿用既有锚点,**不新增**解析方式

- **既有锚点**:`^## 任务总览$`(区段)+ `^\| .* \|$`(表格行)+ 第 1 列(任务编码)
- **新增列**:无
- **修改列**:无
- **结果**:`/code-dashboard` 解析(沿用同样锚点)**字节级兼容**

### D-8 `/code-it` P-1 小步走串行,不并发

- 末尾兜底 commit + P-1 推进看板 **必须串行**(commit 失败 → 不推进看板)
- **理由**:commit 失败说明本次代码未真正落仓,看板不应标"已完成"
- **NFR-6 沿用**:V0.0.2 既有"末尾兜底"模式
