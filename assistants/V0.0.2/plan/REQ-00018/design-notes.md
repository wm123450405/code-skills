## 候选方案权衡(任务拆分)

### 候选 A:2 个任务(本设计选定)

- T-001 `[修改]` SKILL.md 增量追加(实际产出 = SKILL.md 改动)
- T-002 `[文档]` 8 INV 自检 + 5 处看板同步 + 收尾(实际产出 = 看板 5 处同步 + 自检文档)

**优势**:
- 严格按"1 任务 = 1 实际产出"(REQ-00017 强约束)
- 任务粒度适中(0.5-2 天可完成)
- 触发/来源全 = "详细设计",0 派生"更新看板"任务

**劣势**:
- T-002 是"文档"类型而非"修改"类型,略偏轻量

### 候选 B:1 个任务(任务粗化)

- T-001 `[修改]` SKILL.md 增量追加 + INV 自检 + 看板同步

**优势**:
- 任务数最少

**劣势**:
- 1 任务 = 3 件事,粒度过粗(违反"0.5-2 天可完成"原则)
- INV 自检和看板同步本质是"自检 + 同步",不是"实际产出"(违反 REQ-00017 强约束)

**决定**:**否决**

### 候选 C:3+ 个任务(任务细化)

- T-001 写 SKILL.md + T-002 INV 自检 + T-003 看板同步

**优势**:
- 每任务粒度更细

**劣势**:
- T-002 / T-003 不是"实际产出",是"协调/汇报"类操作(违反 REQ-00017 强约束)
- 看板推进职责应由 `/code-it` 末尾 P-1 兜底承担

**决定**:**否决**

## 6 类描述文件解析契约

| 描述文件 | 解析正则 | Edit 替换 |
| --- | --- | --- |
| `package.json` | `"version"\s*:\s*"([^"]+)"` | `"version"\s*:\s*"[^"]+"` → `"version": "<新版本号>"` |
| `manifest.json` | 同 `package.json` | 同 `package.json` |
| `pom.xml` | `<version>([^<]+)<\/version>` | `<version>[^<]+<\/version>` → `<version><新版本号></version>` |
| `Cargo.toml` | `^version\s*=\s*"([^"]+)"` 多行 | `^version\s*=\s*"[^"]+"` 多行 → `version = "<新版本号>"` |
| `pyproject.toml` | 同 `Cargo.toml` | 同 `Cargo.toml` |
| `go.mod` | N/A | N/A(屏幕输出 `⚠ go.mod 无版本号字段`) |
