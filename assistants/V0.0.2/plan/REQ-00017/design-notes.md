## 候选方案权衡

### Q-1 详细权衡

- **A P-1 必须在 commit 成功后执行**(本需求选定):
  - 优点:看板"开发状态"="已完成" 与代码实际落仓严格对应
  - 缺点:commit 失败时 P-1 也不执行,看板可能与代码不同步
- **B P-1 与 commit 并发**:
  - 优点:速度快
  - 缺点:并发写看板风险(单文件写);code-auto 串行驱动原则违反
- **C P-1 总是执行,无视 commit 结果**:
  - 优点:看板总是更新
  - 缺点:commit 失败时看板错误标"已完成",误导用户

**选定 A** 理由:看板"开发状态"="已完成" 的语义是"代码已落仓 + 测试推进待办",与 commit 成功严格对应。

## 选定方案理由

- 锚点 A 选定:在"#### 任务类型"前插入(任务类型紧跟拆任务约束,逻辑通顺)
- 锚点 B 选定:在"第 3 款"前插入第 2.5 款(原编号保持)
- 锚点 C 选定:在"步骤 5 commit 成功"判断后插入 P-1(D-8 串行约束)
- 任务数 2 选定:FR-1 强约束"不拆更新看板任务",故 1 个修改(`/code-plan`) + 1 个修改(`/code-it`) = 2 任务

## 备选方案被否决的理由

- Q-1 选 C 已被否决:看板与代码可能不一致
- Q-3 选 B 已被否决:解析表头动态定位需要新增解析逻辑(增量复杂度)
