## 接口 1:`code-review`(无参)— 整版本模式入口

### 形式
SKILL.md 入口(无参数)

### 触发条件
- 用户调用 `/code-review`(无任何参数)
- 模式选择判定 = 整版本(算法 1)

### 入参
**无**(整版本模式自动从 `.current-version` 读激活版本 + 从主 RESULT.md 读需求清单)

### 出参
**屏幕输出**(stdout):
```
✓ code-review(整版本模式)完成
评审范围:V<版本号> 的 N 个"已完成"需求
评审发现汇总:...
派生任务:K'
输出文件:
  - ./assistants/V<版本号>/REVIEW.md
  - ./assistants/V<版本号>/review/REQ-NNNNN/REVIEW-REPORT.md(单需求 × N 份)
```

**磁盘输出**:
- `./assistants/V<版本号>/REVIEW.md`(聚合,顶层)
- `./assistants/V<版本号>/review/REQ-NNNNN/REVIEW-REPORT.md`(单需求 × N 份)
- `./assistants/V<版本号>/review/<新 TASK>/RESULT.md`(派生改修要求,若有"必须改")
- `./assistants/V<版本号>/plan/REQ-NNNNN/PLAN.md` "任务总览" 追加派生任务行(若有"必须改")

### 错误码
- 退出码 0:完成(含非阻塞警告)
- 退出码非 0:致命错误(沿用模式 1 既有:无 `.current-version` / 上游缺失 / `git` 不可用 / Ctrl+C / 内部错误)

### 示例
**正常完成**:
```
/code-review
→ 步骤 0: 读 .current-version = V0.0.2
→ 步骤 1: 无参,无需解析
→ 步骤 1.5: 模式选择 = 整版本
→ 步骤 2.1: 过滤需求 = [REQ-00005, REQ-00006]
→ 步骤 2.2: 循环评审 2 个需求,产生 3 条发现(2 必须改 + 1 建议改)
→ 步骤 2.3: 写 REVIEW.md(覆盖)
→ 步骤 2.4: 写 2 份单需求 REVIEW-REPORT.md(覆盖)
→ 步骤 2.5: 派生 2 个"审查改修"任务追加到对应 PLAN.md
→ 步骤 2.6: 报告
退出码 = 0
```

**无"已完成"需求(退化)**:
```
/code-review
→ 步骤 0: 读 .current-version = V0.0.2
→ 步骤 1.5: 模式选择 = 整版本
→ 步骤 2.1: 过滤需求 = []
→ 步骤 3: 报告退化
✗ code-review(整版本模式):本版本无可评审的需求
退出码 = 0
```

### 兼容策略
- 与 `code-review REQ-NNNNN` 模式 1 **并存**(FR-1.AC-1.2)
- 多次执行整版本模式**幂等**(NFR-3 + INV-8):REVIEW.md + N 份 REVIEW-REPORT.md 全部覆盖
- 模式 1 与模式 2 互覆盖(NFR-6 允许 + FR-7.AC-7.1 不变)

### 依据规范
- `skill-conventions.md §规则 1`:SKILL.md frontmatter 必含 name + description(既有已合规)
- `encoding-conventions.md §规则 1-4`:派生任务编码双格式兼容 + 由 `code-plan` 生成

## 接口 4:SKILL.md 内部"步骤 2 整版本模式"段(本设计新增,字面契约)

### 字面(锚点 A 插入内容 — 在"步骤 1.5"后)

```markdown
### 步骤 2 整版本模式(本需求新增,完全复用模式 1 步骤 4-15)

**触发条件**:步骤 1.5 模式选择 = 整版本

#### 2.1 过滤需求
1. 读 `./assistants/<版本号>/RESULT.md`,定位 `## 需求清单` 区段(沿用 `code-dashboard` 算法 1)
2. 筛 `状态=已完成`(**不**归一化 "已完成(需求分析)" 到 "已完成",沿用 `code-dashboard` 行为)
3. 空 → 走"步骤 3 报告退化" + 退出
4. 非空 → 打印"评审范围: <N> 个'已完成'需求: <列表>"

**依据**:FR-2.AC-2.1 / FR-2.AC-2.2 / FR-2.AC-2.3

#### 2.2 循环评审(完全复用模式 1 步骤 4-15)
对每个被评审需求 `REQ-NNNNN`,执行既有"单需求模式"评审逻辑(步骤 4-15),产出:
- `review/REQ-NNNNN/REVIEW-REPORT.md`(覆盖)
- 派生任务 `review/<新 TASK>/RESULT.md`(若"必须改")
- PLAN.md "任务总览" 追加派生任务行

**异常处理**(沿用模式 1 既有):
- 评审失败 → 跳过 + 报告 + 继续(E-2)
- 写入失败 → 跳过 + 报告 + 继续(E-4)
- 派生任务编码生成失败 → 跳过该发现 + 报告 + 继续(E-6)

**依据**:FR-6.AC-6.1 / FR-6.AC-6.2 / FR-6.AC-6.3

#### 2.3 写聚合 REVIEW.md
1. 路径:`./assistants/<版本号>/REVIEW.md`(版本顶层,与 `RESULT.md` 同级)
2. 文件已存在 → **覆盖**前次内容
3. 5 章节结构:评审概览 / 各需求评审摘要 / 评审发现汇总(去重)/ 派生任务汇总 / 评审人/AI 备注
4. 去重键:`(需求编码, 描述前 50 字)`(NFR-7 + D-2)
5. 失败:打印 `✗`,**不**整体失败(E-5)

**依据**:FR-3.AC-3.2 / FR-3.AC-3.4 / FR-3.AC-3.5 / FR-4.AC-4.1 ~ AC-4.6 / NFR-3 / NFR-6 / NFR-7

#### 2.4 单需求 REVIEW-REPORT.md(已由 2.2 循环中写)
**说明**:本步骤**不**重复写 — 步骤 2.2 循环中已对每个被评审需求写了 1 份 `REVIEW-REPORT.md`(沿用模式 1 既有路径)

**依据**:FR-3.AC-3.1 / FR-3.AC-3.5

#### 2.5 派生任务追加(沿用模式 1 既有逻辑)
1. 对每个被评审需求的所有"必须改"派生任务,追加到对应 `plan/REQ-NNNNN/PLAN.md` "任务总览" 表的最末行
2. 字段:`触发/来源=审查改修`,`关联任务=被修正原任务`
3. **唯一性检查**:同 `(需求编码, 描述前 50 字)` 已存在 → **不**重复追加(NFR-8)
4. 编码生成:**不**由本设计生成(沿用 `code-plan` 既有规则)

**依据**:FR-5.AC-5.1 / FR-5.AC-5.2 / FR-5.AC-5.3 / NFR-8

#### 2.6 报告
```
✓ code-review(整版本模式)完成
评审范围:V<版本号> 的 N 个"已完成"需求
  - REQ-NNNNN <标题>
  ...
评审发现汇总:
  - REQ-NNNNN:K 必须改 + L 建议改 + O 可选
  - REQ-NNNNN:K 必须改 + L 建议改 + O 可选
  共 M 条发现(必须改 K',建议改 L',可选 O')
派生任务:K'(均已写入对应 PLAN.md;去重后净增 K')
输出文件:
  - ./assistants/V<版本号>/REVIEW.md(聚合,覆盖)
  - ./assistants/V<版本号>/review/REQ-NNNNN/REVIEW-REPORT.md(单需求 × N 份,覆盖)
  - ./assistants/V<版本号>/review/<新 TASK>/RESULT.md(派生改修要求,若有)
```

**S-2 退化(无"已完成"需求)**:走"步骤 3 报告退化"(本节不重复)

---

## 整版本模式 — 评审范围与适用场景

(本节由 REQ-00008 概要设计阶段新增;适用对象:`code-review` 整版本模式)

### 适用场景
- **R-1 发布者**:调 `code-publish` 前的"最后一次评审",用整版本模式
- **R-2 复盘者**:定期调"看本版本健康度"
- **R-3 项目主导者**:在 `code-auto` 完成一轮后,用整版本模式做"额外确认"

### 评审范围(Q-1 锁定 B)
- **筛中**:`状态=已完成` 的需求
- **不**筛中:`待开始` / `进行中` / `已取消` / `阻塞` / `已完成(需求分析)`(状态字面**不**归一化)

### 与模式 1 关系
- **并存**:`code-review`(无参)→ 整版本 / `code-review REQ-NNNNN` → 单需求
- **完全兼容**:整版本模式对每个被评审需求生成的 `REVIEW-REPORT.md` 与模式 1 字面 100% 一致(FR-6.AC-6.1)
- **派生任务字段一致**:与模式 1 派生任务字段 100% 一致(FR-5 + `encoding-conventions §规则 4`)
- **互相覆盖**:整版本模式**覆盖**模式 1 既有 `REVIEW-REPORT.md`(NFR-3 幂等)

### 不触发的区段
- `code-auto` 评审循环**不**用整版本模式(继续用模式 1,NFR-4)
- `code-review` **不**在 REQ-00005 改写范围(继续**不**含"首步拉取"与"末步提交",NFR-5)
- `plugins/code-skills/CLAUDE.md` "AI 工作约定"小节**不**追加(Q-7 采纳默认)
- `marketplace.json` / `plugin.json` / `assistants/rules/` **不**修改(FR-8 全节)

### 待澄清 / 未决项(本节,留作 follow-up)
- **Q-8.1**:用 `code-rule` 沉淀 `review-conventions.md`(整版本模式的检查清单)— **不阻塞**,留作 follow-up
- **Q-8.2**:把 `code-review` 加入 REQ-00005 的"首步拉取+末步提交"改写范围 — **不阻塞**,留作 follow-up
- **Q-8.3**:在 `code-review/templates/` 新增 `REVIEW-ALL.md` 模板 — **本设计倾向不新增**(D-1.B);由 `code-plan` 决定
```
