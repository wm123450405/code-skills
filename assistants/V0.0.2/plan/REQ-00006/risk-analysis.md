## 1. 异常处理

对应 needs §9 E-1 ~ E-10 + design RESULT.md §6.3 + §12 R-1 ~ R-8。

### 异常路径 1:无激活版本(E-5)
- **触发条件**:`.current-version` 文件不存在 ∧ 用户未传位置参数
- **检测手段**:Step 0 中 `ls -la ./assistants/.current-version`
- **处理策略**:文本提示 `请先调 /code-version <版本号> 创建或切换版本。` + 不写任何文件 + 技能结束
- **监控指标**:无(用户即时反馈)
- **对应任务**:T-001(SKILL.md 步骤 0)

### 异常路径 2:目标版本看板不存在(E-2 变体)
- **触发条件**:`<版本号>/RESULT.md` 不存在
- **检测手段**:Step 0 中 `ls -la ./assistants/<版本号>/RESULT.md`
- **处理策略**:文本提示 `版本 <版本号> 的看板不存在,请确认版本号或先调 /code-version` + 不写 + 结束
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 0)

### 异常路径 3:看板缺区段(E-2)
- **触发条件**:RESULT.md 中找不到 `^## 需求清单` / `^## 任务清单` / `^## 缺陷清单` 中任一锚点
- **检测手段**:PreflightChecker 的 extract_section 返回 null
- **处理策略**:该区段视为"全未解决"(保守判定);Step 3 报告"看板不完整,以下区段缺失:[...]" + 不通过分支
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 1 PreflightChecker)

### 异常路径 4:前置检查不通过(E-1)
- **触发条件**:undone 非空
- **检测手段**:PreflightChecker 返回 `passed: false`
- **处理策略**:跳过 Step 2(不生成手册);Step 3 输出 needs §S-2 样式报告;结束
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 1 → 步骤 3 跳转)

### 异常路径 5:publish/ mkdir 失败(E-7)
- **触发条件**:`mkdir -p publish/` 返回非 0(权限/磁盘满)
- **检测手段**:Bash 退出码
- **处理策略**:`报错 + 退出`:文本输出 `❌ 无法创建 publish/ 目录:<原因>` + 不留任何半成品(不进入 Step 2.5 / 2.6 / 3)
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 2 ManualBuilder)

### 异常路径 6:publish/<file>.md Write 失败(E-7 变体)
- **触发条件**:Write 系统调用失败
- **检测手段**:Write 工具返回 error
- **处理策略**:报错 + 立即退出,**不**尝试写下一个文件;已写成功的文件保留(用户决定是否手动清理)
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 2)

### 异常路径 7:qanda/ mkdir 或 README.md Write 失败(E-4)
- **触发条件**:权限不足等
- **检测手段**:Bash 退出码 / Write error
- **处理策略**:**不阻塞**,标记 `qanda_status = 创建失败`;Q&A.md 仍按"占位"生成(QandaAggregator 0 文件分支);Step 3 报告中标 `⚠ qanda/ 创建失败:<原因>`
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 2.5)

### 异常路径 8:qanda 单文件读取失败
- **触发条件**:权限/编码问题
- **检测手段**:Read 工具返回 null/error
- **处理策略**:跳过该文件 + 在 Q&A.md 该段加 `> 警告:无法读取 qanda/<文件>`;不阻塞整体
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 2.6)

### 异常路径 9:看板"任务编号"含旧格式(E-9)
- **触发条件**:任务清单中有 `REQ-00001-001` 等 3 位旧编号
- **检测手段**:无主动检测;字符串处理时自然支持
- **处理策略**:不区分新旧,按字符串原样使用(用作 unique key + 报告显示)
- **监控指标**:无
- **对应任务**:T-001(PreflightChecker 设计)

### 异常路径 10:多次重跑(E-10)
- **触发条件**:用户多次调用 `/code-publish`
- **检测手段**:Step 2 mkdir 前 `ls publish/` 检测已有文件
- **处理策略**:覆盖(D-7)+ Step 3 报告"已覆盖 N 个文件:[...]"
- **监控指标**:无
- **对应任务**:T-001(SKILL.md 步骤 2 ManualBuilder)

## 2. 安全边界

### 2.1 鉴权要求
- **不适用**(本机文件系统操作)

### 2.2 输入校验
- **版本号字符串**:不允许含 `/` `\` `..`(防路径穿越);在 Step 0 校验
- **看板内容**:解析失败 → 退化路径(异常 3)
- **qanda 文件名**:`Glob *.md` 自然过滤;但 Read 内容时不做内容校验(直接渲染到 Q&A.md)

### 2.3 敏感数据处理
- **DEPLOY.md / UPDATE.md placeholder**:含 `<默认账号>` / `<passwd>` 等;在模板首行警告 `⚠ 首次登录后立即修改` + 在 Step 3 报告中提醒"补全后审阅 git diff 再提交"(避免误把口令落 git)
- **Q&A.md 聚合内容**:由用户在 `qanda/` 自行管理;本技能**不**做敏感词过滤(留作 v2)

### 2.4 防注入
- **Markdown 转义**:聚合 qanda 内容时**不做** HTML/JS 转义(因为 Markdown 渲染场景下不会执行 JS;读者是用户/AI,不是浏览器)
- **路径注入**:版本号经 Step 0 校验后用于路径拼装;不允许 `..` / `/`

### 2.5 审计
- **不适用**(技能调用记录由 Claude Code 自身保留)

### 2.6 依据规范
- 无安全规范文件(占位规范不含具体条款)

## 3. 性能与资源

### 3.1 关键路径耗时目标
- 整个技能调用 P95 < 5 秒(Read 看板 ~ 50 KB + Bash ~ 5 次 + Write 3 次)
- 实际测试:V0.0.2 看板 ~ 25 KB,预计 < 1 秒

### 3.2 并发上限
- 不适用(单次调用,无并发)

### 3.3 资源限制
- 内存:单次读取看板字符串 < 1 MB;聚合 qanda < 10 MB
- 文件:写 publish/ 3 个 < 50 KB 总和;qanda/README.md < 5 KB

### 3.4 缓存策略
- 无缓存(每次都重新读取看板;符合幂等要求 NFR-6)

### 3.5 批量/异步
- 同步顺序执行;无批处理需求

### 3.6 降级策略
- 看板缺区段 → 退化"全未解决"(异常 3)
- qanda mkdir 失败 → 退化"占位 Q&A"(异常 7)

## 4. 回退策略

### 4.1 触发条件
- 用户重跑 `code-publish` 后,发现新生成的 publish/ 手册覆盖了自己之前手动补全的内容

### 4.2 步骤
1. `git status` 查看 dirty 文件
2. `git diff publish/` 查看差异
3. `git checkout -- publish/<file>.md` 回退到上次 commit(若已 commit 用户的补全)
4. 或 `git stash` 暂存当前修改,重跑 `code-publish`,然后 `git stash pop` 恢复

### 4.3 验证
- `cat publish/DEPLOY.md` 确认用户补全内容仍在

### 4.4 本技能的"事前避免"
- Step 3 报告中**明示**"已覆盖 N 个文件",用户在每次重跑后立即可见
- 在 DEPLOY/UPDATE 文首加"使用说明"提示"补全后立即 git commit"
