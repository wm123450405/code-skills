### 2. 核心需求拆解
- **痛点**:
  - 当前 `code-require` / `code-plan` / `code-fix` 产出物中,"需求/任务/缺陷"**仅以编号**(如 `REQ-00001` / `TASK-REQ-00001-001` / `BUG-00001`)标识
  - 用户在屏幕上看到 "正在处理 `TASK-REQ-00005-002`" — 不知道这是做什么
  - 看板"任务清单"区段的"标题"列**已存在**,但"进度报告" / "派生任务" / "中止报告"中常出现"本任务" / "本需求"等指代词,无具体上下文
- **本需求**:
  - 为每个需求/任务/缺陷**生成标题**(< 30 字)
  - 标题**概述内容**(不只是"REQ-00001 任务")
  - 在**用户交互**中用"编号+标题"显示(例:`REQ-00001 · Marketplace 改名落地`)
  - 在**看板记录**中用"编号+标题"显示(例:`TASK-REQ-00001-001 · 改 marketplace.json 根 name`,而不是"本任务")
- **本质**:把"标题"提升为**一等公民字段**(从"附注"升为"必须");"编号+标题"成为**唯一**标识符

### 3. 关键边界
- **"标题"字段的位置**:
  - 选项 A:在 `code-require` / `code-plan` / `code-fix` 产物(如 `require/.../RESULT.md`、`plan/.../PLAN.md`、`fix/.../RESULT.md`)中,**显式**生成"标题"小节/字段
  - 选项 B:从**已有内容**(如需求标题/任务标题)派生,**不**新增字段
  - 选项 C:在**看板"需求清单"区段**新增"标题"列(触发 `dashboard-conventions §规则 1` 三方同步)
- **"标题"的字数限制**:
  - 用户原话:"30 字以内"
  - 是"30 个汉字"还是"30 个字符"?默认按"30 个字符"(包含数字/字母/标点)
- **历史回填**:
  - 现有 12 个需求(REQ-00001~00012)**已有"标题"**(标题在 V0.0.1 起就是 `require/.../RESULT.md` 第 1 行)
  - 现有 19 个 V0.0.1 任务**已有"标题"**(在 `plan/.../PLAN.md` 任务总览的"标题"列)
  - 本需求落地后,**这些历史"标题"自动生效** — 无需回填
  - 但用户交互/看板中"显示"行为需在**本轮修改**才能生效
- **"任务"是否含派生任务**:
  - V0.0.1 起 2 个派生任务(REQ-00001-005 / REQ-00002-009)**也有"标题"**(由 `code-review` 写入)
  - 本需求落地后,派生任务的"标题"自动生效
- **与 `code-fix` 的关系**:
  - `code-fix` 是 V0.0.0 起的 7 个技能之一
  - `code-fix` 产出 `fix/.../RESULT.md` 记录缺陷修复
  - 本需求让 `code-fix` 也生成"标题"
- **与 `code-dashboard` / `code-review` / `code-publish` 的关系**:
  - `code-dashboard` 的"任务清单"区段已含"标题"列 — 0 改动
  - `code-review` 的派生任务写入 `PLAN.md` 时有"标题" — 0 改动
  - `code-publish` 的"前置检查"报告中应使用"编号+标题" — **本轮**可能需调整
