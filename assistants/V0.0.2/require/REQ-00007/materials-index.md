## 关键信息抽取(供后续步骤使用)

### 1. 用户原文笔误纠正
- `/code-desgin` → **`/code-design`**(与 REQ-00005 同样的笔误)
- `/code-require` 在本需求中未出现,但 `/code-require` 出现 → **无**
- "如何需要" → "**如需要**"(语义校正)

### 2. 核心需求拆解
- **新增第 10 个 `code-*` 技能**:`/code-auto`
- **输入**:1 个需求内容(自然语言描述)
- **工作流(单次完整执行)**:
  1. 调 `/code-require` 产出需求文档
  2. 调 `/code-design` 产出概要设计
  3. 调 `/code-plan` 产出详细设计与任务计划
  4. **循环**:
     - 对每个任务调 `/code-it` 完成代码
     - 对每个任务调 `/code-unit` 完成单元测试(若需要)
     - 调 `/code-review` 评审
     - 若 `code-review` 产生"必须改"任务 → 调 `code-it` + `code-unit` 完成,再评审
     - 循环往复,直到 `code-review` 结果中**没有需要必须修改的内容**
- **完全自动化**:过程中若遇到"需要用户确认"的场景,**均使用推荐方式自动完成**,**不再需要用户确认**

### 3. "必须修改"的判定
- 来自 `code-review` 的"必须改"级别发现(与 V0.0.1 REQ-00002 / REQ-00003 的"必须改/建议改/可选"三档分类相关)
- 详细阈值见 clarifications.md Q-1

### 4. 关键边界(用户原文未明示)
- **循环终止条件**:
  - 主终止:无"必须改"发现
  - **次终止**(待澄清):超过 N 轮仍存在"必须改" → 终止 + 报告"已陷入循环"
  - **超时终止**:超过 M 小时
- **异常中断**:任何子技能崩溃(非 0 退出码) → 中断 + 报告
- **完全自动化的边界**:
  - 用户原文"使用推荐的方式自动完成" — 但有些场景的"推荐"本身就需要用户指定(例如:某项目有"前端 / 后端"两个候选,默认"后端",但"前端"更合理)
  - 安全相关操作(删除文件、修改外部资源)是否需要二次确认?
- **与 REQ-00005 的协同**:`code-require` / `code-design` / `code-plan` 已被 REQ-00005 加了"首步拉取 + 末步提交";`code-auto` 调它们时,这些步骤会被执行 N 次(N = 任务数)— 是否有问题?
- **与 REQ-00006 的协同**:`code-auto` 是"开发周期自动跑通",`code-publish` 是"开发完成后发布";本需求不触发 `code-publish`,但应在某处提示"开发完成后可调 `code-publish`"
- **与 REQ-00004 的协同**:`code-dashboard` 是"看状态",`code-auto` 是"驱动开发" — 两者不冲突

### 5. 子技能调用机制
- **`code-auto` 如何调子技能**?在 Claude Code 中,跨技能调用的标准方式是 `Skill` 工具
- **上下文隔离**:每次子技能调用是否独立上下文?若共享上下文,可能导致"上下文过长"
- **session 限制**:Claude Code 一次会话的 token 上限可能约束本技能的"循环次数"
