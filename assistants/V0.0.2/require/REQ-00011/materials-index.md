## 关键信息抽取(供后续步骤使用)

### 1. 用户原文笔误纠正
- `/core-desgin` → **`/code-design`**(用户原文笔误)
- `/core-plan` 不需纠正(可能是用户随手写,但 `/code-plan` 才是正确名)
- 涉及技能:`/code-design` / `/code-plan` 2 个(均已在 V0.0.0 起存在,本需求是**优化**而非新增)

### 2. 核心需求拆解
- **现状**:
  - `code-design` 在做概要设计时,无用户确认环节
  - `code-plan` 在做详细设计时,无用户确认环节
- **痛点**:
  - 设计目标(如"少改 vs 灵活")由 AI 默认决定,用户事后才发现"设计过度"或"设计不足"
  - 设计时需在"功能性 / 扩展性 / 健壮性 / 可维护性"间权衡,无显式确认 → 误判风险
- **本需求**:
  - 在 `code-design` / `code-plan` 的"适当环节"增加用户确认
  - 让用户对"设计目标"做选择(少改 / 灵活 / 等)
  - 让用户对"设计维度"做权衡(功能性 / 扩展性 / 健壮性 / 可维护性)

### 3. 关键边界
- **"适当环节"的位置**:
  - `code-design` 选项:
    - 启动时(开始设计前)
    - 概要设计产出后(在写 `design/REQ-NNNNN/RESULT.md` 之前)
    - 概要设计产出后、写完文档之后
  - `code-plan` 选项:
    - 启动时(开始任务拆分前)
    - 详细设计产出后
- **"设计目标"选项**(用户原话):
  - 更少的代码改修(`--minimal`)
  - 更灵活的后期拓展(`--extensible`)
  - 其他目标(由用户新增)
- **"设计维度"**:
  - 功能性(满足需求的程度)
  - 扩展性(为未来需求留空间)
  - 健壮性(异常处理、容错)
  - 可维护性(代码易读易改)
- **确认后是否回写文档**:
  - 选项 A:仅 AI 内部记住,不写文档
  - 选项 B:写入 `design/.../RESULT.md` 或 `plan/.../RESULT.md` 顶部"设计目标"小节
  - 选项 C:写入独立的 `design/.../design-goals.md` 文件
- **与 `code-auto` 协同**:REQ-00007 中 `code-auto` 调 `code-design` / `code-plan` 时,本需求会让 `code-auto` 触发 `AskUserQuestion` — 与 Q-4 `code-auto`"总选推荐项"冲突
- **设计目标与设计维度的关系**:
  - "更少的代码改修"更关注"可维护性 + 健壮性"(少做)
  - "更灵活的后期拓展"更关注"扩展性"(多做)
  - 选项 A:用户选 1 个"设计目标" + 1 个"设计维度"
  - 选项 B:用户选 1 个"设计目标",AI 据此推"设计维度"优先级
  - 选项 C:用户只选"设计目标",设计维度不显式选
