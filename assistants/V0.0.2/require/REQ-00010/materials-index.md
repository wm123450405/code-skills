> > 优化需求: `/core-it`(笔误 → `/code-it`)技能在执行一项任务之前获取当前需求下的任务清单,查看是否有前置的任务没有执行,若有前置任务没有执行应该中止执行操作,并引导用户先执行前置的任务,按顺序执行开发任务。

## 关键信息抽取(供后续步骤使用)

### 1. 用户原文笔误纠正
- `/core-it` → **`/code-it`**(用户原文笔误)
- 涉及技能:`/code-it`(现有技能,V0.0.0 已存在,本需求是**优化**而非新增)

### 2. 核心需求拆解
- **现状**:`code-it` 现有签名 = `code-it TASK-...`,**未**检查任务的前置依赖
- **痛点**:在 `code-plan` 拆分任务时若定义了"前置任务"(`predecessor` / `依赖`),`code-it` 仍可能"乱序执行"
- **本需求**:
  - `code-it` 启动时,先获取"当前需求下的任务清单"(从 `plan/REQ-NNNNN/PLAN.md` 解析)
  - 检查当前任务是否含"前置任务"字段
  - 若前置任务未完成 → **中止** + 引导用户按顺序执行
  - 若无前置或前置已全部完成 → 走原 `code-it` 流程

### 3. 关键边界
- **"前置任务"字段**:
  - 选项 A:看板"任务清单"区段需**新增**"前置任务"列(触发 `dashboard-conventions §规则 1` 三方同步)
  - 选项 B:`PLAN.md` "任务总览"区段需**新增**"前置任务"列(同 A,但不触发看板规范)
  - 选项 C:已有字段(任务行包含 `依赖:` 等子字段)— 需澄清
- **"未完成"判定**:
  - 开发状态 ∈ {`待开始` / `进行中` / `阻塞`} → 未完成
  - 开发状态 = `已完成` → 完成
  - 测试状态如何参与?(若"开发=已完成"但"测试≠已运行-通过/不适用",前置算完成吗?)
- **"按顺序执行"的引导**:
  - 选项 A:仅提示"请先执行 X" + 列出待办
  - 选项 B:直接调 `code-it` 执行前置任务(全自动链式)
  - 选项 C:`AskUserQuestion` 询问用户"现在是否执行前置?"
- **"中止"的退出码**:
  - 退出码 ≠ 0 → `code-auto` 中断
  - 退出码 = 0 → `code-auto` 继续下一个任务(跳过)
- **`code-auto` 协同**:REQ-00007 中 `code-auto` 串行调 `code-it`,本需求守卫会让 `code-auto` 任务循环"卡住" — 是 `code-auto` 的预期行为还是 bug?

### 4. 与现有看板/规范的关系
- **看板"任务清单"区段**(V0.0.1 模板):列含"任务编号 / 需求 / 类型 / 触发/来源 / 标题 / 开发状态 / 测试状态 / 涉及文件 / 完成时间 / 提交哈希 / 关联任务"
  - **无**"前置任务"列
  - 关联任务列已存在,但语义是"派生任务关联到原任务",**不**是"前置依赖"
- **`encoding-conventions.md`**:不直接相关
- **`code-plan` 的职责**:本需求落地后,`code-plan` 在拆分任务时**应**支持"前置任务"字段(否则 `code-it` 无数据可读)

### 5. 待澄清(在 clarifications.md 中展开)
- Q-1:"前置任务"字段的位置(`PLAN.md` 任务总览 / 看板"任务清单"区段 / 现有"关联任务"列复用)
- Q-2:"未完成"判定的精确语义(开发状态?测试状态?两者?)
- Q-3:中止时是否提供"自动链式执行"选项
- Q-4:中止时的退出码(中断 / 跳过)
- Q-5:与 `code-auto` 协同 — `code-auto` 是按"按顺序"调还是按"任务总览"原序调
- Q-6:本需求是否**要求** `code-plan` 同时支持"前置任务"字段
