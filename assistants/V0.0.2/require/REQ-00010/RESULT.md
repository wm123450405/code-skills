- 需求编码:REQ-00010
- 所属版本:V0.0.2

- 责任人:wangmiao

- 最近更新:2026-06-04 14:36
- 当前版本:v1
- **主题**(来自用户输入):
  > 优化需求: `/code-it`(笔误 → `/code-it`)技能在执行一项任务之前获取当前需求下的任务清单,查看是否有前置的任务没有执行,若有前置任务没有执行应该中止执行操作,并引导用户先执行前置的任务,按顺序执行开发任务。

## 1. 需求概述

**为谁**:`code-skills` 仓库的 AI 协作者 + 项目主导者(在 `code-it` 实际执行任务时,期望"按计划顺序执行,前置未完成则中止")。

**解决什么问题**:`code-it` 现有签名 = `code-it TASK-...` — 直接执行,不校验任务的前置依赖。在以下场景会导致问题:
- **手动乱序调用**:用户可能先调 `code-it TASK-002` 再调 `code-it TASK-001`,而 TASK-002 依赖 TASK-001
- **跨需求调用**:用户可能在 A 需求下误调 B 需求的任务
- **任务依赖复杂**:V0.0.2+ 任务粒度可能变大,任务间依赖关系重要

**带来什么价值**:
1. **按计划顺序执行**:`code-it` 启动时按 `PLAN.md` 任务总览区段的**文件登记顺序**判定前置,前置未完成则中止
2. **零规范变更**:**不**新增"前置任务"列(不修改 `PLAN.md` 模板,不修改看板),仅按文件顺序判定
3. **引导用户**:中止时打印"推荐执行 /code-it TASK-NNNNN-NNNNN"命令,让用户复制即可
4. **退出码兼容**:守卫不通过时退出码 ≠ 0,`code-auto` 据此中断

## 3. 用户角色与场景

### 3.1 角色
- **R-1 项目主导者**:手动调 `code-it` 时,期望"按计划顺序执行,前置未完成有提示"
- **R-2 长会话 AI**:在多任务间切换,可能误调乱序
- **R-3 跨需求调用者**:在 A 需求下误调 B 需求的任务

### 3.2 关键场景

#### S-1:前置未完成(主流程,守卫不通过)
- 假设 `plan/REQ-00005/PLAN.md` 任务总览区段登记顺序:`TASK-REQ-00005-001` → `TASK-REQ-00005-002` → `TASK-REQ-00005-003`
- 当前状态:`TASK-001` 已完成,`TASK-002` 待开始
- 用户输入:`/code-it TASK-REQ-00005-003`
- 技能执行:
  1. **步骤 0a 前置任务检查**(新增)— 解析 `plan/REQ-00005/PLAN.md`,找到当前任务 `TASK-003`
  2. 在任务总览区段中,`TASK-003` 前面的所有任务 = [`TASK-001`, `TASK-002`]
  3. 检查每个前置任务的"开发状态":
     - `TASK-001`:`已完成` ✓
     - `TASK-002`:`待开始` ✗
  4. 存在未完成前置 → 中止
- 用户看到:
  ```
  ⛔ code-it 中止(存在未完成的前置任务)
  
  当前任务:TASK-REQ-00005-003
  所在需求:REQ-00005
  PLAN.md 任务总览登记顺序:TASK-001 → TASK-002 → TASK-003
  
  前置任务状态:
    ✓ TASK-REQ-00005-001(开发状态=已完成)
    ✗ TASK-REQ-00005-002(开发状态=待开始)← 未完成
    ✗ TASK-REQ-00005-003(当前任务)
  
  推荐执行 /code-it TASK-REQ-00005-002 完成后,再执行 /code-it TASK-REQ-00005-003
  ```

#### S-2:无前置任务(主流程,守卫通过)
- 假设 `plan/REQ-00001/PLAN.md` 任务总览区段登记顺序:`TASK-REQ-00001-001` → `TASK-REQ-00001-002` → ...
- 当前状态:`TASK-001` 待开始
- 用户输入:`/code-it TASK-REQ-00001-001`
- 技能执行:
  1. **步骤 0a 前置任务检查** — `TASK-001` 在登记顺序中**位于最前**,无前置任务
  2. 守卫通过 → 进入原"步骤 0-N"流程
- (本需求**不**改变该流程,仅新增守卫)

#### S-3:`PLAN.md` 不存在(边界)
- 用户输入:`/code-it TASK-REQ-00005-001`,但 `plan/REQ-00005/PLAN.md` 不存在
- 技能执行:守卫**退化** — 视为"无 PLAN.md,无法判定前置",**通过**守卫,进原流程
- (理由:`PLAN.md` 是 `code-plan` 的产物;若 `code-plan` 未跑过,守卫数据不可用,不应阻止 `code-it`)

#### S-4:跨需求调用(边界)
- 用户输入:`/code-it TASK-REQ-00005-001`,但 `task` 不在当前版本看板中
- 技能执行:按原 `code-it` 行为(报错 "任务编码不存在") — 守卫不触达(由原行为先拦)
- (理由:任务合法性是 `code-it` 原职责,守卫**不**重复)

#### S-5:与 `code-auto` 协同(边界)
- 假设 REQ-00007 `code-auto` 按"任务总览行序"调 `code-it`,本应调 `TASK-001` 之后调 `TASK-002`
- 技能执行:
  1. `code-auto` 调 `code-it TASK-001` → 守卫通过(无前置)→ 完成
  2. `code-auto` 调 `code-it TASK-002` → 守卫通过(前置 TASK-001 已完成)→ 完成
- (本需求守卫**不**改变 `code-auto` 现有"按序"行为;`code-auto` 内部"按序"与本需求"按登记顺序判定前置"**协同**)

#### S-6:用户故意乱序(边界)
- 用户输入:`/code-it TASK-REQ-00005-003`(跳到第 3 任务)
- 守卫不通过 → 中止 + 打印推荐命令(同 S-1)

#### S-7:无激活版本
- 用户输入:`/code-it`,但 `.current-version` 不存在
- 技能执行:提示用户调 `code-version`,退出

## 4. 功能需求(FR)

### FR-1:新增"步骤 0a 前置任务检查"守卫(Q-1 锁定)

- **描述**:`code-it` 在"步骤 0 版本上下文检测"**之前**,新增"步骤 0a 前置任务检查"
- **"前置任务"定义**:**当前任务**在 `PLAN.md` 任务总览区段中**前面**(按文件行序)的所有任务
- **"未完成"判定**(Q-2 锁定 A):开发状态 ∈ {`待开始` / `进行中` / `阻塞`}
- **优先级**:必须
- **AC**:
  - AC-1.1:`code-it/SKILL.md` 在"步骤 0"前显式列出"步骤 0a"
  - AC-1.2:从 `plan/REQ-NNNNN/PLAN.md` 任务总览区段按**文件行序**解析所有任务
  - AC-1.3:当前任务 = 入参 `TASK-...`,从任务总览中找到其位置
  - AC-1.4:前置任务 = 任务总览中**位于当前任务之前**的所有任务
  - AC-1.5:任一前置任务开发状态 ≠ `已完成` → 进入 FR-2 中止流程
  - AC-1.6:全部前置任务开发状态 = `已完成` → 进入原"步骤 0"+ 后续流程
  - AC-1.7:`PLAN.md` 不存在 → 视为"无 PLAN.md,无法判定前置",**通过**守卫(退化)
  - AC-1.8:不修改 frontmatter;不修改"步骤 0"及之后的原有内容(增量追加)

### FR-2:守卫不通过 → 中止 + 打印推荐执行命令(Q-3 锁定)

- **描述**:`code-it` 守卫不通过时:
  1. 中止执行
  2. 屏幕打印"⛔ code-it 中止(存在未完成的前置任务)"
  3. 列出**所有**前置任务 + 各自状态(✓ 已完成 / ✗ 未完成)
  4. **打印推荐执行命令**:`推荐执行 /code-it TASK-NNNNN-NNNNN 完成后,再执行 /code-it TASK-NNNNN-NNNNN`(复制即可)
  5. 退出码 ≠ 0
- **优先级**:必须
- **AC**:
  - AC-2.1:屏幕输出"⛔ code-it 中止(存在未完成的前置任务)"+ 当前任务编码 + 所在需求
  - AC-2.2:列出所有前置任务 + 各自开发状态(✓/✗)
  - AC-2.3:输出"推荐执行 /code-it <第一个未完成前置> 完成后,再执行 /code-it <当前任务>"
  - AC-2.4:退出码 ≠ 0
  - AC-2.5:不调用任何 `Bash` 写代码命令
  - AC-2.6:不写 `code/<任务编码>/RESULT.md`
  - AC-2.7:不修改看板"任务清单"区段

### FR-3:不修改 `code-it` 现有"无前置"流程

- **描述**:守卫通过时,`code-it` 走原流程**不**变
- **优先级**:必须
- **AC**:
  - AC-3.1:守卫通过 → `code-it` 现有"步骤 0 ~ N"全部**不**变

```

  ↓ (通过)                    │ (未通过 → FR-2 中止流程)
[步骤 0:版本上下文检测]     │
  ↓                            │
[步骤 1-?:原有流程]          │
  ↓                            │
[退出]                         │
                               │
[打印中止报告 + 推荐命令]     │
  ↓                            │
[退出码 ≠ 0]                  │
  ↓                            │
[退出] ←───────────────────────┘
```

### 7.2 前置任务检查算法(Q-1/Q-2 锁定)

```
[解析 plan/REQ-NNNNN/PLAN.md 任务总览区段]
  ↓
[按文件行序得到任务列表 [T1, T2, T3, ..., TN]]
  ↓
[找到当前任务 Ti 的位置 i]
  ↓
[前置任务列表 = [T1, T2, ..., T(i-1)]]
  ↓
[对每个前置任务 T_j (j < i)]
```
