## 1. 需求概述

**为谁**:`code-skills` 仓库的用户(在长开发周期末,需要将本工具/或用本工具开发的项目发布到生产环境)。

**解决什么问题**:`code-skills` 7 个技能(`code-version` / `code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-review`)都聚焦**开发周期**;但**没有任何技能**覆盖"开发完成 → 部署上线"这一**最后一步**。用户当前要发布,必须:
- 手动核对看板 3 大区段(需求/任务/缺陷)是否全部"解决"
- 手动撰写 DEPLOY.md / UPDATE.md / Q&A 手册
- 跨多个文档拼装"打包 → 上传 → 初始化 → 启动"步骤

**带来什么价值**:
1. **自动化前置检查**:`code-publish` 调一次即知"是否可发布",并指出"哪个需求/任务/缺陷尚未解决"
2. **3 份手册自动生成**:在 `assistants/<版本号>/publish/` 下产出 `DEPLOY.md` + `UPDATE.md` + `Q&A.md`(基线版本跳过 UPDATE),均含 placeholder 供用户补全
3. **`qanda/` 目录骨架**:本需求顺带在项目级创建 `assistants/qanda/` 目录与 README.md,作为发布 Q&A 的**长期沉淀位置**

## 2. 背景与目标

### 2.1 背景
- `code-skills` 7 个技能中,`code-review` 是开发周期内的"最后一步";**真正意义的"开发完成"由看板"任务清单"区段的"真正可发布"统计锚定**
- V0.0.1 看板已建立"任务清单 / 缺陷清单 / 需求清单"3 大区段的结构化字段,本需求直接消费
- `code-dashboard` (REQ-00004) 提供"展示状态",本需求提供"驱动发布" — 两者共享同一看板数据源
- `code-rule` 维护的项目级规范目录 `assistants/rules/` 中**无**"发布规范";本需求不创建该规范(留作 follow-up)

### 2.2 业务目标
- **G-1**:新增第 9 个 `code-*` 技能 `code-publish`,与现有 8 个平级

- **G-4**:3 份手册内容 = 通用发布部署手册骨架 + 默认内容(Q-4 锁定 C);用户**必须**手动补全
- **G-5**:项目级 `assistants/qanda/` 目录由本需求顺带创建(Q-2 锁定 A)

### 2.3 本次目标

## 3. 用户角色与场景

### 3.1 角色
- **R-1 发布者**:每个开发周期末,调 `code-publish` 完成前置检查 + 生成手册
- **R-2 运维 / 部署工程师**:阅读 `DEPLOY.md` / `UPDATE.md` 执行具体部署
- **R-3 现场支持**:遇到问题时查阅 `Q&A.md` 自行解决

### 3.2 关键场景

#### S-1:可发布版本(主流程,前置检查通过)
- 用户输入:`/code-publish`(当前版本 V0.0.2,所有需求/任务/缺陷已解决)
- 技能执行:
  1. **步骤 0**:版本上下文检测(Q-5 采纳默认 — 不检查里程碑)
  2. **步骤 1 前置检查**:遍历需求清单 / 任务清单 / 缺陷清单 3 区段
     - 需求:全部 `状态=已完成`
     - 任务:全部 `开发状态=已完成` ∧ `测试状态∈{已运行-通过, 不适用}`
     - 缺陷:全部 `状态=已修复`
  3. 通过 → **步骤 2 生成手册**
     - 创建 `assistants/V0.0.2/publish/`
     - 写入 `DEPLOY.md`(通用骨架 + 默认内容)
     - **本版本非基线(V0.0.0 才是基线)** → 写入 `UPDATE.md`(从 V0.0.1 升级到 V0.0.2 的步骤)
     - 写入 `Q&A.md`(聚合 `qanda/` 目录中与发布相关的内容)
  4. **步骤 3 报告**:
     ```
     ✓ 发布前置检查通过(需求 2/2, 任务 N/N, 缺陷 0/0)
     
     已生成 3 份手册:
       - assistants/V0.0.2/publish/DEPLOY.md  (全新部署)
       - assistants/V0.0.2/publish/UPDATE.md  (从 V0.0.1 升级)
       - assistants/V0.0.2/publish/Q&A.md     (常见问题)
     
     ⚠ 提示: 3 份手册均为通用骨架,请手动补全 <软件名>、<版本号>、<服务器地址> 等占位符
     ⚠ 提示: 本技能不自动提交,补全后请手动 git add + commit
     ```

#### S-2:有未完成任务(边界,前置检查不通过)
- 用户输入:`/code-publish`(假设 V0.0.2 有 1 个任务未完成)
- 技能执行:
  1. **步骤 1 前置检查**:发现 1 个未完成
  2. 不通过 → **步骤 2 报告**(不生成手册)
  3. 用户看到:
     ```
     ✗ 发布前置检查未通过
     
     未完成项明细:
       - [需求] REQ-00005 状态=进行中(应该=已完成)
       - [任务] TASK-REQ-00005-005 开发状态=进行中(应该=已完成)
       - [缺陷] BUG-00003 状态=待修复(应该=已修复)
     
     阻塞统计:
       - 需求:1 / 2 未完成
       - 任务:1 / N 未完成
       - 缺陷:1 / K 未修复
     
     ✗ 未生成任何手册。请先解决上述项后重试。
     ```

#### S-3:基线版本(边界,不生成 UPDATE.md)
- 用户输入:`/code-publish V0.0.0`(V0.0.0 是项目基线,无"上一版本")
- 技能执行:
  1. 前置检查通过
  2. 检测本版本 = 基线(`./assistants/V0.0.0` 是 V0.0.0 ~ V0.0.x 中**最早**且为"初始化"里程碑)
  3. **跳过** UPDATE.md 生成(Q-3 锁定 A)
  4. 用户看到:
     ```
     ✓ 发布前置检查通过(本版本 V0.0.0 是基线)
     
     已生成 2 份手册(基线无 UPDATE.md):
       - assistants/V0.0.0/publish/DEPLOY.md
       - assistants/V0.0.0/publish/Q&A.md
     ```

#### S-4:`qanda/` 目录为空(常见)
- 用户输入:`/code-publish`(V0.0.2,检查通过)
- 技能执行:
  1. 前置检查通过
  2. 尝试从 `assistants/qanda/` 读取 Q&A 内容
  3. 目录空(本需求创建的骨架仅含 README.md,无 Q&A 内容)
  4. 生成 `Q&A.md`,含 1 个 placeholder 章节"## 占位:常见问题(待补充)"
  5. 用户看到:
     ```
     ⚠ assistants/qanda/ 目录为空(仅有 README.md),Q&A.md 仅为占位
     请先在 assistants/qanda/ 中添加 Q&A 内容,再重跑 code-publish
     ```

#### S-5:无激活版本(边界)
- 用户输入:`/code-publish`,但 `.current-version` 不存在
- 技能执行:
  1. **步骤 0**:检测失败 → 提示用户调 `code-version` 创建版本
  2. 退出

#### S-6:用户希望强制发布(超越检查)
- 用户输入:`/code-publish --force`
- 技能执行:
  1. **步骤 1 前置检查**:**仍执行**(不跳过)
  2. 不通过 → 打印"未通过 + 详细未完成项"(同 S-2)
  3. 提供 `AskUserQuestion`:
     - A. 仍然生成手册(标记为"未达发布标准")
     - B. 取消
  4. 选 A → 生成手册,但在文件首页加显式警告:
     ```
     > ⚠ 本手册对应版本未通过发布前置检查。
     > 跳过项:<N>
     > 生成时间:<时间>
     ```

## 4. 功能需求(FR)

### FR-1:发布前置检查(全检查最严)

- **描述**:`code-publish` 启动后,首先执行"发布前置检查",遍历当前版本 `RESULT.md` 的 3 大区段
- **判定规则**(Q-1 锁定 A):
  - **需求**:状态 = `已完成`
  - **任务**:开发状态 = `已完成` ∧ 测试状态 ∈ {`已运行-通过`, `不适用`}
  - **缺陷**:状态 = `已修复`
- **不通过** → 打印"未完成项明细 + 阻塞统计",**不**生成任何手册,退出
- **通过** → 进入 FR-2
- **优先级**:必须
- **AC**:
  - AC-1.1:遍历 `需求清单 / 任务清单 / 缺陷清单` 3 区段,**只**用正则/字符串匹配,不修改看板
  - AC-1.2:任一区段存在"未解决"项 → 不通过,打印该区段的全部未解决项(含编码 + 标题 + 当前状态 + 期望状态)
  - AC-1.3:全部通过 → 进入 FR-2
  - AC-1.4:看板"里程碑"区段**不**纳入检查(Q-5 采纳默认)
  - AC-1.5:不修改看板文件本身(纯只读解析)

### FR-2:生成 3 份手册(基线版本跳过 UPDATE)

- **描述**:前置检查通过后,在 `assistants/<版本号>/publish/` 目录生成:
  - `DEPLOY.md`(**始终生成**)
  - `UPDATE.md`(**仅当本版本非基线**)
  - `Q&A.md`(**始终生成**)
- **优先级**:必须
- **AC**:
  - AC-2.1:目录 `assistants/<版本号>/publish/` 不存在 → `mkdir -p` 创建
  - AC-2.2:`DEPLOY.md` 写入,内容为"通用发布部署手册骨架 + 最常见部署方式默认内容"
  - AC-2.3:本版本是基线 → **不**生成 `UPDATE.md`;非基线 → 生成 `UPDATE.md`,内容为"从 <前一版本> 升级到 <本版本>"骨架
  - AC-2.4:`Q&A.md` 写入,内容 = `assistants/qanda/` 目录下所有与发布相关文件的合并;目录为空时,内容仅为占位
  - AC-2.5:每份手册**不**自动 commit(Q-9 采纳默认)— 留 dirty tree 给用户审阅

### FR-3:DEPLOY.md 模板内容(通用骨架)

- **描述**:`DEPLOY.md` 是"通用发布部署手册骨架",内含用户**必须**手动补全的 placeholder + "最常见部署方式"默认内容
- **优先级**:必须
- **章节结构**:
  1. 概述(版本号 / 发布时间 / 目标环境)
  2. 打包(`<打包方式>` 选 1:N 种 / 软件包 / 目录 / 镜像)
  3. 获取成果物
  4. 上传服务器(可选)
  5. 初始化系统
     - 5.1 环境准备(`<环境依赖>`)
     - 5.2 数据库建表语句执行(`<DB 脚本路径>`)
     - 5.3 数据库初始化数据(`<初始化数据脚本>`)
     - 5.4 配置修改(`<配置文件路径>`)
  6. 启动运行(`<启动命令>`)
  7. 首次进入软件系统(`<首次访问 URL / 默认账号>`)
  8. 验证清单(checklist)
- **AC**:
  - AC-3.1:每节用 `<placeholder>` 标记需补全字段
  - AC-3.2:每节**至少**含 1 个"最常见部署方式"示例(占位但可执行),例如:打包节含 `tar -czf <output>.tar.gz <source>` 默认命令
  - AC-3.3:在文首用 1 段"使用说明"提醒用户:必须补全所有 `<placeholder>` 后再按本手册执行

### FR-4:UPDATE.md 模板内容(从上一版本升级)

- **描述**:`UPDATE.md` 是"通用升级部署手册骨架",与 `DEPLOY.md` 类似但聚焦"升级"动作
- **优先级**:必须
- **章节结构**:
  1. 概述(源版本 / 目标版本 / 升级方式:原地 / 蓝绿 / 灰度)
  2. 打包
  3. 获取成果物
  4. 上传服务器(可选)
  5. 初始化系统
     - 5.1 数据库变更语句(`<DB 升级脚本>`)
     - 5.2 配置变更(`<配置 diff>`)
  6. 重新启动软件
  7. 验证清单
  8. **回滚方案**(本节为 `DEPLOY.md` 之外的**专属**内容)
- **AC**:
  - AC-4.1:章节结构同 `DEPLOY.md`,但**新增**"回滚方案"节
  - AC-4.2:含 `<源版本>` / `<目标版本>` placeholder
  - AC-4.3:本版本是基线 → **不**生成(由 AC-2.3 控制)

### FR-5:Q&A.md 模板内容(从 `qanda/` 抽取)

- **描述**:`Q&A.md` 内容来自 `assistants/qanda/` 目录中"与发布部署相关"的文件;目录为空时,生成占位
- **优先级**:必须
- **数据源**:`./assistants/qanda/*.md`
- **筛选规则**:本需求 v1 不实现智能筛选;**全部**聚合(用户自行组织)
- **AC**:
  - AC-5.1:`qanda/` 目录不存在 → 生成 1 段占位 + 提示"请先在 `qanda/` 添加内容,再重跑"
  - AC-5.2:`qanda/` 目录存在但**仅含** README.md → 同 AC-5.1
  - AC-5.3:`qanda/` 目录存在且含**其他** .md → 全部聚合到 `Q&A.md`,每节对应 1 个源文件
  - AC-5.4:聚合时,在每节加 `> 来源:qanda/<文件名>` 标注

### FR-6:创建 `assistants/qanda/` 目录骨架(本需求顺带)

- **描述**:本需求顺带在项目级创建 `assistants/qanda/` 目录与 README.md(Q-2 锁定 A)
- **优先级**:必须
- **AC**:
  - AC-6.1:`mkdir -p assistants/qanda/`
  - AC-6.2:写入 `assistants/qanda/README.md`,说明:
    - 目录用途(Q&A 长期沉淀)
    - 文件命名建议(`<主题>.md`,如 `deploy-faq.md` / `db-init-faq.md`)
    - 引用规范(被 `code-publish` 聚合)
    - 维护方式(暂时人工整理)
  - AC-6.3:README.md 是项目级共享文件,不受任何版本号控制

### FR-7:错误处理与边界(继承 8 技能的约定)

- **描述**:任何异常不崩溃,采用与 `code-dashboard` 相同的"软失败 + 提示"模式
- **优先级**:必须
- **AC**:
  - AC-7.1:无 `.current-version` → 提示调 `code-version`,退出
  - AC-7.2:`RESULT.md` 缺区段 → 退化到"全未解决"判定(不通过,提示看板不完整)
  - AC-7.3:`publish/` 目录写入失败(权限等) → 报错退出,**不**留半成品
  - AC-7.4:`qanda/` 目录创建失败(权限等) → 跳过 Q&A.md 占位,正常完成 DEPLOY/UPDATE,**不**整体失败

### FR-8:不修改其他 8 技能 / 不修改 `code-it` / 不修改 marketplace 与 plugin

- **描述**:本需求只新增 `code-publish` 技能 + 创建 `qanda/` + 创建 `publish/`;不修改其他任何已有产物
- **优先级**:必须
- **AC**:
  - AC-8.1:`marketplace.json` / `plugin.json` **不**被本需求修改
  - AC-8.2:其他 8 个 `code-*` 技能 SKILL.md **不**被本需求修改
  - AC-8.3:`assistants/rules/` 下任何规范文件**不**被本需求修改
  - AC-8.4:`commit-conventions.md` 规则 1 **不**被本需求填充(留作 follow-up)

### FR-9:报告与建议

- **描述**:`code-publish` 执行完毕后,向用户报告:
- **优先级**:必须
- **AC**:
  - AC-9.1:成功时:输出"前置检查通过 + 3 份手册已生成 + 警告 placeholder 需补全 + 提示不自动 commit"
  - AC-9.2:不通过时:输出"未完成项明细 + 阻塞统计 + 不生成任何手册 + 提示去解决"
  - AC-9.3:基线时:输出"已生成 2 份手册(基线无 UPDATE)"

## 5. 非功能需求 / 约束(NFR)

### NFR-1:零新增依赖
- **描述**:不引入新依赖;复用现有 `Bash` / `Read` / `Write` / `Edit` / `Glob` / `Grep` 工具
- **强制级别**:必须

### NFR-2:纯只读检查
- **描述**:前置检查(FR-1)对看板**只读**;**不**修改 `RESULT.md`
- **强制级别**:必须

### NFR-3:不自动 commit
- **描述**:`code-publish` 自身**不**触发 `git commit`(Q-9 采纳默认);手册写入后留 dirty tree 给用户审阅
- **强制级别**:必须
- **理由**:手册内容需用户补全,立刻 commit 会"封死"用户补全空间

### NFR-4:不参与 REQ-00005 的"首步拉取+末步提交"
- **描述**:`code-publish` 不在 REQ-00005 改写范围内(Q-7 采纳默认);用户应在"最新上下文"中调用(由 REQ-00005 之后的会话状态保证)
- **强制级别**:必须

### NFR-5:通用性优先
- **描述**:`code-publish` 是**通用发布脚手架**,不绑定 `code-skills` 自身发布(Q-4 锁定 C);手册内容为"通用骨架 + 默认示例"
- **强制级别**:必须
- **理由**:用户开发软件各异、发布环境不同

### NFR-6:手册骨架可重复生成
- **描述**:重跑 `code-publish` 应**幂等**(再次生成同样的骨架)
  - 若 `DEPLOY.md` / `UPDATE.md` / `Q&A.md` 已存在 → **覆盖**(与 V0.0.1+ 看板模板初始化方式一致)
  - 提示用户已覆盖
- **强制级别**:必须

### NFR-7:基线识别
- **描述**:`code-publish` 识别"本版本是基线"的规则:
  - 规则 1(采纳默认):本版本**无任何**前一版本(如 V0.0.0 在所有版本号中最小)
  - 规则 2(替代):本版本"里程碑"段含"M0:工作空间就绪"且为初始
  - 规则 3(替代):本版本目录下无 `code/` / `test/` 子目录(说明从未编码)
- **强制级别**:必须
- **实现侧**:采用规则 1(最简);规则 2/3 留作 v2

### NFR-8:与 `code-dashboard` 数据源一致
- **描述**:`code-publish` 的"解决状态"判定必须**等于或严于** `code-dashboard` 的"真正可发布"概念
- **强制级别**:必须
- **理由**:避免出现"dashboard 显示全完成,publish 却报有未完成"

### NFR-9:可读性优先
- **描述**:DEPLOY.md / UPDATE.md / Q&A.md 模板**优先** Markdown 排版清晰度(标题层级、表格、代码块),便于用户阅读
- **强制级别**:必须

## 9. 边界与异常

| ID | 场景 | 处理 |
| --- | --- | --- |
| **E-1** | 前置检查不通过 | 打印"未完成项明细 + 阻塞统计" + 不生成任何手册 + 退出(FR-1.AC-1.2) |
| **E-2** | `RESULT.md` 缺区段 | 退化到"全未解决"判定 + 报告"看板不完整" + 退出 |
| **E-3** | 本版本是基线 | 跳过 UPDATE.md(Q-3 锁定 A) |
| **E-4** | `qanda/` 目录不存在 | **本需求创建骨架**(Q-2 锁定 A);若创建失败,跳过 Q&A.md 占位 + 正常完成 |
| **E-5** | 无 `.current-version` | 提示调 `code-version`,退出(同其他 8 技能) |
| **E-6** | `publish/` 目录已存在 + 含 `DEPLOY.md` | **覆盖** + 提示"已覆盖" |
| **E-7** | `publish/` 目录写入失败(权限) | 报错退出,**不**留半成品 |
| **E-8** | 用户希望强制发布(超检查) | 走 S-6 流程;选 A 时在手册首页加显式警告 |
| **E-9** | 看板"任务清单"含旧格式任务编号(`REQ-NNNNN-NNNNN`) | 视为字符串处理(同 NFR-8);不强制要求新格式 |
| **E-10** | 用户多次重跑 | NFR-6 幂等 — 覆盖;提示已覆盖 |
