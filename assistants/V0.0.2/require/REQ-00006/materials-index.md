## 关键信息抽取(供后续步骤使用)

### 1. 技能名与归属
- `/code-publish`(对应技能目录 `code-publish`,遵循 `skill-conventions.md §规则 1` 的 kebab-case 命名)
- 新增第 9 个 `code-*` 技能(继 V0.0.2 REQ-00004 `code-dashboard`、REQ-00005 之后)

### 2. 核心需求拆解
- **步骤 1:发布前置检查**
  - 检查当前版本下"未完全结束的需求、任务、缺陷"
  - **必须全部为"解决状态"** 才允许进入发布流程
- **步骤 2:生成 3 份手册**(均写入 `assistants/<版本号>/publish/`)
  - **`DEPLOY.md`**:全新部署手册
    - 打包(软件包 / 目录 / 镜像等)
    - 获取成果物
    - 上传服务器(如有需要)
    - 初始化系统(环境准备、数据库建表语句、初始化数据、配置修改等)
    - 启动运行
    - 首次进入软件系统
  - **`UPDATE.md`**:从上一版本升级的部署手册
    - 打包 / 获取成果物 / 上传服务器 / 初始化系统(DB 变更、配置变更)/ 重启
  - **Q&A 手册**:可从 `assistants/qanda/` 获取与发布部署相关的内容

### 3. "解决状态"的判定(用户原文"如果所有的需求、任务、缺陷都已是解决状态")
- **需求解决** = 状态 ∈ {`已完成`}(`code-require` 看板"需求清单"列)
- **任务解决** = 开发状态 = `已完成` ∧ 测试状态 ∈ {`已运行-通过`, `不适用`}(看板"任务清单"列,对应"真正可发布")
- **缺陷解决** = 状态 ∈ {`已修复`}(看板"缺陷清单"列)
- **未结束 = 不属于"解决状态"**:任何"待开始 / 进行中 / 阻塞 / 已取消 / 未修复" 都不算解决
- 见 clarifications.md Q-1

### 4. 边界
- **`qanda/` 目录不存在**:用户原文说"可从 `assistants/qanda` 目录中获取" — 该目录目前**不存在**,需要明确处理(Q-2)
  - 选项 A:本次需求顺带创建 `assistants/qanda/` 占位结构(项目级)
  - 选项 B:本次需求**不**创建,仅在 Q&A 手册中提示"qanda 目录为空,请补充"
  - 选项 C:本需求**不**消费 qanda,改为"由用户在 Q&A 手册中自由补充"
- **"上一版本"指代**(DEPLOY.md 与 UPDATE.md 区别)
  - Q-3:本版本是 V0.0.0 / V0.0.1 等"基线"时,UPDATE.md 应如何处理?
  - 选项 A:不生成 UPDATE.md(基线无"上一版本")
  - 选项 B:生成 UPDATE.md,内容为"本版本是初始版本,无升级步骤"
  - 选项 C:列出所有历史版本,让用户选"从哪个版本升级"
- **"软件包/目录/镜像等"的多形态**:用户原话是 OR 关系 — 不同项目形态不同
  - Q-4:本需求的发布形态如何确定?
