## 1. 需求概述

**为谁**:`code-skills` 仓库的 AI 协作者(在长会话或多次调用之间),以及项目主导者(需要在每个工作日/会话快速判断"接下来做什么")。

**解决什么问题**:当前 `code-skills` 技能集合(`code-version` / `code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-review`)都**只往** `./assistants/<版本号>/RESULT.md` **写入**数据,但**没有任何技能"读取"这个看板来汇报全局状态**。用户(或 AI)在长会话中要回答下列问题,必须:
- 手动 `Read` 看板(整文件 200~400 行,上下文开销大)
- 跨多个 `RESULT.md` / `PLAN.md` / `code/.../RESULT.md` 拼接
- 自己汇总"哪些任务未完成 / 哪些缺陷未修复"

**带来什么价值**:
1. **一键总览**:调 `/code-dashboard` 即可看到"当前激活版本"的整体执行情况(需求/任务/缺陷 + 里程碑)
2. **下一步建议**:技能基于状态自动给出"应执行哪个 `code-*` 命令",降低会话中的"路径选择"摩擦
3. **双粒度支持**:
   - 不指定需求编号 → 版本总览(ASCII 进度表 + 文本柱状图,Q-1 锁定 A)
   - 指定需求编号 → 该需求的进度 + 其下任务的进度概览

## 2. 背景与目标

### 2.1 背景
- `code-skills` 已有 7 个 `code-*` 技能,**全是"写入型"**(向 `./assistants/` 沉淀产出)
- 唯一的"读取型"入口是 `code-review`,但它只读 `PLAN.md` 与任务代码,不读整个版本看板
- 看板(`./assistants/<版本号>/RESULT.md`)是**唯一事实来源**,但没有任何技能主动消费它做汇报
- 用户的真实工作流(观察 V0.0.0 → V0.0.1 → V0.0.2 推进)需要频繁回头看"还差什么"

### 2.2 业务目标
- **G-1**:新增第 8 个 `code-*` 技能 `code-dashboard`,与现有 7 个平级
- **G-2**:`code-dashboard` 是**只读型**技能(不修改任何文件,只产出屏幕输出)
- **G-3**:`code-dashboard` 支持**双粒度**输入:无参数(版本总览)/ 指定 `REQ-NNNNN`(需求粒度)
- **G-4**:`code-dashboard` 自动给出"下一步执行建议",推荐具体可执行命令(Q-2 锁定 A)

### 2.3 本次目标
- 新增技能目录 `plugins/code-skills/skills/code-dashboard/`,含 `SKILL.md` + 必要的过程文档
- 不修改 `marketplace.json` / `plugin.json`(继承 V0.0.1 已确立的"不修改范围")
- 不修改其他 7 个 `code-*` 技能的 `SKILL.md`(本技能与其他 7 个**正交**)
- 不修改 `templates/*.md`(本技能**不**产生 `RESULT.md` 文档类输出,仅屏幕打印)
- 若需要,在 `plugins/code-skills/CLAUDE.md` 的"AI 工作约定"小节追加一段(由 `code-rule` 沉淀,非本需求必做)
- 若需要,在 `plugins/code-skills/README.md` + `README.en.md` 的"主要能力"中追加一行(同次提交,遵循 `doc-conventions §规则 1`)

## 3. 用户角色与场景

### 3.1 角色
- **R-1 项目主导者**:每个工作日开头/每次会话开头,调 `/code-dashboard` 看到"还差什么"
- **R-2 AI 协作者**:在长会话中,需要决定"下一步调哪个 `code-*` 命令",调 `code-dashboard` 获得建议
- **R-3 评审/复盘者**:查看某历史版本(切到 V0.0.0 / V0.0.1)的最终状态(Q-5 留作 follow-up)

## 4. 功能需求(FR)

### FR-1:`code-dashboard` 技能定义与元信息

- **描述**:新增 `plugins/code-skills/skills/code-dashboard/SKILL.md`,遵循 `skill-conventions.md §规则 1`:
  - YAML frontmatter 含 `name: code-dashboard` + 一句完整 `description`
  - `name` 与目录名 `code-dashboard` 严格一致
- **优先级**:必须
- **AC**:
  - AC-1.1:`SKILL.md` 顶部 YAML 含 `name: code-dashboard`
  - AC-1.2:`SKILL.md` 顶部 YAML 含 `description`,内容涵盖"展示当前版本需求/任务/缺陷执行情况 + 下一步建议"
  - AC-1.3:`name` 与目录名完全一致

### FR-2:版本总览(无参数模式)

- **描述**:`code-dashboard` 在无参数时,展示"当前激活版本"的整体执行情况,含 4 段:
  1. **需求进度**:总计 / 已完成 / 进行中 / 待开始(从看板"需求清单"区段聚合)
  2. **任务进度**:总计 / 已完成 / 测试通过 / 真正可发布(从看板"任务清单"区段聚合)
  3. **缺陷(高优先级)**:P0 待修复 + P1 待修复(从看板"缺陷清单"区段聚合,Q-3 锁定 B)
  4. **下一步建议**:基于前 3 段自动推断(见 FR-3)
- **优先级**:必须
- **AC**:
  - AC-2.1:无参数时,4 段全部输出,顺序固定
  - AC-2.2:每段含 ASCII 进度表(横线 + 字段名 + 数字),数字与看板数据严格一致
  - AC-2.3:任务进度段含"任务进度比例"的可视化条(例:`[████████░░] 80%`)
  - AC-2.4:缺陷段仅显示 P0 + P1 数量(P2 / P3 计入统计但不单列;若用户需要全部,留作 v2)
  - AC-2.5:所有数字来自看板对应区段的解析,**不**估算、**不**近似

### FR-3:需求粒度查询(指定 `REQ-NNNNN`)

- **描述**:`code-dashboard REQ-NNNNN` 展示指定需求的进度 + 其下任务 + 关联缺陷
- **优先级**:必须
- **AC**:
  - AC-3.1:参数为 `REQ-NNNNN` 格式时,定位 `require/REQ-NNNNN/RESULT.md` + `plan/REQ-NNNNN/PLAN.md`
  - AC-3.2:输出"需求元信息"(标题 / 状态 / 概要设计状态 / 详细设计状态)
  - AC-3.3:输出"任务清单"(从 PLAN.md 解析),列为:任务编码 / 标题 / 开发状态 / 测试状态
  - AC-3.4:输出"关联缺陷"(从看板"缺陷清单"区段筛 `关联任务` 含 `REQ-NNNNN-` 的缺陷)
  - AC-3.5:输出"下一步建议"(见 FR-4)

### FR-4:下一步建议生成

- **描述**:`code-dashboard` 基于当前状态,自动推断"接下来应做什么",输出**可执行命令推荐**(Q-2 锁定 A)
- **优先级**:必须
- **优先级策略**(实现侧决策):
  1. **未完成需求** → `建议: 执行 /code-require 添加首个需求` 或 `建议: 执行 /code-design REQ-NNNNN`(若需求已存在但缺设计)
  2. **未完成任务** → `建议: 执行 /code-it TASK-REQ-NNNNN-NNNNN` 或 `建议: 执行 /code-it REQ-NNNNN-NNNNN`(兼容旧格式,见 NFR-3)
  3. **未修复缺陷(按严重度)** → `建议: 执行 /code-fix BUG-NNNNN`(P0 优先;P1 次之)
  4. **未跑测试** → `建议: 执行 /code-unit TASK-...` 或 `建议: 执行 /code-review REQ-NNNNN`
  5. **全完成** → `该版本已全部完成,可发布/归档` + 提示调 `code-version <新版本号>`
- **AC**:
  - AC-4.1:每条建议含 3 字段:命令 / 依据 / 优先级(高/中/低)
  - AC-4.2:命令格式严格匹配 `code-*` 技能调用的真实语法(以现有 SKILL.md 为准)
  - AC-4.3:建议最多 5 条(超出取优先级最高的前 5 条)
  - AC-4.4:无任何建议触发时,显示"无后续动作" + 版本状态(已发布/可发布)

### FR-5:无激活版本时的引导

- **描述**:检测 `./assistants/.current-version` 不存在时,打印引导信息并退出(Q-4 留作默认,本 FR 落此默认)
- **优先级**:必须
- **AC**:
  - AC-5.1:无 `.current-version` 时,输出 `✗ 未检测到激活的版本工作空间` + 提示 `code-version` 命令
  - AC-5.2:不读任何其他文件,直接退出

### FR-6:参数校验与边界

- **描述**:`code-dashboard REQ-NNNNN` 中需求编号在当前版本不存在时,提示用户(Q-4 同源)
- **优先级**:必须
- **AC**:
  - AC-6.1:需求编号在当前版本不存在时,列出本版本所有需求编号(若有)
  - AC-6.2:无任何需求时,显示 `(无)` 占位
  - AC-6.3:参数格式不匹配 `^REQ-\d{5}$` 时,显示 `✗ 参数格式错误: REQ-NNNNN` + 用法示例

### FR-7:纯只读,不修改任何文件

- **描述**:`code-dashboard` 在执行过程中**不**调用 `Write` / `Edit` / `Bash` 写命令;**只**调用 `Read` / `Glob` / `Grep`(与 `code-review` 行为一致)
- **优先级**:必须
- **AC**:
  - AC-7.1:执行后 `git status` 仍为 `clean`(无新增/修改文件)
  - AC-7.2:不写 `RESULT.md` / `work-log.md` / `deviations.md`(不产生过程文档)

## 5. 非功能需求 / 约束(NFR)

### NFR-1:零外部依赖
- **描述**:`code-dashboard` 不引入新的 npm/pip 依赖;所有输出基于 `Read` / `Glob` / `Grep` 工具 + Markdown/ASCII 字符构造
- **强制级别**:必须
- **理由**:与现有 7 个 `code-*` 技能一致,均不引入运行时依赖(见 `plugin.json` 的 `dependencies` 段)

### NFR-2:错误处理
- **描述**:任何"非预期状态"(看板缺失、字段缺失、表格列错位)均**不崩溃**,而是:
  - 缺文件 → 显示 `✗ 看板文件不存在,可能需要先调 code-version`
  - 字段缺失 → 显示 `?` 或 `(未知)` 占位
  - 表格列错位 → 退化到"原始 markdown 块"显示(读出表格原样)
- **强制级别**:必须

### NFR-3:任务编号格式兼容
- **描述**:`code-dashboard` 解析任务编号时,**同时**支持两种字面格式(见 `related-requirements.md`):
  - **新格式**:`TASK-REQ-NNNNN-NNNNN`(符合 `encoding-conventions.md §规则 3`)
  - **旧格式**:`REQ-NNNNN-NNNNN`(V0.0.1 REQ-00001 起的 4 任务字面,存在于 V0.0.1 看板)
- **强制级别**:必须
- **理由**:不解析旧格式会导致 REQ-00001 任务在 `code-dashboard` 中显示为空
- **新格式优先**:解析出嵌套式后,生成的 `/code-it TASK-...` 命令严格按新格式
- **旧格式透传**:仅作为字符串原样显示,不进入"路径解析"路径

### NFR-4:性能
- **描述**:`code-dashboard` 在 V0.0.x 典型规模(< 50 需求 / < 200 任务 / < 50 缺陷)下,从启动到完整输出 < 5 秒
- **强制级别**:必须
- **实现侧手段**:
  - 单次 `Read` 读整个 `RESULT.md`(避免多次 IO)
  - 解析表格时**只**读区段(行号定位)
  - 需求粒度模式:并行 `Read` 3 个文件(`require/.../RESULT.md` + `plan/.../PLAN.md` + 主 `RESULT.md` 缺陷段)

### NFR-5:看板字段约定扩展需三处同步
- **描述**:若 `code-dashboard` 在执行中发现看板字段不足,**不**主动修改看板;若必须扩展字段(例如想新增"最近一次 dashboard 调用时间"区段),按 `dashboard-conventions.md §规则 1`:
  - 同一次提交同步 3 处:看板模板 + CLAUDE.md"看板字段约定"段 + dashboard-conventions.md 自身
  - 本需求**不**触发此约束(`code-dashboard` 纯只读,不写看板)

### NFR-6:不修改其他技能
- **描述**:`code-dashboard` 不修改 `marketplace.json` / `plugin.json` / 其他 7 个 `code-*` 技能的 `SKILL.md`(继承 V0.0.1 REQ-00003 的"FR-9 边界"精神)
- **强制级别**:必须

### NFR-7:可重复执行
- **描述**:`code-dashboard` 多次执行结果**幂等**(纯只读)
- **强制级别**:必须

## 9. 边界与异常

| 场景 | 触发条件 | 处理 |
| --- | --- | --- |
| E-1:无 `.current-version` | 文件不存在 | 打印 `✗ 未检测到激活的版本工作空间`,提示 `code-version`,退出(FR-5) |
| E-2:`.current-version` 指向不存在的版本 | 目录缺失 | 提示 `版本 <X> 工作空间不存在,请先调 code-version 修复` |
| E-3:需求编号在当前版本不存在 | `require/<REQ>/` 不存在 | 列出本版本所有需求,提示用户(FR-6) |
| E-4:参数格式不匹配 `^REQ-\d{5}$` | 用户输入 `/code-dashboard xxx` | 打印用法示例,退出(FR-6.AC-6.3) |
| E-5:看板文件缺失区段 | `## 需求清单` 等区段不存在 | 显示 `(无)`,不报错 |
| E-6:看板表格列错位 | 列数与期望不符 | 退化到"原始 markdown 块"显示,不解析 |
| E-7:全版本无任何需求 | V0.0.2 初始化态 | 建议 `code-require 添加首个需求`(高优先级) |
| E-8:全版本已完成 | 所有任务+需求已完成 | 建议 `code-version <新版本号>`(高优先级) |
| E-9:旧格式任务编号 | `REQ-NNNNN-NNNNN` 字面 | 透传显示,不解析(见 NFR-3) |
| E-10:`code-dashboard` 自身被异常调用 | 递归或越界 | 退出(无内部状态可污染) |
