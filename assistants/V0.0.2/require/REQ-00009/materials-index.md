## 关键信息抽取(供后续步骤使用)

### 1. 用户原文笔误纠正
- 本轮**无笔误**

### 2. 核心需求拆解
- **现状**:`code-unit` 现有签名 = `code-unit TASK-...` — 强制执行单元测试
- **痛点**:
  - 某些项目**不适合**单测(如纯文档项目 / 配置仓库 / 单文件脚本)
  - 某些项目**不能**单测(如无测试框架 / 无构建工具 / 不可执行环境)
  - 这些场景下,`code-unit` 应该"跳过"而非"强行跑"
- **本需求**:
  - `code-unit` 启动时增加"项目可测性检查"守卫
  - 守卫不通过 → 跳过单测过程 + 将测试状态标记为"已完成"(**不是**"已运行-通过",**不是**"失败")
  - 守卫通过 → 走原 `code-unit` 流程

### 3. "适合/能够进行单元测试"的判定信号(用户原文未明示)
- **"适合"信号**:
  - 项目类型(代码项目 vs 文档项目 vs 纯配置项目)
  - 任务内容(代码任务 vs 文档任务 vs 配置任务)
- **"能够"信号**:
  - 是否存在测试框架配置(`pytest.ini` / `pyproject.toml` / `package.json` 脚本 / `Cargo.toml` 等)
  - 是否存在测试运行命令(可执行)
  - 是否有可测试的代码产物(`code-it` 阶段产出的代码)

### 4. 关键边界
- **判定时机**:何时执行守卫?
  - 选项 A:`code-unit` 启动**第一步**(类似 REQ-00005 的"首步拉取"位置)
  - 选项 B:在 `code-unit` 主流程中,**写测试前**判断
  - 选项 C:由上游(`code-plan` / `code-it`)提供"是否需要单测"标记,`code-unit` 据此决定
- **"已完成"标记的语义**:
  - 选项 A:复用 V0.0.1 已有的"不适用"状态(看板"任务清单"列已有此枚举)
  - 选项 B:新增"已完成(跳过)"状态(扩枚举值,触发 `dashboard-conventions §规则 1` 三方同步)
  - 选项 C:标记为"已运行-通过"+ 备注"无测试" — 隐藏跳过语义
- **与 `code-auto` 协同**:REQ-00007 中 `code-auto` 调 `code-unit` 时,若守卫不通过会**自动跳过**(原 `code-auto` FR-4.AC-4.2 中"若 `code-it` 输出含'测试需要=Y'"的判断逻辑会被本需求**覆盖**)
- **报告留痕**:跳过时应写 `test/<任务编码>/RESULT.md` 吗?见 clarifications.md Q-4
- **`code-unit` 自身是否仍可被"强制调用"**:用户可能希望"我就要写单测",即使守卫不通过也跑?

### 5. 待澄清(在 clarifications.md 中展开)

- Q-3:"跳过时"的测试状态值(复用"不适用" / 新增"已完成(跳过)" / 标记"已运行-通过")
- Q-4:跳过时是否写 `test/<任务编码>/RESULT.md`(留痕 / 不写 / 写但内容为"跳过原因")
- Q-5:是否提供"强制调用"参数 `--force`(覆盖守卫)
