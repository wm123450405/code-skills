- **主题**(来自用户输入):
  > 优化需求: `/code-unit` 技能调用时需要检查当前项目是否适合编写单元测试或是否能够进行单元测试,若不需要或不能进行单元测试,则跳过单元测试过程,并将单元测试结果标记为已完成。

## 1. 需求概述

**为谁**:`code-skills` 仓库的 AI 协作者 + 项目主导者(在使用 `code-unit` 跑单测时,期望"不适用场景自动跳过"而非"强行失败")。

**解决什么问题**:`code-unit` 现有签名 = `code-unit TASK-...` — 强制执行单元测试。但在以下场景会导致问题:
- **纯文档/规范项目**:无代码可测,`code-unit` 会因"无测试文件"失败
- **无构建/测试框架的项目**:没有 `pytest` / `npm test` / `cargo test` 等命令
- **不可执行环境**:CI 沙箱、临时工作区等没有测试运行环境

**带来什么价值**:
1. **自动跳过**:`code-unit` 启动时检测"项目可测性",不可测则跳过单测过程
2. **看板兼容**:跳过时把测试状态标记为现有"不适用"枚举(与 V0.0.1 看板"任务清单"区段完全兼容)
3. **零新增产物**:跳过时**不**写 `test/<任务编码>/RESULT.md`,**不**新增任何文件
4. **`code-dashboard` 零改动**:复用"不适用"枚举,`code-dashboard` 现有统计逻辑无变化

## 2. 背景与目标

## 3. 用户角色与场景

### 3.1 角色
- **R-1 文档/规范项目主导者**:项目只有 .md,无代码 → `code-unit` 自动跳过
- **R-2 无测试框架项目主导者**:项目有代码但无测试配置 → `code-unit` 自动跳过
- **R-3 真实代码项目主导者**:项目有完整测试栈 → `code-unit` 走原流程

### 3.2 关键场景

#### S-1:不可测项目(主流程,守卫不通过)
- 用户输入:`/code-unit TASK-REQ-00001-001`(项目根无 `package.json` / `pyproject.toml` 等)
- 技能执行:
  1. **步骤 0a 项目可测性检查**(新增)— 检查项目根:无 `package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod` / `pom.xml` / `build.gradle` 任一文件,且无 `test/` 目录
  2. 守卫不通过 → **跳过单测过程**
  3. 写入看板"任务清单"区段:测试状态 = `不适用`
  4. **不**写 `test/<任务编码>/RESULT.md`
  5. 报告"已跳过"
- 用户看到:
  ```
  ⏭ code-unit 跳过(项目不可测)
  
  任务:TASK-REQ-00001-001
  守卫检查:项目根无 package.json / pyproject.toml / Cargo.toml / go.mod / pom.xml / build.gradle / test/ 目录
  状态:不适用
  
  看板"任务清单"区段:测试状态 → 不适用
  ```

#### S-2:可测项目(主流程,守卫通过)
- 用户输入:`/code-unit TASK-REQ-00010-001`(项目根有 `package.json` 含 `test` 脚本)
- 技能执行:
  1. **步骤 0a 项目可测性检查** — 项目根有 `package.json` → 守卫通过
  2. 进入原有"步骤 1-N 单测流程"
  3. 写 `test/<任务编码>/RESULT.md`
  4. 看板"测试状态" → `已运行-通过` / `已运行-失败`
- (本需求**不**改变该流程,仅新增守卫)

#### S-3:边界 — `code-auto` 调用时守卫不通过
- 假设 REQ-00007 `code-auto` 在任务循环中调 `code-unit TASK-REQ-00005-001`
- 技能执行:
  1. `code-auto` 调 `code-unit` — 触发守卫
  2. 守卫不通过 → `code-unit` 跳过 + 看板"不适用"
  3. `code-auto` 收到 `code-unit` 正常退出(退出码 0)→ 继续下一个任务
- 用户看到(`code-auto` 内部):
  ```
  [code-auto] 2/5:code-unit TASK-REQ-00005-001 ⏭(项目不可测)
  ```

#### S-4:`code-it` 输出"测试需要=Y"但守卫不通过
- 假设 `code-it` 在某任务中输出 `测试需要=Y`,但项目根无构建/测试文件
- 技能执行:
  1. `code-auto` 据"测试需要=Y"调 `code-unit`
  2. `code-unit` 守卫不通过 → 跳过
  3. 看板"测试状态" → `不适用`
- **影响**:`code-auto` 现行 FR-4.AC-4.2 的"测试需要=Y"判断**不**变;但本需求让"即使要测也测不了"

#### S-5:无激活版本(边界)
- 用户输入:`/code-unit`,但 `.current-version` 不存在
- 技能执行:提示用户调 `code-version`,退出

#### S-6:`code-unit` 自身崩溃(边界)
- 守卫通过,但单测过程崩溃
- 技能执行:按原 `code-unit` 行为处理(退出码 ≠ 0,看板"测试状态" → `已运行-失败`)

## 4. 功能需求(FR)

### FR-1:新增"步骤 0a 项目可测性检查"守卫

- **描述**:`code-unit` 在"步骤 0 版本上下文检测"**之前**,新增"步骤 0a 项目可测性检查"
- **检查项**(Q-1 锁定 A):
  - 项目根存在以下任一文件 → **可测**:
    - `package.json`(含 `scripts.test`)
    - `pyproject.toml`(含 `[tool.pytest]` 等测试配置)
    - `Cargo.toml`
    - `go.mod`
    - `pom.xml`
    - `build.gradle` / `build.gradle.kts`
    - `test/` 目录
  - 否则 → **不可测**
- **优先级**:必须
- **AC**:
  - AC-1.1:`code-unit/SKILL.md` 在"步骤 0"前显式列出"步骤 0a"
  - AC-1.2:检查项清单与本 FR 描述一致
  - AC-1.3:可测 → 进入原"步骤 0"+ 后续流程
  - AC-1.4:不可测 → 进入 FR-2 跳过流程
  - AC-1.5:不修改 frontmatter;不修改"步骤 0"及之后的原有内容(增量追加)

### FR-2:守卫不通过 → 跳过 + 写入看板(Q-2 锁定 A)

- **描述**:`code-unit` 守卫不通过时:
  1. 跳过单测过程(不写测试文件 / 不跑测试命令)
  2. 写入看板"任务清单"区段:测试状态 = `不适用`
  3. 退出码 = 0(正常退出)
- **优先级**:必须
- **AC**:
  - AC-2.1:守卫不通过时,看板"任务清单"区段的"测试状态"列更新为 `不适用`
  - AC-2.2:`code-unit` 退出码 = 0(让 `code-auto` 继续)
  - AC-2.3:屏幕打印"⏭ code-unit 跳过(项目不可测)"+ 守卫检查结果
  - AC-2.4:不调用任何 `Bash` 跑测试命令
  - AC-2.5:不写 `test/<任务编码>/RESULT.md`(Q-3 锁定 A)

### FR-3:不留痕(Q-3 锁定 A)

- **描述**:`code-unit` 跳过时**不**产生任何新文件
- **优先级**:必须
- **AC**:
  - AC-3.1:不写 `test/<任务编码>/RESULT.md`
  - AC-3.2:不写 `test/<任务编码>/work-log.md` / `deviations.md` 等
  - AC-3.3:仅更新看板"任务清单"区段(由 `code-unit` 写,与现有"测试状态"字段写入路径一致)

### FR-4:不修改 `code-unit` 现有"可测"流程

- **描述**:守卫通过时,`code-unit` 走原流程**不**变
- **优先级**:必须
- **AC**:
  - AC-4.1:守卫通过 → `code-unit` 现有"步骤 0 ~ N"全部**不**变
  - AC-4.2:写 `test/<任务编码>/RESULT.md` 的逻辑**不**变
  - AC-4.3:`Bash` 跑测试命令(如 `npm test`)的逻辑**不**变
  - AC-4.4:看板"测试状态"列写入 `已运行-通过` / `已运行-失败` 的逻辑**不**变

### FR-5:与 9 个其他 `code-*` 技能正交

- **描述**:本需求**不**修改其他 9 个技能;其他技能**不**感知"项目可测性"概念
- **优先级**:必须
- **AC**:
  - AC-5.1:`code-version` / `code-require` / `code-design` / `code-plan` / `code-it` / `code-review` / `code-dashboard` / `code-publish` / `code-auto` **不**被本需求修改
  - AC-5.2:`code-auto` 现行 FR-4.AC-4.2 的"测试需要=Y"判断**不**变(Q-5 采纳默认)
  - AC-5.3:`code-dashboard` / `code-publish` / `code-review` 现有统计逻辑**不**变

### FR-6:不修改 marketplace 与 plugin

- **描述**:本需求不触发 `marketplace-protocol.md` 任何规则
- **优先级**:必须
- **AC**:
  - AC-6.1:`marketplace.json` / `plugin.json` **不**被本需求修改
  - AC-6.2:`commit-conventions.md` 规则 1 **不**被本需求填充
  - AC-6.3:`assistants/rules/` 下任何规范文件**不**被本需求修改
  - AC-6.4:中英 README 若需更新"code-unit 增加守卫",按 `doc-conventions §规则 1` 同次提交中英 — 但本需求**不**主动写 README(由 `code-rule` 沉淀)

### FR-7:报告与建议

- **描述**:`code-unit` 跳过时向用户报告
- **优先级**:必须
- **AC**:
  - AC-7.1:守卫不通过时,屏幕输出"⏭ code-unit 跳过(项目不可测)" + 任务编码 + 守卫检查结果(具体哪些文件缺失)
  - AC-7.2:守卫通过时,按原 `code-unit` 报告格式输出(无新增)
  - AC-7.3:**不**生成独立报告文件(S-1 报告即全部内容)

## 5. 非功能需求 / 约束(NFR)

### NFR-1:零新增依赖
- **描述**:不引入新依赖;复用现有 `Bash` / `Read` / `Glob` 工具
- **强制级别**:必须

### NFR-2:增量修改 SKILL.md
- **描述**:`code-unit/SKILL.md` 修改方式为 **Edit 工具追加"步骤 0a"**,不重写稳定章节
- **强制级别**:必须
- **理由**:避免破坏现有 frontmatter / 既有"可测"流程

### NFR-3:与 V0.0.1 看板"任务清单"区段枚举完全兼容
- **描述**:`code-unit` 跳过时写入的"测试状态"枚举 = `不适用`(V0.0.1 既有);**不**新增枚举值
- **强制级别**:必须
- **理由**:避免触发 `dashboard-conventions §规则 1` 三方同步(枚举值不变,只复用)

### NFR-4:与 `code-dashboard` 现有统计逻辑 0 冲突
- **描述**:`code-dashboard` 现有"测试通过" / "真正可发布" 统计逻辑**不**因本需求变化
- **强制级别**:必须
- **理由**:复用"不适用"枚举,`code-dashboard` 已正确处理该状态

### NFR-5:与 `code-publish` 前置检查 0 冲突
- **描述**:`code-publish` 现有"任务解决"判定 = "开发=已完成 ∧ 测试∈{已运行-通过, 不适用}";本需求**不**破坏该判定
- **强制级别**:必须
- **理由**:本需求恰好让更多任务"测试=不适用",仍属 `code-publish` 前置检查通过

### NFR-6:与 `code-auto` 退出码兼容
- **描述**:`code-unit` 守卫不通过时退出码 = 0;`code-auto` 据此继续
- **强制级别**:必须
- **理由**:`code-auto` FR-7.AC-7.1 要求"子技能退出码 ≠ 0 才中断",本需求确保守卫不通过**不**触发中断

### NFR-7:性能
- **描述**:守卫检查(< 10 个文件检测)耗时 < 1 秒
- **强制级别**:必须

### NFR-8:不提供"强制调用"参数
- **描述**:本需求**不**实现 `--force` 等强制跳过守卫的参数(Q-4 采纳默认)
- **强制级别**:必须
- **回退路径**:v2 可加

## 8. 数据与状态

### 8.1 守卫检查项

| 序号 | 文件/目录 | 类型 | 备注 |
| --- | --- | --- | --- |
| 1 | `package.json` | Node.js / 前端 | 需含 `scripts.test` |
| 2 | `pyproject.toml` | Python | 需含测试配置(如 `[tool.pytest]`) |
| 3 | `Cargo.toml` | Rust | — |
| 4 | `go.mod` | Go | — |

### 8.3 跳过 vs 不可测 vs 不适用语义对照

| 术语 | 含义 | 来源 |
| --- | --- | --- |
| 跳过 | `code-unit` 内部判定不跑单测 | 本需求新增(本技能内部状态) |

**关系**:`跳过 + 不可测 → 写入看板"不适用"`(本需求的核心动作)
