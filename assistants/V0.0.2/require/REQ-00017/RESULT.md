## 1. 需求概述

**当前问题**:`/code-plan` 技能在拆分任务阶段,会为"更新版本看板"单独拆出一个**派生任务**,导致 PLAN.md 的"任务总览"中多出一条"派生任务"行,触发/来源=**详细设计**。但**版本看板的状态推进**(把"待开发"→"开发中"→"已完成"等)在每个任务**自身执行**时就已经发生——即 `/code-it` 自身消费计划数据并产出会触发看板更新。这是典型的"职责越位"。

**本需求目标**:
1. **FR-1**:`/code-plan` 不再为"更新看板"拆分独立任务
2. **FR-2**:`/code-it` 自身在执行任务时,根据本任务在 PLAN.md 任务总览的"开发状态"列的当前值,把状态推进到"开发中"或"已完成";同时把对应的关联行(若本任务影响"测试状态"、"关联需求"等)也做相应的状态推进
3. **FR-3**:`/code-plan` 拆分任务的判定准则改为"是否会带来新的代码/测试/文档/数据/配置变更",不再混入"更新看板"这种纯文档操作
4. **FR-4**:看板"任务清单"区段中,本任务的状态推进,完全由 `/code-it` 在成功完成后一次写入;`/code-plan` 不再向看板追加"待开发"行(因为 `/code-it` 执行时自然会写)

**非目标**:
- 不重写 `/code-plan` 整体拆分逻辑
- 不动 `/code-unit` / `/code-review` 的看板更新职责
- 不修改 9 个 `code-*` 技能中其他 7 个的看板更新行为
- 不修改 `dashboard-conventions.md` 解析锚点(看板 3 区段保持)

(来源:用户在 `code-auto` 步骤 1 调用 `code-require` 时直接给出的需求描述)

## 2. 背景与目标

### 2.1 背景

`/code-plan` 技能当前在拆分任务步骤中,会为"更新看板"单独生成一个任务行(典型标题:`更新 <版本> 看板:<需求编码> 任务总览 + 变更记录`),并把它的"触发/来源"标为"详细设计"。这条任务行:

- 不产生代码变更
- 不产生测试
- 不产生数据/配置变更
- 唯一的"产出"是改写 `./assistants/<版本号>/RESULT.md` 的"任务清单"区段

而真正的代码改/测试在它的"前置/被依赖"任务中已经完成,在 `/code-it` 自身完成时,看板**本应**同步推进——只是当前设计把它推给了"看板更新任务"去补。

### 2.2 目标

- **G-1 任务粒度更纯粹**:`/code-plan` 拆出的每个任务都对应**实际产出**(代码/测试/文档/数据/配置/部署脚本)之一
- **G-2 看板更新与任务执行同构**:`/code-it` 完成一个任务时,本任务的看板状态由 `/code-it` 自身一次性推进(开发状态:待开发→开发中→已完成;若带测试则同时推进测试状态:待编写→...;若影响其他行则同步)
- **G-3 派生任务减少**:`code-auto` 步骤 4 任务循环数量减少(看板更新行消失),步骤 6 评审派生任务不受影响(独立)
- **G-4 看板总量更稳定**:`/code-plan` 不再追加"待开发"行(由 `/code-it` 一次性写"已完成"或对应状态)

### 2.3 非目标

- 不删除 `/code-plan` 末尾的"更新看板步骤"——它仍然需要把本需求的"任务总览"行(全部任务)写入看板(那是"详细设计完成"的事件,不是"每个任务完成"的事件)
- 不修改 `/code-it` 的代码改写职责,只在末尾兜底提交之后**新增**"推进看板状态"小步
- 不破坏 `code-review` 派生任务往 PLAN.md 追加"审查改修"行的能力

## 4. 功能需求(FR)

### FR-1 `/code-plan` 不再为"更新看板"拆任务

**位置**:`/code-plan` 步骤 4A(拆分任务)。

**当前行为**:
- `/code-plan` 拆分出 N 个真实任务 + 1 个"更新看板"派生任务
- 派生任务的"触发/来源"="详细设计","关联任务"=本需求所有真实任务

**目标行为**:
- `/code-plan` 拆分出 N 个真实任务,**不再**追加"更新看板"派生任务
- `PLAN.md` 任务总览中**只**包含真实任务(代码/测试/文档/数据/配置/部署脚本之一)
- 真实任务的"触发/来源"="详细设计","关联任务"=本需求其他任务(若有依赖)

**AC**:
- AC-1.1 给定一个完整需求,跑完 `/code-plan` 后,`PLAN.md` 任务总览的"触发/来源"列**全部**="详细设计",**不出现**="更新看板"
- AC-1.2 `PLAN.md` 任务总览的"产出物"列不再出现"看板变更"或类似项

### FR-2 `/code-it` 自行推进本任务看板状态

**位置**:`/code-it` 末尾兜底提交之后,新增小步"P-1 推进看板"。

**当前行为**:
- `/code-it` 完成后,`./assistants/<版本号>/RESULT.md` 中本任务的"开发状态"还停留在"待开发"(由 `/code-plan` 写入),要等"更新看板"任务(派生)执行才会推进

**目标行为**:
- `/code-it` 在末尾兜底提交成功之后,自动读取 `./assistants/<版本号>/RESULT.md`,定位"任务清单"区段
- 把本任务(任务编码匹配)的"开发状态"列从"待开发" → "已完成"(若该任务**不**带测试)
- 若该任务带测试(由 `/code-plan` 标注"测试需要=Y"),则:
  - "开发状态" → "已完成"
  - "测试状态"列保持"待编写"(由 `/code-unit` 推进)
- 把更新后的看板一行写回(原子操作)
- 在"变更记录"区段追加一行:
  ```
  YYYY-MM-DD HH:mm  任务完成  <任务编码> <任务标题>(开发状态:已完成)  <任务编码>
  ```

**AC**:
- AC-2.1 给定一个无测试任务,跑完 `/code-it` 后,看板"任务清单"区段本任务的"开发状态"="已完成",**无需**再跑任何"看板更新"任务
- AC-2.2 给定一个带测试任务,跑完 `/code-it` 后,看板"任务清单"区段本任务的"开发状态"="已完成","测试状态"="待编写"
- AC-2.3 看板"变更记录"区段在 `/code-it` 完成后追加一条"任务完成"行

### FR-3 `/code-plan` 拆分任务的判定准则

**位置**:`/code-plan` 步骤 4A 的"拆任务"决策点。

**目标行为**:
- 拆分准则改为:**一个任务 = 一个"实际产出"**
- 实际产出候选集 = {代码改写, 测试编写, 文档改写, 数据迁移, 配置变更, 部署脚本}
- "看板更新" **不**在实际产出候选集中
- "代码评审"是派生任务的"触发/来源",但**不**是 `/code-plan` 拆任务的依据

**AC**:
- AC-3.1 拆分准则写进 `/code-plan` SKILL.md 步骤 4A 注释,作为强约束
- AC-3.2 `/code-plan` 的 `code-auto` 步骤 4 任务循环中,任务数 == 拆分出的真实任务数(无冗余)

### FR-4 `/code-plan` 不再向看板追加"待开发"行

**位置**:`/code-plan` 步骤 9A 同步看板。

**当前行为**:
- `/code-plan` 完成时,把本需求所有任务(代码 + 派生"更新看板")以"待开发"状态写入看板

**目标行为**:
- `/code-plan` 完成时,只把本需求所有**真实任务**(FR-1 后)以"待开发"状态写入看板
- "看板变更"事件本身不产生任务行

**AC**:
- AC-4.1 看板"任务清单"区段中,本需求所有任务的"触发/来源"="详细设计",**无**"更新看板"来源

## 7. 交互逻辑

### 7.1 状态机:任务的看板状态推进

```
[/code-plan 完成]
  看板行: 开发状态=待开发, 测试状态=待编写/不适用
  ↓
[/code-it 完成 → 末尾兜底提交成功]
  看板行: 开发状态=已完成, 测试状态=待编写/不适用   ← FR-2
  ↓
[/code-unit 完成(若带测试)]
  看板行: 开发状态=已完成, 测试状态=已运行-通过/不适用
```

### 7.2 关键流程

```
[/code-plan]
  步骤 4A: 拆任务(FR-3 准则)→ PLAN.md
  步骤 9A: 同步看板(FR-4)→ 看板"任务清单"加 N 行(待开发)
  步骤 N:  末尾兜底提交
  ↓
[/code-it <任务编码>]
  步骤 0a: git pull
  步骤 1-9: 代码改写
  步骤 10: 末尾兜底提交
  步骤 11: 推进看板(FR-2 新增小步 P-1)→ 看板"任务清单"本任务行"开发状态"=已完成
```

## 8. 数据与状态

### 8.1 看板"任务清单"区段字段

| 字段 | 含义 | 取值 | 推进时机 |
| --- | --- | --- | --- |
| 任务编码 | 唯一编码 | `TASK-REQ-NNNNN-NNNNN` | `/code-plan` 写入 |
| 任务标题 | 简述 | 文本 | `/code-plan` 写入 |
| 关联需求 | 所属需求编码 | `REQ-NNNNN` | `/code-plan` 写入 |
| 触发/来源 | 派生任务的来源 | `详细设计` / `审查改修` | `/code-plan` 或 `/code-review` 写入 |
| 开发状态 | 进度 | `待开发` / `开发中` / `已完成` | `/code-it` 推进 |
| 测试状态 | 进度 | `待编写` / `已编写` / `已运行-通过` / `已运行-失败` / `不适用` / `阻塞` | `/code-unit` 推进 |
| 备注 | 自由文本 | 文本 | 任意技能可写 |

### 8.2 看板"变更记录"区段

每条事件一行,字段:
- 时间(`YYYY-MM-DD HH:mm`)
- 事件类型(`需求新增` / `需求变更` / `概要设计完成` / `详细设计完成` / `任务完成` / `测试完成` / `审查完成` / `派生任务完成` / `缺陷修复` / `发布`)
- 对象(`<需求编码>` / `<任务编码>` / `<缺陷编码>`)
- 描述
- 关联(`<需求编码>` / `<任务编码>` / 空)

新增事件类型 `任务完成`:
```
YYYY-MM-DD HH:mm  任务完成  <任务编码> <任务标题>(开发状态:已完成)  <任务编码>
```

## 9. 边界与异常

| ID | 场景 | 处理 |
| --- | --- | --- |
| E-1 | `/code-it` 末尾兜底提交**失败** | 不推进看板(stderr 告警);后续由"更新看板"任务(若有遗留)或 `/code-review` 派生任务补 |
| E-2 | `/code-it` 末尾兜底提交成功但**推进看板失败**(权限/磁盘) | stderr 告警 `⚠ 看板推进失败(<原因>),请人工核对`,**不重试**,**不阻断**;`/code-it` 整体仍视为完成 |
| E-3 | 看板中**找不到**本任务行 | `/code-it` 写 stderr `⚠ 看板未找到本任务行,跳过推进`;**不阻断** |
| E-4 | 看板中本任务行"开发状态"**已经是**"已完成" | 幂等:不重复写,只跳过;不报错 |
| E-5 | 同一任务被 `/code-it` 多次执行(如修正) | 第二次执行末尾兜底成功后,推进看板"开发状态"="已完成"(幂等) |

## 10. 验收标准(AC)
