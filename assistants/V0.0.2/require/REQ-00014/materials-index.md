## 上游需求

- **来源**:用户口头/文本 ARGUMENTS
- **完整原文**:
  > 优化技能 `/code-plan`,现在的拆分任务的维度有问题,在拆分任务时应该按功能点拆分,同一个功能点都应该包含一套完整的展示效果、功能逻辑、使用说明的完全内容,若需求内容较多应该包含单独的架构设计开发任务作为首个任务。
- **关键信息抽取**(在 `analysis-notes.md` 中详细展开):
  - 优化目标:`/code-plan` 任务拆分维度
  - 拆分原则:**按"功能点"拆分**(每个功能点 = 完整一套 = 展示效果 + 功能逻辑 + 使用说明)
  - 架构任务:需求内容较多时,**首个任务 = 架构设计开发任务**
  - 现状问题:目前 `/code-plan` SKILL.md §10A "任务拆分原则"是按"底层依赖/数据结构 → 核心逻辑 → 集成层 → 测试 → 文档"分层拆分,不是按"功能点"

## 关键观察(供 analysis-notes.md 展开)

- **REQ-00007 PLAN 偶然"功能点化"**:5 任务 = 写 SKILL.md / marketplace.json / README / 看板同步 / 自检 — **每个任务都独立完整**(T-001 含完整 SKILL.md 的展示/逻辑/说明;T-002 含完整的 JSON 修改/JSON Schema 校验/JSON 静态自检)。**这是"按功能点"拆分的范例**,但**没有显式规则化**
- **REQ-00004/06 PLAN 严格"分层"**:T-001 写算法核心 + T-002 写模板 + T-003 写自检文档 — **每个任务不独立**(T-001 不含模板,T-002 不含算法)。这是"分层"模式,本需求要优化的就是这种
- **优化预期**:把"分层拆分"改为"功能点拆分",并显式规则化(避免再次偶然);当需求"较高扩展性/可维护性/可配置性 + 较高复杂度"或涉及"未来可能的不同实现方式/对接不同三方组件"时,插入"架构设计开发任务"作为首个任务
