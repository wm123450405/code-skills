### Q-A1:本优化中"同一个功能点"的粒度是?

- **问题**:用户原文说"按功能点拆分",但"功能点"是用户可见能力、技术模块、还是交付单元?
- **选项**:
  - A. **按用户可见能力(狭义)**:1 个完整的用户可见能力 = 1 个功能点。例:"用户登录功能" = 登录页 UI + 鉴权逻辑 + README 入门说明。
  - B. 按技术模块(广义):1 个技术模块 = 1 个功能点。例:"鉴权模块" = 登录 + 注册 + 密码重置 + OAuth 接入。粒度较粗。
  - C. 按交付单元(可包含多个能力):1 个接口或 1 个完整特性 = 1 个功能点。适合内部工具。
- **用户回答**:A — **按用户可见能力(狭义)**
- **影响**:RESULT.md §4 FR-1;§10 验收标准;影响 `code-plan` SKILL.md §10A 改写
- **解读**:与"功能完整"原则一致(每个功能点 = 1 个用户完整可用的能力);粒度细,但保持"完整一套"
- **决策依据**:用户原文"同一个功能点都应该包含一套完整的展示效果、功能逻辑、使用说明" — 强指向"用户可见能力"

### Q-A2:"首个任务 = 架构设计"的触发条件是?

- **问题**:用户原文"若需求内容较多应该包含单独的架构设计开发任务作为首个任务",但"内容较多"的客观判断标准是?
- **选项**:
  - A. 需求多模块/多接口时才插架构任务
  - B. 所有需求都强制插架构任务
  - C. 需求含 ≥3 个功能点时才插架构任务
  - D. (用户在原选项外的补充判断)
- **用户回答**:D — **"在 `/code-design` 概要设计过程中,会对开发的扩展性、可维护性、可配置性目标进行设计,根据较高的扩展性、可维护性、可配置性,以及需求的较高的复杂度,应当插入架构开发任务,或功能涉及未来可能的不同实现方式或对接不同三方组件时考虑引入架构开发任务。架构任务是为未来可能的需求调整、环境变化来准备的。"**
- **影响**:RESULT.md §4 FR-2;§10 验收标准;影响 `code-plan` SKILL.md §10A 改写
- **解读**:**架构任务是面向未来的** — 不是"当前需求大就要架构",而是"未来可能调整/变化/扩展就要架构"。这是更精炼的判断标准
- **决策依据**:用户在原选项外的补充(更精炼,语义更准)
- **关键提炼**:
  - 触发 1:**较高扩展性 / 较高可维护性 / 较高可配置性** + **较高复杂度** → 需架构任务
  - 触发 2:**涉及未来可能的不同实现方式** → 需架构任务
  - 触发 3:**对接不同三方组件** → 需架构任务
  - **不**触发:单一简单功能点 + 低扩展性 + 无三方组件

### Q-A3:现有 10 个需求中 7 个有 PLAN。本优化仅修改 `/code-plan` 拆分规则,是否要追溯重新拆分?

- **问题**:本优化是规则升级,要不要对现有 7 个 PLAN 做追溯重拆?
- **选项**:
  - A. 仅未来需求生效(已拆分 PLAN 不追溯)
  - B. 未来需求 + 现有 PLANS 加追溯说明
  - C. 未来需求 + 现有 PLANS 全部重拆
- **用户回答**:A — **仅未来需求生效(已拆分 PLAN 不追溯)**
- **影响**:RESULT.md §4 FR-3;§11 关联需求;§12 待澄清
- **解读**:
  - 重拆 7 个既有 PLAN 会破坏历史状态(开发状态、测试状态、提交哈希) — 成本高
  - 既有 PLAN 已完成实施,无追溯必要
  - 仅未来需求应用新规则 — 最小变更 + 最大收益
- **决策依据**:`code-review` 评审的"状态字段不可逆"原则 + 实施成本
