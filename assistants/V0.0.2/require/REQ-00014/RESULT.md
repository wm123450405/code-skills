- **主题**(来自用户输入):
  > 优化技能 `/code-plan`,现在的拆分任务的维度有问题,在拆分任务时应该按功能点拆分,同一个功能点都应该包含一套完整的展示效果、功能逻辑、使用说明的完全内容,若需求内容较多应该包含单独的架构设计开发任务作为首个任务。

---

## 1. 需求概述

**为谁**:`code-plan` 技能的未来使用者(用户/AI 协作者)+ V0.0.2 未来所有需求的拆分者。

**解决什么问题**:当前 `/code-plan` SKILL.md §10A 的"任务拆分原则"是按"**底层依赖 → 核心逻辑 → 集成层 → 测试 → 文档**"**分层拆分**,导致**单个功能点被拆为多个跨层任务**,每个任务**不独立完整**(例如"发布部署"功能被拆为 8 个模板任务)。这与用户期望的"按功能点拆分 + 完整一套"不符。

**带来什么价值**:
1. **拆分维度清晰**:**按"用户可见能力"拆分**,1 个功能点 = 1 个任务
2. **任务独立完整**:**展示效果 + 功能逻辑 + 使用说明** 必须在**同一个任务**中完整呈现
3. **架构任务可选前置**:当需求具有较高扩展性/可维护性/可配置性 + 较高复杂度,或涉及未来可能的不同实现方式,或对接不同三方组件时,**首个任务 = 架构设计开发任务**
4. **向下兼容**:既有 7 个 PLAN 不追溯重拆(Q-A3 锁定 A);仅未来需求应用新规则

## 2. 背景与目标

### 2.1 背景

- `code-plan` 是 `code-skills` 9 技能中的"详细设计与任务拆分"步骤;产出 `PLAN.md` 给 `code-it` 实施
- 既有 7 个 PLAN(REQ-00004/05/06/07 + 部分其他)使用了"分层拆分"模式 — **偶然失败**
- 1 个反例:REQ-00007 自己的 PLAN.md 偶然"功能点化"(5 任务 = 写 SKILL.md / 改 JSON / 改 README / 改看板 / 自检) — **每个任务独立完整**;但**没有显式规则化**

### 2.2 业务目标

- **G-1**:把"分层拆分"改为"按功能点拆分",**显式规则化**
- **G-2**:当需求内容较多时,**首个任务 = 架构设计开发任务**(为未来可能的扩展/变化预留)
- **G-3**:**不破坏**既有 `code-plan` 输出的字段(任务编号/双状态/触发来源)
- **G-4**:**不触发** `dashboard-conventions §规则 1` 三同步(不修改字段,只改"拆分"行为)
- **G-5**:**不追溯**重拆既有 7 个 PLAN(Q-A3 锁定 A)

### 2.3 本次目标

- 修改 `plugins/code-skills/skills/code-plan/SKILL.md` §10A "任务拆分原则"段
- 仅修改 1 个章节(约 30 行);其他 18 章节保留
- 修改生效范围:V0.0.2 未来所有需求(REQ-00008+)
- 修改不**涉及** `code-it` / `code-unit` / `code-review` / `code-design` 等其他 8 个技能
- 修改不**涉及** `code-fix` / `code-init` / `code-rule` / `code-version` / `code-dashboard` / `code-publish` / `code-require` / `code-auto`

## 4. 功能需求(FR)

### FR-1:按"功能点"拆分任务

- **描述**:`code-plan` 拆分任务时,按"用户可见能力"拆分;**1 个任务 = 1 个完整的功能点**
- **触发**:Q-A1 锁定(用户确认"按用户可见能力(狭义)")
- **优先级**:必须
- **AC**:
  - AC-1.1:`code-plan` SKILL.md §10A 显式声明"按功能点拆分"原则
  - AC-1.2:**1 个任务必须包含"展示效果 + 功能逻辑 + 使用说明"的完整一套**(在同一个任务的"关键变更"中)
  - AC-1.3:**同一个功能点不拆分到多个任务**(若用户期望"用户登录功能"应包含 UI/鉴权/README,则这 3 个子项必须在 1 个任务中)
  - AC-1.4:每个任务有清晰的"完成定义" — 包含"产物是什么"和"产物如何被验证"

### FR-2:架构设计开发任务作为首个任务(条件性)

- **描述**:当需求满足以下**任一**触发条件时,`code-plan` 拆分任务时,**首个任务 = 架构设计开发任务**
- **触发**:Q-A2 锁定(用户确认"较高扩展性 + 较高复杂度"或"未来不同实现方式"或"对接不同三方组件")
- **优先级**:必须
- **AC**:
  - AC-2.1:**触发条件 1** — 需求具有**较高扩展性 / 较高可维护性 / 较高可配置性** + **较高复杂度** → 需架构任务
  - AC-2.2:**触发条件 2** — 需求涉及**未来可能的不同实现方式** → 需架构任务
  - AC-2.3:**触发条件 3** — 需求**对接不同三方组件** → 需架构任务
  - AC-2.4:架构任务内容 = **架构设计 + 架构骨架** = 抽象层 + 接口契约 + 扩展点(**不**实现具体业务)
  - AC-2.5:架构任务位置 = 任务的**首个**(TASK-XXX-00001)
  - AC-2.6:简单需求(单一功能点 + 低扩展性 + 无三方组件)可**不**插入架构任务

### FR-3:仅未来需求生效,既有 7 个 PLAN 不追溯

- **描述**:本优化生效范围 = V0.0.2 未来所有需求(REQ-00008+);既有 7 个 PLAN(REQ-00004/05/06/07 等)**不**追溯重拆
- **触发**:Q-A3 锁定 A
- **优先级**:必须
- **AC**:
  - AC-3.1:V0.0.2 已有 PLANS(REQ-00004/05/06/07 + 部分其他)保持原样
  - AC-3.2:V0.0.2 未来 PLANS(REQ-00008+)按新规则拆分
  - AC-3.3:`code-plan` SKILL.md §10A 显式声明"本规则仅适用于未来需求"

### FR-4:不破坏其他 8 个 `code-*` 技能

- **描述**:本优化**只**修改 `code-plan` SKILL.md §10A 任务拆分原则;**不**影响其他 8 个技能

## 7. 交互逻辑

### 7.1 `code-plan` 拆分流程(优化后)

```
用户调 /code-plan <需求编码>
  ↓
[code-plan 步骤 0-6] 读上游 + 项目代码 + 加载清单(与既有同)
  ↓
[步骤 7A-9A] 详细化设计 + 澄清 + 测试策略推导(与既有同)
  ↓
[步骤 10A 拆分任务] ← 本优化作用点
  ├─ 步骤 10A.0: 评估需求(满足架构任务触发条件?)
  │   ├─ 是 → 首个任务 = 架构设计开发任务(FR-2)
  │   └─ 否 → 直接按功能点拆分(FR-1)
  ├─ 步骤 10A.1: 识别功能点(用户可见能力,1 个 = 1 个任务)
  ├─ 步骤 10A.2: 对每个功能点,1 个任务 = 1 个完整一套
  │   ├─ 展示效果(UI 截图/界面描述/示例)
  │   ├─ 功能逻辑(算法/接口/数据结构)
  │   └─ 使用说明(README/入门文档/API 文档)
  └─ 步骤 10A.3: 任务编号 + 双状态 + 触发/来源(与既有同,沿用)
  ↓
[步骤 11A-18A] 依赖图 + 里程碑 + 撰写 RESULT.md/PLAN.md + 看板同步(与既有同)
```

### 7.2 拆分示例(优化后)

#### 示例 1:简单需求(无架构任务)

需求:优化 `code-plan` SKILL.md §10A

任务拆分(优化后):
- **TASK-001 写 `code-plan` SKILL.md §10A 优化**(展示 = §10A 全文;逻辑 = "按功能点拆分"原则 + "架构任务触发条件";说明 = 在 SKILL.md 内的"工作流步骤"参考链)

#### 示例 2:多模块需求(有架构任务)

需求:开发 1 个 LLM 聊天机器人

任务拆分(优化后):
- **TASK-001 架构设计 + LLM 抽象层骨架**(展示 = 架构图 + 抽象层接口;逻辑 = 抽象层 + 接口契约 + 扩展点;说明 = 架构决策记录 ADR)

- **TASK-003 上下文管理功能**(展示 = 上下文 UI;逻辑 = 上下文窗口管理 + token 计数;说明 = 上下文配置文档)
- **TASK-004 工具调用功能**(展示 = 工具选择 UI;逻辑 = 工具注册 + 调度 + 结果处理;说明 = 工具开发文档)
- **TASK-005 README 同步 + 发布准备**(展示 = README 技能表;逻辑 = 注册到 marketplace.json;说明 = 入门指南)
