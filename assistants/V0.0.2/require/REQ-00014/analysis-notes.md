# 分析笔记 — REQ-00014

更新时间:2026-06-05 12:15
版本:V0.0.2
需求:REQ-00014 优化技能 `/code-plan` 的任务拆分维度

## 现状分析(为什么需要优化)

### 1. 既有 `/code-plan` SKILL.md §10A "任务拆分原则"(PL160-200)

```text
任务拆分原则:
- 最小可执行单元:每条任务 0.5–2 天内可完成
- 单一职责:每条任务只做一件事
- 可独立验证:每条任务有清晰的"完成定义"
- 顺序友好:从底层依赖/数据结构 → 核心逻辑 → 集成层 → 测试 → 文档
```

**关键问题**:**"顺序友好"原则 = 按"底层 → 核心 → 集成 → 测试 → 文档"分层拆分**,这导致**单个功能点被拆为多个跨层任务**,每个任务**不独立完整**。

### 2. 现状的典型问题(以 REQ-00006 publish 技能为例)

`REQ-00006` 的 PLAN.md 拆分为 9 任务:
- T-001 写 `code-publish/SKILL.md`
- T-002 ~ T-006 写 5 份模板(DEPLOY/UPDATE/Q&A/qanda-README/assistants-layout)
- T-007 不变量自检 + 同步看板
- T-008 同步双 README

**问题诊断**:
- 用户视角看 `/code-publish` 是"1 个功能"(发布部署)
- 但被拆为 8 个任务(模板各自分离)
- T-001 SKILL.md 引用模板路径,但模板在 T-002~T-006 之后才写完
- 任务之间存在**强依赖**(T-007 依赖 T-001~T-006)
- **单个任务不"独立完整"**:T-002 只有 DEPLOY.md,没有"完整的发布部署"概念

### 3. 反例:REQ-00007 auto 技能的 PLAN.md(偶然功能点化)

`REQ-00007` 的 PLAN.md 拆分为 5 任务:
- T-001 写 `code-auto/SKILL.md`(frontmatter + 17 章节 + 7 步状态机)
- T-002 `marketplace.json` 追加
- T-003 中英 README 追加
- T-004 同步 V0.0.2 看板
- T-005 8 项自检 + 收尾

**为什么"功能点化"了**:
- 每个任务都"独立完整" — T-001 含完整 SKILL.md(展示/逻辑/说明),T-002 含完整 JSON 校验/Schema 校验,T-003 含完整中英对仗校验,T-004 含完整 5 处同步校验,T-005 含完整 8 项不变量校验
- **每个任务本身就是 1 个完整可验证的能力**(虽然 T-002/T-003/T-004/T-005 都不是"用户可见能力",但每个都是"独立完整的功能点")

**结论**:`code-plan` 现状的"分层拆分"是**偶然失败**的(REQ-00007 偶然成功);**没有显式规则化** — 所以下次可能又失败

## 优化目标

把"分层拆分"改为"**按功能点拆分**",**显式规则化**:

### 目标 1:按用户可见能力(狭义)拆分

- **规则**:**1 个功能点 = 1 个用户完整可用的能力** = 1 个完整一套 = (展示效果 + 功能逻辑 + 使用说明)
- **关键**:**同一个功能点**不拆分到多个任务;**展示/逻辑/说明** 应该在**同一个任务**中完整呈现
- **意义**:用户视角的"功能"是 1 个,任务也是 1 个;**心智模型对齐**

### 目标 2:需求内容较多时,首个任务 = 架构设计开发任务

- **规则**:当需求满足以下任一条件时,插入"架构设计开发任务"作为**首个任务**:
  - 触发 1:**较高扩展性 / 较高可维护性 / 较高可配置性** + **较高复杂度** → 需架构任务
  - 触发 2:**涉及未来可能的不同实现方式** → 需架构任务(为未来切换实现方式预留扩展点)
  - 触发 3:**对接不同三方组件** → 需架构任务(为未来替换/升级三方组件预留抽象层)
- **架构任务的边界**:**架构设计 + 架构骨架** = 抽象层 + 接口契约 + 扩展点(不实现具体业务)
- **意义**:**架构任务是为未来可能的需求调整、环境变化来准备的** — 提前预留扩展点,降低未来变更成本

## 关键决策

### 决策 1:不引入"分层拆分"和"功能点拆分"的二选一,而是"功能点拆分"为默认 + "架构任务"为可选前置

- **决策**:**只**采用"按功能点拆分"为唯一原则;**当**需要时插入架构任务
- **理由**:
  - "功能点拆分"是**粒度原则**(如何拆)
  - "架构任务"是**可选的前置任务**(是否需要先做架构)
  - 两者**正交**,不互斥
- **影响**:0

### 决策 2:不修改 `code-plan` SKILL.md 的其他 18 章节,只改 §10A

- **决策**:**只**修改 §10A "任务拆分原则"
- **理由**:
  - 其他章节(步骤 0-9 / 11-18 / 缺陷分支)的工作流正确,无需修改
  - 最小变更原则 — 改的越少,引入 bug 风险越低
- **影响**:0

### 决策 3:不修改 `code-plan` SKILL.md 的任务编号/双状态字段/触发来源枚举

- **决策**:**只**改 §10A 中"任务拆分原则"4 行 + 加 §10A.1 "功能点拆分定义"
- **理由**:
  - 任务编号(`TASK-(REQ|BUG)-NNNNN-NNNNN`)、双状态(开发/测试)、触发来源枚举都是**既有已落地**的字段
  - `code-it` / `code-unit` / `code-review` 都依赖这些字段;改它们会引发级联修改
- **影响**:0

### 决策 4:既有 7 个 PLAN 不追溯重拆(Q-A3 锁定 A)

- **决策**:**只**对未来需求应用新规则;既有 7 个 PLAN 保持原样
- **理由**:
  - 重拆会破坏历史状态(开发/测试状态/提交哈希) — 不可逆
  - 既有 PLAN 已完成实施,无追溯必要
  - 实施成本高
- **影响**:0

## 与既有"code-plan SKILL.md" 的兼容性

- **`code-it` 消费 PLAN.md 的"任务编号 / 双状态 / 触发/来源"字段** — 本优化**不**修改
- **`code-unit` 消费 PLAN.md 的"测试状态"字段** — 本优化**不**修改
- **`code-review` 消费 PLAN.md 的"任务详情" / "关联任务" / "对应设计章节"字段** — 本优化**不**修改
- **`code-dashboard` 展示 PLAN.md 的"任务清单"区段** — 本优化**不**修改字段,只改"任务是怎么产生的"
- **`code-publish` 消费 PLAN.md 的"任务状态"作为前置检查** — 本优化**不**修改
- **结论:8 个 `code-*` 技能均无影响**(本优化只改 `code-plan` 自身的"拆分"行为,不改产出格式)

## 优化后的"任务拆分原则"草稿(供 `code-plan` SKILL.md §10A 修改用)

```text
### 任务拆分原则(REQ-00014 优化后)

#### 核心原则:按"功能点"拆分

- **1 个任务 = 1 个完整的功能点** = 1 个用户完整可用的能力
- **完整一套**:同一个功能点的**展示效果 + 功能逻辑 + 使用说明** 必须在**同一个任务**中完整呈现
- **不拆分到多个任务**:"用户登录功能" 的 UI / 鉴权 / README 入门说明 必须在 1 个任务中
- **可独立验证**:每个任务有清晰的"完成定义" — 既包含"产物是什么"也包含"产物如何被验证"
- **0.5–2 天内可完成**(与既有约束一致)

#### 架构任务作为首个任务(条件性)

- **触发条件**(任一满足):
  - 需求具有**较高扩展性 / 较高可维护性 / 较高可配置性** + **较高复杂度**
  - 需求涉及**未来可能的不同实现方式**(需预留扩展点)
  - 需求**对接不同三方组件**(需预留抽象层)
- **架构任务内容**:**架构设计 + 架构骨架** = 抽象层 + 接口契约 + 扩展点
- **不**实现具体业务逻辑(只设计 + 留扩展点)
- **位置**:**首个任务**(TASK-XXX-00001),后续任务在此架构上填充功能点
- **可省略**:简单需求(单一功能点 + 低扩展性 + 无三方组件)可**不**插入架构任务
```
