## 关键信息抽取(供后续步骤使用)

### 1. 用户原文笔误纠正
- `/core-review` → **`/code-review`**(用户原文笔误)
- 涉及技能:`/code-review`(现有技能,V0.0.1 已存在,本需求是**优化**而非新增)

### 2. 核心需求拆解
- **现状**:`code-review` 现有签名 = `code-review REQ-NNNNN`(必须传入需求编号)
- **本需求**:`code-review`(无参数)= 评审**当前激活版本**的"整个需求及修改的代码"
- **"整个需求"**:当前版本 `require/<REQ>/*` 下所有需求(状态不限)
- **"修改的代码"**:当前版本已产生的代码(`code/<任务编码>/RESULT.md` 描述的代码改动) + 需求文档本身 + 设计/计划文档
- **本质**:**"整个版本"的代码审查** — 类似 V0.0.1 起"REVIEW-REPORT.md"模式,但 scope 从"单需求"扩展到"整版本"

### 3. 关键边界
- **"当前版本"的判定**:沿用 `.current-version`(与其他 8 个 code-* 技能一致)
- **"整个需求"的边界**:
  - 选项 A:全部 `require/<REQ>/RESULT.md` 都评审
  - 选项 B:仅评审"已完成"状态的需求
  - 选项 C:仅评审"已完成"需求 + "进行中"需求(排除"待开始/已取消/阻塞")
- **"修改的代码"的边界**:
  - 选项 A:本版本所有 `code/<任务编码>/RESULT.md` 描述的代码改动
  - 选项 B:仅 `git diff` 显示的代码改动(本版本分支相对基线)
  - 选项 C:本版本所有 `code/` 下文件 + `plan/` 下文件
- **与现有 `code-review REQ-NNNNN` 模式的关系**:
  - 选项 A:两种模式输出格式完全一致(派生任务逻辑、REVIEW-REPORT 路径都按单需求模式)
  - 选项 B:整版本模式输出"整版本"专属 REPORT 路径(如 `review/ALL/REVIEW-REPORT.md`),派生任务针对每个被评审的需求
  - 选项 C:整版本模式**不**派生任务,只输出发现列表(让用户决定)
- **无任务可评审的边界**:本版本无任何任务(只有需求分析) → 退化行为
- **与 `code-auto` 的协同**:本需求落地后,`code-auto` 评审循环是否应优先用"整版本"模式?见 clarifications.md Q-5
