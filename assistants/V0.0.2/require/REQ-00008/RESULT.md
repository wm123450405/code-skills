## 1. 需求概述

**为谁**:`code-skills` 仓库的 AI 协作者 + 项目主导者(在发布前,需要"全版本最后一次评审"的场景)。

**解决什么问题**:`code-review` 现有签名 = `code-review REQ-NNNNN`(必须传入需求编号),**只能**评审单个需求。用户在以下场景有痛点:
- **发布前全量评审**:调 `code-publish` 前的"全版本全量审查",需逐个 `code-review` 调 N 次
- **定期复盘**:每隔一段时间回头看"本版本所有需求都达发布标准了吗"
- **跨需求一致性检查**:同一份规范在 N 个需求中的应用是否一致

**带来什么价值**:
1. **一键整版本评审**:`code-review`(无参)即对"所有'已完成'需求"做评审
2. **聚合报告**:`./assistants/<版本号>/REVIEW.md` 顶层文件,聚合 N 个需求的发现
3. **双写兼容**:同时为每个被评审需求生成 1 份独立 `review/REQ-NNNNN/REVIEW-REPORT.md`,与现有"单需求模式"完全兼容
4. **派生任务沿用**:派生任务行为与模式 1 完全一致(写入对应 `PLAN.md` 任务总览)

## 2. 背景与目标

### 2.1 背景
- `code-review` 在 V0.0.1 起就是"单需求模式" — 现有 3 个评审产物(`./assistants/V0.0.1/review/REQ-00001~3/REVIEW-REPORT.md`)都是单需求模式
- V0.0.2 REQ-00007 引入 `code-auto` 编排者,会**反复**调 `code-review REQ-NNNNN`(单需求模式)
- V0.0.2 REQ-00006 引入 `code-publish`,前置检查"全版本已解决"是看板层判定,**未**触发代码层全版本评审
- 本需求是 **`code-review` 的第 2 种调用模式**:"整版本模式"

### 2.2 业务目标
- **G-1**:`code-review` 增加"无参模式" = 整版本评审
- **G-2**:整版本模式**只**评审"已完成"状态的需求(Q-1 锁定 B)
- **G-3**:整版本模式生成 2 份输出:
  - **聚合**:`./assistants/<版本号>/REVIEW.md`(顶层)
  - **单需求**(双写):`./assistants/<版本号>/review/REQ-NNNNN/REVIEW-REPORT.md`(与模式 1 同路径)
- **G-4**:多次执行整版本模式 → 2 份输出**更新**(覆盖前次)
- **G-5**:派生任务能力沿用模式 1(Q-4 默认)

### 2.3 本次目标
- 修改 `plugins/code-skills/skills/code-review/SKILL.md` 正文(不改 frontmatter,**不**改模式 1 行为)
- 严格遵循 `skill-conventions.md §规则 1`(frontmatter 不变)
- 不修改 `marketplace.json` / `plugin.json` / 其他 9 个 `code-*` 技能

## 3. 用户角色与场景

### 3.1 角色
- **R-1 发布者**:调 `code-publish` 前的"最后一次评审",用整版本模式
- **R-2 复盘者**:定期调"看本版本健康度"
- **R-3 项目主导者**:在 `code-auto` 完成一轮后,用整版本模式做"额外确认"

### 3.2 关键场景

#### S-1:整版本评审(主流程,有 2 个"已完成"需求)
- 用户输入:`/code-review`(V0.0.2 假设已有 REQ-00005 / REQ-00006 状态=已完成)
- 技能执行:
  1. **步骤 0**:版本上下文检测(沿用)
  2. **步骤 1 解析参数**:无参 → 走"整版本模式"
  3. **步骤 2 过滤需求**:从看板"需求清单"区段,筛 `状态=已完成` → 得到 [REQ-00005, REQ-00006]
  4. **步骤 3 评审循环**:对每个被评审需求,执行原有"单需求模式"评审逻辑
  5. **步骤 4 双写**:
     - 单需求:`./assistants/V0.0.2/review/REQ-00005/REVIEW-REPORT.md` + `REQ-00006/REVIEW-REPORT.md`(**覆盖**已有)
     - 聚合:`./assistants/V0.0.2/REVIEW.md`(新建)
  6. **步骤 5 派生任务**:每条"必须改"按其所属需求追加到对应 `PLAN.md` 任务总览
  7. **步骤 6 报告**
- 用户看到:
  ```
  ✓ code-review(整版本模式)完成
  
  评审范围:V0.0.2 的 2 个"已完成"需求
    - REQ-00005 优化 3 技能
    - REQ-00006 增加 /code-publish
  
  评审发现汇总:
    - REQ-00005:F-1(必须改),F-2(建议改)
    - REQ-00006:F-3(必须改)
    - 共 3 条发现(必须改 2,建议改 1)
  
  派生任务:2(均已写入对应 PLAN.md)
  
  输出文件:
    - ./assistants/V0.0.2/REVIEW.md(聚合)
    - ./assistants/V0.0.2/review/REQ-00005/REVIEW-REPORT.md
    - ./assistants/V0.0.2/review/REQ-00006/REVIEW-REPORT.md
  ```

#### S-2:无"已完成"需求(边界)
- 用户输入:`/code-review`(V0.0.2 当前所有需求都是"已完成(需求分析)",但无任何"已完成"全周期)
- 假设状态:整版本模式认为 `已完成` = 需求分析 + 概要设计 + 详细设计 + 编码 + 测试 + 评审全完成;若看板"需求清单"区段无此状态需求 → 退化
- 技能执行:
  1. 过滤需求 → 空
  2. 报告"本版本无可评审的需求"
  3. 退出(不生成任何文件)
- 用户看到:
  ```
  ✗ code-review(整版本模式):本版本无可评审的需求
  
  看板"需求清单"区段中,无任何 需求状态=已完成 的需求。
  请先完成需求全周期(需求 → 设计 → 计划 → 编码 → 测试)后再评审。
  ```

#### S-3:多次执行整版本模式
- 用户输入:`/code-review` → 完成 → 修复 F-1 → `/code-review` 再次
- 技能执行:
  1. 第二次执行时,聚合文件 `REVIEW.md` 覆盖前次
  2. 单需求文件 `REVIEW-REPORT.md` 也覆盖前次
  3. 派生任务列表基于"最新评审发现"生成(前次派生的任务**不**自动标记完成)
- 用户看到:
  ```
  ✓ code-review(整版本模式)完成(第 2 次)
  
  与前次比较:
    - F-1(必须改):已处理 → 不再出现
    - F-2(建议改):仍存在
    - F-3(必须改):仍存在
  ```

#### S-4:与模式 1 共存(同会话中)
- 用户输入:`/code-review REQ-00005` → 评审单需求(模式 1)
- 之后:`/code-review` → 评审整版本(模式 2)
- 技能执行:
  - 模式 1 完成后:`./assistants/V0.0.2/review/REQ-00005/REVIEW-REPORT.md` 已存在
  - 模式 2 评审时,**也覆盖** REQ-00005 的单需求文件
  - 模式 2 生成的 `REVIEW.md` **也覆盖**前次

#### S-5:无激活版本
- 用户输入:`/code-review`,但 `.current-version` 不存在
- 技能执行:
  1. 提示用户调 `code-version`
  2. 退出

## 4. 功能需求(FR)

### FR-1:增加"无参"入口

- **描述**:`code-review` 增加"无参模式"作为新入口,与现有"传入 `REQ-NNNNN`"模式**并存**
- **优先级**:必须
- **AC**:
  - AC-1.1:`/code-review`(无参)→ 走"整版本模式"
  - AC-1.2:`/code-review REQ-NNNNN`(有参)→ 走"单需求模式"(沿用现有行为,**不**变)
  - AC-1.3:`/code-review` 后跟其他无效参数 → 走"整版本模式" + 警告"忽略参数"

### FR-2:整版本模式 — 评审范围过滤(Q-1 锁定 B)

- **描述**:整版本模式只评审"已完成"状态的需求
- **判定**:
  - 需求状态 ∈ {`已完成`} → 评审
  - 需求状态 ∈ {`待开始` / `进行中` / `已取消` / `阻塞`} → **不**评审
- **优先级**:必须
- **AC**:
  - AC-2.1:从看板"需求清单"区段解析,筛 `状态=已完成`
  - AC-2.2:被筛中的需求列表打印给用户(可见性)
  - AC-2.3:被筛中数量 = 0 → 走 S-2 退化流程

### FR-3:整版本模式 — 双写 REPORT(Q-2 锁定 B+C 混合)

- **描述**:整版本模式输出 2 类文件:
  - **聚合**:`./assistants/<版本号>/REVIEW.md`
  - **单需求**:`./assistants/<版本号>/review/REQ-NNNNN/REVIEW-REPORT.md`
- **优先级**:必须
- **AC**:
  - AC-3.1:每个被评审需求生成 1 份 `REVIEW-REPORT.md`,内容与"单需求模式"**完全一致**(复用模式 1 的输出)
  - AC-3.2:聚合文件 `REVIEW.md` 位于 `./assistants/<版本号>/` 顶层(与 `RESULT.md` 同级)
  - AC-3.3:`REVIEW.md` 内容 = 各被评审需求评审发现的**并集 + 概要**(含需求标题 / 发现数 / 必须改派生任务数)
  - AC-3.4:文件已存在 → **覆盖**前次内容(整版本模式多次执行可幂等)
  - AC-3.5:单需求文件路径与模式 1 完全相同(允许模式 1 和模式 2 互相覆盖)

### FR-4:`REVIEW.md` 聚合文件结构(详细)

- **描述**:`REVIEW.md` 是"整版本评审"的聚合视图
- **优先级**:必须
- **AC**:
  - AC-4.1:文件标题 = `# 整版本代码审查报告 — <版本号>`
  - AC-4.2:含 "## 1. 评审概览" 段(评审时间 / 评审范围 / 评审人 / 发现总数 / 必须改数)
  - AC-4.3:含 "## 2. 各需求评审摘要" 段(按需求列出)
  - AC-4.4:含 "## 3. 评审发现汇总(去重)" 段(跨需求去重后的发现列表;按级别排序)
  - AC-4.5:含 "## 4. 派生任务汇总" 段(跨需求的所有派生任务)
  - AC-4.6:含 "## 5. 评审人/AI 备注" 段(可选;预留)

### FR-5:派生任务(沿用模式 1)

- **描述**:整版本模式的"必须改"派生任务,按其所属需求追加到对应 `PLAN.md` 任务总览
- **优先级**:必须
- **AC**:
  - AC-5.1:每条"必须改"派生任务的 `触发/来源` 列填"整版本评审",`关联任务` 列填被修正原任务编码
  - AC-5.2:派生任务编码生成沿用 `encoding-conventions.md §规则 4`(`code-plan` 阶段生成);`code-review` 自身**不**生成编码,而是**追加**到 `PLAN.md` 任务总览的最末行
  - AC-5.3:`PLAN.md` 中"任务总览"区段被追加任务行(由 `code-review` 写,而不是 `code-plan`)

### FR-6:与模式 1 完全兼容(Q-2 锁定 B+C 混合)

- **描述**:整版本模式的输出**严格兼容**模式 1;`code-auto` 评审循环可继续使用模式 1
- **优先级**:必须
- **AC**:
  - AC-6.1:整版本模式生成的 `REVIEW-REPORT.md` 内容与模式 1 完全相同(无字段差异)
  - AC-6.2:`code-auto` 解析"必须改列表"的逻辑**不**变 — 解析 `REVIEW-REPORT.md` 的"评审发现汇总"区段
  - AC-6.3:`code-auto` 评审循环**继续**走模式 1(单需求);整版本模式由用户手动调

### FR-7:不修改 `code-review` 现有行为

- **描述**:`code-review REQ-NNNNN` 模式 1 行为**完全不变**
- **优先级**:必须
- **AC**:
  - AC-7.1:`code-review REQ-NNNNN` 输出的 `REVIEW-REPORT.md` 格式与 V0.0.1 完全一致
  - AC-7.2:`code-review` 现有 frontmatter **不**变
  - AC-7.3:模式 1 的派生任务逻辑**不**变
  - AC-7.4:本需求**仅**新增"无参入口"的处理逻辑,模式 1 路径**不**重写

### FR-8:不修改其他 9 个 `code-*` 技能

- **描述**:本需求**只**扩展 `code-review` 自身的 SKILL.md;其他 9 个技能**不**被本需求修改
- **优先级**:必须
- **AC**:
  - AC-8.1:`marketplace.json` / `plugin.json` **不**被本需求修改
  - AC-8.2:`code-auto` / `code-dashboard` / `code-publish` / `code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-version` **不**被本需求修改
  - AC-8.3:`assistants/rules/` 下任何规范文件**不**被本需求修改
  - AC-8.4:中英 README 若需更新"code-review 增加整版本模式",按 `doc-conventions §规则 1` 同次提交中英 — 但本需求**不**主动写 README(由 `code-rule` 沉淀)

### FR-9:报告与建议

- **描述**:整版本模式完成时,向用户报告
- **优先级**:必须
- **AC**:
  - AC-9.1:完成时:输出"评审范围 + 各需求发现数 + 派生任务数 + 输出文件路径"
  - AC-9.2:无评审目标时:输出"本版本无可评审的需求" + 提示
  - AC-9.3:多次执行时:输出"与前次比较"(哪些发现已消失 / 新增)

## 9. 边界与异常

| ID | 场景 | 处理 |
| --- | --- | --- |
| **E-1** | 无 `.current-version` | 提示调 `code-version`,退出 |
| **E-2** | 某需求的 `code-review` 内部失败 | 跳过该需求 + 报告"REQ-NNNNN 评审失败:<err>"+ 继续 |
| **E-3** | 无"已完成"需求 | 报告"本版本无可评审的需求" + 退出(Q-3 采纳默认) |
| **E-4** | 单需求 REVIEW-REPORT 写入失败 | 跳过该需求 + 报告 + 继续 |
| **E-5** | 聚合 REVIEW.md 写入失败 | 报告 + 不整体失败(单需求文件已写) |
| **E-6** | 派生任务编码生成失败(需求内 TASK 已满 99999) | 跳过该发现 + 报告 + 继续 |
| **E-7** | 多次执行整版本模式 | 全部覆盖(Q-2 锁定 + NFR-3 幂等) |
| **E-8** | 模式 1 与模式 2 互覆盖 | 单需求 REPORT 互相覆盖(NFR-6 允许) |
| **E-9** | `code-auto` 评审循环触发模式 2 | **不**触发(Q-5 锁定 — `code-auto` 继续用模式 1) |
