## 4. 关键决策与权衡

### 4.1 末尾兜底 commit 的特殊处理
- **决策**:本任务**不**含末尾兜底 commit 步骤
- **理由**(`PLAN.md §9.4`):
  - T-001 ~ T-003 已各自 commit 自己的 SKILL.md
  - 本任务的 dirty 文件是 `V0.0.2/RESULT.md`(看板)
  - 末尾兜底步骤的语义是"git status --porcelain 收集 + 弹窗确认 + commit",但 T-004 阶段已**无**用户参与(`code-it` 是自动化执行),弹窗不再适用
- **替代方案**:本任务**手工** `git add` + `git commit`(无弹窗)
- **commit message scope**:`dashboard`(因本任务改的是看板)
- **范围**:**只**暂存 `V0.0.2/RESULT.md`,**不**"补 commit" `code/TASK-REQ-00005-00004/` 目录的过程文档(由 `code-review` 阶段统一管理)

### 4.2 M1:可发布 vs M1.REQ-00005 里程碑
- **决策**:不修改既有 M1:可发布(本版本整体),**追加** M1.REQ-00005(per-requirement 粒度)
- **理由**:
  - M1:可发布 覆盖整个 V0.0.2 的所有任务(包括 REQ-00004/00006-00013 等),状态仍为 `待开始`
  - M1.REQ-00005 仅覆盖本需求的 4 任务,状态 = `已完成`
  - 两个里程碑**并存**,粒度不同

### 4.3 任务清单提交哈希字段
- **决策**:本任务的提交哈希先填 `<TBD>`,在手工 `git commit` 后**回填**
- **理由**:
  - 看板"任务清单"需要立即反映本任务状态推进(用户查询时可见)
  - 提交哈希是 commit 后的产物,需要**先** commit **再**回填
  - 这是 1 个 commit 的"两步走"模式

### 4.4 不触发 `dashboard-conventions §规则 1` 三方同步
- **决策**:本任务**不**触发三同步
- **理由**:
  - 本任务**不**扩展看板字段,只追加常规条目
  - 三方同步(`templates/version-RESULT.md` + `CLAUDE.md` + `dashboard-conventions.md`)只在新字段/枚举/语义变化时触发
  - 任务清单的 4 行 + 统计 + 里程碑 + 变更记录 都是**既有字段**的常规追加
