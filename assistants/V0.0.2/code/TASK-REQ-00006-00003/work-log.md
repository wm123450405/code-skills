### 关键决策与权衡

#### 决策 IT-1:H1 标题格式
- **选定**:`# 升级部署手册 — 从 <源版本> 升级到 <本版本号>`
- **理由**:对照 DEPLOY.md 的 `# 发布部署手册 — <本版本号>`,**强调"升级"而非"全新"**;`code-publish` 替换后一目了然
- **依据**:data-changes.md §4.2

#### 决策 IT-2:复用 DEPLOY.md 的 §2-§7 结构
- **选定**:§2-§7 章节与 DEPLOY.md 几乎相同,但**措辞聚焦"升级"**:
  - §2 打包:沿用(可能 90% 相同)
  - §3 获取成果物:沿用
  - §4 上传:沿用
  - §5 初始化:**改为"升级"的语境**:
    - §5.1 环境变更(若运行时依赖变化)
    - §5.2 数据库**升级**脚本(DB 升级 vs DB 建表)
    - §5.3 数据库迁移数据(若有)
    - §5.4 配置变更(diff vs 全新)
  - §6 重启:措辞改"重启现有服务"
  - §7 验证清单:沿用 + 强调"升级后功能"验证
- **理由**:与 DEPLOY.md 区分"全新部署" vs "升级";但**不**完全相同(否则用户会困惑"两份手册有何区别")
- **依据**:NFR-5 通用性 + FR-4 AC-4.1

#### 决策 IT-3:§8 回滚方案为"小节+3 子点"
- **选定**:`## 8. 回滚方案(新增)` + 3 子点:
  - `回滚到 <源版本> 的步骤:<...>`
  - `数据回滚(如有):<DB 回滚脚本>`
  - `触发回滚的条件:<...>`
- **理由**:3 子点覆盖"代码回滚 / 数据回滚 / 决策点";运维人员可一目了然
- **依据**:needs §6.2 + FR-4 AC-4.1

#### 决策 IT-4:§5.2 "DB 升级脚本"默认示例用 SQL 转储
- **选定**:`mysqldump --no-data <source_db> > schema-backup.sql` + `mysql <target_db> < upgrade.sql`
- **理由**:DB 升级的常见模式 = "先备份旧 schema → 应用新 schema"
- **依据**:NFR-5 + AC-3.2

#### 决策 IT-5:§1 概述加入"升级方式"3 选 1
- **选定**:`<原地 / 蓝绿 / 灰度>` 三选 1 placeholder
- **理由**:升级方式显著影响后续操作(蓝绿/灰度有"流量切换"步骤);必须显式
- **依据**:data-changes.md §4.2 章节 1 + needs §6.2

#### 决策 IT-6:§8 回滚方案**不**给详细命令,只给"步骤骨架"
- **选定**:仅列"回滚步骤"+"数据回滚"+"触发条件"3 项,每项用 placeholder
- **理由**:回滚步骤**强依赖具体环境**;给详细命令可能误导;留 placeholder 让用户补全
- **依据**:FR-4 AC-4.1(只要求"步骤"骨架)

#### 决策 IT-7:不与 DEPLOY.md "共享太多"
- **选定**:不引入 DEPLOY.md 的附录"发布后通知";不引入"其他启动方式"列表(因为升级时通常**沿用**已有服务)
- **理由**:升级手册聚焦"变更";"通用但冗长"反而稀释重点
- **依据**:NFR-9(可读性优先)

#### 决策 IT-8:不预设"升级窗口" / "维护模式"
- **选定**:不写"维护模式"或"升级窗口"placeholder
- **理由**:本模板是"通用骨架";具体维护策略由用户决定
- **依据**:NFR-5
