- **主题**(来自用户输入):
  > 优化 `/code-rule` 技能,增加不同类型的核心编码规范的解析或引导。示例:对 CLAUDE.md 增加说明、对模板(requirements/design/plan/templates)提供内容提示等。

## 1. 需求概述

**为谁**:`code-rule` 技能的当前用户(项目主导者 / 规范沉淀人),以及所有依赖规范落地的下游技能(`code-design` / `code-plan` / `code-it` / `code-unit` / `code-review`)与本仓库内部的 AI 工作流(读 `CLAUDE.md`)与模板消费方(读 `templates/*.md`)。

**解决什么问题**:当前 `code-rule` 技能**只能把规范写到 `./assistants/rules/<分类>.md` 这一个目标**。但项目的"核心编码规范"实际上分散在多个目标位置:

| 维度 | 当前位置 | 缺口 |
| --- | --- | --- |
| 项目级规则条款 | `./assistants/rules/<分类>.md`(5 个现有文件) | 缺"框架/三方依赖/语言与命名/目录与模块/代码书写/提交合并"等 6 类常用分类,且"未来可能需要"的分类没有占位文件 |
| AI 工作指引(给 Claude Code 读取) | `plugins/code-skills/CLAUDE.md`(当前为静态内容) | 没有"由 code-rule 维护的 AI 工作约定"小节;新规范无法以 AI 可理解的方式追加到 CLAUDE.md |
| 模板内容提示(给消费模板的技能读取) | `plugins/code-skills/skills/<技能>/templates/*.md`(当前为静态模板) | 没有"内容提示"段落,AI 在使用模板时缺少对必填字段/格式的明确引导 |

**带来什么价值**:
1. **统一入口**:`code-rule` 成为所有"核心编码规范"的唯一添加入口(规则 / AI 指引 / 模板提示)
2. **可扩展性**:6 类常用规范分类 + 占位文件机制,使项目在需要新规范时"开箱即用"
3. **下游协同**:AI 在 CLAUDE.md 中读到工作约定、在模板中读到字段提示,产出物质量更稳定

## 2. 背景与目标

### 2.1 背景
- `code-rule` 技能当前仅支持"识别用户自然语言 → 归类 → 追加到 `./assistants/rules/<分类>.md`"的单一流程
- 项目演进中,用户(参见 V0.0.0 EXISTING-003 code-rule)多次提出"规范不止是规则文件"——它还包括 AI 工作约定、模板字段提示等
- 当前 5 个规范文件(`dashboard-conventions` / `doc-conventions` / `marketplace-protocol` / `module-conventions` / `skill-conventions`)覆盖不全;常见规范(命名、代码风格、提交规范)未沉淀
- 部分"未来可能需要"的规范(如框架/三方依赖/提交合并)若到时再建,会经历"先发现缺 → 走完整 code-require → code-design → code-plan → code-it 流程"的 5 步延迟

### 2.2 业务目标
- **G-1**:`code-rule` 支持 3 种"目标类型"(规则文件 / CLAUDE.md / 模板)
- **G-2**:Type A 规则文件覆盖 6 类常用规范(部分条件性,部分默认)
- **G-3**:6 类中"当前不需要但未来可能需要"的类型,先建空占位文件,降低未来启用成本
- **G-4**:Type B(CLAUDE.md)与 Type C(模板)有明确的"内容结构"和"插入位置"约定

### 2.3 本次目标
- 扩展 `code-rule` 技能本身(SKILL.md + 新增过程文档 + 扩展现有模板)
- 不修改其他 9 个 `code-*` 技能的 SKILL.md(保持边界)
- 不修改 `marketplace.json` / `plugin.json`(继承现有"不修改范围")
- 不回填 6 类规范的"内容"(本次只解决"能写",具体规则内容由用户在后续调 `code-rule` 时逐条添加)

## 3. 用户角色与场景

### 3.1 角色
- **R-1 项目主导者**:在 V0.0.x 启动时,先用 `code-rule` 建占位规范(6 类空文件);后续按需填充
- **R-2 规范沉淀人**:在开发过程中,发现需要新规范,通过 `code-rule` 添加(可能是 Type A/B/C 任一类型)
- **R-3 模板作者**:为某个技能维护 `templates/*.md` 时,需要在模板中加"内容提示"以引导 AI 正确使用
- **R-4 AI 协作者**:在使用 `code-require` / `code-design` 等技能时,会读 `CLAUDE.md`(AI 工作约定)和 `templates/*.md`(模板提示),从而产出更规范的文档

### 3.2 关键场景
- **S-1 项目初始化**:用户调 `code-rule` 一键建 6 类 Type A 规范的占位文件(其中 3 类为空,3 类预填 1-2 条示例规则)
- **S-2 添加命名规范**:用户说"函数命名统一 camelCase",`code-rule` 识别为 Type A / 分类=语言与命名 / 文件=`naming-conventions.md`,追加规则
- **S-3 给 CLAUDE.md 加 AI 约定**:用户说"AI 写 RESULT.md 时必须先读 requirements 模板",`code-rule` 识别为 Type B,追加到 CLAUDE.md 的"## AI 工作约定"小节
- **S-4 给 templates/plan.md 加字段提示**:用户说"plan.md 必须含任务依赖图",`code-rule` 识别为 Type C,在 plan.md 末尾追加"## 提示: 任务依赖图"
- **S-5 内联模板提示**:用户说"在 plan.md 的'## 4. 模块详细化'下加一个提示,说明每个模块都要写'依据规范'",`code-rule` 识别为 Type C + 内联模式,在指定小节下追加
- **S-6 未来启用占位文件**:项目演化到需要"提交合并规范"时,用户调 `code-rule`,技能检测到 `commit-conventions.md` 是空占位,直接追加规则;无需先建文件

## 4. 功能需求(FR)

### FR-1:扩展 `code-rule` 技能支持 3 种"目标类型"(核心扩展)

- **描述**:`code-rule` 在识别用户描述后,根据"目标位置"分为 3 种类型,每种类型有不同的处理逻辑:
  - **Type A:项目级规范条款** → 写入 `./assistants/rules/<分类>.md`(已有,需扩展)
  - **Type B:AI 工作指引** → 写入 `plugins/code-skills/CLAUDE.md` 的"## AI 工作约定(由 code-rule 维护)"小节
  - **Type C:模板内容提示** → 写入 `plugins/code-skills/skills/<技能>/templates/<name>.md` 的末尾追加或内联
- **入口**:用户调用 `code-rule` 时,可隐式(自动推断)或显式(说明"对 X 文件加 Y")指定类型
- **前置条件**:技能必须先识别类型,再走对应子流程
- **主流程**:
  1. `code-rule` 收到用户描述
  2. 走"类型识别"(自动 + 显式确认,详见 FR-7)
  3. 根据类型走对应子流程(FR-2 ~ FR-6)
  4. 写入目标文件
  5. 汇报
- **分支/异常**:
  - 类型推断置信度低 → 显式追问用户
  - 目标文件不存在 → Type A 创建;Type B/C 需先确认(避免误建)
- **数据变化**:
  - `./assistants/rules/` 下增加新分类文件
  - `CLAUDE.md` 末尾增加"AI 工作约定"小节(可能由本次需求首次创建)
  - `templates/*.md` 末尾或内联增加"提示"
- **来源**:(用户原始描述 §1)

### FR-2:扩展 Type A 的"分类"识别,支持 6 类核心规范

- **描述**:`code-rule` 在 Type A 流程中,识别以下 6 类(基于关键词 + 用户描述语义):
  - **C-1 框架规范**(`framework-conventions.md`,**条件性** — 仅当用户选择"现在需要"或"未来占位")
  - **C-2 三方依赖规范**(`dependency-conventions.md`,**条件性**)
  - **C-3 语言与命名规范**(`naming-conventions.md`,**默认**)
  - **C-4 目录与模块规范**(`directory-conventions.md`,**默认**;或与现有 `module-conventions.md` 合并 —— 详见 §12 Q-4)
  - **C-5 代码书写规范**(`coding-style.md`,**默认**)
  - **C-6 提交与合并规范**(`commit-conventions.md`,**条件性**)
- **入口**:Type A 流程的"分类确认"步骤
- **主流程**:
  1. 对每条规则,基于关键词给出**候选分类**;若同时命中多个 → 展示给用户选
  2. 若用户描述明确命中"条件性"分类(C-1/C-2/C-6) → 主动追问"现在需要 / 未来占位 / 跳过"
  3. 若用户描述命中"默认"分类(C-3/C-4/C-5) → 直接进入现有分类确认步骤
- **数据变化**:`./assistants/rules/` 下增加新分类文件
- **来源**:(用户澄清回答 Q-1,2026-06-03 20:35)

### FR-3:Type A 条件性分类的"占位"模式

- **描述**:对 C-1 / C-2 / C-6(条件性分类),若用户选择"未来占位"或"未来可能需要",`code-rule` 创建对应文件但**不填任何规则内容**;仅含最小骨架(分类标题 + "本规范由 code-rule 维护" + "## 规则 1: (待添加)"占位)
- **前置条件**:用户明确选"未来占位"
- **主流程**:
  1. 走 Type A 流程
  2. 用户选分类后,追问"现在需要 / 未来占位 / 跳过"
  3. 若选"未来占位" → 创建文件,仅骨架
  4. 汇报:文件已创建,等待未来填充
- **分支/异常**:用户选"跳过" → 不创建任何文件
- **数据变化**:`./assistants/rules/<新分类>.md` 新建
- **来源**:(用户澄清回答 Q-1 备注)

### FR-4:Type A 默认分类的"引导"模式

- **描述**:对 C-3 / C-4 / C-5(默认分类),若用户首次添加该类规则,`code-rule` 引导用户填 1-2 条示例规则(若用户愿意);若用户只提供 1 条 → 至少补全"规则 1"小节
- **前置条件**:用户选"现在需要"或首次添加
- **主流程**:
  1. 检测目标分类文件是否存在
  2. 不存在 → 走"新建"流程,提示用户"是否同时补 1-2 条示例规则"
  3. 存在 → 走现有"追加"流程(行为不变)
- **数据变化**:新分类文件含 1+ 条规则
- **来源**:本需求派生(NFR-1 向后兼容 + 用户期望"减少未来启用成本")

### FR-5:Type B(AI 工作指引)的标准结构与插入位置

- **描述**:Type B 处理时,目标文件 = `plugins/code-skills/CLAUDE.md`,插入位置 = 文件末尾追加"## AI 工作约定(由 code-rule 维护)"小节(若不存在则创建)
- **每个"指引"的标准字段**:
  - **指引标题**:`### 指引 N:<指引简称>`
  - **描述**:一句话或一段说明该指引做什么
  - **适用场景**:什么情况下 AI 应遵守
  - **期望行为**:AI 应如何行动(可含"读哪个文件/调哪个技能/避免什么")
  - **来源**:用户原始描述 + 添加时间戳
- **主流程**:
  1. 检测 CLAUDE.md 是否含"## AI 工作约定(由 code-rule 维护)"小节
  2. 不存在 → 用 `Edit` 在文件末尾追加(若文件末尾无空行,先补空行)
  3. 存在 → 在该小节末尾追加新的"指引 N"
  4. **绝不**修改 CLAUDE.md 其它任何小节
- **数据变化**:`CLAUDE.md` 末尾追加小节
- **来源**:(用户原始描述示例 §1)

### FR-6:Type C(模板内容提示)的两种插入位置

- **描述**:Type C 处理时,支持两种插入位置:
  - **C-a 末尾追加**:在模板文件末尾追加"## 提示: <主题>"小节(若不存在则新建"## 提示"二级小节,作为"内容提示区")
  - **C-b 内联**:在模板文件的指定二级小节(如"## 4. 模块详细化")下追加"### 提示: <字段>"或"#### 提示: <字段>"
- **每个"提示"的标准字段**:
  - **字段/小节名**:提示针对的模板字段或小节
  - **必填**?是/否
  - **简明说明**:1-2 句话,告诉使用者(主要是 AI)该写什么
  - **错误示例**(可选):常见的错误填写
- **主流程**:
  1. 询问用户:追加位置 = 末尾 / 内联
  2. 若末尾 → 在模板文件末尾追加
  3. 若内联 → 用户指定目标二级小节(用二级标题精确匹配);若用户未指定 → 追问
  4. 写入提示
- **数据变化**:`templates/*.md` 增加"提示"
- **来源**:(用户澄清回答 Q-3,2026-06-03 20:35;用户原始描述示例 §1)

### FR-7:类型识别方式(自动 + 显式)

- **描述**:`code-rule` 优先从用户描述的关键词/语义自动推断类型;推断不出或置信度低时,显式追问用户
- **自动推断的关键词线索**:
  - 提到"规则"/"规范"/"约定"(无特定文件提示) → Type A
  - 提到"CLAUDE.md"/"AI 工作"/"AI 约定" → Type B
  - 提到"模板"/"templates"/具体模板文件名 → Type C
  - 提到具体规则分类名(如"命名"/"提交规范") → Type A + 分类
- **主流程**:
  1. 收到用户描述
  2. 关键词匹配 → 候选类型
  3. 若候选唯一 + 置信度高 → 跳过追问,进入对应类型子流程
  4. 若候选多 / 置信度低 / 关键词不命中 → 用 `AskUserQuestion` 列出候选(让用户确认)
- **分支/异常**:用户可显式覆盖(`"对 X 文件加 Y"` 形式)
- **来源**:(用户澄清回答 Q-2,2026-06-03 20:35)

### FR-8:保持现有 Type A 流程不变(向后兼容)

- **描述**:对 Type A 的"识别 → 分类 → 澄清字段 → 写文件"现有 4 步流程,**完全保持不变**;仅在分类识别时多支持 6 个新分类
- **数据变化**:无
- **来源**:NFR-1 向后兼容

### FR-9:`code-rule` 不得修改的边界

- **描述**:`code-rule` 在处理任何类型时,均**不得**修改:
  - `.claude-plugin/marketplace.json` 任何字段
  - `plugins/code-skills/.claude-plugin/plugin.json` 任何字段
  - 其他 9 个 `code-*` 技能的 SKILL.md(包括 `name` / `description` frontmatter)
  - `plugins/code-skills/skills/<技能>/SKILL.md` 的 frontmatter
  - `plugins/code-skills/README.md` / `README.en.md` 中描述技能用途的章节
  - `assistants/V0.0.0/` 与 `assistants/V0.0.1/` 下的任何工作文件
- **来源**:继承 `code-rule` 现有"不修改范围"(`SKILL.md` §不要做的事)

### FR-10:Type B / Type C 严禁重写既有内容

- **描述**:`code-rule` 处理 Type B / Type C 时:
  - Type B:只**追加**到 CLAUDE.md 末尾的"AI 工作约定"小节;不修改其它任何小节
  - Type C:只**追加**到模板末尾的"提示"小节 或 内联到指定小节末尾;不修改其它任何内容
- **验证手段**:`git diff` 应只显示"纯追加",无任何"删除行"
- **来源**:NFR-3 作用域边界

## 9. 边界与异常

| 场景 | 处理方式 |
| --- | --- |
| 用户描述同时命中 Type A 与 Type B(如"对 CLAUDE.md 增加一条命名规则") | `code-rule` 优先按字面"目标文件"判断(用户说"对 CLAUDE.md 增加" → Type B);若仍模糊,追问 |
| 目标文件不存在但用户没明确"创建"意图 | Type A:直接创建(Type A 的常规行为);Type B/C:先确认"是否创建" |
| 模板文件有多个匹配(用户描述不精确) | 列出所有匹配,让用户选 |
| 用户想批量添加多条规则 | 沿用现有"逐条处理 + 询问是否继续"流程(FR-2 分类逐条确认) |
| 占位文件被用户手动修改了 | 不感知差异;`code-rule` 仍按"目标分类文件存在 → 追加"流程(FR-4) |
| `code-rule` 处理 Type B 时,CLAUDE.md 已被用户外部修改 | 仅追加"## AI 工作约定"小节;不重写任何其它小节 |
| 用户在 `code-rule` 处理中途中止 | 已写入的部分保留;后续调用 `code-rule` 可继续追加(但**不**自动检测"中断点") |
