### Q-7(本轮锁定):TASK 编码结构
- **用户原始追加指令**(`/code-require REQ-00002 ...`):
  > 修改其中的任务编码命名规范:需求任务为 `TASK-REQ-需求编码-NNNNN`,修复任务为 `TASK-BUG-缺陷编码-NNNNN`
- **解读**:TASK 不再采用全局独立计数(G1),也不沿用旧嵌套(G3),而是采用**新嵌套式**:
  - 需求任务:`TASK-REQ-<REQ 数字部分>-NNNNN`,如 `TASK-REQ-00001-00001`(REQ-00001 的第 1 个任务)
  - 修复任务:`TASK-BUG-<BUG 数字部分>-NNNNN`,如 `TASK-BUG-00001-00001`(BUG-00001 的第 1 个修复任务)
  - 正则:`^TASK-(REQ|BUG)-\d{5}-\d{5}$`
  - 任务序号在**父级**内独立递增(REQ-00001 的 TASK 编号从 1 起;BUG-00001 的 TASK 编号也从 1 起,二者不冲突)
- **影响**:
  - **关闭**Q-7 选项 G1 / G2 / G3,采用**第四方案 G4(新嵌套)**
  - 修订 FR-1:正则增加 TASK 嵌套式
  - 修订 FR-2:code-it 的 regex 从原 `^REQ-\d{4}-\d{4}-\d{3}$` 改为 `^TASK-(REQ|BUG)-\d{5}-\d{5}$`,且需双路径解析(REQ 路径 / BUG 路径,从 TASK 编码中第 2 段判断)
  - 修订 FR-3:模板示例 `REQ-2026-0001-001` → `TASK-REQ-00001-00001`,`BUG-001-fix-001`(若有) → `TASK-BUG-00001-00001`
  - **新增 Q-12 待澄清**:"需求编码"在 TASK 编码中是否含 `REQ-` 前缀?
    - (a) 仅数字段(`TASK-REQ-00001-00001`)— 推荐,字面与正则均最简洁
    - (b) 含 REQ- 前缀(`TASK-REQ-REQ-00001-00001`)— 严格按字面"需求编码=REQ-NNNNN"嵌套
  - 当前 v2 文档**默认采用 (a) 解读**(仅数字段),用户确认 (b) 后再做增量
