## 1. 关键详细化决策

### D-1:任务拆分粒度
- **选定**:按"文件类型 + 验证阶段"拆分 4 个任务
  - `REQ-00001-001`:改 `.claude-plugin/marketplace.json` 根 name
  - `REQ-00001-002`:同步中英 README(`README.md` + `README.en.md`)
  - `REQ-00001-003`:核查 `plugins/code-skills/CLAUDE.md`
  - `REQ-00001-004`:全仓库穷举式 Grep + 偏差日志 + 不变量自检 + commit
- **理由**:
  - 边界清晰,code-review 可独立验收
  - T-001/T-002 是"写",T-003/T-004 是"读"——粒度对称
  - 失败重试粒度合适(若 T-002 中英文不一致,可只回退 README)
  - T-004 兼做"最终 commit 节点",把"所有写"与"最后审 + 提交"分离
- **依据规范**:无直接对应;遵循 `code-plan` SKILL.md 步骤 10A"任务拆分原则"

### D-2:任务编号格式
- **选定**:`REQ-00001-NNN`(3 位补零,与 `code-plan` SKILL.md 模板一致)
- **依据规范**:`code-plan` SKILL.md 步骤 10A"任务编号"

### D-3:任务编码与 REQ-00002 v2 新格式的并存策略
- **现状**:REQ-00002 v2 锁定 TASK 编码为 `TASK-REQ-<REQ 数字段>-NNNNN`
- **本计划采用**:`REQ-00001-NNN` 旧格式
- **理由**:
  - REQ-00001 概要设计阶段在 REQ-00002 v2 锁定**之前**完成,采用旧格式
  - 本需求 4 个任务的全部产出物在 REQ-00002 实施前完成
  - 后续如需切换为新格式,由 REQ-00002 `code-it` 阶段统一清理
- **依据规范**:概要设计 §12.2 + REQ-00002 v2 G4

### D-4:commit 粒度
- **选定**:**全部 4 个文件(含 T-003 默认 0 变更)在 1 个 commit 内**(继承概要设计 Q-4 决策)
- **理由**:
  - `doc-conventions.md §规则 1` 强约束(README 中英同次提交)
  - 避免中间态(JSON 改但 README 未改 → install 命令指向不存在的 marketplace)
  - git revert 友好
- **依据规范**:`doc-conventions.md §规则 1`

### D-5:实施工具
- **Edit**(3 个文件) + **Grep**(验证) + **Bash: git diff --stat**(整体审阅)
- **理由**:继承概要设计 Q-1/Q-2 决策(diff 最小化)
- **T-003 工具**:仅 Grep + Read,无 Edit(默认 0 变更)

## 3. 测试策略推导

> 本需求**纯文档/字符串层变更**,无应用代码,无单元测试。`code-unit` 阶段对本需求所有任务标记为 `不适用`。

- **不适用理由**:无应用代码,无测试框架,无可运行单元
- **替代验证手段**(覆盖原"测试要点"角色):
  - 字节级 diff 审阅(`git diff --stat` + `git diff`)
  - Grep 关键短语验证
  - JSON 语法验证(Claude Code 加载时自动校验)
  - 中英 README 并列 diff 对照
  - 字段逐项核对(`grep "name"` 全文)
  - 11 条不变量自检
