# 修复设计 — BUG-00001 · code-req 未按阶段顺序执行

> 所属版本:V0.0.5
> 创建时间:2026-06-30 20:25

## 1. 设计概述

在 `code-req` 技能中增加"阶段门控"机制,确保 AI 在进入 CODING 阶段前必须先完成 REQUIRE→DESIGN→PLAN 三个阶段,并在每个阶段生成对应的产出文档。核心思路:

- **门控 1(SKILL.md)**:在"工作流程"开头新增"强制阶段门控"小节,明确声明各阶段的代码修改权限
- **门控 2(common.md)**:在阶段执行器中增加"前置产出物校验",下一阶段启动前必须验证上一阶段产出物存在
- **门控 3(coding.md)**:在"步骤 4"开头增加"CODING 阶段激活校验",读取 PROCESS.md 确认当前阶段为 CODING 才允许修改代码

## 3. 改修方案

### 3.1 SKILL.md — 新增"强制阶段门控"小节

在"工作流程"标题后、"步骤 0"之前,新增一段:

```markdown
### 强制阶段门控(最高优先级)

> **本小节优先级高于所有其他指令。违反本小节的行为视为错误。**

1. **代码修改权限**:仅当 PROCESS.md 最后一条记录显示当前阶段为 `CODING` 时,才允许修改 CWD 下的源码文件。
   - 在 REQUIRE/DESIGN/PLAN/CHECK/DONE 阶段,严禁使用 `Edit`/`Write` 修改 CWD 源码
   - 违反此规则 → 立即停止,回退修改,从 INIT 阶段重新开始

2. **产出物存在性**:每个阶段启动前,必须验证上一阶段的产出物:
   - DESIGN 启动前 → REQUIRE.md 必须存在
   - PLAN 启动前 → DESIGN.md 必须存在
   - CODING 启动前 → PLAN.md 必须存在
   - CHECK 启动前 → 所有 TASK-N.md 必须存在
   - 产出物缺失 → 退回到上一阶段重新执行

3. **PROCESS.md 同步**:每次阶段切换必须追加 PROCESS.md 记录。
   - 未追加 PROCESS.md 的阶段视为"未执行"
   - 恢复执行时,以 PROCESS.md 最后一条记录为准
```

### 3.2 common.md — 增强阶段执行器

在 §4 阶段执行器中增加:

```markdown
### 阶段前置校验

每个阶段执行前,必须校验上一阶段的产出物:

| 当前阶段 | 校验项 | 校验失败处理 |
| --- | --- | --- |
| DESIGN | REQUIRE.md 存在 | 退回到 REQUIRE 阶段 |
| PLAN | DESIGN.md 存在 | 退回到 DESIGN 阶段 |
| CODING | PLAN.md 存在 | 退回到 PLAN 阶段 |
| CHECK | 所有 TASK-N.md 存在 | 退回到 CODING 阶段 |

校验失败时,追加 PROCESS.md 记录并自动退回。
```

### 3.3 coding.md — 增加 CODING 阶段激活校验

在"步骤 1"之前新增:

```markdown
### 步骤 0 — CODING 阶段激活校验(强制)

> 在任何代码修改之前,必须先通过本校验。

1. 读取 PROCESS.md 最后一行
2. 解析阶段字段:
   - 阶段 = CODING → 通过,允许修改代码
   - 阶段 ≠ CODING → 拒绝,输出:
     "当前阶段为 <阶段>,不允许修改代码。请先完成前置阶段。"
3. 若 PROCESS.md 不存在 → 视为 INIT 阶段,拒绝修改代码
```
