# 修复设计 — BUG-00007 · REQ-00049 执行中遗漏用户确认环节

> 所属版本:V0.0.5
> 创建时间:2026-06-30

## 1. 设计概述

修复 REQ-00049 执行过程中遗漏的 3 个用户确认环节。核心思路:回补 REQUIRE.md 和 DESIGN.md 中缺失的用户确认内容，通过 AskUserQuestion 向用户确认后更新对应文档。

## 2. 模块拆分

| 模块 | 职责 | 涉及文件 | 依赖 |
| --- | --- | --- | --- |
| 回补 REQUIRE 确认 | 向用户确认 --confirm 与 --auto 关系 | `req/REQ-00049/REQUIRE.md` | 无 |
| 回补 DESIGN 确认 | 向用户确认默认行为变更 + 步骤 9 三确认 | `req/REQ-00049/DESIGN.md` | REQUIRE 确认结果 |

## 3. 修复方案

### 3.1 修复 1:确认 --confirm 与 --auto 关系

**当前状态**:REQUIRE.md FR-6 标注"设计推断"，未向用户确认
**修复**:向用户确认 `--confirm` 与 `--auto` 的具体关系，更新 FR-6

需确认的问题:
- `--confirm` 与 `--auto` 是互斥(同时传入报错)还是某种组合?
- 若互斥，哪个优先?

### 3.2 修复 2:确认默认行为变更

**当前状态**:DESIGN.md 决策 2 直接选择"去除默认 AskUserQuestion"，未向用户确认
**修复**:向用户确认是否移除原有默认 AskUserQuestion

需确认的问题:
- 无 flag 时，阶段间是否自动流转?
- 原有的阶段内 AskUserQuestion(需求澄清、方案选择)是否保留?
- `--confirm` 确认与原有 AskUserQuestion 的关系是什么?

### 3.3 修复 3:回补步骤 9 用户确认

**当前状态**:DESIGN 阶段完全跳过了 design.md reference 步骤 9
**修复**:向用户确认以下三个维度:

- **9a 扩展性确认**:是否需要预留扩展点(如后续支持更多 flag 组合)?
- **9b 方案选型确认**:三个决策点(互斥/默认行为/重读时机)的方案是否合理?
- **9c 改修方案确认**:修改 3 个文件(SKILL.md ×2 + common.md)的范围是否合适?

## 4. 关键流程

```
1. 向用户确认 --confirm 与 --auto 关系 → 更新 REQUIRE.md FR-6
2. 向用户确认默认行为变更方案 → 更新 DESIGN.md 决策 2
3. 向用户确认步骤 9 三确认 → 追加 DESIGN.md 步骤 9 记录
4. 更新 DESIGN.md 变更记录
```

## 5. 方案选型

### 决策:修复方式
- **选择**:在当前 BUG-00007 的 DESIGN 阶段直接向用户确认，然后修改 REQ-00049 的 REQUIRE.md/DESIGN.md
- **备选**:取消 REQ-00049 重新执行
- **选择理由**:REQ-00049 已完成 REQUIRE 和 DESIGN 阶段，只需回补缺失的确认环节，无需重做
- **权衡**:REQ-00049 的 PROCESS.md 记录的是"DESIGN 取消"，回补确认后需手动更新 PROCESS.md
