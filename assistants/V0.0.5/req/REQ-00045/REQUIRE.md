# 需求分析 — REQ-00045 · 补充 REQ-00044 重构后丢失的旧技能能力

> 所属版本:V0.0.5
> 创建时间:2026-06-30 19:31

## 1. 需求概述

REQ-00044 将 14 个旧技能重构为 7 个新技能,在合并过程中丢失了旧技能的若干核心能力。本需求旨在系统性地找回并补充这些丢失的能力,确保新技能 `code-req`(及与其共享 references 的 `code-fix`)具备与原技能体系同等的功能完整度。

(源自:用户对话输入)

## 2. 背景与目标

### 背景

REQ-00044 技能系统 v2 大改版将 `code-require`/`code-design`/`code-plan`/`code-it`/`code-check` 合并为 `code-req` 单一入口,将 `code-auto` 废弃为 `--auto` 参数。重构后新技能产出了 5 个阶段 references(require.md/design.md/plan.md/coding.md/check.md + common.md),但较原技能丢失了以下能力:

1. **语言体系特性适配**:旧技能有 `references/go.md`/`python.md`/`nodejs.md`/`rust.md`/`java-gradle.md`/`java-maven.md` 等语言适配文件,提供项目结构识别、构建/测试/运行命令检测、编码约定、工具链检测等能力,新技能 coding.md/check.md 完全缺失此类能力
2. **评审-编码循环**:旧 `code-auto` 有"code-check → 派生任务 → code-it → 再 code-check → 循环直到无必须改"的完整闭环,新 `code-req` 的 CHECK 阶段仅内联修复"必须改",不形成循环
3. **用户确认环节**:旧技能在需求分析(code-require)、概要设计(code-design)、详细设计(code-plan)各阶段均有详细的用户确认环节(缺失需求细节、设计扩展性、改修方案细节等),新技能仅在各阶段结束时弹出简单确认
4. **要求与限制**:旧技能有大量细节要求(过程文档自适应判定、前置任务守卫、项目可测性守卫、逻辑行统计、逻辑行阈值、错误修复循环、标题解析、编码规范等),新技能中部分缺失

### 目标

1. 为 `code-req` 补充语言体系特性适配能力(6 种语言/构建系统)
2. 为 `code-req` 的 CHECK 阶段补充评审-编码循环机制(CODING↔CHECK 循环,直到无"必须改")
3. 为 `code-req` 的 REQUIRE/DESIGN/PLAN 阶段补充用户确认环节(需求细节澄清、设计扩展性确认、改修方案确认)
4. 找回并补充旧技能中的各项要求与限制(过程文档自适应判定、可测性守卫、逻辑行统计、阈值检测、错误修复循环、标题解析等)

## 3. 功能需求(FR)

### FR-1: 语言体系特性适配能力

**描述**:在 `code-req` 的 references 目录下新增 `languages/` 子目录,包含 6 个语言适配文件,为 CODING 和 CHECK 阶段提供语言感知能力。

**语言适配文件清单**:
| 文件 | 对应语言/构建系统 | 提供能力 |
| --- | --- | --- |
| `references/languages/go.md` | Go | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |
| `references/languages/python.md` | Python | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |
| `references/languages/nodejs.md` | Node.js | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |
| `references/languages/rust.md` | Rust | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |
| `references/languages/java-gradle.md` | Java (Gradle) | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |
| `references/languages/java-maven.md` | Java (Maven) | 项目结构识别、构建/测试/运行命令、编码约定、工具链 |

**每个语言文件包含的章节**:
- §1 项目结构识别(描述文件、源码目录、测试文件约定)
- §2 构建命令检测(编译命令、构建产物)
- §3 测试框架识别(内置/常见扩展、运行命令)
- §4 启动/运行命令检测
- §5 Monorepo 声明文件解析(若适用)
- §6 编码约定(命名规范、文件组织)
- §7 工具链检测(代码行数统计、Lint、格式化)

**输入**:CODING/CHECK 阶段自动检测项目语言类型,加载对应语言文件
**输出**:CODING 阶段使用语言感知的构建/测试/运行命令;CHECK 阶段使用语言感知的编码约定检查
**来源**:旧 `code-it/references/{go, python, nodejs, rust, java-gradle, java-maven}.md` 和旧 `code-check/references/{go, python, nodejs, rust, java-gradle, java-maven}.md`

### FR-2: 评审-编码循环机制(CODING↔CHECK 循环)

**描述**:在 `code-req` 的 CHECK 阶段增加循环机制:CHECK 审查完成后,若存在"必须改"发现,自动生成改修任务、进入 CODING 阶段修复、修复后重新进入 CHECK 阶段审查,循环往复直到无"必须改"发现为止。

**循环流程**:
```
CODING(全部任务完成) → CHECK(审查)
  ├─ 无"必须改" → DONE
  └─ 有"必须改" → 生成改修任务 → CODING(执行改修任务) → CHECK(重新审查)
       └─ 循环直到无"必须改"
```

**关键约束**:
- 循环次数上限:最多 5 轮(防止无限循环),超过后停下询问用户(非 `--auto`)或报告( `--auto`)
- 改修任务编号:使用递增序号,在 PLAN.md 中追加
- 每轮 CHECK 必须重新审查所有变更(非仅改修任务)
- `--auto` 模式下自动循环,无需确认
- 非 `--auto` 模式下,每轮循环完成后确认(继续改修/暂停/取消)

**来源**:旧 `code-auto` 的步骤 5-6 循环(CODE-CHECK → 解析必须改列表 → 派生任务循环 → 回到 CODE-CHECK)

### FR-3: 需求分析阶段用户确认环节

**描述**:在 REQUIRE 阶段增加详细的用户确认环节,包括:

**确认环节 1 — 需求细节澄清**(步骤 5 增强):
- 对模糊、歧义、缺失的功能点或交互,向用户确认
- 能给出 2-4 个具体选项 → 使用 `AskUserQuestion`
- 需要开放式描述 → 自然语言提问
- 每轮最多 1-3 个最阻塞的问题
- 追加 `clarifications.md` 记录问答

**确认环节 2 — 边界条件确认**(新增):
- 确认异常路径/边界值处理方案
- 确认用户角色与权限边界
- 确认与现有功能的交互方式

**输出增强**:REQUIRE.md 中增加"待澄清"标注(明确标注哪些是假设、哪些是用户确认的结果)

**来源**:旧 `code-require` 的步骤 5(与用户澄清)和 clarifications.md 机制

### FR-4: 设计阶段用户确认环节

**描述**:在 DESIGN 阶段增加详细的用户确认环节,包括:

**确认环节 1 — 扩展性确认**(步骤 8 后新增):
- 确认设计方案的可扩展性:是否需要预留扩展点、是否考虑未来变更
- 确认模块拆分的粒度:是否过细/过粗
- 确认接口设计的灵活性:是否需要版本化、是否需要向后兼容

**确认环节 2 — 方案选型确认**(步骤 8 后新增):
- 每个技术决策点确认用户偏好
- 存在多方案时给出选项让用户选择
- 记录决策理由和权衡

**确认环节 3 — 改修方案确认**(步骤 4-7 增强):
- 确认每个模块的修改范围:新增/修改/删除
- 确认接口变更的影响范围
- 确认数据结构变更的兼容性

**来源**:旧 `code-design` 的交互流程和旧 `code-require` 的 AskUserQuestion 模式

### FR-5: 任务排期阶段用户确认环节

**描述**:在 PLAN 阶段增加详细的用户确认环节,包括:

**确认环节 1 — 任务拆分确认**(步骤 2 后新增):
- 确认任务拆分粒度是否合理
- 确认任务间依赖关系是否准确
- 确认里程碑划分是否合理

**确认环节 2 — 优先级确认**(步骤 4 后新增):
- 确认任务优先级排序
- 确认关键路径识别

**来源**:旧 `code-plan` 的交互流程

### FR-6: 找回旧技能要求与限制

**描述**:从旧技能中找回并补充以下要求与限制:

**子需求 FR-6.1 — 过程文档自适应判定**:
- 在 CODING 阶段增加"过程文档自适应判定"机制
- 判定准则:编译/运行/测试/偏差 4 类过程文档是否生成
- 仅当有"不生成"判定时才写 `process-doc-decisions.md`
- 来源:旧 `code-it/references/common.md §9`

**子需求 FR-6.2 — 项目可测性守卫**:
- 在 CODING 阶段增加"项目可测性守卫"(7 项检查)
- 检查:package.json/pyproject.toml/Cargo.toml/go.mod/pom.xml/build.gradle[.kts]/test/ 目录
- 至少 1 个模块命中 → 需写单测;否则 → 跳过单测
- 来源:旧 `code-it/references/common.md §6`

**子需求 FR-6.3 — 按需写单测(3 类自动判定)**:
- 文档类型 → 不写单测,写占位
- 函数级代码类 → 写单测并跑通
- 配置/类型定义 → 不写单测,写占位
- 来源:旧 `code-it/references/common.md §7`

**子需求 FR-6.4 — 逻辑行统计与阈值**:
- 在 CODING 阶段增加逻辑行统计(检测工具:tokei > cloc > heuristic)
- 在 CHECK 阶段增加"代码行数超标"检查维度
- 阈值:单文件总逻辑行 500、单文件新增逻辑行 200(可从 REQUIRE.md 覆盖)
- 来源:旧 `code-it/references/common.md §8` 和 `code-it/lib/logic-loc-defaults.md`

**子需求 FR-6.5 — 前置任务守卫**:
- 在 CODING 阶段增强前置任务守卫
- 按 PLAN.md 任务总览文件行序判定"当前任务的前置任务是否全部完成"
- 前置任务 = 当前任务之前的所有任务
- 未完成 → 中止 + 打印推荐命令
- 来源:旧 `code-it/references/common.md §1`

**子需求 FR-6.6 — 错误修复循环增强**:
- 当前 coding.md 已有"最多连续失败 5 次",需增强:
- 区分错误类型:代码 bug → 修代码;设计缺陷 → 记入 LOG.md,停下询问;环境/依赖 → 尝试解决,记入 LOG.md
- 来源:旧 `code-it/references/common.md §13`

**子需求 FR-6.7 — 标题解析**:
- 在 REQUIRE/DESIGN/PLAN/CODING/CHECK 各阶段增加标题解析
- 从产出文档中提取标题,用于屏幕输出(启动/完成/中止/错误)
- 格式:`REQ-NNNNN · <标题>`
- 来源:旧 `code-require` 的"标题解析"章节

**子需求 FR-6.8 — 追踪编号禁用规则补全**:
- 当前 coding.md 已有追踪编号禁用规则,需补全:
- 增加 commit message 中允许使用追踪编号的说明
- 增加 `./assistants/` 工作产物中允许使用追踪编号的说明
- 来源:旧 `code-it/references/common.md §5`

**子需求 FR-6.9 — 审查改修任务特殊规则**:
- 在 CODING 阶段增加"审查改修任务特殊规则"
- 按 CHECK.md 的发现清单展开改修
- 严格遵守"不需要做的"部分,不越界
- 发现其他问题 → 记入 LOG.md,不擅自修复
- 来源:旧 `code-it/references/common.md §5`

## 4. 非功能需求(NFR)

| 编号 | 类型 | 描述 | 约束 |
| --- | --- | --- | --- |
| NFR-1 | 兼容性 | 新能力不破坏现有 `code-req`/`code-fix` 的 SKILL.md 结构 | 仅追加,不修改现有章节锚点 |
| NFR-2 | 兼容性 | 新语言文件不影响未使用对应语言的项目 | 仅在检测到对应语言时加载 |
| NFR-3 | 兼容性 | 评审-编码循环与现有 PROCESS.md 断点续跑兼容 | 循环内每轮追加 PROCESS.md 记录 |
| NFR-4 | 可维护性 | 语言适配文件遵循统一章节结构 | 7 个章节,每个文件独立自包含 |
| NFR-5 | 可维护性 | 用户确认环节与 `--auto` 模式兼容 | `--auto` 模式下自动选推荐项 |
| NFR-6 | 复用性 | 新增 references 可被 `code-fix` 复用 | 存放在 `code-req/references/` 下 |
