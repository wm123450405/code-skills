# 编码格式规范(encoding-conventions)

> 本规范文件由 `code-rule` 技能维护,所有 `code-*` 技能在执行时会读取本文件作为强制约束。
> 最后更新:2026-06-04 10:05
> 适用版本:跨所有版本共享(项目级)

## 适用场景

本规范定义了 `code-skills` 仓库内**所有**编码格式(需求 / 任务 / 缺陷)的**权威定义**。当:

- AI 协作者准备调用 `code-require` / `code-design` / `code-plan` / `code-it` 产出新编码时
- 用户调 `code-fix` 登记新缺陷时
- 任何技能在示例、文档、模板中引用编码格式时
- 团队讨论"为什么这个需求叫 `REQ-00001` 而不是 `REQ-2026-0001`"时

请查阅本文件作为**唯一权威源**,不要参考各 SKILL.md 或模板中的硬编码示例(那些可能因历史原因未及时更新)。

## 强制级别约定

本文件中各规则的强制级别逐条标注(参见各"规则 N"小节的"强制级别"字段),本文件头部不统一声明。

## 规则 1:三类编码格式定义(权威源)

### 条款

仓库内所有"项目级"编码严格遵循以下三类格式:

| 维度 | 格式 | 正则 | 容量上限 | 备注 |
| --- | --- | --- | --- | --- |
| **需求** | `REQ-<后缀>` | `^REQ-[A-Za-z0-9.\-_]+$` | 按后缀长度(无硬上限,仅受 OS 文件系统 255 字符约束) | 默认 5 位纯数字;接收可放宽,前缀固定 `REQ-` |
| **缺陷** | `BUG-<后缀>` | `^BUG-[A-Za-z0-9.\-_]+$` | 按后缀长度(无硬上限,仅受 OS 文件系统 255 字符约束) | 默认 5 位纯数字;接收可放宽,前缀固定 `BUG-` |
| **任务** | `TASK-(REQ\|BUG)-<父级>-<任务序号>` | `^TASK-(REQ\|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$` | 父级内按后缀长度(无硬上限) | 嵌套式;生成端默认两段均为 5 位纯数字,接收端放宽为 `[A-Za-z0-9.\-_]+` 字符集 |

### 强制级别
- 必须

### 适用范围
- 所有 `code-*` 技能产出的新编码(自本规范生效起)
- 所有 `code-rule` 添加的新规范(自本规范生效起)
- **不**追溯:已有的 `EXISTING-NNN` 形式(V0.0.0 基线特例)保持原样

### 正面示例

```text
REQ-00001              # 第一个需求
REQ-00050              # 第 50 个需求
BUG-00001              # 第一个缺陷
BUG-00005              # 第 5 个缺陷
TASK-REQ-00001-00001   # REQ-00001 的第 1 个任务
TASK-REQ-00001-00005   # REQ-00001 的第 5 个任务
TASK-BUG-00001-00001   # BUG-00001 的第 1 个修复任务
```

### 反面示例

```text
REQ-2026-0001          # ❌ 含年份段(已淘汰)
REQ-2025-0099          # ❌ 含年份段(已淘汰)
BUG-001                # ❌ 仅 3 位(已淘汰,容量不足)
TASK-REQ-00001-001     # ❌ 任务序号 3 位(已淘汰)
TASK-00001-00001       # ❌ 缺父级前缀(REQ/BUG)
REQ-1                  # ❌ 数字段不足 5 位
```

### 例外
- `EXISTING-NNN` 形式(V0.0.0 基线产物)继续使用,不追溯重命名
- 历史 V0.0.1 `REQ-2026-0001` 形式已在 REQ-00002 中追溯重命名为 `REQ-00001`(2026-06-03 20:20 部分落地 + REQ-00002 code-it 阶段完成)

### 关联规范
- `./assistants/rules/migration-mapping.md` §规则 1(新旧编码追溯表)
- 实施流程:见 §规则 4

## 规则 1.5:第三方平台接入指南

> 本小节为新接入(2026-06-08 起,REQ-00025 引入)规则,定义"如何把第三方平台(JIRA / Azure DevOps / GitHub Issues / 飞书项目 / 钉钉项目 / 自研系统 等)的编号格式接入本仓库的 REQ/BUG 体系"。

### 条款

1. **用户登记新前缀**:用户调 `code-rule` 登记新前缀(例如 `JIRA- → 等价 REQ-`、`ADOPROJ- → 等价 BUG-`)。
   - 登记位置:本节"第三方平台前缀登记表"。
   - 登记内容:平台名 / 前缀 / 等价类别(`REQ` / `BUG` / `TASK`)/ 登记时间 / 登记人 / 备注。
2. **AI 自动识别**:登记完成后,`code-require JIRA-123` 自动识别为"需求"(以 `JIRA-` 为前缀,等价于 `REQ-` 路径);`code-fix ADOPROJ-456` 自动识别为"缺陷"(等价于 `BUG-` 路径)。
3. **路径映射**:所有 `code-*` 技能在解析用户输入时,先按 §规则 1 的默认正则(`^REQ-\d{5}$` / `^BUG-\d{5}$` / `^TASK-(REQ|BUG)-\d{5}-\d{5}$`);未命中时回退到本表查询等价类别,再用等价类别的"宽松正则"(`^REQ-[A-Za-z0-9.\-_]+$` / 同 BUG / 同 TASK)兜底匹配。
4. **生成端 vs 接收端**:
   - **生成端**(本仓库主动产出的新编号):仍沿用 5 位纯数字(§规则 2 不变)。
   - **接收端**(解析用户输入):按本表 + §规则 1 宽松正则匹配,**不**强制 5 位。
5. **登记表初始为空**:本节表格初次登记为"无"。

### 强制级别
- 必须(若用户使用第三方平台编号,必须先登记)

### 适用范围
- 所有 `code-*` 技能解析用户输入的"需求 / 缺陷 / 任务"编号时
- 所有 `code-rule` 添加的新第三方平台前缀

### 第三方平台前缀登记表

| 平台名 | 前缀 | 等价类别 | 登记时间 | 登记人 | 备注 |
| --- | --- | --- | --- | --- | --- |
| (无) | — | — | — | — | 第三方平台接入按需登记;首次登记需调 `code-rule` |

### 正面示例
- 用户调 `code-rule "登记 JIRA- 前缀,等价 REQ-"` → 表格新增 1 行(`JIRA-` / `REQ` / 2026-06-XX / wangmiao)
- 用户调 `code-require JIRA-V0.0.1.001` → 命中登记表 → 等价 `REQ-` 路径 → 进入 `require/JIRA-V0.0.1.001/RESULT.md`
- 用户调 `code-require REQ-00001` → 未命中登记表(默认前缀)→ 直接走 `require/REQ-00001/RESULT.md`

### 反面示例
- 用户调 `code-require JIRA-123` 但**未**调 `code-rule` 登记 → AI 应提示"请先调 `code-rule` 登记 `JIRA-` 前缀"(沿用既有"格式不匹配"错误信息 + 引导登记)
- 用户**不**经 `code-rule` 私自修改本表 → 违反"登记表是 `code-rule` 唯一职责"硬约束

### 例外
- 无(本小节自 V0.0.3 REQ-00025 实施日起生效)

### 关联规范
- §规则 1(编码格式定义 / 接收端宽松正则)
- §规则 2(生成端 5 位纯数字)
- `code-rule` 技能(本表唯一写入方)

## 规则 2:5 位纯数字格式约束

### 条款

1. **生成端 5 位纯数字(默认);接收端任意 1+ 位后缀,字符集 `A-Za-z0-9.\-_`**:本仓库主动产出新编号时仍走 5 位纯数字 + 前导补零(默认),如 `1` → `00001`、`50` → `00050`;**接收端**(解析用户输入)按 §规则 1 宽松正则兜底匹配,后缀长度不强制 5 位,字符集为 `A-Za-z0-9.\-_`(可含字母 / 数字 / `.` / `-` / `_`)。
2. **纯数字**:数字段仅含 `0-9`,**不**含年份、字母、连接符
3. **父级内递增**:需求/缺陷 ID 在全仓库内严格递增(每次取最大+1);任务 ID 在父级(REQ/BUG)内独立递增
4. **不重用**:已废弃的编码**不**重新分配给新对象(避免历史引用断裂)

### 强制级别
- 必须

### 适用范围
- 规则 1 中定义的 3 类编码

### 正面示例
- `REQ-00001`(前导补零 ✅)
- `BUG-00010`(5 位纯数字 ✅)
- `TASK-REQ-00001-00005`(父级内独立递增 ✅)

### 反面示例
- `REQ-1`(数字段不足 ✅)
- `REQ-01`(前导补零不足 ✅)
- `REQ-A0001`(含字母 ✅)

### 例外
- 无

### 关联规范
- 规则 1(编码格式定义)

## 规则 3:嵌套式 TASK 编码规则

### 条款

任务 ID(`TASK-...`)采用**嵌套式**结构,体现任务与父级(REQ/BUG)的关系:

1. **父级前缀**:`TASK-REQ-` 表示需求任务,`TASK-BUG-` 表示缺陷修复任务
2. **父级数字段**:紧接着父级前缀的 5 位数字,直接是父级 ID 的数字段(无 `REQ-` 前缀)
3. **任务序号**:末尾 5 位数字,父级内独立递增(从 00001 起)
4. **解析规则**(AI 在 `code-it` / `code-plan` 步骤 1 解析任务编码时):
   - 匹配 `^TASK-REQ-(\d{5})-(\d{5})$` → 需求任务,父级 = `REQ-<第 1 组>`,任务序号 = `<第 2 组>`
   - 匹配 `^TASK-BUG-(\d{5})-(\d{5})$` → 缺陷任务,父级 = `BUG-<第 1 组>`,任务序号 = `<第 2 组>`

### 强制级别
- 必须

### 适用范围
- 所有 `code-plan` 拆分的任务
- 所有 `code-fix` 派生的修复任务

### 正面示例

```text
TASK-REQ-00001-00001  # ✅ 需求任务,父级 REQ-00001,序号 00001
TASK-BUG-00001-00001  # ✅ 缺陷任务,父级 BUG-00001,序号 00001
TASK-REQ-00050-00010  # ✅ 需求任务,父级 REQ-00050,序号 00010
```

### 反面示例

```text
TASK-00001-00001      # ❌ 缺父级前缀
TASK-REQ-1-1          # ❌ 数字段不足 5 位
TASK-REQ-00001        # ❌ 缺任务序号
```

### 例外
- 无(本规则自 V0.0.1 REQ-00002 实施日起生效)

### 关联规范
- 规则 1(编码格式定义)
- 规则 2(5 位纯数字约束)

## 规则 4:实施流程

### 条款

新编码的产出流程:

1. **由 `code-plan` 生成任务 ID**:
   - 用户告知父级(REQ-NNNNN 或 BUG-NNNNN)
   - 扫描该父级下 `plan/<父级>/PLAN.md` 与 `code/TASK-(REQ|BUG)-<父级>-*/` 已有 TASK 编码
   - 取该父级内最大 TASK 编号 + 1
   - 拼装新编码 `TASK-REQ-<父级>-NNNNN` 或 `TASK-BUG-<父级>-NNNNN`,在 PLAN.md 追加任务行

2. **由 `code-fix` 生成缺陷 ID**:
   - 扫描 `fix/RESULT.md`(若存在)→ 解析所有 `BUG-NNNNN`,取最大 N
   - 新编号 = `BUG-{N+1 零填充到 5 位}`
   - 若 `fix/RESULT.md` 不存在 → 编号 = `BUG-00001`

3. **由 `code-require` / `code-version` 创版本时同步生成需求 ID**:
   - 取仓库内最大 REQ 编号 + 1
   - 5 位零填充

4. **解析入口**:
   - `code-it` / `code-plan` / `code-unit` / `code-check` 在步骤 1 解析任务编码时,**优先**使用本规范 §规则 1 定义的"接收端宽松正则"(`^TASK-(REQ|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$`,后缀任意长度,字符集 `A-Za-z0-9.\-_`);生成端仍默认 5 位纯数字(向后兼容)
   - 解析结果用于确定工作目录 + 上游输入源

### 强制级别
- 必须

### 适用范围
- 所有 `code-*` 技能

### 正面示例
- `code-plan REQ-00001` → 扫描 `plan/REQ-00001/PLAN.md` + 已有 TASK → 生成下一个 TASK 编码
- `code-fix "用户报告:..."` → 扫描 `fix/RESULT.md` → 生成 BUG-00001 或下一个
- `code-require "添加功能 X"` → 取 max REQ-NNNNN + 1

### 反面示例
- 在 `code-plan` 内部硬编码 `001` 起始序号(应扫描父级)
- 在 `code-fix` 内部硬编码 `BUG-001` 起始(应扫描 `fix/RESULT.md`)

### 例外
- 无

### 关联规范
- 规则 1 / 规则 2 / 规则 3
