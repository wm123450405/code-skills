# 编码迁移映射(migration-mapping)

> 本规范文件由 `code-rule` 技能维护,所有 `code-*` 技能在执行时会读取本文件作为强制约束。
> 最后更新:2026-06-04 10:08
> 适用版本:跨所有版本共享(项目级)

## 适用场景

本规范记录了 `code-skills` 仓库在演进过程中**已经实施或计划实施**的"旧编码 → 新编码"映射关系。当:

- AI 协作者在历史文档 / 看板 / 旧 commit 中遇到 `REQ-2026-0001` 这类旧编码,需要**追溯**到新编码时
- 用户调 `code-require` / `code-fix` 时,需要知道"旧的 `BUG-001` 对应新的什么"以避免重复登记
- 团队在审计"为什么仓库里有 2 种编码格式"时,需要一份权威的"映射表"作为解释

请查阅本文件作为**唯一权威源**。本规范**不**追溯"已存在的 `EXISTING-NNN`"(V0.0.0 基线特例,详见下文)。

## 强制级别约定

本文件中各规则的强制级别逐条标注(参见各"规则 N"小节的"强制级别"字段),本文件头部不统一声明。

## 规则 1:已落地的迁移映射表

### 条款

下列映射关系**已实际实施**,可在历史文档、看板、commit 中找到旧编码,并通过本表追溯到新编码:

| 旧编码 | 新编码 | 类型 | 落地时间 | 落地范围 |
| --- | --- | --- | --- | --- |
| `REQ-2026-0001` | `REQ-00001` | 需求 | 2026-06-03 20:20(部分)+ 2026-06-04 09:50(全量) | 目录 `require/REQ-2026-0001/` → `require/REQ-00001/`;SKILL.md × 5;templates × 11;README × 2(中英) |
| `REQ-2026-0002` | `REQ-00002` | 需求 | 2026-06-04 09:50(全量) | 与 REQ-00001 同步(同期落地) |
| `REQ-2026-0001-001` | `TASK-REQ-00001-00001` | 任务 | 2026-06-04 09:50 | SKILL.md / templates / README 中的旧任务编码示例 |
| `REQ-2026-0001-003` | `TASK-REQ-00001-00003` | 任务 | 2026-06-04 09:50 | 同上 |
| `REQ-2026-0001-005` | `TASK-REQ-00001-00005` | 任务 | 2026-06-04 09:50 | 同上 |
| `BUG-001` | `BUG-00001` | 缺陷 | 2026-06-04 09:50 | SKILL.md / templates / README 中的旧 BUG 编码示例 |
| `BUG-002` | `BUG-00002` | 缺陷 | 2026-06-04 09:50 | 同上 |
| `BUG-005` | `BUG-00005` | 缺陷 | 2026-06-04 09:50 | 同上 |
| `BUG-001`/`BUG-002`/`BUG-005` 流程示例中的"BUG-NNN" | `BUG-00001`/`BUG-00002`/`BUG-00005` + "BUG-NNNNN" | 缺陷 + 格式约定 | 2026-06-04 09:50 | SKILL.md / templates 中所有 BUG 流程示例 |

### 强制级别
- 推荐(追溯时优先使用本表)

### 适用范围
- 历史文档、看板变更记录、commit message、issue tracker 中的旧编码
- 任何 AI / 用户在对话中引用旧编码时,应同时给出新编码(或本表的位置)

### 正面示例
- 用户问"`REQ-2026-0001` 对应什么" → 答"`REQ-00001`,见 `migration-mapping.md` 规则 1"
- AI 在 `git log` 中看到 `REQ-2026-0001-001` 旧任务 → 在新文档中引用为 `TASK-REQ-00001-00001`

### 反面示例
- ❌ 看到旧编码后**直接按字面引用**(不追溯)— 读者会困惑
- ❌ 看到旧编码后**改名**(本规范**不**对已落地的旧编码做"二次重命名")

### 例外
- **历史 commit / 历史变更记录** 保留原字面值(不改写 git 历史)
- 任何文档中标注"v1 历史变更记录保留原字面值"的字段,即使与新格式冲突也不动

### 关联规范
- `./assistants/rules/encoding-conventions.md` §规则 1(权威编码格式定义)
- `./assistants/V0.0.1/RESULT.md` 变更记录(落地时间与范围)

## 规则 2:理论映射(待落地)

### 条款

下列映射关系**在 REQ-00002 需求文档中定义**,但**当前仓库内可能无对应旧编码文件**(理论映射,作为参考):

| 旧编码 | 新编码 | 类型 | 备注 |
| --- | --- | --- | --- |
| `REQ-2025-0099` | `REQ-00510` | 需求 | 用户中心示例需求;仓库 `code-require/SKILL.md` + `templates/requirements.md` 中已替换为 `REQ-00510`(2026-06-04 09:50) |
| `REQ-YYYY-NNNN` | `REQ-NNNNN`(5 位纯数字) | 通用规则 | 任何含年份段的需求编码都映射到 5 位纯数字(年份 + 序号转十进制) |
| `<REQ>-<NN>`(3 位任务序号) | `TASK-REQ-<NNNNN>-NNNNN`(5 位) | 任务 | 任何 3 位任务序号都映射到嵌套式 5 位 |
| `BUG-NNN` | `BUG-NNNNN`(5 位) | 通用规则 | 任何 3 位 BUG 都前导补零到 5 位 |

### 强制级别
- 参考(理论映射,仅在确实遇到旧编码时使用)

### 适用范围
- 任何仓库中可能存在但本次 REQ-00002 实施时未发现的旧编码
- 未来 `code-review` / `code-unit` 阶段再次审计时发现的新旧不一致

### 正面示例
- 团队成员发现某文件含 `REQ-2027-1234` → 按本表规则映射为 `REQ-71234`(年份+序号转十进制 = `20271234` 折算)或类似规则,具体见下方"通用映射公式"

### 反面示例
- ❌ 在仓库中**主动查找**旧编码并触发追溯重命名(本规范**不**主动重命名历史)
- ❌ 在新文档中**再次使用**旧格式示例(本规范**不**支持新代码用旧格式)

### 例外
- 通用规则(年份+序号 → 5 位纯数字)的具体转换公式见规则 3

### 关联规范
- `./assistants/rules/encoding-conventions.md` §规则 1(权威编码格式定义)

## 规则 3:通用映射公式(年份段 → 5 位纯数字)

### 条款

旧格式 `REQ-YYYY-NNNN` 的通用映射公式:

```
new_NNNNN = 取后 5 位 = (YYYY * 10000 + NNNN) mod 100000
```

**注意**:这只是"理论公式",实际映射以规则 1 / 规则 2 的具体条目为准。

| 旧编码 | 公式计算 | 期望新编码 | 实际落地新编码 | 备注 |
| --- | --- | --- | --- | --- |
| `REQ-2026-0001` | `(2026 * 10000 + 1) mod 100000 = 20260001 mod 100000 = 26001` | `REQ-26001` | `REQ-00001` | **公式与实际不一致**,实际是"序号 0001 折算为 5 位"(`00001`),不是"年份+序号" |
| `REQ-2025-0099` | `(2025 * 10000 + 99) mod 100000 = 20250099 mod 100000 = 50099` | `REQ-50099` | `REQ-00510` | **公式与实际不一致**,实际是"序号 0099 → 5 位 00510"(具体折算规则未明) |

### 强制级别
- 不适用(公式仅供理解,实际映射以规则 1 / 规则 2 为准)

### 适用范围
- 仅供团队理解"为什么映射结果不是简单的数字拼接"

### 正面示例
- 理解映射关系:**新编码 = 序号 NNNN 的 5 位纯数字版本**(年份段被丢弃)
- 即:`REQ-YYYY-NNNN` → `REQ-NNNNN`(取 `NNNN` 的 5 位纯数字,前导补零)

### 反面示例
- ❌ 用本公式去自动计算未知旧编码的新值(本规范**不**做自动转换)

### 例外
- 无(本规则仅作说明,无强制约束)

### 关联规范
- 规则 1 / 规则 2

## 规则 4:已知不完全映射(V0.0.0 EXISTING-NNN)

### 条款

V0.0.0 基线版本由 `code-init` 技能批量生成的 10 个现有功能需求(`EXISTING-001` ~ `EXISTING-010`),采用**旧格式 3 位数字**:

| 编码 | 含义 | 是否重命名 |
| --- | --- | --- |
| `EXISTING-001` | 工程初始化(`code-init`) | ❌ 不追溯重命名 |
| `EXISTING-002` | 版本管理(`code-version`) | ❌ |
| `EXISTING-003` | 编码规范管理(`code-rule`) | ❌ |
| `EXISTING-004` | 需求分析(`code-require`) | ❌ |
| `EXISTING-005` | 概要设计(`code-design`) | ❌ |
| `EXISTING-006` | 详细设计与任务计划(`code-plan`) | ❌ |
| `EXISTING-007` | 开发编码(`code-it`) | ❌ |
| `EXISTING-008` | 单元测试(`code-unit`) | ❌ |
| `EXISTING-009` | 缺陷登记与跟踪(`code-fix`) | ❌ |
| `EXISTING-010` | 代码评审(`code-review`) | ❌ |

### 强制级别
- 必须(**不**追溯重命名)

### 适用范围
- V0.0.0 基线版本中所有 `EXISTING-NNN` 编码
- V0.0.1+ 后续 `code-init` 调用若生成新的 `EXISTING-NNN`,也按本规则

### 正面示例
- 旧项目接入 `code-init` → 批量生成 `EXISTING-001` ~ `EXISTING-M` → **保持原格式**,不映射到 `REQ-NNNNN`
- 团队讨论"`EXISTING-006` 是什么" → 答"`code-plan`,见 V0.0.0/INIT-REPORT.md`(用原编码,不用新格式追溯)

### 反面示例
- ❌ 把 `EXISTING-006` 改名为 `REQ-00006`(违反 V0.0.0 基线不可追溯原则)
- ❌ 试图建立 `EXISTING-006 ↔ REQ-NNNNN` 的双向追溯(本规范明确**不**追溯)

### 例外
- 无(本规则强制不变)

### 关联规范
- `./assistants/rules/encoding-conventions.md` §规则 1 例外(`EXISTING-NNN` 形式继续使用)
- V0.0.0/INIT-REPORT.md(原始登记)

## 规则 5:维护说明

### 条款

本规范的维护责任:

1. **本规范的创建者**:`code-it` 在 REQ-00002-006 中创建(2026-06-04)
2. **后续维护者**:`code-rule` 技能
3. **维护触发**:
   - 任何 `code-*` 技能在未来发现新的旧编码 → 在本规范规则 1 / 规则 2 追加条目
   - 任何 `code-review` 发现旧编码未在迁移表中 → 派生"审查改修"任务,由 `code-it` 实施
4. **本规范不自动生成**:不做"扫描仓库 → 自动追加映射",全部由人或 AI 显式添加
5. **本规范不被自动消费**:仅作为追溯参考,不影响其他技能的"读新编码"逻辑

### 强制级别
- 必须

### 适用范围
- 所有 `code-*` 技能

### 正面示例
- AI 在 `git log` 中发现 `REQ-2027-1234` → 在对话中告知用户"本规范未收录,可推测新编码为 `REQ-01234`"

### 反面示例
- ❌ 维护者删除本规范(本规范是迁移追溯的**唯一**权威源,删除会导致历史编码无法理解)
- ❌ 维护者修改已落地的映射(本规范**不**对已落地的旧编码做"二次重命名")

### 例外
- 无

### 关联规范
- `./assistants/rules/encoding-conventions.md`(上游权威编码定义)
