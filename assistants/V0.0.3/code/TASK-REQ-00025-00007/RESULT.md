## 3. 详细改动

### 3.1 §输入(修改)

- **需求编码**(追加 2 子项):
  - **默认**格式:5 位纯数字 `^REQ-\d{5}$`
  - **接收**可放宽:`^REQ-[A-Za-z0-9.\-_]+$`(沿用 `encoding-conventions §规则 1` 接收端宽松正则,新规则 REQ-00025)
- **任务编码**(派生任务评审时,**新增**项,2 子项):
  - **生成端**沿用 5 位纯数字格式:`^TASK-(REQ|BUG)-\d{5}-\d{5}$`(默认)
  - **接收端**放宽为:`^TASK-(REQ|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$`(后缀自由)

---

## 4. 关键决策与权衡

### 4.1 为什么新增"任务编码"项?

- `code-check` 既评审**整体需求**(`review/<需求编号>/REVIEW-REPORT.md`),也派生**审查改修任务**(`review/<任务编码>/RESULT.md`)
- 既有 §输入**只**列"需求编码"+"上游..."等,未明示"任务编码"格式
- 显式列"任务编码"项让用户在派生任务评审时识别"任务编码也按新规则放宽"

### 4.2 为什么不修改 §工作流程 各步骤?

- `code-check` 工作流程是"读 PLAN.md → 列任务 → 读 code/<TASK>/RESULT.md → 评审 → 派生",**不**涉及具体正则硬编码
- §输入 显式声明"接收端放宽"后,所有"读 <任务编码> 路径"操作**自动**沿用新规则
- 0 触及 30+ 步骤既有逻辑
