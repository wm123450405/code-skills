## 4. 关键决策与权衡

### 4.1 为什么 §工作目录约定 显式声明"需求编号路径沿用新规则"?

- `code-design` 工作目录粒度是"需求",路径模板 `design/<需求编号>/` 直接拼接用户输入
- 若**不**显式声明"沿用新规则",AI 在解析 `code-design JIRA-V0.0.1.001` 时可能误判"目录路径格式不匹配"
- 显式声明后,`code-design` 与 `code-require`(已由 T-2 同步)行为一致,统一接收端语义

### 4.2 为什么不修改 `design/<需求编号>/` 的目录结构图?

- 原目录结构图(L28-50)用 `<需求编码>` 占位符,**不**限定具体格式
- 路径模板的"放宽"在文字层(本任务追加的 1 项)声明即可,**不**需要重画图
- 保持图示与 T-2 / T-9 的"声明式说明"风格一致

### 4.3 为什么不修改 §输入 的"上游需求"项?

- "上游需求"是**目录路径引用**,其格式约束已在 §工作目录约定中显式声明
- `code-design` 步骤 0(读 `.current-version`)/ 步骤 1(读上游需求)/ 步骤 2(读 `rules/`) 等核心逻辑**未**触及
