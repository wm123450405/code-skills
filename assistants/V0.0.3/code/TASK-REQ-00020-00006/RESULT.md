## 4. 关键决策与权衡

### 4.1 为什么"PLAN.md 存在,RESULT.md 不存在"可合并?

- 该分支的语义:"计划已就绪,补做详细设计(不常见;步骤 7D)"
- 与"都不存在"分支语义重叠:**两者都从零开始构建**,只是其中一个先有 PLAN.md 骨架
- 合并后统一走"步骤 7A 首次设计",LLM 看到 PLAN.md 骨架后,自然会采用"补做详细设计"路径
- 节省 1 个独立分支 + 简化逻辑(4 → 3)

### 4.2 为什么 E-1 / E-4 / E-8 / E-9 合并为表格?

- 4 个 E 边界都是"职责不归属本守卫"—— 都由其他步骤处理或由其他流程分支处理
- 表格化后,读者一眼看到"哪个边界由谁处理",避免重复 4 次相似描述
- 节省 8 行 → 5 行(净减 3 行),职责归属更清晰
- 体现"封装性"维度(本需求 7 维度之一,优先级=高)—— 抽象出"职责归属"概念

### 4.3 为什么保留 E-2 / E-3 / E-5 / E-6 / E-7 / E-10 独立小节?

- 这 6 个 E 边界都涉及守卫的"具体决策逻辑",**不**是"职责归属"
- 例如 E-2(守卫不通过)需要打印详细报告;E-3(PLAN.md 不存在)需要走退化路径
- 表格化会丢失关键决策细节,故保留独立小节
- M-4 仅合并 4 个职责归属型边界,**不**影响 6 个决策型边界
