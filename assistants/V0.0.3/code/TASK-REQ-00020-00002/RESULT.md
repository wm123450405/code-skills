## 4. 关键决策与权衡

### 4.1 为什么大需求 8 问而非 7 问?

- 用户原文"在 code-plan 阶段要问全所有维度"= 整体(1)+ 7 维度(7)= 8 问
- 中需求 4 问:整体 + 沿用 3 维度(功能性 / 扩展性 / 健壮性);新增 3 维度按需问
- 小需求 1 问:只问整体设计目标;7 维度默认按"中"或"--balanced"推断

### 4.2 为什么维度选项扩展为"高/中/低/不适用"?

- 沿用 REQ-00011 的"高/中/低"3 选项
- 本需求新增 3 维度对"自然语言项目"(如本仓库 Markdown)无意义 → 加"不适用"选项
- 屏显模板按"不适用"显示,任务粒度调整规则按"不适用"跳过

### 4.3 为什么不在 design 阶段问全 7 维度?(职责分离)

- `code-design` 是概设阶段,过早问"未来怎么扩"会让 AI 倾向过度设计
- 用户原文"不在概设阶段考虑架构设计目标"= design 阶段只问 1 维度(功能性)
- `code-plan` 是详设阶段,详细到具体编码时必须考虑全 7 维度
- 故本任务扩展 `code-plan` 而非 `code-design`(后者由任务 T-1 简化)
