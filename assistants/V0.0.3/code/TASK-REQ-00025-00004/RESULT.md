## 4. 关键决策与权衡

### 4.1 为什么 §输入 显式列"缺陷编码"项?

- 既有 §输入**只**列"需求编码",缺陷路径隐含在 `<需求编码>` 中
- 但 `code-plan` 既接受 REQ 路径也接受 BUG 路径(双路径设计,沿用 `code-fix` + `code-plan` 协同)
- 显式列"缺陷编码"项让用户/AI 都能识别"本技能双路径支持",避免在 BUG 路径调用时被误判

### 4.2 为什么 §步骤 9B 任务编号分配 仍用 5 位纯数字?

- 生成端是 `code-plan` 的**主动产出**,仍沿用"项目内自增 + 5 位纯数字"契约(沿用 REQ-00002)
- 接收端放宽是**消费**用户输入的兼容,不影响 `code-plan` 自身产生的任务编号
- 用户携带的第三方平台编号(如 `TASK-JIRA-V0.0.1.001-00001`)由 `code-plan` 接收,但新生成的"任务序号"段仍按 5 位纯数字(`00001` 起递增)

### 4.3 为什么不修改 §步骤 10A 末尾的"任务粒度调整规则"段?

- 任务粒度调整规则是**纯项目级**内容,沿用 REQ-00020 锁定
- 本任务**只**改"任务编号"段(7 维度不变 / 任务类型不变 / 关联任务不变 / 触发/来源不变 / 双状态不变)
