## 4. 关键决策与权衡

### 4.1 为什么用 `> 引用:` 块而非简单折叠?

- `> 引用:` 块在 Markdown 中**视觉独立**,读者一眼可识别"这是引用 / 说明 / 责任归属"
- 与原"## 工作流程"小节的步骤描述清晰区分
- 引用风格与 `code-design/SKILL.md` §步骤 0b.0 完全对齐(本任务做的是"原版 → 引用版")

### 4.2 为什么保留"公共段定义"4 子段而非纯引用?

- 维护者读 `code-plan` 步骤 0b.0 时**需要**知道本步骤对应的 4 个具体决策(检测到 / 未检测到 / 检测失败 / 约束)
- 纯引用会让 `code-plan` 读起来"断章",需跳到 `code-design` 才能继续阅读
- 折中:保留 4 子段摘要(检测机制一行 + 24h 一行 + 决策三选一 + 约束一行)+ 末尾"详见原版"提示

### 4.3 为什么"原 18 行重复文字删除"是 M-1 关键收益?

- 原 18 行中,12 行是**与 code-design 完全相同**的字面重复(检测机制 / 24h / 决策 / 约束)
- 删除后,18 行 → 12 行(净减 6 行,**减少 33%**)
- 长期维护:若 BUG-00001 修复方案 A3 的细节调整,只改 `code-design` 即可,`code-plan` 自动跟随
- 体现"封装性"维度(本需求 7 维度之一,优先级=高)
