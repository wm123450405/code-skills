## 4. 关键决策与权衡

### 4.1 为什么"封装性/可复用性/可读性=高"各只加 1 个任务?

- 沿用 REQ-00011 的"扩展性=高 → 至少 1 个扩展性任务"基线
- 用户原文"封装性/可复用性/可读性不要无限拆分"= 各加 1 个代表任务即可
- 但"扩展性=高 + 整体=--extensible"叠加 → 至少 3 个扩展性任务(沿用 REQ-00011 的强度规则)
- 三个新维度**不**沿用"至少 3 个"的强度规则,避免过度拆分(用户的 0.5-2 天任务粒度要求)

### 4.2 为什么"可读性=高"需"编程语言非自然语言"条件?

- 本仓库是 Markdown 自然语言,语义即注释
- "可读性=高 + 自然语言项目"无实际意义 → 不触发任务拆分
- 仅当"可读性=高 + 编程语言项目"(如 TypeScript / Python)才加"关键代码注释"任务
- 避免给 Markdown 项目强加无意义的"加注释"任务

### 4.3 为什么表格列结构从 2 列变 4 列?

- 原 2 列只能表达"扩展性优先级 → 任务粒度调整"
- 新 4 列(整体设计目标 / 维度=高 / 至少任务数 / 任务类型)能清晰表达:
  - 哪个**维度**对应(扩展性 / 封装性 / 可复用性 / 可读性)
  - 至少多少个任务
  - 任务类型是什么
- 改后表格**自解释**,不再需要表外说明
