## 3. 详细改动

### 3.1 §输入(修改)

- **原**:`- **缺陷编号**(\`BUG-NNN\`,N 从 001 起,3 位补零)或**缺陷描述**...`

## 4. 关键决策与权衡

### 4.1 为什么 §步骤 1.2 校验正则放宽(原 3 位数字 → 接收端放宽)?

- 原 `^BUG-\d{3}$`(3 位)实际是 §输入 父项 "BUG-NNN,N 从 001 起,3 位补零" 的"快速校验"——这与 §规则 1 锁定 5 位纯数字(`^BUG-\d{5}$`)不一致
- 既有的 `BUG-00001`(5 位纯数字)按原 §步骤 1.2 的 3 位校验**会失败**(5 > 3),这是潜在 bug
- 本任务把"已有编号"分支校验统一到 §规则 1 接收端宽松正则(向后兼容 3 位/5 位/任意 1+ 位)

### 4.2 为什么不修改 §输入 "BUG-NNN,N 从 001 起,3 位补零" 字面?

- §输入 的 "BUG-NNN" 是"展示性"描述(说明命名规则),**不**是"硬性校验"规则
- 既有 BUG-00001 已用 5 位纯数字(沿用 §规则 1 锁定),"3 位补零"是历史叙述,**不**影响实际接收
- 显式字面更新会破坏"展示性"语义;**不**改

### 4.3 为什么不修改 §步骤 1.3 "新建缺陷"分支?

- §步骤 1.3 是"自动生成新编号",沿用 §规则 1 生成端 5 位纯数字(沿用既有 5 位零填充)
- 接收端放宽**不**影响"生成端",本任务**不**触及
