## 4. 关键决策与权衡

### 4.1 为什么"生成端 5 位纯数字(默认) + 接收端任意 1+ 位后缀"?

- **生成端**沿用 5 位纯数字,保留 REQ-00002 既定契约(项目内自增 + 前导补零),不破坏既有 21 个 REQ 编号 + 1 个 BUG 编号 + 45 个 TASK 编号
- **接收端**放宽为 `[A-Za-z0-9.\-_]+`,允许用户携带第三方平台前缀(如 `JIRA-V0.0.1.001`、`AzureDevOps-2025-Q4-001` 等)直接调用 `code-require` / `code-fix` 而无需"先转译为 REQ-NNNNN"
- 字符集 `A-Za-z0-9.\-_` 涵盖:英文字母(平台名缩写)+ 数字(原数字段)+ `.` / `-` / `_`(版本号 / 短横线 / 下划线连接符)

### 4.2 为什么新增 §规则 1.5 而不是改 §规则 1 的"适用范围"?

- §规则 1 是"编码格式定义"的**权威源**,其表格只列 3 类内生编码(REQ/BUG/TASK)
- 第三方平台接入是"**外生**编码如何被本仓库识别"的扩展问题,属于不同的"职责面",独立成节避免污染主规则
- 后续 `code-rule` 维护登记表时,§规则 1.5 是唯一写入点,职责单一(单一职责原则)

### 4.3 为什么"接收端"加超集正则(0 破坏性变更)?

- 既有 5 位纯数字(`REQ-00020` / `TASK-REQ-00020-00001`)**不**被新规则拒收(新正则 `^REQ-[A-Za-z0-9.\-_]+$` 是 `^REQ-\d{5}$` 的超集)
- 用户调 `code-require REQ-00020` / `code-fix BUG-00001` 等既有编号**继续**工作
- 这与 REQ-00025 NFR-1 "0 破坏性变更" 强约束对齐

### 4.4 为什么不引入"接收端编号 vs 生成端编号"的双轨数据结构?

- 仓库内的"接收端编号"仅在 `require/<外部前缀>-<外部后缀>/` 目录命名上有所体现
- `code-require` 在内部把外部编号"原样保留"作为目录名(不归一化为 `REQ-NNNNN`),但所有**逻辑判定**(工作目录路径 / 关联需求)按本表查询等价类别
- 这样既保持"目录名 = 外部 ID 字面值"(可读性),又保持"逻辑判定 = REQ/BUG 等价类别"(一致性)
