## 4. 关键决策与权衡

1. **步骤 8.7 位置选择**:放在步骤 8 末尾(步骤 8.6 之后)而不是步骤 0 之前
 - 备选 1:步骤 0 之前(类似 `code-require` / `code-design` 那样)
 - 选择理由:步骤 0 之前**没有** `taskType` / `changedFiles` / `testable` 信息,无法判定 `decisions`;步骤 8 末尾信息完备,可立即物化
 - 权衡:与其他 6 个技能的"步骤 0 之前判定"(实际是空操作,因为信息不足)不一致,但本任务是**修复** code-it,**信息完备**是关键
2. **`decisions` 字典双轨**:运行时变量(步骤 9/10/11 守卫立即查询)+ 持久化(`process-doc-decisions.md` 留档)
 - 备选 1:仅运行时变量(不持久化)
 - 选择理由:`process-doc-decisions.md` 是评审要查的文档,必须留档;运行时变量是步骤 9/10/11 守卫立即查询的需要
 - 权衡:两者**不矛盾**,步骤 8.7 同时物化运行时 `decisions` + 写 `process-doc-decisions.md`(若需要)
3. **守卫粒度:步骤开头而非动作级**:守卫加在步骤 9/10/11 开头(整步跳过)而不是动作级(检测/执行/记录三段各自守卫)
 - 备选 1:动作级守卫
 - 选择理由:步骤 9/10/11 的 3 个动作(检测/执行/记录)**强耦合** — 没命令可执行就没结果可记录;步骤开头守卫简洁,后续动作不需要重复判定
 - 权衡:字节级保留既有 3 段(检测/执行/记录)逻辑,只在步骤开头加 if-else
