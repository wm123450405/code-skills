## 4. 关键决策与权衡

### 4.1 为什么"接收"与"默认"分离?

- **生成端**(本仓库主动产出,沿用 `code-require` 内部 `max(REQ-NNNNN) + 1` 逻辑):仍 5 位纯数字
- **接收端**(解析用户输入,可能是 `code-auto` 路径感知模式下的第三方平台前缀):放宽为 `[A-Za-z0-9.\-_]+` 字符集
- 两者**不**冲突:生成端产生 5 位,接收端兼容更宽的字符集(超集包含 5 位)

### 4.2 为什么 §输入 写 4 行而非 1 行紧凑?

- `code-require` 技能是用户**第一接触点**,需要明确表达"5 位默认 + 放宽"的语义
- 与既有"需求材料"项的展开风格保持一致(原 L45-51 同样是 1 行父项 + 多个子项的列表)
- 让用户在调用前就能识别"是否需要先调 `code-rule` 登记第三方前缀"

### 4.3 为什么不修改 §工具使用约定的"读激活版本"等核心步骤?

- 本任务**不**触及 `code-require` 步骤 0(读 `.current-version`)/ 步骤 1(判定输入)/ 步骤 2(校验需求存在)/ 步骤 3(定位工作目录) 等核心逻辑
- 仅在**字面**层面把"5 位纯数字"软化为"默认 5 位 + 接收放宽"
- INV-10/INV-16 守卫("code-require 加'不修改 SKILL.md'硬约束"由 BUG-00001 T-1 落地):本任务为**例外**(触发/来源=详细设计,PLAN.md 显式授权)
