# 设计笔记 — REQ-00035

## 1. 关键设计问题清单

### Q-1:判定器放在哪里?

- 选项 A:每个技能 SKILL.md 步骤 0a(版本检测后)
- 选项 B:抽出共享库 / 脚本
- 选项 C:`code-rule` 维护的共享规则

**选定 A**。理由:与"自含"原则一致(沿用 REQ-00034);B 在 markdown-as-skill 范式下不适用;C 违反"code-rule 跨版本共享"约束(NFR-9)。

### Q-2:决策记录文件是否始终生成?

- 选项 A:仅当有"不生成"判定时生成
- 选项 B:始终生成
- 选项 C:始终生成(精简版)

**选定 A**。理由:反向节省 token(若所有过程文档都生成,无判定可记录);不违反 NFR-1。

### Q-3:看板"变更记录"何时不追加?

- 选项 A:严格语义(仅当有变更)
- 选项 B:宽松语义(始终追加)

**选定 A**。沿用"看板是状态总览,不是日志"语义。

### Q-4:8.13 评审维度派生任务级别?

- 选项 A:派生"建议改"
- 选项 B:派生"必须改"
- 选项 C:不派生任务,仅记录

**选定 A**。AI 判定本身有合理自由度;B 阻断; C 失去纠错。

## 4. 设计权衡

### 4.1 收益 vs 复杂度

- 收益:单需求过程文档总量预计减少 30%+
- 复杂度:5 主流程技能各加 ~30 行(步骤 0a + 过程文档收尾)+ 5 模板新增 + 1 编排微调 + 1 dashboard 兼容

**结论**:复杂度可控(线性扩展),收益显著(30%+ token 减少)。

### 4.2 AI 自由度过大风险

- 风险:AI 把所有过程文档都判定"不生成"以节省 token
- 缓解:NFR-1 零空白文件 + 8.13 评审维度派生"建议改"

### 4.3 模板未改可能"对不齐"

- 风险:模板未引入"过程文档决策"字段(本需求**不**改既有模板)
- 缓解:每个技能的过程文档由 SKILL.md 步骤约定,不依赖模板字段(沿用既有模式)
