## 2. 候选方案(关键问题展开)

### Q1 候选展开:改造哪个文件

- **方案 A(选定)**:改 `plugins/code-skills/skills/code-require/SKILL.md`
  - **优点**:屏幕日志"建议"本质是步骤说明的一部分;一处修改,既改 10A 又改 10B(对称)
  - **缺点**:SKILL.md 文档会变长(NFR 估算 +2 段 ≈ +12 行)
- **方案 B**:改 `plugins/code-skills/skills/code-require/templates/requirements.md`
  - **优点**:模板是 RESULT.md 产出源
  - **缺点**:**严重违背 NFR-2 零规范变更**;模板不应该承载"屏幕日志建议"(模板生成的是文档,不是屏幕)
- **方案 C**:新建 helper 脚本
  - **优点**:职责分离
  - **缺点**:**严重违背本仓库无脚本约定**;NFR-5 0 新增依赖 / 0 新增脚本;helper 脚本无法被"主流程末尾追加"调用

### Q2 候选展开:在哪一段追加

- **方案 A(选定)**:步骤 10A / 10B 段内文末追加 1 段"### 下一步建议(本需求 REQ-00032 新增)"
  - **优点**:**最小化变更**;纯文末追加;既有"向用户汇报"段字节级保留
  - **缺点**:SKILL.md 中步骤 10A / 10B 各 +1 段(共 2 段)
- **方案 B**:新增"### 步骤 10A.x — 下一步建议"
  - **优点**:结构清晰
  - **缺点**:**引入新步骤编号**,破坏既有编号体系;下游消费方(如 code-auto 任务循环的步骤 4)可能需更新;**过度工程化**
- **方案 C**:步骤 N(末尾兜底提交)内追加
  - **优点**:与"完成"动作绑定
  - **缺点**:**严重违背职责分离**;步骤 N 是 commit 流程,不是用户汇报流程;用户在"提交前"和"提交后"看到的是不同建议(混乱)

### Q3 候选展开:isTiny 判定位置

- **方案 A**:步骤 5A / 5B 末尾
  - **优点**:判定与"材料/FR/AC 提取"同处,语义连贯
  - **缺点**:**判定结果未在 5A / 5B 内使用**,要传到 10A / 10B 才用(跨步骤变量传递,易腐化)
- **方案 B(选定)**:步骤 10A / 10B 末尾
  - **优点**:**判定与使用同处**;`isTiny` 是 10A / 10B 局部变量(无需跨步骤传递);提示词维护在 1 处
  - **缺点**:**判定位置与"材料提取"分处**,需要 AI 在 10A / 10B 末尾"回头看"提取的 FR/AC(实操可行,AI 在同一次执行内可读 session memory)
- **方案 C**:步骤 9A / 9B 末尾
  - **优点**:与"同步版本看板"同处
  - **缺点**:**判定与使用分处**,且 9A / 9B 末尾还要同步看板,职责过载

## 3. 选定方案与理由(整体)

### 3.1 方案:在 `code-require/SKILL.md` 步骤 10A / 10B 段内文末各追加 1 段"### 下一步建议(本需求 REQ-00032 新增,2026-06-12 起生效)"

- **理由 1**(Q1):屏幕日志建议是步骤说明的一部分,写在 SKILL.md 最自然
- **理由 2**(Q2):文末追加最小化变更,不引入新步骤编号
- **理由 3**(Q3):判定与使用合并,降低跨步骤变量传递风险
- **理由 4**(Q4):沿用既有"→"起首风格,与 `code-dashboard` 引导 N 字符数契约一致
- **理由 5**(Q5):`isTiny` 不持久化,保持"零文档变更"
