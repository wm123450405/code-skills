## 2. 候选方案(已锁定)

### 2.1 Q-1 扩展性三阶段下沉

- **方案 A(本设计采用)**:概设只标 / 详设展开 / 编码拆任务
- 方案 B:概设就展开扩展点位置 + 抽象层 + 接口契约
- 方案 C:全部下沉到 `code-it` 阶段,概设/详设都不涉及
- **选定 A 的理由**:用户澄清记录(REQ-00030/clarifications.md Q-3 + Q-4):概设只做概设、详设必须补做;"过早问怎么扩"会让 AI 倾向过度设计(沿用 REQ-00020 既有职责分离声明)

### 2.2 Q-2/3/4 概设深度(硬切)

- **方案 A(本设计采用)**:模块表 5 列 / 接口 4 项 / 实体 4 项;模板顶部加 `> 本模板只覆盖概要设计...` 一行
- 方案 B:字段名 + 类型要写,完整约束与示例下沉到 plan
- 方案 C:保留现 §7-§11 全部细节,仅在模板顶部加说明
- **选定 A 的理由**:用户澄清 Q-3 选 A;"软降权"治标不治本,SKILL.md 步骤定义 + templates 章节**双管齐下**才能硬切

### 2.3 Q-5 详设补做力度

- **方案 A(本设计采用)**:§4-§10 全展开(伪代码/字段/接口 schema);4 份过程文档由可选改必选
- 方案 B:密度可调 — 概设没写的详设必须补
- 方案 C:只补"概设未覆盖的"关键空白
- **选定 A 的理由**:用户澄清 Q-4 选 A;"弹性"语义会留下 AI 偷懒空间;配合 FR-7 `code-check` 评审清单新增"详设完整性"校验点,形成"必须做 + 必须检"闭环

### 2.4 Q-6 架构骨架触发条件收紧

- **方案 A(本设计采用)**:沿用 REQ-00014 既有 3 条件,新增第 4 / 5 条件:
  - 条件 4:满足 FR-1 扩展性触发条件(新增三方依赖 OR 跨 ≥3 模块 OR 对接多套实现)
  - 条件 5:整体设计目标 = `--extensible`
- 方案 B:仅沿用 REQ-00014 既有 3 条件
- **选定 A 的理由**:与本需求"扩展性三阶段下沉"(FR-8)对齐;若概设不标 `--extensible`,编码不应自动拆架构骨架(避免过度设计)

## 3. 选定方案与理由(总结)

| 设计问题 | 选定方案 | 关键理由 |
| --- | --- | --- |
| Q-1 扩展性三阶段 | A 概设标 / 详设展 / 编码拆 | 概设过早问"未来怎么扩"会倾向过度设计 |
| Q-2 模块表字段 | 5 列(名/路径/状态/职责/依赖) | 双管齐下(SKILL.md + templates)硬切 |

| Q-4 数据结构字段 | 4 项(名/状态/关键字段/关系) | 同上 |
| Q-5 详设补做 | §4-§10 必填 + 4 过程文档必选 | 用户澄清 Q-4 锁定;`code-check` 闭环 |
| Q-6 架构骨架触发 | 5 条件"或" | 与扩展性三阶段对齐 |
| Q-7 code-check 校验 | 3 校验点(完整性/越界/比例) | "必须做 + 必须检"闭环 |
| Q-8 文件边界 | 4 改 + 1 新增;frontmatter L1-3 保留 | 沿用 `skill-conventions §规则 1` + NFR-3 |

## 4. 备选方案被否决的理由

| 备选 | 否决理由 |
| --- | --- |
| 概设就展开扩展点(2.1 B) | 概设"过早问"导致 AI 倾向过度设计(用户原文) |
| 软降权(2.2 B / C) | 治标不治本,SKILL.md 步骤定义没变,AI 仍会按步骤生成详设级别内容 |
