## 1. 设计概述

本需求是**一次性文档级清理**:把 14 个 `code-*` 技能 SKILL.md + 同目录 templates/ 中所有"开发痕迹"(REQ/BUG 编号尾注、退场功能回溯注释、决策记录 Q-N、生效日标记、退场文件名引用等)剥离,只保留技能的实际工作流。

**核心架构思路**:`--minimal` 硬编码清理策略。在 `code-it` 阶段以**确定性**方式落地 6 条清理规则(每条规则 = 1 个正则 + 1 个替换动作 + 1 组"例外保留"边界),不引入规则引擎、不抽象"清理器"接口。所有规则在一个执行循环里跑完,**单次提交**,完事即弃(NFR-1)。

**新增/复用/修改范围**:
- 新增:无(纯文本清理,0 模块、0 接口、0 数据结构、0 三方依赖)
- 复用既有:`code-it` 的"读取文件 → 正则匹配 → 替换/删除 → 写回" 能力(每条任务 = 一个 `code-it` 任务,沿用既有步骤 7-12)
- 修改既有:`plugins/code-skills/skills/<14 个技能>/SKILL.md` + `templates/*.md` 中相应段落(行级)

```
└──────────────────────────────────────────────────────────────────┘
                                  │

        ▼                         ▼                         ▼
  ┌──────────────┐         ┌──────────────┐         ┌──────────────┐
  │ 文件清单阶段 │ ──────▶ │ 逐文件清理   │ ──────▶ │ 验证 + 提交  │
  │ (步骤 1-2)   │         │ (步骤 3-4)   │         │ (步骤 5-6)   │
  └──────────────┘         └──────────────┘         └──────────────┘
        │                         │                         │
        ▼                         ▼                         ▼
   Glob 14 技能              6 条规则循环            AC-1 ~ AC-8
   目录 SKILL.md            (每条 = 1 正则 +          + git status
   + templates/*.md          1 替换动作 +              + git diff
   输出待清理文件表          1 例外保留白名单)          + git commit
```

**数据流**(单文件级别):

```
输入文件 → Read 整文件 → 按规则 1 ~ 6 顺序扫描 →
  ├─ 命中 + 不在例外白名单 → 删除/替换
  ├─ 命中 + 在例外白名单 → 保留
  └─ 未命中 → 保留
→ 写回文件 → diff 校验(若改动后剩大段残缺 → 跳过该文件,记日志)
```

## 6. 功能架构

### 功能域 1:文档级痕迹清理(唯一域)

- **目的**:剥离 SKILL.md / templates/ 中 6 类开发痕迹,保留功能契约
- **涉及模块**:`code-it` 的"读-改-写"能力(已有,直接复用);**不**新增模块
- **协作方式**:`code-it` 单一任务循环(每技能 1 任务),无并发、无依赖

**6 条清理规则**(硬编码,`--minimal` 路线):

| 规则号 | 规则名 | 触发正则(示意) | 替换动作 | 例外保留白名单 |
| --- | --- | --- | --- | --- |
| R-1 | 段尾「本需求 REQ-NNNNN」清理 | `\(\s*本需求\s+(REQ\|BUG)-\d{5}[^)]*\)\s*[,、)]\s*$` | 删除整串括号(若括号内只这一串)或仅删除括号内文 | 「本仓库主动产出 ...」类、跨需求引用(如「沿用本仓库 NFR-X」) |
| R-2 | 「原 code-unit / 原 fix-plan」清理 | `\(\s*(原\|沿用原)\s+(code-unit\|fix-plan\|fix-work-log\|fix-compile-and-run\|fix-test-results)[^)]*\)` | 删除整串;若在段首且无其他内容,删除整段 | 「既有 N 需求 NFR-5 字节级保留」类合规条款 |
| R-3 | 「Q-N 锁定 / Q-PN 锁定」决策记录清理 | `\(\s*Q-?\d*\s*(锁定\|采纳\|隐含答复)[^)]*\)` | 删除整串 | 「(沿用 V0.0.X NFR-Y)」类版本继承引用 |
| R-4 | 「YYYY-MM-DD 起生效」标记清理 | `\(\s*(\d{4}-\d{2}-\d{2})\s*起生效\s*\)` | 删除整串 | 个位数合规性条款(≤ 3 处) |
| R-5 | 退场文件名引用清理 | `fix-plan\.md\|fix-work-log\.md\|fix-compile-and-run\.md\|fix-test-results\.md` | 删除该词(不替换为新名,直接删除);若引用在描述性句子中,整句可保留,只删除文件名 | 目录级引用(`fix/`)保留;`code-fix` 技能本身 SKILL.md 中对自身目录的引用保留 |
| R-6 | 杂项清理 | 「变更人:<具体人名>」类自然人名 → 改为 `<责任人>` 占位 | 替换或删除 | — |

**执行顺序**:R-1 → R-2 → R-3 → R-4 → R-5 → R-6(顺序固定,前序规则可能消除后续规则的上下文)

**例外保留白名单实现**:`code-it` 任务在执行每条规则前,先 grep 该文件的"白名单匹配区",跳过这些区域。简化方案(本设计采用):
- 每条规则的正则已包含"不在括号内" / "非合规条款"的反向匹配(由正则负向断言承担)
- 不维护独立白名单文件 → 减少误判、简化流程

**输入/输出**:
- 输入:`plugins/code-skills/skills/<技能名>/{SKILL.md, templates/*}`(共 14 技能 × N 文件)
- 输出:被修改的文件(行级改动)+ 1 个 `process-doc-decisions.md`(本设计内)+ 1 个 commit

### 功能域 2:验证(隐含,不展开)

执行 AC-1 ~ AC-8(grep 正则、抽查、人工 review)。不在 `code-it` 主流程内,作为 `code-check` 阶段的输入。

## 13. 备选方案

### 备选 A:`--extensible`(用户选项中已否决)

- **方案**:抽象为"清理规则表 + 匹配引擎 + 多个独立清理器";规则表 JSON 配置化;每个清理器独立可扩展
- **否决理由**:
  - 本轮实际只有 6 条硬编码规则,引擎化是过度设计
  - 配置文件需要文档化(违反 `--minimal`)
  - 增加出错面(规则引擎本身的 bug)
  - 未来若需要新清理类型,直接改 6 条规则为 7 条 + commit(线性,简单)

### 备选 B:逐文件单独写规则(`--extensible` 变种)

- **方案**:为 14 个技能各写 1 套清理规则(共 14 套)
- **否决理由**:
  - 痕迹**形态**是跨技能统一的(都来自本仓库开发过程),不需要 14 套
  - 维护成本翻 14 倍
  - 误判面翻 14 倍

### 备选 C:不清理,留作后续(用户选项外)

- **方案**:不做
- **否决理由**:用户明确诉求要清
