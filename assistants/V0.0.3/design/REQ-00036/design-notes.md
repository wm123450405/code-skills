# 设计笔记 — REQ-00036

更新时间:2026-06-16 17:33
版本:V0.0.3

## 关键设计问题清单

1. **Q-1**:清理规则是"硬编码 6 条"还是"配置化规则引擎"?
2. **Q-2**:清理范围是"SKILL.md 单层"还是"SKILL.md + templates/"?
3. **Q-3**:占位符(`REQ-00001` / `BUG-00001`)算痕迹还是功能性内容?
4. **Q-4**:跨技能契约(章节号、字段名)如何保证不误伤?
5. **Q-5**:历史合规性条款(如「既有 N 需求 NFR-5 字节级保留」)保留还是清理?
6. **Q-6**:执行粒度 — 1 次大 commit 还是按技能分多个 commit?
7. **Q-7**:不动 `checklists/` / `guidelines/` 的边界划定依据?
8. **Q-8**:清理后是否重跑 `code-check`?

## 候选方案(对每个关键问题)

### Q-1 清理规则形态

- **方案 A**(`--minimal`):硬编码 6 条规则,1 次扫完
- **方案 B**(`--extensible`):抽象为规则表 + 引擎 + 多个独立清理器
- **方案 C**(`--balanced`):硬编码 6 条 + 1 份"例外白名单"配置文件

**选定:A(`--minimal`)**
- 理由:本轮 6 条规则**足够**,无证据未来会频繁扩;引擎化是过度设计;白名单配置文件会变成"需要文档化的新规范"(违反 `--minimal` 精神)

### Q-2 清理范围

- **方案 A**:只清 SKILL.md
- **方案 B**:SKILL.md + templates/
- **方案 C**:SKILL.md + templates/ + checklists/ + guidelines/ + assistants/

**选定:B(用户已确认)**
- 理由:用户明确选择"连同 templates/ 一起清";C 选项超出本轮范围(用户已确认 checklists/guidelines 不动)

### Q-3 占位符处理

- **方案 A**:全部删除
- **方案 B**:全部保留
- **方案 C**:按文件类型区分 — SKILL.md 删除,模板里保留

**选定:C(用户已确认)**
- 理由:模板占位符是**功能性**的(用户填空用),SKILL.md 里不出现占位符(已验证 — 全文 grep `REQ-00001` 0 命中)

### Q-4 跨技能契约保护

- **方案 A**:不做保护,事后 grep 验证
- **方案 B**:正则负向断言(不匹配契约字面)
- **方案 C**:人工逐文件 review

**选定:B + C(双保险)**
- 理由:B 在执行层拦截误伤(负向断言避开「步骤 N」「§X.Y」「文件路径」等);C 在验证层兜底(`code-check` AC-5 人工抽查 5 处)

### Q-5 历史合规性条款

- **方案 A**:全部清理,不管是否合规
- **方案 B**:全部保留
- **方案 C**:逐条判断 — 真合规条款保留,伪合规清理

**选定:C(默认保留,`code-it` 阶段细化)**
- 理由:合规性条款的"合规性"需逐条判断,无法靠通用正则;`code-it` 步骤 9 实施时人工/AI 评估;AC-2/AC-4 允许 ≤ 2-3 处例外

### Q-6 执行粒度

- **方案 A**:1 次大 commit
- **方案 B**:按 14 技能分 14 commit
- **方案 C**:按 6 条规则分 6 commit

**选定:A(NFR-8 锁定)**
- 理由:1 次 commit 便于 git revert 整次回退(NFR-10);14 或 6 次 commit 容易在部分回退时遗漏依赖

### Q-7 checklists/guidelines 边界

- **方案 A**:本设计**不**清
- **方案 B**:本设计清,加注释
- **方案 C**:本设计只清 SKILL.md 的"指向 checklists/guidelines"的引用

**选定:A(用户已确认)**
- 理由:checklists/guidelines 是结构化资产(`review-checklist.md` / `coding-style.md` 等),不是开发痕迹源;留作后续需求

### Q-8 清理后 review

- **方案 A**:不重跑 `code-check`(本设计已通过 AC 自检)
- **方案 B**:重跑 `code-check`

**选定:A(默认;用户可在 code-it 完成后手动调)**
- 理由:本设计的 6 条 AC 是 grep 验证,已自包含;`code-check` 是更深入的"任务级"评审,与本设计"文档级清理"目标不一致

## 选定方案与理由汇总

**整体方案**:`--minimal` 路线,1 次 commit,清 14 技能 × (SKILL.md + templates/),0 新增模块/接口/数据结构/三方依赖,6 条硬编码清理规则 + 1 套基于正则负向断言的误伤防护 + `code-check` 阶段 AC-5 人工兜底。

**理由**:
1. 用户明确选 `--minimal`
2. 本轮实际只有 6 类痕迹(经 Grep 扫描已确认),规则化抽象无收益
3. 单次 commit 便于 git revert 整次回退(NFR-10)
4. 0 新增依赖(符合"项目级规范"§dependency-conventions 精神)

## 备选方案被否决的理由(汇总)

- `--extensible` 路线:6 条规则硬编码足够,引擎化是过度设计(详见 §13 备选)
- 按 14 技能分 14 commit:回退粒度过细,容易遗漏依赖
- 按 6 条规则分 6 commit:同 1,回退粒度过细
- 引入 `ripgrep` 等 CLI 工具:沿用 `Grep` 工具集(Claude Code 工具集),无新依赖
- 引入 `prettier`/`remark` 格式化工具:本设计只删不改,无格式化需求
- 清理 `checklists/` / `guidelines/`:用户已确认不动
- 清理 `assistants/` 工作空间:用户已确认不动(史料价值)
