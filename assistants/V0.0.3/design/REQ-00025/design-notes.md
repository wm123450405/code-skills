# 设计笔记 — REQ-00025
更新时间:2026-06-07
版本:V0.0.3

## 关键设计问题与候选

### Q1:编号格式软化方式?
- **候选 A**:保持"5 位纯数字"强约束,新增"特殊编码"标识
- **候选 B**:软化接收端正则(前缀固定 + 后缀自由字符集)
- **选定**:B(用户原始描述第 1 段明确)

### Q2:后缀字符集范围?
- **候选 A**:`A-Za-z0-9`(仅字母数字)
- **候选 B**:`A-Za-z0-9.\-_`(字母数字 + `.` / `-` / `_`)
- **选定**:B(覆盖典型场景;`\.` 允许 `V0.0.1.001`,`-` 允许 `2025-Q4-001`)

### Q3:第三方平台前缀未登记如何处理?
- **候选 A**:拒绝(屏显未登记)
- **候选 B**:自动登记(无需用户操作)
- **选定**:A(用户原始描述第 3 段隐含;"方便用户使用其他平台登记...再通过技能进行代码编写"指用户主动登记,非自动)

### Q4:屏显是否截断 30 字符?
- **候选 A**:标题 + 编号都截断 30 字符
- **候选 B**:仅截断标题,编号完整保留
- **选定**:B(编号需完整保留用于审计 + 跨平台追溯)

### Q5:`code-auto` 是否同步改造?
- **候选 A**:同步(`code-auto` 解析关键字 `from`,沿用 v2 解析)
- **候选 B**:**不**同步(BUG-00001 修复路径独立,`code-auto` v2 模式 B 沿用)
- **选定**:B(本需求 NFR 严守边界;`code-auto` v2 已识别 REQ-NNNNN,本需求放宽后兼容性继续)

## 备选方案被否决的理由

- 候选 A(保持 5 位强约束):违反用户原始描述
- 候选 A(仅字母数字):覆盖不全(`.` / `-` 不能用)
- 候选 B(自动登记):违反用户"主动登记"隐含意图
- 候选 A(屏显截断 30):编号审计需要完整字面值
- 候选 A(同步 code-auto):本需求边界限定 8 个技能,不含 code-auto

- **影响**:0 既有编号受影响

- **理由**:`code-fix` 自身有 `from REQ-NNNNN` / `from BUG-NNNNN` 双模式(本需求 0 触发)
- **影响**:0
