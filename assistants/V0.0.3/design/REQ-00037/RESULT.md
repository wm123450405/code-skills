## 1. 设计概述

本需求是**缺陷状态推进职责的重新分工**:把"修复过程中需复跑 `/code-fix` 来推进缺陷状态"的痛点,通过 4 个 SKILL.md(`code-fix` / `code-plan`(BUG 路径)/ `code-it`(BUG 路径)/ `code-check`)的协同自动推进来消除。

**核心架构思路**:`--balanced` 路线,5 个新状态 + 既有 8 状态字面共存。`code-fix` 仅负责登记与维护"前 5 段状态"(`报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞` + 新增入口 `待处理`),**不再**参与"待开发及之后"的状态推进;`code-plan` 完成时自动推 `待开发`;`code-it` 启动第 1 个任务时推 `开发中`,全部任务完成时推 `待审查`;`code-check` 完成评审时推 `已完成`。整个流程 4 步,无中途复跑。

**新增/复用/修改范围**:
- 新增:**无**(0 模块、0 接口、0 数据结构、0 三方依赖)
- 复用既有:`code-fix` / `code-plan` / `code-it` / `code-check` 的核心工作流 + 状态字段写回工具(`Edit` 工具)
- 修改既有:4 个 SKILL.md + 1 个看板显示技能(`code-dashboard`)— 详见 §7.3

## 3. 设计目标与非目标

### 目标

- **G-1 — 流程瘦身**:典型完整流程由 10 步压缩为 4 步,无中途复跑 `/code-fix`(FR-1 ~ FR-5)
- **G-2 — 状态字面共存**:5 新状态 + 8 老状态字面共存;历史 BUG 状态保留(NFR-3)
- **G-3 — 自动推进幂等**:任一技能在推进状态时,若当前状态已为目标状态,跳过(NFR-1)
- **G-4 — 失败回退**:任一同步动作失败时,`fix/<BUG-NNN>/RESULT.md` 状态字段写回旧值(NFR-2)
- **G-5 — 看板同步**:4 个技能的"状态回写"动作**都**同步版本看板"缺陷清单"区段(NFR-7)
- **G-6 — 不破坏既有契约**:`code-fix` 前段状态(报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞)推进路径**字节级不变**;4 个 SKILL.md 的"## 工作流程"既有小节**不重排**;frontmatter 字节级保留(NFR-5/6)

### 非目标(Out of Scope)

**状态推进点**(本设计核心):

| 推进点 | 触发条件 | 旧 → 新 | 同步文件 |
| --- | --- | --- | --- |
| P-1:`code-fix` 步骤 6 新建分支 | 文档头"状态"字段初始值 | — → `待处理` | `fix/<BUG-NNN>/RESULT.md` + `fix/RESULT.md` + 看板 |
| P-2:`code-plan` 步骤 27A/28A 末尾 | 步骤 27A 写 `fix/<BUG-NNN>/RESULT.md` 文档头后 | `待处理` → `待开发` | 同上(若当前状态 = `待开发` / `报告` / `调查中` 等老字面,跳过 / 维持) |
| P-3:`code-it` 步骤 21(缺陷分支)| 本任务 = `PLAN.md` 任务总览首条 + 当前状态 = `待开发` | `待开发` → `开发中` | 同上 |
| P-4:`code-it` 步骤 24(缺陷分支)| 全部任务已完成+已取消 == 总任务数 + 当前状态 = `开发中` 或 `待开发`(单任务) | → `待审查` | 同上 |
| P-5:`code-check` 步骤 13(扩展 BUG 路径)| 评审完成 + 当前状态 = `待审查` | `待审查` → `已完成` | 同上 |

**幂等与失败回退**(每个推进点都遵循):
1. 读 `fix/<BUG-NNN>/RESULT.md` 当前状态
2. 检查是否已为目标状态 → 是则跳过(幂等)
3. 用 `Edit` 写新状态到文档头
4. 同步 `fix/RESULT.md` 缺陷清单 + 看板"缺陷清单"区段
5. 任一同步失败 → 写回旧状态 + 屏显 `⚠`

## 6. 功能架构

### 功能域 1:`code-fix` 登记与状态推进收敛

- **目的**:本技能**仅**负责登记(初始状态 `待处理`)+ 维护"前 5 段状态"(`报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞`);不参与"待开发及之后"的状态推进
- **涉及模块**:`code-fix` SKILL.md
- **协作方式**:`code-fix` 步骤 4 状态推进表新增 1 行(`待处理` 行) + 步骤 9 引导下一步表新增 4 行(标记"本技能不推进")

**关键修改**(`code-fix` SKILL.md):
- 步骤 6 "新建分支":文档头"状态"字段写 `待处理`(替代 V0.0.3 当前的 `报告`)
- 步骤 4 状态推进表:新增 6 行(`待处理 / 待开发 / 开发中 / 待审查 / 已完成`)
  - `待处理` 行:候选目标 = `报告 / 调查中 / 已关闭-非缺陷 / 已取消`(本技能可推进)
  - `待开发 / 开发中 / 待审查 / 已完成` 行:候选目标 = `(本技能不推进;由 code-plan / code-it / code-check 推进)`
- 步骤 5 注脚:新增"本技能不推进"待开发 / 开发中 / 待审查"等状态"说明
- 步骤 9 引导下一步表:4 行(`待开发 / 开发中 / 待审查 / 已完成`)改为"(由 ... 推进)"
- ## 衔接 段"典型完整流程"举例由 10 步压缩为 4 步(FR-8)
- ## 不要做的事 段新增:`不推进"待开发 / 开发中 / 待审查 / 已完成" 状态 — 这些由 code-plan / code-it / code-check 自动推进`

**输入/输出**:
- 输入:用户原始缺陷描述 / 已有 BUG-NNN
- 输出:`fix/<BUG-NNN>/RESULT.md`(状态=`待处理`)+ `fix/RESULT.md` 缺陷清单行 + 看板"缺陷清单"行

### 功能域 2:`code-plan` 完成时状态回写

- **目的**:`code-plan` 步骤 27A / 28A 末尾追加"状态回写"动作:当前状态 = `待处理` → 推进为 `待开发`;若当前状态为 `报告 / 调查中 / 修复规划中` 等老字面,维持原字面(不强迁)
- **涉及模块**:`code-plan` SKILL.md §"缺陷分支" 步骤 27A
- **协作方式**:`code-plan` 步骤 27A 用 `Edit` 写 `fix/<BUG-NNN>/RESULT.md` 文档头"当前状态"字段 + 步骤 28A 同步看板

**关键修改**(`code-plan` SKILL.md):

## 10. 集成点 / 风险

### 10.1 集成点(每个技能的状态回写动作汇总)

| `code-fix` | 步骤 6(更新)| `fix/<BUG-NNN>/RESULT.md` 当前状态 | 文档头"状态"字段(前 5 段) | `fix/RESULT.md` + 看板 |

### 10.2 风险

- **R-1:状态回写失败回退**:`code-it` 步骤 24 写 `fix/<BUG-NNN>/RESULT.md` 状态 `开发中 → 待审查` 成功,但同步 `fix/RESULT.md` 或看板失败。**缓解**:NFR-2 写回旧状态 + 屏显 `⚠` + 不阻断。
- **R-2:`code-check` 评审派生改修任务**:BUG 状态不回退(沿用 E-3);保持 `待审查`;改修完成后再调一次 `code-check`。
- **R-3:多任务 BUG 中用户跳过中间任务**:`code-it` 步骤 0a 前置任务守卫拦截(沿用既有);BUG 状态保持 `待开发`。
- **R-4:历史 BUG 被新流程推过时**:`code-plan / code-it / code-check` 检测到当前状态为老字面,维持原字面继续走老路径(沿用 E-5)。
- **R-5:`code-dashboard` 状态分类不归一化**:看板"缺陷清单"区段可能同时出现新老字面;`code-dashboard` 步骤 4 段 2 分类规则扩展后,屏显"待修复"数量 = 4 个新字面 BUG 数量 + 老字面 BUG 数量(沿用 PD-2 锁定)。

## 11. 备选方案

### 备选 1:5 新状态完全替换 8 老状态(否决)

- **方案**:历史 BUG 一次性迁移;`code-dashboard` 重写分类。
- **否决理由**:迁移成本高(需写迁移脚本 + 历史 commit 调整);与"PD-2 锁定"不兼容;沿用既有 8 状态机的迁移负担由"老路径"承担更经济。

### 备选 2:`code-check` BUG 路径写独立简化模板(否决)

- **方案**:BUG 路径**仅**评审 `fix/<BUG-NNN>/PLAN.md` + `code/<TASK-BUG-...>/RESULT.md`;产出简化版 `REVIEW-REPORT.md`。
- **否决理由**:长期维护 2 份模板成本高;与 REQ 路径评审结果格式不对齐;`code-dashboard` 难以统一展示。**选定**:沿用既有"单需求评审"骨架(字节级),BUG 路径**复用** REQ 路径的评审维度。

### 备选 3:引入"状态推进器"抽象层(否决)

- **方案**:新增 1 个 `code-bug-state-pusher` 技能或 1 个共享配置文件,4 个 SKILL.md 调用它推进状态。
- **否决理由**:抽象层收益(未来新增状态字段)不抵其实施复杂度(4 个 SKILL.md 都要改 + 新增 1 个技能目录);与"每个 SKILL.md 自含"原则相悖(沿用 `module-conventions §规则 1`)。

### 备选 4:单任务 BUG 完全跳过 `开发中` 状态(否决)

- **方案**:`code-it` 步骤 21 判定"总任务数"为 1 时,跳过 `开发中` 状态;`待开发 → 待审查` 直接。
- **否决理由**:`code-it` 步骤 21 需要读 `PLAN.md` 任务总览才能判定,增加复杂度;与多任务 BUG 的"无差别处理"原则相悖(用户希望"流程 4 步无复跑",多任务 vs 单任务的差异**越少越好**);状态字段在单任务 BUG 持续时间极短,屏显影响可忽略。
