# 设计笔记 — REQ-00026

更新时间:2026-06-08 12:30
版本:V0.0.3

## 1. 关键设计问题清单

### Q1:如何系统性识别"描述性段落" vs "硬约束段落"?
- 候选 A:按"行级 grep + 人工逐行判断"
- 候选 B:按"段落级语义切片"(以 `## / ### / 表格行 / 代码块` 为边界)
- 候选 C:基于规则预筛 + 段落级复核
- **选定:C**(在 `code-it` 阶段以"行号范围标注"形式执行,先 grep 全部命中点,再按上下文(段落标题、表格列、`> 不修改` 段)二分类)

### Q2:`<本仓库>` 占位符的语义边界
- 候选 A:在 SKILL.md 中首次出现时显式声明"`<本仓库>` = 该 marketplace 仓库的根目录"
- 候选 B:不显式声明,读者自悟
- **选定:A**(在每个被改 SKILL.md 的概述段加一句"`<本仓库>` 指代本 marketplace 仓库的根目录,如下 `<本仓库>/skills/...`")

### Q3:对 `code-merge` 等本轮未列出的 SKILL.md 是否也应同步?
- **选定:不动**(本需求 FR-1 范围明确列出 10 个,`code-merge` 是新增未纳入本需求,留待后续 REQ)

### Q4:本次扫除是否需要新增 lint 规则防回归?
- 候选 A:在 `assistants/rules/` 新增 `skill-conventions.md §规则 2`,禁止 SKILL.md 描述段出现 `plugins/code-skills` 字面
- 候选 B:不新增,留待 code-check 评审时人工 / Grep 校验
- **选定:B**(不新增规则,因 `./assistants/rules/` 在本需求范围外(NFR-1),且无用户授权)

## 2. 候选方案权衡

### 方案 A(选定):纯文案扫除,不引入自动化工具
- 优点:改动面小,风险可控,符合本需求"通用化"的精神("技能契约可被人工维护")
- 缺点:依赖 `code-it` 阶段人工逐条 Edit
- 适用:本需求

### 方案 B(否决):写一个 `normalize-skill-descriptions.js` 脚本
- 优点:可复用、可批量
- 缺点:本仓库"不写工程代码"(`CLAUDE.md` 明确),脚本会违反仓库自我定位
- 否决理由:本项目是技能库,不是工具集

## 3. 选定方案与理由

**方案 A + Q1 选 C + Q2 选 A + Q3 不动 + Q4 不新增规则**。

理由:
- 本项目是技能库,不放工程代码
- 改动面集中在 13 个文件,人工可控
- 占位符 `<本仓库>` 在每个被改 SKILL.md 概述段中显式声明,避免歧义
- 不修改规范文件(NFR-1 + Q4 选定 B)

## 4. 备选方案被否决的理由
- 方案 B:违反"本项目不写工程代码"自我定位
- Q1 选 A:行级粒度过细,改完后行号漂移导致半改半留
- Q1 选 B:段落边界在 SKILL.md 中不统一(有的用 `##` 有的用 `###`),边界识别成本高
- Q4 选 A:本需求范围外,需新需求登记
