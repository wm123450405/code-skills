## 2. 候选方案权衡(关键决策)

### 2.1 启动能力探测 vs 显式参数

- **A. 自动探测**(选定):
 - 优点:零用户介入,沿用 NFR-3;不引入 CLI 参数
 - 缺点:配置文件类型多,需枚举(`package.json` / `pyproject.toml` / `Makefile` / `docker-compose.yml` / `Cargo.toml` / `go.mod` / `pom.xml` / `build.gradle`)
 - 缓解:每种语言 1 个主判定,FR-1 列出 6 步伪代码;`canStart = false` 沿用 `E-1` 降级
- **B. 显式参数**:
 - 优点:精确控制
 - 缺点:增加用户负担;**违反** NFR-3(不新增 CLI 参数)

### 2.2 失败降级链(A 选定)

- **链式降级**:
 1. 启动失败 → 屏显 `⚠ 启动失败:<stderr 前 500 字符>` + 写 `### 复现结论 = 复现失败(<原因>)` + 继续登记
 2. 进程崩溃(exit ≠ 0)→ 屏显 `⚠ 进程异常退出:<exit code>` + 写产物 + 继续登记
 3. 超时(> 60s)→ 屏显 `⚠ 启动超时` + 终止进程(SIGTERM 5s 后 SIGKILL)+ 降级为文本复现
 4. 截图工具不可用 → 跳过截图,屏显 `⚠ 截图工具不可用`
 5. 日志过大(> 10MB)→ 截断到前 1MB + 屏显 `⚠ 日志过大,已截断`
- **原则**:**任何失败**都**不**阻断 `code-fix` 主流程;产物缺失仅影响"### 复现结论" 字段,**不**影响"当前状态" 字段(仍为 `待处理`)

## 3. 选定方案与理由(总结)

- **架构**:`code-fix` 步骤 0 末尾 + 步骤 6 末尾追加子节(纯追加,既有结构字节级保留)
- **产物**:`fix/<BUG-NNN>/reproduce/` 子目录,4 类文件(`run.log` / `screenshot-N.png` / `interaction-N.{json|txt}` / `RESULT-meta.json`)
- **模板**:`bug.md` 新增"## 复现产物登记" 区段 + 文档头新增 2 字段(复现方式 / 产物路径)
- **辅助**:`assistants-layout.md` 同步追加 `reproduce/` 子目录说明
- **降级**:链式降级,屏显 `⚠` + 记录原因 + 继续登记
- **状态**:**不**改 REQ-00037 已确定的初始态(`待处理`)与状态推进路径
- **工具**:`playwright` → `puppeteer` → `headless-chrome` 链式降级(运行时探测,**不**作为项目级依赖)
