## 2. 候选方案与选定

### Q-1:模块识别是否引入新三方依赖?

- **候选 A**:引入 `workspace-tools` / `find-packages` 等 npm 包 → **否决**(违反 NFR-3 零规范变更;本仓库是 Claude Code 技能插件集合,无 npm 依赖)
- **候选 B**:**只读** monorepo 声明文件(pnpm-workspace.yaml / package.json#workspaces / pom.xml#modules / Cargo.toml#workspace.members / go.mod 等)→ **选定**(0 新增依赖,纯 IO 操作,NFR-3 满足)
- **候选 C**:完全靠 git diff 公共子目录 → **否决**(无法处理无 git 仓库场景;无法识别 monorepo 显式声明)

### Q-2:模块识别优先级链如何排序?

- **候选 A**:**声明文件 > git diff > CWD 根** → **选定**(声明文件权威;git diff 退化;CWD 根兜底)
- **候选 B**:git diff > 声明文件 > CWD 根 → **否决**(git diff 在非 git 仓库场景退化为 CWD 根,绕过声明文件)
- **候选 C**:CWD 根(单模块 = 单工程,字节级沿用 REQ-00034)→ **否决**(违反 FR-1 模块识别要求)

### Q-3:7 项守卫检查本身是否需要改?

- **候选 A**:**7 项检查字节级沿用 REQ-00034,仅位置从 CWD 根 → 模块目录** → **选定**(NFR-4 不归一化;INV-3 字节级保留;FR-2 锁定)
- **候选 B**:7 项检查改写以支持多语言混合识别 → **否决**(需求超范围;FR-2 字面要求"字节级沿用")
- **候选 C**:扩展为 10+ 项检查(覆盖 Bazel / Meson / SCons 等)→ **否决**(违反 INV-3 + 性能约束)

### Q-4:模板"## 单元测试"小节是否需要改?

- **候选 A**:**既有"## 单元测试"小节字节级保留 + 新增"## 各模块单测结果"小节** → **选定**(NFR-4 字节级保留;FR-4 锁定;100% 兼容既有 RESULT.md)
- **候选 B**:改写"## 单元测试"小节为多模块支持 → **否决**(破坏既有 7 字段语义)
- **候选 C**:仅扩展"## 单元测试"小节,新增字段行 → **否决**(破坏既有字段语义;不可扩展性差)

### Q-5:是否需要扩展看板字段?

- **候选 A**:**不扩展看板字段,仅在既有"任务清单"区段追加 1 行(本需求登记)** → **选定**(dashboard-conventions §规则 1 0 触发;NFR-3 零规范变更)
- **候选 B**:新增"模块级单测记录"区段到看板 → **否决**(违反 dashboard-conventions §规则 1 三方同步;本需求范围超出)
- **候选 C**:扩展"任务清单"区段"测试状态"列 → **否决**(破坏既有 2 状态语义;`code-plan` 既定)

### Q-6:FR-5 `code-plan` 字面改写是必须还是可选?

- **候选 A**:**FR-5 字面改写为 1 句(任务粒度描述对齐模块识别)** → **选定**(任务粒度描述需要反映"模块识别"新能力;字面增量不破坏既有结构)
- **候选 B**:FR-5 跳过,`code-plan` 不改 → **否决**(任务粒度描述与新行为脱节,后续 code-auto 编排可见性差)
- **候选 C**:FR-5 字面改写 + 新增字段(触发 `dashboard-conventions §规则 1`)→ **否决**(违反 NFR-3 零规范变更)

### Q-7:步骤 8a.0 模块识别放在哪里?

- **候选 A**:**锚点 = "## 步骤 8 实施开发"之后,"## 步骤 8a 守卫"之前** → **选定**(模块识别是守卫的前置;FR-1 字面要求)
- **候选 B**:锚点 = "## 步骤 8a.1 守卫检查项清单"之前(作为 8a.1 的子节) → **否决**(8a.0 是 7 项检查之外的独立逻辑,作为独立步骤更清晰)
- **候选 C**:锚点 = "## 步骤 0a 前置任务守卫"之后(全局前置) → **否决**(模块识别仅在守卫阶段需要;过早执行浪费性能)

### Q-8:单测输出位置扩展为多模块后,屏显契约是否需要改?

- **候选 A**:**屏显契约 0 改,仅增加"模块级守卫检查详情"模块级输出** → **选定**(NFR-3 既有屏显契约 0 改;步骤 8a.4 屏显格式扩展为模块级)
- **候选 B**:屏显契约完全改写为"多模块 + 多语言"格式 → **否决**(破坏既有屏显契约;`code-auto` 解析逻辑可能受影响)
- **候选 C**:屏显契约保持不变,模块级细节仅写 work-log.md → **否决**(屏显是首要可观测性入口,work-log 是次要)
