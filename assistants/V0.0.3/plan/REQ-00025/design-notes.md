# 设计笔记 — REQ-00025
版本:V0.0.3

## 关键设计问题

### 问题 1:任务 1 (encoding-conventions.md) 是不是必须排在首位?

- **决策**:**是**
- **理由**:`encoding-conventions.md` 是编码权威源,8 个 SKILL.md 任务全部"读规范"模式,任务 1 修订后,8 个 SKILL.md 才有"读什么"的依据
- **候选否决**:
  - 候选 A:并行执行 9 任务 → 否决,8 个 SKILL.md 任务的"字面更新"无依据
  - 候选 B:8 个 SKILL.md 任务在前,规范任务在后 → 否决,顺序颠倒

### 问题 2:8 个 SKILL.md 任务的顺序如何定?

- **决策**:`code-require` → `code-design` → `code-plan` → `code-it` → `code-unit` → `code-check` → `code-fix` → `code-dashboard`
- **理由**:沿用概要设计 D-6 给定的顺序(主流程管道:require → design → plan → it → unit → check,辅以 fix 和 dashboard)
- **候选否决**:
  - 候选 A:按文件大小 → 否决,与业务流程无关联
  - 候选 B:字母顺序 → 否决,无业务语义

### 问题 3:8 个 SKILL.md 任务之间是否互相依赖?

- **决策**:**否**(互相独立)
- **理由**:8 个 SKILL.md 之间无相互引用,无共享代码块,字面更新可独立完成
- **候选否决**:
  - 候选 A:依赖链式执行 → 否决,无依据

### 问题 4:是否需要"里程碑"区分?

- **决策**:**是**,2 个里程碑
  - M1:软化编号正则上线(规范 + 8 SKILL.md = 9 任务全部完成)
  - M2:验证既有 5 位纯数字 + 新格式编号均可解析(手动调用 U-1~U-10)
- **理由**:M1 是"完成态",M2 是"验证态",M2 是发布前的最后一道关卡
- **候选否决**:
  - 候选 A:单里程碑 → 否决,缺失验证阶段
  - 候选 B:3+ 里程碑 → 否决,过度细分

## 候选方案

### 候选 A:本方案(已采纳)

- 1 规范 + 8 SKILL.md = 9 任务
- 任务 1 规范修订在前,8 SKILL.md 任务在后
- 触发/来源=详细设计,测试状态=不适用
- 2 里程碑(M1 完成态 + M2 验证态)

### 候选 B:1 规范 + 8 SKILL.md 各拆 2-3 子任务 = 20+ 任务

- 否决理由:违反 REQ-00017 强约束("一个任务 = 一个'实际产出'");8 SKILL.md 各自的字面更新是"一个实际产出",**不**应拆分
- 任务粒度:**过细**(每个 SKILL.md 拆 2-3 子任务 → 25+ 任务)

### 候选 C:9 任务合 1

- 否决理由:违反"0.5-2 天内可完成"约束;9 文件合 1 → 单任务 3-5 天,粒度过粗
- 任务粒度:**过粗**
