## 关键设计问题

### Q1:任务拆分粒度
- 候选 A:14 个模块 = 14 个任务(每文件 1 任务)
- 候选 B:按"功能点"聚合为 3-4 个任务(例如"全部 10 SKILL.md 改完"为 1 任务)
- 候选 C:按"高内聚"聚合(描述段 + 1 模板 = 1 任务;code-rule 单独 1 任务)

**选定:C**(按"职责聚合"切分):
1. T-001:[修改] 通用占位符声明 + 9 个 SKILL.md 描述段去专属化(`code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` / `code-fix` / `code-publish` / `code-init`)
2. T-002:[修改] code-rule/SKILL.md 描述段 + L336/363/370 三处 CLAUDE.md 字面替换
3. T-003:[修改] code-publish/templates/(DEPLOY.md / UPDATE.md / qanda-README.md) 字面替换
4. T-004:[修改] code-init/templates/INIT-REPORT.md 字面替换
5. T-005:[文档] 同步版本看板"任务清单" + "变更记录"

理由:
- T-001 聚合 9 个 SKILL.md,因为它们的"描述段去专属化"模式完全一致(占位符 + 概述段声明),可一次性 Edit 模板 + 9 个文件
- T-002 独立是因为 `code-rule` 涉及 L336/363/370 的特殊硬替换(不只"描述段"那么简单)
- T-003 独立是因为 3 个 templates 都在同一目录下,可集中
- T-004 独立 1 任务因为体量小且不与其他聚合
- T-005 是本仓库"不写代码"导致的`code-it` 副作用,详见 `commit-conventions.md` 沿用

### Q2:本需求是否触发"架构任务"?
- `code-plan` §10A 架构任务触发条件:
  - 触发 1:较高扩展性 + 较高复杂度 → **不触发**(本需求是纯文案)
  - 触发 2:涉及未来可能的不同实现方式 → **不触发**
  - 触发 3:对接不同三方组件 → **不触发**
- **不**插入 T-000 架构任务

### Q3:是否派生"更新看板"任务?
- **不派生**(沿用 REQ-00017 强约束)
- 看板推进由 `code-it` 末尾兜底 P-1 小步承担(本仓库纯文档,此职责可由 `code-it` 步骤 N 末尾承担)
