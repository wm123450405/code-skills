## 详细设计的关键设计问题

### Q-1:`code-fix` 步骤 6 "新建分支" 状态字段的写入路径

- **方案 A(直接 Write)**:用 `Write` 工具一次性写 `fix/<BUG-NNN>/RESULT.md`,状态字段直接写 `待处理`。
- **方案 B(模板 + Edit)**:用 `Write` 写模板(状态字段默认 `报告`),再用 `Edit` 工具替换为 `待处理`。
- **选定 A**:
  - 方案 A 减少 1 次写操作,降低写回失败概率(沿用 NFR-2)
  - 方案 B 需要 2 次写 + 校验,实施成本高
  - 方案 A 与既有 `Write` 工具调用模式一致(沿用 `code-fix` 步骤 6 "新建分支" 既有结构)

### Q-2:`code-plan` 步骤 27A 状态回写的判定粒度

- **方案 A(仅字面匹配)**:判断旧状态字面是否 = `待处理`,是则推 `待开发`;否则跳过。
- **方案 B(枚举集合判定)**:判断旧状态字面 ∈ {`待处理`},是则推 `待开发`;若是老字面(`报告 / 调查中 / 修复规划中`),维持原字面;若是新字面(`开发中 / 待审查 / 已完成`),跳过。
- **选定 B**:
  - 方案 B 显式区分"老字面"与"新字面已推进过",避免重复推进
  - 方案 B 强健性更高(状态字面集合明确,而不是依赖字面匹配)
- **落地**:步骤 27A 末尾新增"状态回写"子步骤,伪代码:
  ```
  readStatus = readFixDocState(bugNum)  // 读 fix/<BUG-NNN>/RESULT.md 文档头
  if readStatus == "待处理":
    writeFixDocState(bugNum, "待开发")
    syncFixRegistry(bugNum, "待开发")
  elif readStatus in ["报告", "调查中", "修复规划中"]:
    pass  // 维持原字面
  elif readStatus in ["开发中", "待审查", "已完成"]:
    pass  // 已推进过,跳过
  else:
    log("⚠ code-plan 状态回写:无法识别当前状态字面 " + readStatus)
  ```

### Q-3:`code-it` 步骤 21 的"首条任务"判定语义

- **方案 A(按文件行序)**:读 `fix/<缺陷编号>/PLAN.md` 任务总览(沿用 `dashboard-conventions §规则 1` 锚点 `^## 任务总览$`),找**首行表格行**(按文件行序,排除表头分隔行)。
- **方案 B(按任务序号)**:解析所有 `TASK-BUG-NNNNN-NNNNN`,找**最小任务序号**。
- **选定 A**:
  - 方案 A 与既有 `code-it` 步骤 7"前置任务守卫"的判定逻辑一致(沿用既有"按文件行序")
  - 方案 B 需要额外解析(从任务编号字符串中提取末 5 位),复杂度高
- **落地**:`grep -n "^| TASK-" fix/<bugNum>/PLAN.md` 取首行 → 解析任务编号列 → 与本任务编号对比

### Q-4:`code-it` 步骤 24 的"全部任务已完成"判定语义

- **方案 A(严格等于)**:已完成+已取消任务数 == 总任务数。
- **方案 B(已完成等于总任务数)**:已完成任务数 == 总任务数(忽略已取消)。
- **选定 A**:
  - 方案 A 与"任务真正可发布"语义一致(已取消任务不再阻塞后续推进)
  - 方案 B 在某些 BUG 因故取消 1 个任务时,BUG 状态会卡在 `开发中` 永远不推进
- **落地**:读 `fix/<缺陷编号>/PLAN.md` 任务总览表格行,统计 `开发状态` 列 ∈ {`已完成`, `已取消`} 的任务数;与"任务总览表格行数 - 表头行数 - 分隔行数"对比;相等时推 `待审查`

### Q-5:`code-check` 步骤 1.5 模式选择的优先级

- **方案 A(RE 优先)**:先匹配 `^REQ-\d{5}$`,再匹配 `^BUG-\d{5}$`。
- **方案 B(BUG 优先)**:先匹配 `^BUG-\d{5}$`,再匹配 `^REQ-\d{5}$`。
- **选定 A**:
  - `code-check` 历史一直以 REQ 路径为主(沿用 V0.0.1 ~ V0.0.3 既有)
  - BUG 路径是**新增**(本设计),优先级应**低于** REQ 路径(避免破坏既有 REQ 调用习惯)
- **落地**:步骤 1.5 三态机扩展:
  1. `^REQ-\d{5}$` → REQ 路径(既有,字节级不变)
  2. `^BUG-\d{5}$` → BUG 路径(新增,本设计新增)
  3. 其他 → 整版本模式 + 警告

### Q-6:`code-dashboard` 步骤 4 段 3 扩展的字面集合

- **方案 A(仅新字面)**:段 3 判定逻辑只把 5 新状态纳入"待修复" / "已修复"。
- **方案 B(新老字面并存)**:段 3 判定逻辑把 5 新状态 + 既有 8 老字面都纳入"待修复" / "已修复"统计(老字面沿用既有路径)。
- **选定 B**:
  - 方案 B 保证"PD-2 锁定"(不归一化字面)延伸到 `code-dashboard` 段 3 统计
  - 方案 A 会让"待修复"统计在历史 BUG 上"少算"(老字面 BUG 不会被算入),与既有"已修复"判定不一致
  - 方案 B 屏显 `P0 待修复: N` 中 N = 5 新字面 BUG 数 + 老字面 BUG 数(沿用既有)

### Q-7:`code-check` 评审发现"必须改"派生改修任务的状态行为

- **场景**:`code-check BUG-NNN` 评审发现"必须改"项,派生 `TASK-BUG-NNNNN-NNNNN` 任务。
- **方案 A(不回退状态)**:BUG 状态保持 `待审查`(沿用 REQ-00027 E-3);改修完成后再调 `code-check` 重跑。
- **方案 B(回退到 `开发中`)**:BUG 状态回退到 `开发中`;改修完成后由 `code-it` 重新推 `待审查`。
- **选定 A**:
  - 方案 A 与用户原始诉求(流程 4 步无复跑)更接近:评审不通过 → 改修 → 再评审 → 完成,这条路径**不**反复经历 `开发中 → 待审查`
  - 方案 B 让"待审查"成为短暂状态,容易被跳过;方案 A 让"待审查"成为稳定状态,语义清晰
