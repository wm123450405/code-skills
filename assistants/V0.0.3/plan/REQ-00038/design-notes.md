# 设计权衡笔记 — REQ-00038

版本:V0.0.3

## 关键设计问题与决策

### 问题 1:模块识别要不要新增 monorepo 检测三方库?

**候选方案**:
- A. 新增 `pnpm-workspace-yaml` / `workspace-tools` 等三方库解析声明文件
- B. 手写解析器,只读声明文件,无新依赖
- C. 调 `git diff` + 启发式推断,不读声明文件

**决策**:B(手写解析器)

**理由**:
- REQ-00034 NFR-3 锁定"0 新增三方依赖";本需求 NFR-3 沿用
- 声明文件格式固定(pnpm-workspace.yaml / package.json#workspaces 等),手写解析器工作量 < 50 行
- 候选 C 退化为 git diff 公共子目录 → E-1 处理

**否决**:
- A 违反 NFR-3
- C 在没有变更路径的极端场景下退化为 CWD 根,无识别能力

### 问题 2:7 项守卫的检查位置从 CWD 根改到模块目录,会不会破坏单模块工程?

**候选方案**:
- A. 模块目录检查(每个模块独立执行 7 项)
- B. 工程根 + 模块目录同时检查(双层守卫)
- C. 仅在工程根无守卫命中时启用模块级守卫(分层守卫)

**决策**:A(模块目录检查)

**理由**:
- 单模块工程 = 1 模块 = 1 整工程,模块目录 = CWD 根,7 项守卫命中字面 = 字节级沿用 REQ-00034(0 回归,AC-4 锁定)
- B / C 增加守卫位置状态机,违反 INV-3 步骤 8a 既有逻辑字节级保留
- 沿用 REQ-00034 7 项守卫字面,仅位置扩展(NFR-4 锁定)

**否决**:
- B 引入新分支,违反 INV-3
- C 工程根守卫命中 → 直接 testable = True,不进入模块级守卫(能力退化为工程根级,FR-2 锁定不允许)

### 问题 3:`unit-test-results.md` 模板多模块支持的字段粒度?

**候选方案**:
- A. 每个模块独立小节,字段:模块路径 / 守卫检查 / 检查位置 / 测试框架 / 测试文件 / 跑通情况(7 字段,FR-4 锁定)
- B. 仅追加 1 个表格行,字段:模块路径 / 守卫结果(2 字段)
- C. 沿用既有 0 改,在 RESULT.md 末尾追加 1 个文件清单

**决策**:A(每模块独立小节,7 字段)

**理由**:
- FR-4 字段明确锁定 7 字段(模板已字面定义)
- B 字段过少,无法满足 `code-check` 评审的"模块级守卫检查详情"要求
- C 不修改既有 `## 单元测试(由 code-it 内化)` 小节(NFR-4 沿用),但 0 信息量

**否决**:
- B 字段不足
- C 信息量过少

### 问题 4:`code-plan` 任务粒度描述字面改写要不要新增字段?

**候选方案**:
- A. 仅字面改写 1 句(由 `code-it` 内化(`code-it` 步骤 8a 守卫 + 步骤 8.5 按需写单测) → 由 `code-it` 内化(`code-it` 步骤 8a.0 模块识别 + 步骤 8a 守卫 + 步骤 8.5 按模块写单测))
- B. 新增"模块识别产物"字段
- C. 新增"模块级守卫结果"字段

**决策**:A(仅字面改写)

**理由**:
- FR-5 字面改写 1 句,锁定不新增字段
- 任务粒度描述与"模块识别"是**正交**两件事(模块识别是 `code-it` 内部细节,任务粒度是 `code-plan` 抽象)
- 沿用既有 V0.0.3 REQ-00031 任务粒度原则(0 新增测试任务 / 0 新增字段)

**否决**:
- B 引入新字段,违反 INV-3(任务粒度原则字节级沿用)
- C 同 B

### 问题 5:模块级单测输出位置退化策略?

**候选方案**:
- A. 7 层测试目录识别优先级链(FR-3 锁定)+ 无约定 → CWD 根 `test/` 退化(原 REQ-00034 行为)
- B. 强制写到模块根 `test/` 目录(简化)
- C. 不退化,跳过(单测功能放弃)

**决策**:A(7 层优先级链 + CWD 根退化)

**理由**:
- 各语言测试约定差异大(Go `*_test.go` 同包 / Python `test_*.py` / Node `__tests__/` / Rust `tests/`),无统一约定
- 候选 B 在 Go / Rust 工程下违反语言惯例
- 候选 C 违反 NFR-2 兼容性(原 REQ-00034 行为退化)

**否决**:
- B 违反语言惯例
- C 破坏兼容性

### 问题 6:模块识别是否要支持 `.gitignore` / `.git/info/exclude` 排除的路径?

**候选方案**:
- A. 仅 `git diff --name-only` 输出,不二次过滤
- B. 调 `git check-ignore` 二次过滤
- C. 配置文件覆盖(用户可配忽略列表)

**决策**:A(不二次过滤)

**理由**:
- `git diff --name-only` 已是 git 仓库的权威输出,包含所有变更文件(无论是否 .gitignore)
- 候选 B 引入新工具调用,增加复杂度;`git diff` 的输出就是 source of truth
- 候选 C 引入新配置,与 FR-1 锁定"不引入用户问路"冲突

**否决**:
- B 复杂度,无明确收益
- C 违反 NFR-3(不引入用户问路 / 新 CLI 参数)

### 问题 7:模块识别结果是否要缓存?

**候选方案**:
- A. 缓存到 `code-it` 内部,整个任务生命周期内复用
- B. 每次步骤 8a.0 都重新识别
- C. 持久化到文件,跨任务复用

**决策**:A(任务级内存缓存)

**理由**:
- 单次任务内步骤 8a.0 一次执行,缓存到 `code-it` 内部(沿用既有"模块识别:步骤 8a.0 一次执行"模式)
- 候选 B 重复扫描,无必要(典型 monorepo 100 模块下 < 2 秒,NFR-1 锁定)
- 候选 C 跨任务缓存增加状态管理复杂度,任务边界之外模块可能已被删除/重命名,缓存失效

**否决**:
- B 性能浪费
- C 状态管理复杂

### 问题 8:`code-check` 是否需要同步适配?

**候选方案**:
- A. `code-check` 评审读取 `unit-test-results.md` 时按模块拆结果(沿用既有结构 + 字段兼容)
- B. `code-check` 新增"模块级守卫评审"维度
- C. `code-check` 0 适配,沿用既有"按需写单测"评审

**决策**:A(`code-check` 仅适配判定文件位置,0 新增字段)

**理由**:
- FR-1 ~ FR-4 不涉及 `code-check` 改造
- 既有 `code-check` 评审"按需写单测"维度 = 读 `unit-test-results.md` + 判定各模块守卫情况,本身已模块级(新增小节"## 各模块单测结果"是新增信息源,**不**改变评审结构)
- INV-5 锁定:11 个其他 `code-*` 技能 SKILL.md 核心工作流 0 改

**否决**:
- B 引入新评审维度,违反 INV-5
- C 失去新增的"模块级守卫详情"信息源,信息丢失
