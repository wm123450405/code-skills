# REQ-00029 — 优化 /code-dashboard 屏显报告(只展示结果 + 去除不必要换行)

- 需求编码:REQ-00029
- 所属版本:V0.0.3
- 状态:已锁定
- 责任人:用户
- 创建:2026-06-10
- 最近更新:2026-06-10
- 当前版本:v1

## 1. 需求概述

为 `code-dashboard` 技能的屏显报告做"瘦身"改造:删除"数据来源 / 计算过程 / 分析过程"等元描述,只展示结果数据;同时**去除不必要的换行**,让每条建议、每段说明占用的屏显空间最小化。目标:用户一眼看到"现在版本什么状态 + 接下来做什么",不被渲染层的解释性文字干扰。

## 2. 背景与目标

- **背景**:`code-dashboard` 当前的屏显报告沿用 REQ-00023 的 4 段契约(总开发进度 + 5 类状态占比 + 高优缺陷 + ≤5 条建议),每段都附带"解释性说明 / 段标题 / 数据来源"等元描述(总行数 ≤ 12 行)。长会话中用户反复调用,这些元描述占用屏显空间且不增加信息密度。
- **业务目标**:让 `code-dashboard` 屏显更"密度高 + 无废话",单次调用屏显 ≤ 8 行(总览模式)/ ≤ 15 行(需求模式)。
- **本次目标**:
  1. 删除所有"数据来源 / 计算过程 / 分析过程"等元描述(算法 1/2/3/4/5 字节级保留,只改渲染层)
  2. 5 类状态占比段从 N 行表格压缩为 1 行摘要
  3. 建议段从"3 行模板(建议/依据/优先级)"压缩为"1 行(命令+优先级+依据)"
  4. 进度条解释行删除(ASCII 比例条自带可读性)
  5. 总览模式 + 需求模式**都压**(沿用用户澄清答复)

## 3. 用户角色与场景

- **角色 1:开发者(本仓库的协作者)**
  - **场景 1.1**:在长会话中频繁调 `/code-dashboard` → 每次只看到精简的"进度 + 缺陷 + 建议" → 屏显滚动更快
  - **场景 1.2**:在 CI 脚本中解析 `code-dashboard` 输出 → 单行格式更易 grep / awk 提取
  - **场景 1.3**:在做 code-check 评审前快速摸底 → 总览 + 需求两种模式都压,屏显更聚焦

- **角色 2:新成员(包括 AI Agent)**
  - **场景 2.1**:快速理解项目"进度 + 阻塞" → 1 行状态摘要 + 1 行缺陷 + 5 行建议,信息密度高
  - **场景 2.2**:与 LLM 协同时(把 code-dashboard 输出喂给另一个 LLM) → 单行格式更易被 LLM 解析

## 4. 功能需求(FR)

### FR-1:删除数据来源/计算过程/分析过程说明
- **描述**:屏显报告中**不**出现"算法 X 沿用自 §X" / "数据来源:`RESULT.md` 第 X 段" / "分类依据:任务'开发状态'≠'已完成'"等元描述。
- **入口**:code-dashboard §工作流程 步骤 4 渲染 + 步骤 6 屏显
- **前置条件**:无
- **主流程**:
  1. 重构步骤 4 段 1(总开发进度)渲染:删除"`算法 1 renderBar` 沿用 V0.0.2"等说明
  2. 重构步骤 4 段 2(5 类状态)渲染:删除"按 5 类状态分类,数据来自 §任务清单 + §评审发现汇总"等说明
  3. 重构步骤 4 段 3(高优缺陷)渲染:删除"P0/P1 醒目标记,只列数量"等说明
  4. 重构步骤 4 段 4(建议)渲染:删除"每类状态 1 条推荐,共 ≤ 5 条"等说明
- **数据变化**:**无**(屏显只删不改数据)
- **来源**:用户 args"无需显示数据来源、计算过程、分析过程"

### FR-2:5 类状态占比段压缩为单行摘要
- **描述**:5 类状态占比(待概要设计 / 待详细设计 / 待代码开发 / 待单元测试 / 待代码评审)从 N 行表格(每类一行)压缩为 1 行摘要。
- **入口**:code-dashboard §工作流程 步骤 4 段 2 渲染
- **前置条件**:FR-1 已实施
- **主流程**:
  1. 沿用既有 5 类状态判定函数(算法 2,字节级保留)
  2. 渲染层从"每类一行"压缩为"1 行字符串摘要"
  3. 格式:`状态: 待代码开发 2 / 待代码评审 1` 或 `(无)`(无状态时)
- **数据变化**:**无**
- **来源**:用户澄清记录 2026-06-10 §问题 2 答复"压缩为单行摘要"

### FR-3:建议段压缩为单行格式
- **描述**:每条建议从"3 行模板(建议/依据/优先级)"压缩为"1 行(命令+优先级+依据)"。
- **入口**:code-dashboard §工作流程 步骤 4 段 4 渲染
- **前置条件**:FR-1 已实施
- **主流程**:
  1. 沿用既有 5 类状态映射(算法 4 字节级保留)
  2. 渲染层从"3 行"压缩为"1 行"
  3. 格式:`> /code-fix BUG-00001 [高] (依据: P0 待修复 2 个)`
  4. ≤ 5 条建议(沿用 AC-4.3)
- **数据变化**:**无**
- **来源**:用户澄清记录 2026-06-10 §问题 3 答复"含命令+优先级+依据(单行)"

### FR-4:进度条解释行删除
- **描述**:总开发进度的"`总开发进度` / `──────` / `[████████░░░░] 67%`"3 行中,删除前 2 行(段标题 + 分隔线),只保留 ASCII 比例条行。
- **入口**:code-dashboard §工作流程 步骤 4 段 1 渲染
- **前置条件**:FR-1 已实施
- **主流程**:
  1. 沿用既有 ASCII 比例条(算法 5 字节级保留,12 字符)
  2. 渲染层从"3 行"压缩为"1 行"
  3. 格式:`[████████░░░░] 67%`
- **数据变化**:**无**
- **来源**:用户 args"进度条的解释性说明也无需独占一行"

### FR-5:总览 + 需求两种模式都压
- **描述**:总览模式(4 段 → 4 段结构保留但每段都压)+ 需求模式(5 段 → 5 段结构保留但每段都压)。
- **入口**:code-dashboard §工作流程 步骤 4 段 5(需求模式)
- **前置条件**:FR-1 ~ FR-4 已实施
- **主流程**:
  1. 总览模式段 1(总开发进度):FR-4 应用
  2. 总览模式段 2(5 类状态占比):FR-2 应用
  3. 总览模式段 3(高优缺陷):FR-1 应用(屏显 `P0 待修复: █ 2` / `P1 待修复: ░ 0` 1 行)
  4. 总览模式段 4(建议):FR-3 应用
  5. 需求模式段 1(元信息):删除"标题 / 状态 / 概要设计 / 详细设计"独立行,合并为 1 行
  6. 需求模式段 2(任务清单):删除表头独立行,压缩为 1 行字符串摘要
  7. 需求模式段 3(关联缺陷):沿用段 3 范式
  8. 需求模式段 4(任务进度):合并到建议段
  9. 需求模式段 5(建议):FR-3 应用
- **数据变化**:**无**
- **来源**:用户澄清记录 2026-06-10 §问题 1 答复"两种模式都压(推荐)"

### FR-6:屏显总行数约束
- **描述**:总览模式屏显 ≤ 8 行;需求模式屏显 ≤ 15 行(从既有 ≤ 12 / 5 段收紧)。
- **入口**:code-dashboard §工作流程 步骤 4 渲染 + 步骤 6 屏显
- **前置条件**:FR-1 ~ FR-5 已实施
- **主流程**:
  1. 总览模式满负载(段 1 + 段 2 + 段 3 + 5 条建议)= 1 + 1 + 1 + 5 = 8 行 ✅
  2. 需求模式满负载(元信息 + 任务清单 + 关联缺陷 + 5 条建议)= 1 + 1 + 1 + 1 + 5 = 9 行(实际会更少,任务清单/关联缺陷可能为空)
- **数据变化**:**无**
- **来源**:用户 args"去除不必要的换行" + 业务目标"单次调用屏显 ≤ 8 行 / ≤ 15 行"

### FR-7:工具集契约保持
- **描述**:code-dashboard 仍只调用 `Read` / `Glob` / `Grep`,**不**调用 `Write` / `Edit` / `Bash`。
- **入口**:code-dashboard §工具使用约定
- **前置条件**:无
- **主流程**:**不**改工具集
- **数据变化**:**无**
- **来源**:沿用既有(NFR-7 严守)

### FR-8:无副作用验证
- **描述**:执行 `/code-dashboard` 后,`git status` 保持调用前状态。
- **入口**:AC-1 验收
- **前置条件**:无
- **主流程**:**不**写任何文件
- **数据变化**:**无**
- **来源**:沿用既有(NFR-7 严守)

## 5. 非功能需求 / 约束(NFR)

- **可维护性**:FR-1 ~ FR-5 全部为"渲染层压缩",算法 1/2/3/4/5 字节级保留(NFR-1 强约束)
- **兼容性**:
  - 与既有 REQ-00023 改造兼容:总览模式"4 段结构"保留,仅段内行数压缩
  - 与既有 REQ-00026 通用化兼容:0 改 frontmatter / 0 改 marketplace.json / 0 改 README
  - 不改其他 10 个 `code-*` 技能 SKILL.md frontmatter(NFR-2 严守)
- **可读性**:单行信息密度提高,但仍可读(命令 + 优先级 + 依据 = 一目了然)
- **可重复性**(沿用):ASCII 比例条 + Glob/Grep 顺序稳定 + 无时间戳 = 多次执行输出 byte-for-byte 一致
- **可解析性**:单行格式更易被 CI / awk / grep 提取(用户场景 1.2)

## 9. 边界与异常

### L1 启动错误(必须退出,无副作用)
- **E-1**:无 `.current-version` → 屏显错误 + 引导调 `code-version` + 退出
- **E-2**:版本工作空间不存在 → 屏显错误 + 引导 + 退出
- **E-3**:需求模式 + `require/<REQ>/` 不在列表 → 屏显错误 + 列出本版本所有需求 + 退出
- **E-4**:参数格式错 → 屏显用法 + 退出
- **E-5**:看板文件缺失 → 屏显错误 + 引导调 `code-version` + 退出

### L2 数据错误(可降级)
- **E-6**:区段缺失 → 屏显 `(无)`(沿用既有)
- **E-7**:表格列错位 → 退化到原始 markdown 块原样输出(沿用既有)

### L3 异常兜底
- **E-8**:内部异常 → 屏显 `✗ 内部错误: <msg>` + 退出(沿用既有)

### 退化场景

| 场景 | 屏显(改后) |
| --- | --- |
| 无需求无缺陷 | `(无需求无缺陷,无需进度)`(单行) |
| 5 类状态全 0 | `状态: (无)`(单行) |
| 高优缺陷全 0 | `P0 待修复: ░ 0 | P1 待修复: ░ 0`(1 行) |
| 5 类建议无触发 | `> 无后续动作`(1 行) |
| 命中 > 5 条建议 | 屏显前 5 条 + `> 另有 N 条相关建议,可通过执行 `/code-require` 补录` 触发 |

## 10. 验收标准(AC)

### AC-1:无副作用验证
- **对应需求**:FR-8
- **验证方式**:自动化(检查 `git status`)

- **优先级**:高

- **优先级**:高

### AC-3:5 类状态占比段压缩为单行
- **对应需求**:FR-2

### AC-4:建议段单行格式
