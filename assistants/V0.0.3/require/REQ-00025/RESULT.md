## 1. 需求概述

**为谁**:`code-skills` 用户(本项目自身使用者)+ 通过第三方平台(Jira / 禅道 / Redmine / GitLab 等)接入需求 / 任务 / 缺陷的工作流。

**解决什么问题**:当前 `encoding-conventions.md` 强制 3 类编码格式(`REQ-NNNNN` / `BUG-NNNNN` / `TASK-(REQ|BUG)-NNNNN-NNNNN`)必须 5 位纯数字,过于严苛,无法承接第三方平台生成的非纯数字编号(如 `REQ-V0.0.1.001` / `JIRA-1234` / `JIRA-456`)。

**带来什么价值**:保持 `code-skills` 体系**生成逻辑**不变(5 位纯数字作为默认),同时**放宽接收逻辑**(允许任何前缀 + 任意后缀的编号),第三方平台只需保证前缀部分一致(`REQ-` / `BUG-` / `TASK-` + 父级前缀)。

---

## 2. 背景与目标

- **背景**:
  - 当前 `encoding-conventions.md` §规则 1 强约束:`^REQ-\d{5}$` / `^BUG-\d{5}$` / `^TASK-(REQ|BUG)-\d{5}-\d{5}$`
  - §规则 2:5 位固定宽度 + 前导补零
  - §规则 4:解析入口必须使用上述正则
  - 实际场景:用户可能用 Jira / 禅道 / 内部平台生成 `PROJ-123` / `REQ-2025-Q4-001` / `JIRA-456` 等编号
  - 限制:用户希望"前缀部分(REQ-/BUG-/TASK-)保持规则,后缀部分自由"
- **业务目标**:
  - **生成逻辑** = 5 位纯数字(沿用,默认)
  - **接收逻辑** = 仅校验前缀(REQ-/BUG-/TASK-),后缀自由
  - 5 个相关技能 `code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-fix` / `code-check` / `code-dashboard` 全部放宽接收端正则
  - **不**改 `code-init` / `code-version` / `code-rule` / `code-publish` / `code-auto`(非编号相关)
- **本次目标**:本轮交付:
  1. `encoding-conventions.md` §规则 1 + §规则 2 + §规则 4 软化条款
  2. 5 个 `code-*` 技能 SKILL.md 中所有"解析 / 校验"编号的正则放宽
  3. 既有 5 位纯数字编号继续工作(向后兼容,无破坏性)

## 3. 用户角色与场景

### 角色 1:`code-skills` 用户(本项目开发者)
- **场景 A — 沿用 5 位纯数字**:
  - 作为本项目开发者,我想要继续使用 `REQ-00001` / `BUG-00001` 编号,新规则不破坏既有行为。
  - **前置条件**:无
  - **主流程**:用户调 `code-require REQ-00025` → 解析为"5 位纯数字"格式 → 走既有逻辑

### 角色 2:第三方平台接入用户
- **场景 B — Jira 编号**:
  - 作为接入 Jira 的项目用户,我想要在 `code-skills` 中使用 `JIRA-123` 作为需求编号,无需先在 `code-skills` 内手动改名为 `REQ-00123`。
  - **前置条件**:用户需在 `./assistants/rules/encoding-conventions.md` 中显式登记"Jira 编号前缀 = `JIRA-`"(或新规则"其他平台"允许的 3 类前缀 REQ-/BUG-/TASK-+ 父级)
  - **主流程**:用户调 `code-require JIRA-123` → 解析为"非 5 位纯数字"格式 → 按新规则处理(后缀自由)
- **场景 C — 跨平台 JIRA 数字编号**:
  - 用户调 `code-require JIRA-0001` → 同上(数字后缀仍可解析)
  - 用户调 `code-require JIRA-V0.0.1.001` → 同上(`V0.0.1.001` 后缀自由)
- **场景 D — 任务编码**:
  - 用户调 `code-it JIRA-123-456` → 解析为"父级=JIRA-123, 任务序号=456" → 任务创建
  - 用户调 `code-it JIRA-V0.0.1.001-456` → 同上(任务序号部分自由)

### 角色 3:AI 协作者(Claude Code 模型层)
- **场景 E — 解析失败兜底**:
  - 收到非法格式编号(既不匹配 `^REQ-` / `^BUG-` / `^TASK-` 任意前缀,也不在 `require/<input>/` / `fix/<input>/` 目录)
  - 屏显 `⚠ 编号格式不合法: <input>(不在 REQ-/BUG-/TASK- 任何前缀下)`,提示用户用 AskUserQuestion 确认

## 4. 功能需求(FR)

### FR-1:解析端正则放宽(5 个 code-* 技能)

- **描述**:`code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` / `code-fix` / `code-dashboard` 中,所有"解析用户输入编号" / "校验已有编号" / "生成子路径" 的正则,放宽"前缀必须 5 位纯数字"约束,改为"前缀必须 3 类之一(REQ-/BUG-/TASK-)+ 后缀自由"。
- **入口**:8 个相关技能 SKILL.md
- **前置条件**:无
- **主流程**:
  1. **新正则约束**(统一为下表,所有技能沿用):
     | 维度 | 旧正则 | 新正则 |
     | --- | --- | --- |
     | 需求编号 | `^REQ-\d{5}$` | `^REQ-[A-Za-z0-9.\-_]+$`(前缀 `REQ-` + 后缀 1+ 个字母数字 / `.` / `-` / `_`) |
     | 缺陷编号 | `^BUG-\d{5}$` | `^BUG-[A-Za-z0-9.\-_]+$` |
     | 任务编号(REQ) | `^TASK-REQ-\d{5}-\d{5}$` | `^TASK-REQ-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$` |
     | 任务编号(BUG) | `^TASK-BUG-\d{5}-\d{5}$` | `^TASK-BUG-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$` |
  2. **生成端不变**:所有 5 个 `code-*` 技能继续按"5 位纯数字前导补零"生成(沿用既有 INV)
  3. **既有编号继续可用**:`REQ-00001` / `BUG-00001` / `TASK-REQ-00001-00001` 等全部通过新正则(向后兼容)
- **分支/异常**:
  - 编号后缀含非法字符(如 `!@#$`)→ 沿用既有"非法字符"处理(屏显 `⚠ 编号含非法字符`)
  - 编号后缀为空(如 `REQ-` / `BUG-` / `TASK-REQ--1`)→ 屏显 `⚠ 编号后缀为空: <input>`
- **数据变化**:**不**新增字段,**不**改字段语义(仅放宽约束)
- **来源**:用户原始描述第 1-2 段

### FR-2:`encoding-conventions.md` 软化条款

- **描述**:`encoding-conventions.md` §规则 1 + §规则 2 + §规则 4 软化:**"默认生成"** 与 **"接收"** 分开。
- **入口**:`./assistants/rules/encoding-conventions.md`
- **前置条件**:无
- **主流程**:
  1. **§规则 1 修订**:从"权威源定义 3 类格式"改为"权威源定义 3 类**生成格式**(默认) + 3 类**接收格式**(放宽)"
     - 旧:需求 = `REQ-NNNNN`(`^REQ-\d{5}$`),容量 99999
     - 新:需求 = `REQ-<1+位后缀>`(`^REQ-[A-Za-z0-9.\-_]+$`),容量"按后缀长度"(无限制,仅目录名长度限制)
  2. **§规则 2 修订**:从"5 位纯数字固定"改为"生成端 5 位纯数字 + 接收端任意 1+ 位后缀"
  3. **§规则 4 修订**:
     - 旧:`code-plan` / `code-it` / `code-unit` / `code-check` 解析任务编码时**必须**使用 `^TASK-(REQ|BUG)-\d{5}-\d{5}$`
     - 新:解析时**优先**用 `^TASK-(REQ|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$`,**生成**仍用 5 位
  4. **新增 §规则 1.5**:**第三方平台接入指南**(说明如何登记新前缀,如 `JIRA-`、`TAPD-` 等)
- **分支/异常**:
  - 用户未读软化条款直接调 `code-require PROJ-123` → 旧逻辑下报错(前缀 `PROJ-` 不在 REQ-/BUG-/TASK- 之列),新逻辑下屏显"不在登记前缀列表中"并提示调 `code-rule` 登记
- **数据变化**:文档修订,0 字段新增
- **来源**:用户原始描述第 1 段

## 7. 交互逻辑

### 交互 1:`code-require <编码>` 解析

- **触发**:用户调 `code-require <编码>`
- **流程**:
  1. 读取 `<编码>` 字符串
  2. 校验前缀:
     - 以 `REQ-` 开头 → 视为需求
     - 以 `BUG-` 开头 → 视为缺陷
     - 以 `TASK-REQ-` 开头 → 视为需求任务
     - 以 `TASK-BUG-` 开头 → 视为缺陷任务
     - 其他 → 屏显 `⚠ 前缀 <X>- 未登记,请先调 code-rule 登记或使用 REQ-/BUG-/TASK- 之一`
  3. 提取后缀(`<前缀>` 之后的所有字符)
  4. 校验后缀字符集:`A-Za-z0-9.\-_` 1+ 位
     - 命中 → 进入既有需求 / 缺陷 / 任务流程
     - 未命中(空 / 含非法字符)→ 屏显 `⚠ 编号后缀非法: <后缀>`
- **状态机**:
  ```mermaid
  stateDiagram-v2
    [*] --> ParseInput: 用户调 code-require <input>
    ParseInput --> ExtractPrefix: 提取前缀部分
    ExtractPrefix --> ReqPrefix: 前缀 = REQ-
    ExtractPrefix --> BugPrefix: 前缀 = BUG-
    ExtractPrefix --> TaskReqPrefix: 前缀 = TASK-REQ-
    ExtractPrefix --> TaskBugPrefix: 前缀 = TASK-BUG-
    ExtractPrefix --> OtherPrefix: 其他前缀
    OtherPrefix --> [*]: 屏显 + 退出(未登记)
    ReqPrefix --> ValidateSuffix: 校验后缀
    BugPrefix --> ValidateSuffix
    TaskReqPrefix --> ValidateSuffix
    TaskBugPrefix --> ValidateSuffix
    ValidateSuffix --> PathDetect: 后缀合法 → 进入 REQ-00024 路径感知
    ValidateSuffix --> [*]: 后缀非法(屏显 + 退出)
  ```
- **关键规则**:
  - 解析端只校验"前缀 + 后缀字符集",**不**校验"5 位纯数字"
  - 生成端仍生成 5 位纯数字(沿用 INV)
  - 屏显保留完整编号(不截断)

### 交互 2:`code-rule` 登记新前缀(FR-6 配套)

- **触发**:用户调 `code-rule` 登记新前缀
- **流程**:
  1. 用户输入"前缀名" + "等价类别"(REQ-/BUG-/TASK- 之一)
  2. `code-rule` 追加 `encoding-conventions.md` §规则 1.5 一行:

     ```
     JIRA- → 等价 REQ-(2026-06-07 登记)
     ```
  3. 后续 `code-require JIRA-123` 自动识别为"需求"

## 9. 边界与异常

- **极端输入**:
  - 编号含 `..` 路径遍历(如 `REQ-../etc/passwd`)→ 后缀字符 `..` 是允许的,但 OS 文件系统层会拒绝创建 → 屏显 `⚠ 文件系统拒绝: <path>`
  - 编号含 `/` 路径分隔(如 `REQ-a/b`)→ 后缀正则排除 `/` → 屏显 `⚠ 编号含非法字符: /`
  - 编号超长(> 200 字符)→ 接受(OS 文件系统层有 255 字符限制,大数字才触发)
  - 编号后缀全 0(如 `REQ-0000`)→ 接受(后缀"0000"是合法 4 位字符)
- **文件系统异常**:
  - `<编码>` 包含 OS 不支持的字符(如 Windows 的 `<>|:*?`)→ 屏显 `⚠ 文件系统不支持字符: <char>`
- **并发场景**:
  - 路径检测 + `mkdir` 之间有竞态(理论可能)→ 本需求**不**修复(沿用既有"无并发控制"语义)
- **登记缺失**:
  - 第三方平台前缀未在 `code-rule` 登记 → 屏显 `⚠ 前缀 <X>- 未登记` + 引导调 `code-rule`
- **第三方依赖**:无
