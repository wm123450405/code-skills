## 关键决策

### D-1:生成 vs 接收分离

- **决策**:生成端继续 5 位纯数字,接收端放宽前缀+后缀两段式
- **理由**:
  - 生成端:本仓库 `code-require` / `code-plan` 等自动生成,5 位纯数字是默认值
  - 接收端:第三方平台生成的编号无规律(可含 `.` / `-` / 字母),仅校验前缀
  - 2 段式解析:`id-prefix` + `id-suffix`(后缀字符集限定 `A-Za-z0-9.\-_`)
- **影响**:0(纯加法,既有 5 位继续可用)

### D-2:后缀字符集 `A-Za-z0-9.\-_`

- **决策**:后缀只允许字母 / 数字 / `.` / `-` / `_`
- **理由**:
  - 排除 `/` / `\` 防止路径遍历漏洞
  - 排除 `!@#$%^&*()` 避免 OS 文件系统差异
  - 覆盖典型场景:`00001` / `V0.0.1.001` / `2025-Q4-001` / `XYZ_123`
- **影响**:0 既有编号受影响(纯数字 + `.` / `-` / `_` 全部允许)

### D-3:第三方平台前缀登记流程

- **决策**:用户调 `code-rule` 登记新前缀(如 `JIRA- → 等价 REQ-`)
- **理由**:
  - 不在 `code-rule` 中**新增**契约(本需求 NFR 强约束)
  - 既有 `code-rule` 追加规范时,只需在 `encoding-conventions.md` §规则 1.5 写 1 行
  - 沿用既有"登记即生效"模式
- **影响**:0 既有流程破坏

### D-4:8 个相关技能 SKILL.md 字面更新

- **决策**:仅更新 8 个相关技能的 SKILL.md(不更新 `code-init` / `code-version` / `code-rule` / `code-publish` / `code-auto`)
- **理由**:
  - 8 个技能 = `code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` / `code-fix` / `code-dashboard`
  - 其他 5 个技能**不**涉及编号解析
  - 沿用 NFR-6 强约束
- **影响**:0

### D-5:屏显保留完整编号(不截断 30 字符)

- **决策**:标题若 > 30 字符仍按 `truncateTitle` 截断;编号本身保留完整(不截断)
- **理由**:
  - 30 字符限制是针对**标题**的可读性约束,不是**编号**的合规约束
  - 编号本身需要完整保留(便于审计 + 跨平台追溯)
- **影响**:0
