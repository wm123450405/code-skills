# 需求提示词文档 — 优化 /code-plan 任务粒度(内化编译/运行,外移单元测试)

- 需求编码:REQ-00031
- 需求标题:优化 /code-plan 任务粒度(内化编译/运行,外移单元测试)
- 所属版本:V0.0.3
- 需求状态:已完成(需求分析)
- 创建:2026-06-12 15:13
- 最近更新:2026-06-12 15:18

> 本文件由 `code-require` 在 `code-auto` 编排下产出,作为 `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` 的共同基线。

## 1. 需求概述

优化 `/code-plan` 拆分任务时的两条核心规则:

1. **任务完成定义内化编译/运行**:每个任务**自身**必须保证"编译成功"**或**"运行成功"(按任务性质二选一),不依赖后续任务/独立"编译运行检测"任务兜底;`/code-plan` 不再产出独立的"编译运行检测"任务。
2. **任务规划不包含单元测试部分**:`/code-plan` 只规划"编码 + 编译/运行验证",**不**规划"单元测试编写"或"单元测试执行"任务;`/code-unit` 是**独立、可选**的技能(由项目/用户按需在 `code-it` 之后自行调用),其职责是"编写单元测试 + 执行测试",与 `/code-it` 职责正交。

**关键边界(用户原文,2026-06-12 15:18)**:
- `/code-unit` 的作用**就是**编写单元测试 + 执行测试,功能独立清晰
- `/code-it` 的作用**就是**编码 + 确保正常编译/运行,功能独立清晰
- `/code-plan` 任务规划**只规划编码部分**,不包含规划单元测试部分
- 单元测试技能**根据不同项目可选使用**,不由 `code-plan` 强制

## 2. 背景与目标

### 2.1 背景(沿用 REQ-00030 + 本次新增)

V0.0.2 → V0.0.3 期间,`code-plan` 在任务拆分上存在以下痛点:

- **任务粒度与"完成定义"不一致**:既有任务"完成定义"沿用 V0.0.2 的"`code-it` 末尾兜底后做编译/运行/测试",但 `code-it` 步骤 9-12 已强制每任务必须迭代修复到"编译通过 / 启动正常 / 测试通过",导致 `code-plan` 既未在任务"完成定义"中显式声明"编译/运行成功"是任务的交付物,又可能在任务清单中混入"编译运行检测"独立任务 — 双重冗余。
- **"测试"语义扩散**:任务"双状态"中"测试状态"字段含 `未编写 / 已编写 / 已运行-通过 / 已运行-失败` 4 个粒度,容易被误解为"任务必须自带单元测试";但本仓库(纯 Markdown 元技能仓库)无测试框架,`code-unit` 调不到测试命令 → `code-auto` 步骤 4 中"code-unit 是否调用"判 "测试需要=Y" 经常是 false 退化路径。
- **`/code-unit` 职责与 `/code-it` 模糊**:既有时序是"code-it → code-unit → code-check",但 code-plan 拆任务时**也**可能规划"测试任务"或"测试补齐"任务 → 三方责任分散。

### 2.2 目标

- **目标 1**:`code-plan` 任务粒度原则中,任务"完成定义"显式含"编译/运行成功";不产出独立"编译运行检测"任务。
- **目标 2**:`code-plan` 不再产出"单元测试编写 / 单元测试执行 / 测试补齐"等任务类型;`code-plan` 任务"双状态"中的"测试状态"字段**对所有任务统一收窄为 `不适用`**,由 `code-unit`(可选)或 `code-check` 在 `code-it` 之后**另起流程**判定。
- **目标 3**:`/code-it` 职责文档显式声明"本技能**只**保证编译/运行成功,不含单元测试";`/code-unit` 职责文档显式声明"本技能是**独立、可选**的技能,由项目/用户按需调用,非 `code-plan` 任务的强制环节"。

### 2.3 非目标

- **不**改 `code-unit` 内部行为(步骤 1-15 字节级保留)
- **不**改 `code-check` 评审清单(8.7 测试质量维度字节级保留)
- **不**改既有 11 个 `code-*` 技能(除 `code-plan` / `code-it` / `code-unit` / `code-auto` 4 个被改技能)
- **不**重写 `code-plan` 全部步骤,只修订任务粒度相关章节

## 3. 用户角色与场景

### 3.1 角色

- **元技能维护者**(本需求主要受益方):希望 `code-plan` 任务粒度清晰、职责单一,不混"测试"语义
- **项目开发者**:在 `code-it` 之后,根据项目需要**自行**决定是否调 `code-unit`(纯单元测试项目必调,纯元技能仓库不调)
- **`code-auto` 编排者**:任务循环步骤 4 中,本需求修订后**永不再调** `code-unit`(本需求"code-plan 不再产出测试任务" → 任务清单"测试状态"全为 `不适用` → `code-auto` 步骤 4.b 永远跳过)

### 3.2 场景

| 场景 | 触发 | 期望行为 |
| --- | --- | --- |
| 元技能改动 REQ(纯 Markdown) | `code-plan` 拆 5 任务,全为 SKILL.md 改动 | 任务清单**不**含"测试"类任务;`code-auto` 任务循环**不**调 `code-unit` |
| 业务项目 REQ(有 Python/JS 测试框架) | `code-plan` 拆 8 任务,含 1 个测试任务 | 任务清单**仍不**含"测试"任务(由用户在 `code-it` 完成后**自行**调 `code-unit`);`code-auto` 任务循环**不**调 `code-unit`(`code-plan` 已不产出) |
| 旧 REQ(RE-00001 ~ REQ-00030)重跑 | `code-plan` 增量更新 PLAN.md | 既有任务"测试状态"统一收窄为 `不适用`(NFR-2 沿用) |
| `code-check` 评审 | 任务"双状态"显示"测试=`不适用`" | 不再派生"测试补齐"任务(`code-check` §8.7 字节级沿用 — `不适用` 不派生任务) |
