# 分析笔记 — REQ-00033

版本:V0.0.3
需求编码:REQ-00033
时间:2026-06-15 11:10

## 1. 需求本质判定

用户的输入是 1 段**技能职能边界澄清**:
- `code-require` = 需求澄清(只确定功能点)
- `code-design` = 概要设计(确定技术选型)
- "没有必要进行技术选型的需求无需分析技术选型选项" = **反向确认** `code-require` 的零选型责任

**判定**:本需求是**职责边界显式化**(契约化),**非**新功能 / 新模块 / 跨版本需求。

## 2. 现状分析

### 2.1 隐含 vs 显式

| 维度 | 现状 | 本需求后 |
| --- | --- | --- |
| code-require 是否涉及技术选型 | **隐含不涉及**(0 命中;由 §目标 段"需求提示词文档"语义推导) | **显式不涉及**(§目标 / §工作流程 段内 1 行硬约束) |
| code-design 是否涉及技术选型 | **显式涉及**(模板含 3+ 字段) | 保持不变 |
| 跨技能边界 | 隐含 | 显式(双向:`code-require` 不做 / `code-design` 做) |

### 2.2 改造点候选(本需求唯一改造对象 = `code-require/SKILL.md`)

候选 1:**§"目标" 段内追加 1 行**
- 优点:位置最显眼(读者首段就看到)
- 缺点:"目标"段语义偏泛,放"边界"语义略偏

候选 2:**§"工作流程 → 步骤 5A 首次分析" 段内追加 1 行**
- 优点:与"提取候选需求"段同位置,直接约束提取范围
- 缺点:本需求不止约束"提取",也约束"输出"与"判据"

候选 3:**§"不要做的事" 小节追加 1 条**
- 优点:沿用 BUG-00001 范式(同属"职责边界显式化");语义最清晰
- 缺点:本需求**不是**"不要做",而是"范围外",措辞需微调(如"## 范围外"或"## 不涉及")

候选 4:**新增"## 范围外"小节(独立于"## 不要做的事")**
- 优点:语义最精准(范围外 ≠ 不要做)
- 缺点:新增小节,净增行数最多

**推荐**:候选 3(沿用既有 §"不要做的事" 范式,只改 1 行的措辞)——见 RESULT.md FR-2 锁定

## 3. 候选 3 的措辞建议

候选 3 措辞 1(强制式):
> `code-require` **不**涉及技术选型 / 技术栈 / 技术方案的确定;技术选型归 `code-design`;本技能仅确定功能点(FR / NFR / 页面 / 交互 / 数据 / 边界)。

候选 3 措辞 2(简短式):
> `code-require` 输出 = 需求提示词文档(**不**含技术选型 / 技术栈 / 技术方案);技术选型归 `code-design`。

候选 3 措辞 3(列表式):
> `code-require` **不**产出以下内容(职责归 `code-design` / `code-plan`):
> - 技术选型(语言 / 框架 / 库 / 存储 / 中间件)
> - 架构风格(分层 / 微服务 / 事件驱动 / 六边形)
> - 接口风格(REST / GraphQL / gRPC / 消息队列)
> - 数据模型(库表 / 字段类型 / 索引)
> - 部署形态(单机 / 集群 / 容器 / K8s)

**推荐**:措辞 1(强制式)—— 见 RESULT.md FR-2 锁定
