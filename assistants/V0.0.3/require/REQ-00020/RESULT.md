  > 2. 优化 `/code-plan` 技能的架构设计目标增加封装性、可复用性目标,用于归并相同、相似、相近的代码逻辑,通过复用性减少冗余代码量

## 1. 需求概述

**为谁**:`code-skills` 仓库的 AI 协作者 + 项目主导者(在 `code-design` / `code-plan` / `code-it` 实际产出设计/计划/代码时,期望"按用户的真实需求定位架构目标,避免 AI 自由裁量过度,减少 token 浪费")。

**解决什么问题**(4 个子痛点):
- **P-1 职责错位**:`code-design` 概设阶段问"扩展性 / 可维护性 / 健壮性"等架构目标,但这些目标本应属于**详设阶段**(`code-plan`)的关切。概设阶段关注的是"系统长什么样",过早问"未来怎么扩"会让 AI 倾向过度设计。
- **P-2 缺封装/可复用**:现有"## 设计目标"小节只有 4 维度(功能性 / 扩展性 / 健壮性 / 可维护性),**没有**"封装性 / 可复用性 / 可读性"。导致 AI 写代码时倾向复制粘贴,产生大量重复代码。
- **P-3 缺可读性显式目标**:对非自然语言编程语言(如强类型 / 编译型),AI 倾向写"代码即注释",但**缺少显式的"可读性"目标**导致关键逻辑无注释。
- **P-4 token 浪费**:`code-plan` / `code-it` SKILL.md 经过 REQ-00005 / 00010 / 00011 / 00019 / 00013 多轮叠加,出现**大量相同/相似的执行步骤**和**多余的逻辑分支**,在每次技能调用时都被加载,浪费大量 token。

**带来什么价值**:
1. **职责正位**:`code-design` 只问"功能性"1 个目标(纯业务维度);架构目标(扩展性 / 健壮性 / 可维护性 / **封装性 / 可复用性 / 可读性**)统一下沉到 `code-plan` 阶段。
2. **新增 3 维度**:`code-plan` "## 设计目标"小节从 4 维度扩为 7 维度(原 4 + 封装性 / 可复用性 / 可读性),用户可显式选择优先级。
3. **归并相似步骤**:`code-plan` / `code-it` SKILL.md 中相同/相似的步骤合并,逻辑分支收敛,token 消耗下降。
4. **完全可独立执行**:不依赖 `code-auto` / `code-review` 等其他技能;沿用 REQ-00005 / 00010 / 00011 / 00019 既有步骤命名规范。
