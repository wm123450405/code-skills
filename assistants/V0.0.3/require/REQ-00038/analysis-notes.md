# 分析笔记 — REQ-00038

更新时间:2026-06-22 13:00

## 1. 核心痛点(用户原始输入)

> 现有的 `/code-it` 技能在判断是否需要编写单元测试的逻辑上存在遗漏,有些工程下的子模块需要单独编写单元测试;优化 `/code-it` 技能,判断逻辑不能以整个工程为为主,而是要判断改修的代码属于哪个模块的,单独判断模块是否应该编写单元测试。

**关键短语**:
- "有些工程下的子模块" → **多模块工程 / monorepo**
- "需要单独编写单元测试" → **单测写到子模块,不是工程根**
- "判断逻辑不能以整个工程为为主" → **粒度从工程 → 模块**
- "单独判断模块是否应该编写单元测试" → **模块级守卫**

## 2. 现状分析(沿用 REQ-00034 落地结果)

### 2.1 `code-it` 步骤 8a 工程根级守卫(7 项)

| 序号 | 文件/目录 | 检查位置 |
| --- | --- | --- |
| 1 | `package.json` 含 `scripts.test` | CWD 根 |
| 2 | `pyproject.toml` 含测试配置 | CWD 根 |
| 3 | `Cargo.toml` | CWD 根 |
| 4 | `go.mod` | CWD 根 |
| 5 | `pom.xml` | CWD 根 |
| 6 | `build.gradle` / `build.gradle.kts` | CWD 根 |
| 7 | `test/` 目录 | CWD 根 |

**问题**:monorepo 工程(根级无 `package.json` 的 `scripts.test` 或根级 `package.json` 仅有 `private: true` 无测试)→ 7 项全不命中 → 守卫不通过 → 整工程跳过单测 → **子模块的 `packages/*/package.json` 含 `scripts.test` 也被忽略**。

### 2.2 `code-it` 步骤 8.5 工程根级单测输出

**问题**:
- 单测写到 CWD 下的项目测试目录(通常 `test/` / `__tests__/` / `src/` 同包)
- monorepo 下所有子模块的单测挤到工程根 → **模块边界破坏**
- 多语言工程下(Node.js + Go + Python 混合)→ 单测框架混乱

## 3. 优化方案(用户 1 轮澄清后定稿)

### 3.1 模块识别(FR-1)

**模块识别优先级链**(从高到低):
1. **声明文件**(monorepo workspace 声明):
 - `pnpm-workspace.yaml`(`packages` / `pnpm-workspace`)
 - `package.json` 的 `workspaces` 字段(NPM / Yarn workspace)
 - `lerna.json` 的 `packages` 字段
 - `nx.json` / `turbo.json` 的 workspace 配置
 - `pom.xml` 的 `<modules>` 字段(Maven 多模块)
 - `Cargo.toml` 的 `[workspace] members` 字段
 - `go.mod` 的 `module` 路径 + 子目录(Go 不原生支持 monorepo,需约定)
 - 等等
2. **git diff 退化**:无声明文件 → 取变更文件路径的最长公共子目录
3. **CWD 根退化**:无变更路径(理论不可能)→ 退化为 CWD 根(原 REQ-00034 行为)

**输出**:`modules: string[]`(模块路径列表,相对 CWD)

### 3.2 模块可测性守卫(FR-2,沿用原 7 项)

每个模块独立执行 7 项检查(在模块目录而非 CWD 根):

| 序号 | 文件/目录 | 检查位置 |
| --- | --- | --- |
| 1 | `package.json` 含 `scripts.test` | **模块目录** |
| 2 | `pyproject.toml` 含测试配置 | **模块目录** |
| 3 | `Cargo.toml` | **模块目录** |
| 4 | `go.mod` | **模块目录** |
| 5 | `pom.xml` | **模块目录** |
| 6 | `build.gradle` / `build.gradle.kts` | **模块目录** |
| 7 | `test/` 目录 | **模块目录** |

**判定逻辑**:
- 至少 1 个模块通过 → 整体 `testable = True` → 进入步骤 8.5
- 全部模块不通过 → 整体 `testable = False` → 跳过步骤 8.5 → 任务"测试状态"列 = `不适用`

### 3.3 单测输出位置(FR-3,模块级测试目录)

**测试目录识别优先级链**(每个通过的模块独立):
1. 模块内 `package.json#scripts.test` 含框架配置 → 沿用 `testMatch` / `testRegex`
2. 模块内 `pyproject.toml` 含 `[tool.pytest.ini_options]` → 沿用 `testpaths`
3. 模块内 `Cargo.toml` → 约定 `src/` 同包 `#[cfg(test)]` 或 `tests/`
4. 模块内 `go.mod` → 约定同包 `*_test.go`
5. 模块内 `pom.xml` / `build.gradle` → 约定 `src/test/`
6. 无约定 → 模块根 `test/` 目录
7. 仍无 → CWD 根 `test/` 目录(原 REQ-00034 行为)

**判定逻辑**:
- 多个模块通过 → 每个模块分别写单测到各自测试目录
- 1 个模块通过 → 该模块测试目录
- 全部不通过 → 跳过(已由步骤 8a 判定)

### 3.4 `unit-test-results.md` 多模块支持(FR-4)

- 模板新增"## 各模块单测结果"小节
- 每个通过的模块独立记录:
 - 模块路径
 - 测试框架
 - 守卫检查详情
 - 新增/修改的测试文件
 - 跑通情况
- 沿用原"## 单元测试(由 code-it 内化)"小节
- 单一通过的模块 → 单模块记录
- 全部不通过 → 占位"本任务不涉及单元测试(模块级守卫不通过)"

## 4. 边界与异常

### 4.1 退化路径

- **声明文件不存在** → 退化为 git diff 公共子目录
- **git diff 失败(非 git 仓库)** → 退化为 CWD 根
- **变更路径跨越多个模块** → 多个模块分别写单测
- **模块目录无法访问(权限/不存在)** → 跳过该模块,屏显警告
- **多模块通过但只有一个有变更** → 只给该模块写单测(其他模块占位)

### 4.2 性能

- 模块识别:O(M),M = 模块数(典型 1-20,<< 1 秒)
- 守卫检查:O(M × 7),每个模块 7 项检查(Glob + Read 验证)
- 总耗时:< 2 秒(实际 0.1-0.5 秒,monorepo 100 模块下 < 2 秒)

### 4.3 与既有约束的兼容

- **不修改 frontmatter**:`code-it/SKILL.md` L1-3 字节级保留
- **不修改既有"## 工作流程"小节**:沿用既有结构,仅在步骤 8a / 8.5 内部扩展
- **不修改"## 不要做的事"小节**:沿用既有 N-1 ~ N-N
- **不触发 `AskUserQuestion`**:本技能沿用 NFR-3 强约束,模块识别全部自动判定
- **不新增 CLI 参数**:模块识别完全自动(无 `--module` 等)
- **沿用 REQ-00034 字节级沿用**:`code-unit` 退场后所有单测能力通过 `code-it` 接管,模块级细化是 `code-it` 内部的扩展

## 5. 候选方案权衡(沿用概设 §6 待补充)

| 维度 | 工程根级(原) | 模块级(本方案) |
| --- | --- | --- |
| 适用场景 | 单一工程 | monorepo / 多语言 / 多模块 |
| 守卫粒度 | 整工程 | 每模块独立 |
| 单测输出 | 工程根 `test/` | 各模块独立测试目录 |
| 性能 | < 1 秒 | < 2 秒(可接受) |
| 复杂度 | 低 | 中 |
| 兼容性 | 100% 沿用原 REQ-00034 | 100% 兼容 + 扩展 |
| 误判风险 | monorepo 子模块被忽略 | 几乎无(每模块独立判定) |
