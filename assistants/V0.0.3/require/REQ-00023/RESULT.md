# 需求提示词文档 — REQ-00023(简化 code-dashboard 技能的输出,仅展示当前版本的总开发进度 + 状态占比 + 后续操作建议)

- 需求编码:REQ-00023
- 所属版本:V0.0.3
- 创建时间:2026-06-07
- 状态:待分析 → 分析中 → 待评审

## 1. 需求概述

**简化 `/code-dashboard` 技能的输出,仅展示当前版本的总开发进度(将需求、缺陷各阶段都记录到进度中),以及各状态下需求、缺陷数量占比,每种状态下只展示一条推荐的后续操作建议。**

当前 `code-dashboard` 屏幕输出(总览模式)有 4 段:
- 段 1:需求进度(状态枚举 + 12 字符比例条 + 完成率)
- 段 2:任务进度(总览 + 测试通过数 + 真正可发布数 + 比例条)
- 段 3:缺陷(高优先级)(P0 / P1)
- 段 4:下一步建议(最多 5 条,3 类优先级 + 1 特殊)

本需求改造方向:
1. **段 1 改造**:`需求进度` 段重定义为"**总开发进度**"(加权平均,需求 6 步 / 缺陷 5 步,不适用的环节按已完成计算)
2. **段 2 改造**:`任务进度` 段重定义为"**各状态下需求、缺陷数量占比**"(5 类状态:待概要设计 / 待详细设计 / 待代码开发 / 待单元测试 / 待代码评审)
3. **段 3 保留**:缺陷(高优先级)段保留(P0 / P1 醒目标记),但只列 P0 / P1 数量,**不**展示具体缺陷内容
4. **段 4 改造**:`下一步建议` 段由"最多 5 条"收紧为"**每种状态 1 条推荐,5 类状态共 5 行**",**不**超过 5 条

## 2. 背景与目标

### 2.1 背景
- 当前 `code-dashboard` 输出信息密度较高(4 段共 10+ 行),用户每次会话开头调用时需要 1-2 秒扫读
- 用户希望聚焦"还差什么"——即**总进度 + 5 类状态占比 + 下一步操作**
- 当前"下一步建议"基于 3 类优先级(高/中/低)+ 1 类特殊,信息过散,用户希望按"状态"维度直接对齐

### 2.2 目标
- **G-1**(主):总开发进度可一眼看明白 — 需求 + 缺陷各阶段加权的百分比(用户原话:"需求分析完成进度+1,需求概设完成进度+1...")
- **G-2**(主):5 类状态下需求 + 缺陷的数量分布清楚 — 让用户知道"现在卡在哪一阶段"
- **G-3**(主):每种状态 1 条可执行命令,直接复制可执行 — 不超过 5 条,不冗余
- **G-4**(辅):保留 P0 / P1 缺陷高亮 — 阻断发布的核心信号
- **G-5**(辅):NFR-6 严守,不动其他 10 个 `code-*` 技能 SKILL.md frontmatter

### 2.3 非目标(本需求**不**做)
- 不实现跨版本汇总(留 v2 follow-up,沿用 `code-dashboard` §"不适用"段)
- 不实现 `--json` / `--filter` / `--since` 等 CLI flag(留 v2,沿用)
- 不实现"切到历史版本"能力(沿用)
- 不实现"自动登记缺陷 / 推进任务状态"(沿用,`code-dashboard` 是只读)
- 不修改 V0.0.2 引入的状态子状态 `已完成(需求分析)` → 归一化为 `已完成`(沿用 NFR-3 双格式兼容 + §"不要做的事"段)
- 不修改 `marketplace.json` / `plugin.json`(沿用 NFR-6)
- **不**对需求 / 任务的"进度"做 ORM 式自动判定 — 全部从 `RESULT.md` 表格列字面解析(沿用既有 parseTaskId 算法)

## 3. 用户角色与场景

| 角色 | 场景 | 痛点 | 期望 |
| --- | --- | --- | --- |
| **主用户**(AI 协作者 / 项目开发者) | 每次会话开头 / 长会话中段 | 当前输出信息过散,要 1-2 秒扫读才知道"还差什么" | 1 秒扫读即可知:"总进度 N% / 卡在 X 状态 / 下一步做 Y" |
| **项目负责人** | 周报 / 进度汇报 | 需要一个 0-100% 的总进度数字 + 5 类状态分布 | 数字 + 状态表,直接可贴周报 |
| **代码审查员** | 评审前快速摸底 | 想知道"还有多少任务在 X 状态" | 5 类状态占比表 |

## 4. 功能需求(FR)

### FR-1:`code-dashboard` 总览模式改为 4 段精简输出
- **总览模式**(无参数)输出顺序固定为:
  1. **总开发进度** — 单行百分比 + 12 字符 ASCII 比例条
  2. **各状态数量占比** — 5 类状态(待概要设计 / 待详细设计 / 待代码开发 / 待单元测试 / 待代码评审)+ 需求数 / 缺陷数 / 占比
  3. **高优先级缺陷** — P0 / P1 待修复数量(醒目标记)
  4. **后续操作建议** — 5 类状态各 1 条命令(共 ≤ 5 条,无触发状态不展示)
- **需求模式**(`REQ-NNNNN`)输出**不变**(沿用既有 5 段)
- **错误模式**(参数错)输出**不变**(沿用既有 E-4)

### FR-2:总开发进度计算公式(加权平均)
- **分子分母定义**:
  - 需求类(6 步):需求分析 / 概要设计 / 详细设计 / 编码 / 单元测试 / 代码评审
  - 缺陷类(5 步):缺陷登记 / 详细设计 / 编码 / 单元测试 / 代码评审
- **每条需求贡献**:每完成 1 步 +1,6 步全完成为 +6
- **每条缺陷贡献**:每完成 1 步 +1,5 步全完成为 +5
- **不适用的环节**:按已完成计入(分母 + 1,分子 + 1)
- **总进度**(百分比):
  ```
  total_done = Σ 需求已完步骤数 + Σ 缺陷已完步骤数
  total_expected = (需求数 × 6) + (缺陷数 × 5)
  总进度 = round(total_done / total_expected × 100)%
  ```
- **退化**(分母为 0,即无需求无缺陷):显示 `—` + 屏显"无需求无缺陷,无需进度"

### FR-3:5 类状态判定规则
- 状态从 `RESULT.md` 看板的"需求清单" / "缺陷清单" / "任务清单" + `plan/<REQ>/PLAN.md` 状态字面解析,**不**新增字段
- 5 类状态:
  | 状态 | 判定条件 |
  | --- | --- |
  | **待概要设计** | 需求清单"概要设计"列为 `—` 或空 + 不属于缺陷 |
  | **待详细设计** | 概要设计已完成 + 详细设计未开始(需求: `plan/<REQ>/RESULT.md` 不存在;缺陷: `fix/<BUG>/PLAN.md` 不存在) |
  | **待代码开发** | 详细设计已完成 + 任务清单有任务"开发状态≠已完成"(按 `plan/<REQ>/PLAN.md` 任务总览或 `fix/<BUG>/PLAN.md`) |
  | **待单元测试** | 任务"开发状态=已完成 ∧ 测试状态∈{未编写, 已编写, 已运行-失败}" |
  | **待代码评审** | 所有任务真正可发布(`开发状态=已完成 ∧ 测试状态∈{已运行-通过, 不适用}`)+ 评审发现汇总"必须改"未处理 |
- **缺陷路径映射**:
  - 缺陷的"待详细设计"对应 `fix/<BUG>/PLAN.md` 不存在
  - 缺陷的"待代码开发 / 单元测试 / 评审"判定同需求

### FR-4:各状态数量占比(5 类状态 × 需求/缺陷 2 类对象)
- 输出格式(每行 1 个状态):
  ```
  待概要设计: 需求 2 / 缺陷 0(占比 18.2%)
  待详细设计: 需求 1 / 缺陷 0(占比 9.1%)
  待代码开发: 需求 0 / 缺陷 1(占比 9.1%)
  待单元测试: 需求 0 / 缺陷 0(占比 0%)
  待代码评审: 需求 0 / 缺陷 0(占比 0%)
  ```
- **占比分母**:本版本内"需求 + 缺陷"总数
- **无需求无缺陷** → 该段退化显示 `(无)`
- **某状态无需求无缺陷** → 该行不展示(0 / 0 = 0%,无信息量)

### FR-5:每状态 1 条推荐后续操作(共 ≤ 5 条)
- 每类状态映射 1 条可执行命令:
  | 状态 | 推荐命令(需求路径) | 推荐命令(缺陷路径) |
  | --- | --- | --- |
  | **待概要设计** | `/code-design <REQ>` | (缺陷不经过此状态) |
  | **待详细设计** | `/code-plan <REQ>` | `/code-plan <BUG>` |
  | **待代码开发** | `/code-it TASK-REQ-...-00001`(首个未完成任务) | `/code-it TASK-BUG-...-00001` |
  | **待单元测试** | `/code-unit TASK-REQ-...-00001` | `/code-unit TASK-BUG-...-00001` |
  | **待代码评审** | `/code-check <REQ>` | `/code-check <BUG>`(缺陷走 review 子流程) |
- **展示规则**:
  - 同状态需求 + 缺陷并存 → **只展示 1 条**(需求路径优先,因为需求是主流程)
  - 同状态只有缺陷 → 展示缺陷路径
  - 状态无触发 → **该行不展示**(沿用 AC-4.4 "无后续动作" 段退化为"按状态隐藏")
- **严格 5 行上限**:即使有需求 + 缺陷并存,5 类状态最多 5 条(沿用既有 AC-4.3)

### FR-6:高优先级缺陷段(P0 / P1)保留
- 沿用既有段 3 实现,**不**简化:
  ```
  高优先级缺陷
  ────────────
  P0 待修复: █ 1
  P1 待修复: ▓ 0
  ```
- **不**展示具体缺陷标题(避免屏显膨胀)
- **不**展示 P2 / P3 数量(沿用既有 §"步骤 4 段 3" 段,只突出 P0 / P1)

## 6. 验收标准(AC)

### AC-1:总开发进度计算正确
- **AC-1.1**:1 项需求,5 步已完成(缺"代码评审"),缺陷 0 → 总进度 = 5/6 = 83%
- **AC-1.2**:1 项需求全 6 步完成,1 项缺陷 3 步完成 → 总进度 = (6+3)/(6+5) = 82%
- **AC-1.3**:1 项需求"单元测试=不适用"(纯文档),分母 = 6(不适用的环节按已完成计入)→ 总进度按 6 步算
- **AC-1.4**:分母 = 0(无需求无缺陷)→ 屏显 `— / 无需求无缺陷,无需进度`

### AC-2:5 类状态判定正确
- **AC-2.1**:需求"概要设计"列= `—` 或空 → 归入"待概要设计"
- **AC-2.2**:需求"概要设计"=已链接 + `plan/<REQ>/RESULT.md` 不存在 → 归入"待详细设计"
- **AC-2.3**:详细设计已链接 + PLAN.md 存在 + 任务清单有任务"开发状态≠已完成" → 归入"待代码开发"
- **AC-2.4**:所有任务"开发状态=已完成 ∧ 测试状态∈{未编写, 已编写, 已运行-失败}" → 归入"待单元测试"
- **AC-2.5**:所有任务真正可发布 + 评审发现"必须改"未处理 → 归入"待代码评审"

### AC-3:5 类状态数量 + 占比正确
- **AC-3.1**:每行格式 `待概要设计: 需求 N / 缺陷 M(占比 X.X%)`
- **AC-3.2**:占比 = (N + M) / (本版本需求+缺陷总数)× 100%(保留 1 位小数)
- **AC-3.3**:某状态 (N + M) = 0 → 该行不展示
- **AC-3.4**:本版本内某需求属于"待代码开发",且无任何缺陷 → 该状态行展示"需求 1 / 缺陷 0(占比 100%)"
- **AC-3.5**:占比加和 = 100%(5 行内合计 = 100%,允许 0.1% 四舍五入误差)

### AC-4:后续操作建议 ≤ 5 条
- **AC-4.1**:5 类状态全部有触发 → 5 条命令(各 1 行)
- **AC-4.2**:同状态需求 + 缺陷并存 → 1 条(需求路径优先,缺陷路径不展开)
- **AC-4.3**:某状态无触发 → 该行不展示
- **AC-4.4**:全部 5 类状态无触发 → 显示"无后续动作" 1 行(沿用既有 §"步骤 5 算法 3" 段 AC-4.4)
- **AC-4.5**:每条命令**严格**按既有 10 个 `code-*` SKILL.md frontmatter 真实语法(沿用既有 §"附录 C 建议数据结构" 段 AC-4.2)

### AC-5:高优先级缺陷段保留
- **AC-5.1**:P0 待修复 N → 屏显 `P0 待修复: █ N`
- **AC-5.2**:P1 待修复 N → 屏显 `P1 待修复: ▓ N`
- **AC-5.3**:P0 / P1 均为 0 → 屏显 `P0 待修复: ░ 0` / `P1 待修复: ░ 0`(不省略,保留骨架)

### AC-6:不修改其他 12 个 `code-*` 技能 SKILL.md
- **AC-6.1**:`git diff --name-only` 改动文件清单只含 `plugins/code-skills/skills/code-dashboard/SKILL.md` 1 个
- **AC-6.2**:其他 12 个技能 SKILL.md frontmatter L1-3 字节级保留
- **AC-6.3**:其他 12 个技能 SKILL.md 正文无变化(`git diff` 退出码 0)

### AC-7:屏显总行数符合 NFR-1
- **AC-7.1**:有需求有缺陷 + 5 类状态全部有触发 → ≤ 12 行
- **AC-7.2**:无需求无缺陷 → ≤ 6 行

### AC-8:解析幂等
- **AC-8.1**:同一版本连续调 2 次 `/code-dashboard` 输出**完全一致**
- **AC-8.2**:`RESULT.md` 无变化时,百分比 / 比例条**不**抖动
