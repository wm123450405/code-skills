- 需求编码:REQ-00035
- 所属版本:V0.0.3
- 状态:草稿
- 责任人:wm(用户)
- 创建:2026-06-15
- 最近更新:2026-06-15
- 当前版本:v1

## 1. 需求概述

当前 `code-skills` 的 9 个主流程技能(`code-require` / `code-design` / `code-plan` / `code-it` / `code-check` / `code-fix` / `code-init` / `code-version` / `code-rule` + `code-auto` / `code-dashboard` 编排)在执行时会**无条件**生成大量"过程文档"(位于 `require/` / `design/` / `plan/` / `code/` / `review/` / `fix/` 子目录的辅助 `*.md`,以及主 `RESULT.md` 内的"变更记录"小节)。

**痛点**(基于 V0.0.3 REQ-00020~REQ-00034 实测数据):
- 单个需求过程文档行数范围:**181 ~ 1840 行**(均值 ≈ 750 行)
- 折合 **~30K-60K 字符 / 需求**(约 10K-20K tokens,占完整 `code-auto` 会话 ~15-25%)
- 其中**显著比例的过程文档在特定需求类型下不涉及**:
  - `clarifications.md` 无用户问路时仅 1-2 条或空文件
  - `findings-no-task.md` 无轻微发现时常为 "0 条"骨架
  - `deviations.md` 无设计偏离时常为 "无偏离" 1 行
  - `dependencies.md` 元技能改场景(改 SKILL.md)时常空
  - `data-changes.md` 无数据库 / 持久化场景时常空
  - `test-results.md` 测试状态=`不适用` 时常只写 "不适用" 1 行
  - `risk-analysis.md` 微小需求 / 元技能改场景下常流于模板
  - `related-requirements.md` / `related-designs.md` 跨需求无关联时仅为模板骨架
  - `analysis-notes.md` 自由形式,内容质量与是否真用高度相关

**目标**:**为每个过程文档定义"不涉及/不适用"判定标准;技能启动时 AI 自主判定本轮是否生成对应过程文档;只生成真有必要的过程文档(零空白文件、零模板骨架、零"不适用"占位)。**

**用户决策要点**(来自 2026-06-15 需求澄清会):
1. **改造范围**:**所有**过程文档都重新判定(包括 `require/` / `design/` / `plan/` / `code/` / `review/` / `fix/` 子目录的辅助 `*.md` + 看板 `RESULT.md` 内的"变更记录"小节)
2. **触发方式**:**AI 自适应判定**(基于需求类型/规模/工程类型/材料),不新增 `AskUserQuestion` 不新增 CLI 参数
3. **历史处理**:**不处理历史**(V0.0.2 / V0.0.3 既有档案字节级保留),只改规则;从 REQ-00035 开始按新约定生成
4. **判定语义**:**判断是否生成**对应过程文档;**不**生成"包含不涉及/不适用说明内容"的占位文档

## 2. 背景与目标

### 2.1 业务背景

- `code-skills` 是元技能集合(本仓库本身开发本仓库),大部分历史需求都是"改 SKILL.md / 改模板 / 改文档"类型的**元技能改**,与"真实业务工程改"的差别巨大:
  - 元技能改 → 无数据库、无单元测试、无 GUI、无用户故事 → 多个过程文档强制写但内容全空
  - 微小需求(1 文件 + 1 函数)→ 详设 `module-details` / `interface-specs` 等深度过程文档内容过浅,价值低
  - 大型需求(跨多模块) → 现有过程文档粒度往往不够,反而要细化
- 累计 **15 个需求 × 750 行平均 = ~11K 行过程文档**,折合 ~3M 字符,占全仓库 ~5%
- 这些过程文档的**可追溯价值**主要由"可派工给 `code-it` 消费"决定,与文档存在与否关系不大

### 2.2 业务目标

完成后要达到的可衡量效果:
- 单个需求过程文档总行数**减少 ≥ 30%**(目标:均值 750 → ≤ 500)
- 零空白过程文档文件(0 字节过程文档)
- 零"不涉及/不适用/无偏离"占位过程文档文件
- 过程文档**存在即有内容**:`Glob` + `Read` 任意过程文档都有 ≥ 5 行实质内容

### 2.3 本次目标

本次迭代要交付的范围:
- 9 个主流程技能 + `code-auto` / `code-dashboard` 编排 + 模板文档,过程文档生成规则全部改造
- 为每类过程文档明确"不涉及/不适用"判定标准
- AI 自主判定,无 `AskUserQuestion` / 无 CLI 参数新增
- 历史档案字节级保留(不动 V0.0.2 / V0.0.3 既有需求的过程文档)
- 新规则仅对 REQ-00035 及之后的新需求生效

### 2.4 非目标(明确不做)

- **不**改主 `RESULT.md` 主体结构(需求/概要设计/详细设计/计划/任务/缺陷/评审等主产出物)
- **不**改 `code-auto` 7 步状态机结构
- **不**改看板(`./assistants/<版本>/RESULT.md`)的区段清单(变更记录仍是区段,只是某些场景下不追加)
- **不**改 `dashboard-conventions.md` 字段约定(`code-dashboard` 仍能正常解析看板)
- **不**为过程文档生成引入新 CLI 参数(用户决策要点 #2)
- **不**清理历史过程文档(用户决策要点 #3)
- **不**改 `code-init`(项目级一次性引导,过程文档固定为 `INIT-REPORT.md` + `EXISTING-NNN/RESULT.md`,不属本改造范围)

## 3. 用户角色与场景

### 3.1 角色 1:需求开发者(wm,本仓库维护者)

**场景 1.1**:提交新需求 `REQ-00035`(本需求),"修改所有技能的过程文档记录逻辑" → 走完整 `code-require` → `code-design` → `code-plan` → `code-it` → `code-check` → 完成。
- 前置条件:V0.0.3 已激活
- 主流程:`code-require` 启动 → AI 读用户输入 → AI 自适应判定每类过程文档是否生成 → 只写需要的过程文档
- 异常/备选:若判定结果有疑问,AI 仍可不生成(由 `code-check` 评审时反馈"应生成但未生成"作为"建议改")

**场景 1.2**:提交微小需求 `REQ-NNNNN`(元技能改 1-2 文件) → 走 `code-auto`。
- 前置条件:过程文档自适应规则已就位
- 主流程:`code-auto` 调 `code-it` 时,任务级过程文档(`work-log` / `compile-and-run` / `deviations` / `test-results`)按规则判定
- 期望:小任务的过程文档总量从 ~130 行降至 ~50 行

**场景 1.3**:提交大型需求 `REQ-NNNNN`(跨 5+ 模块、3+ 子系统) → 走完整流水线。
- 期望:现有过程文档**不减少**(反而可能需要细化),但**不增加**纯模板占位

### 3.2 角色 2:`code-check` 评审人(AI 自动化)

**场景 2.1**:评审 `REQ-00035` 整体 → 检查过程文档**完整性** + **未生成的过程文档是否合理**。
- 评审维度新增 `8.13 过程文档适配性`(详见 §7):若 AI 判定"不生成 X"但评审认为应生成 → 派生"建议改"任务(非必须改)

**场景 2.2**:评审大需求 → 现有 12 维度不受影响,过程文档内容**可能更精炼**但**可追溯性不降低**

## 4. 功能需求(FR)

### FR-1:过程文档"不涉及"判定准则建立

- **描述**:为 9 个主流程技能中所有现存过程文档建立"不涉及/不适用"判定准则。每类过程文档**有且仅有一个明确判定标准**,AI 按标准自适应判定本轮是否生成。
- **入口**:`code-require` / `code-design` / `code-plan` / `code-it` / `code-check` 启动时,作为步骤 0a(版本检测后)的前置判定
- **前置条件**:本需求已落地(新规则对 REQ-00035+ 生效)
- **主流程**:
  1. AI 读上游输入(需求材料 / 设计章节 / 任务清单 / 评审范围)
  2. 对每类过程文档,按其"不涉及判定准则"判定本轮
  3. 判定结果 = `生成` / `不生成`
  4. 写入 `<版本>/require/<REQ>/process-doc-decisions.md`(本技能唯一新增的"过程判定结果"文件,详见 FR-2)
- **分支/异常**:判定本身不出错;若 AI 对某类过程文档"是否生成"犹豫 → 倾向**生成**(因空白文件最易踩坑,详见 NFR-2)
- **数据变化**:`process-doc-decisions.md` 是**唯一**过程文档"决策记录"文件,**仅当本类过程文档被判定为"不生成"时才记录**该判定及理由(避免新增一个全需求都必须写的"判定日志"反向增加 token)
- **来源**:(材料 §1.1 / 用户决策要点 #1+#2)

### FR-2:新增 `process-doc-decisions.md` 决策记录文件

- **描述**:在 `require/` / `design/` / `plan/` / `code/` / `review/` 各技能目录**仅当有"不生成"判定时**生成 `process-doc-decisions.md`,记录"本轮本技能判定了哪些过程文档不生成,理由是什么"。
- **入口**:本技能步骤 0a(FR-1 判定后)
- **前置条件**:本轮至少有 1 个过程文档被判定"不生成"
- **主流程**:
  1. 列出"不生成"的过程文档(类型 + 判定理由)
  2. 列出"生成"的过程文档(供下游校验)
  3. 末尾追加"## 变更记录"小节
- **分支/异常**:若所有过程文档都被判定"生成" → **不写** `process-doc-decisions.md`(避免反向增加)
- **数据变化**:仅追加,无修改既有内容
- **来源**:(材料 §1.1)

### FR-3:过程文档自适应规则纳入 SKILL.md

- **描述**:每个技能的 `SKILL.md` 中明确:
  - 列出本技能产出的所有过程文档(含主 RESULT.md 的"变更记录"小节)
  - 对每类过程文档,给出"不涉及/不适用"判定准则
  - 在"## 工作流程"中加 1 个新步骤(步骤 0a,版本检测之后),执行 FR-1 的判定
- **入口**:`code-require` / `code-design` / `code-plan` / `code-it` / `code-check` 启动时
- **前置条件**:本需求已落地
- **主流程**:步骤 0a(版本检测) → 步骤 0b(过程文档判定,新增,FR-1) → 既有步骤 1-N
- **分支/异常**:不适用(新增步骤是"判定 + 决策记录"线性流程)
- **数据变化**:仅在每个 SKILL.md 中**追加** 1 个新小节,不改既有内容
- **来源**:(材料 §2.3 / 用户决策要点 #1)

### FR-4:模板改造:`process-doc-decisions.md` 模板新增

- **描述**:在 `code-require` / `code-design` / `code-plan` / `code-it` / `code-check` 各自的 `templates/` 目录下新增 `process-doc-decisions.md` 模板,描述判定记录的章节结构。
- **入口**:`SKILL.md` 引用此模板作为依据
- **前置条件**:FR-3 落地
- **主流程**:本需求 `code-it` 阶段写模板文件(以 `code-require` / `code-design` / `code-plan` 三个改写技能为任务,见 §10 任务拆分)
- **数据变化**:`plugins/code-skills/skills/<skill>/templates/process-doc-decisions.md` × 5 个技能
- **来源**:(材料 §4)

### FR-5:`code-auto` 编排同步改造

- **描述**:`code-auto` 在调 `code-require` / `code-design` / `code-plan` / `code-it` / `code-check` 时,沿用子技能的过程文档自适应规则(零修改子技能语义),不需要做特殊判断。
- **入口**:`code-auto` 步骤 1-6
- **前置条件**:FR-1~FR-4 落地
- **主流程**:`code-auto` 7 步状态机不变;子技能各自执行 FR-1
- **分支/异常**:无
- **数据变化**:`code-auto/SKILL.md` 中"子技能调用表"备注列加 1 行"过程文档自适应:已纳入(沿用本技能约定)"
- **来源**:(材料 §2.3)

### FR-6:`code-dashboard` 解析兼容

- **描述**:`code-dashboard` 解析看板 `RESULT.md` 的"变更记录"区段时,新版"短变更记录"(本需求可能不生成某些行)与旧版"长变更记录"**解析兼容**。
- **入口**:`code-dashboard` 启动
- **前置条件**:FR-7 落地(看板变更记录策略)
- **主流程**:沿用 `dashboard-conventions.md §规则 1` 三同步,**不**新增字段
- **分支/异常**:看板的"任务清单/需求清单/缺陷清单"等强约束区段**仍必须存在**(只是"变更记录"行数可能变少)
- **数据变化**:`code-dashboard/SKILL.md` 中"## 解析锚点"小节加 1 条说明:"变更记录行数自适应(本需求 REQ-00035 起生效)"
- **来源**:(材料 §2.3)

### FR-7:看板 `RESULT.md`"变更记录"自适应

- **描述**:看板 `RESULT.md` 的"变更记录"小节,在某些场景下**不追加**:
  - 技能为幂等空跑(`git status --porcelain` 空)→ 不追加
  - 技能"步骤 N — 末尾兜底提交"被用户取消 → 不追加
  - 子技能判定"本轮无任何改动" → 不追加
- **入口**:`code-require` 步骤 9A / 步骤 9B / 步骤 N 等所有写"变更记录"的位置
- **前置条件**:本需求落地
- **主流程**:沿用既有"只在有变更时追加"语义,**显式**写入 SKILL.md 步骤约定
- **分支/异常**:看板的"需求清单/任务清单/缺陷清单"等其他区段**不**受影响(继续按既有规则追加)
- **数据变化**:`code-require` / `code-design` / `code-plan` / `code-it` / `code-check` / `code-fix` / `code-version` / `code-init` 各自的 SKILL.md 中"步骤 N — 末尾兜底提交"小节加 1 段"幂等空跑不追加变更记录"
- **来源**:(用户决策要点 #1)

### FR-8:过程文档不生成时 SKILL.md 步骤跳过

- **描述**:当某类过程文档被判定为"不生成"时,SKILL.md 中对应的"步骤 X — 写该过程文档"小节**整体跳过**(包括步骤内的 Read/Write/Edit 调用),并在末尾的"过程文档收尾"小节列一行"本轮跳过:X.md(理由:...)"。
- **入口**:本技能步骤 0b(FR-1 判定后)
- **前置条件**:FR-3 落地
- **主流程**:每个技能的 SKILL.md 中,所有"写过程文档"步骤前置 1 行条件 `if 过程文档决策.类型 === '生成'`(伪代码),不满足则跳过
- **分支/异常**:主 `RESULT.md` 始终生成(本需求**不**改主产出物),不参与此跳过逻辑
- **数据变化**:SKILL.md 中**仅在每个"写过程文档"步骤前置 1 行"如不生成则跳过"**,不改步骤内部文本
- **来源**:(材料 §2.3)

## 5. 非功能需求 / 约束(NFR)

### NFR-1:零空白文件

- 改造后,任意需求下 `Glob` 任意过程文档,**不存在 0 字节过程文档**
- 任意需求下 `Glob` 任意过程文档,**不存在只含 1 行"不涉及/不适用"占位的过程文档**
- 验证手段:在 `code-check` 步骤 0a 加 1 个**轻量校验**(不阻断):`for (每个过程文档): Read → 若文件行数 < 3 行 → 派生"建议改"任务:文件 `<path>` 疑似占位(实际行数 N)`;`N >= 3` 才算"有内容"

### NFR-2:可追溯性不降低

- 主 `RESULT.md`(需求/概要设计/详细设计/计划/任务/缺陷/评审)**不**在本改造范围内
- 任务的"触发/来源"字段、`PLAN.md` 的任务总览表、看板的"任务清单/缺陷清单"**不**参与"不生成"判定
- 评估手段:对本需求本身(REQ-00035)做 `code-check` 时,检查"任务清单/缺陷清单/需求清单"三区段行数 ≥ 改造前同类型需求

### NFR-3:不引入 CLI 参数 / 不引入 `AskUserQuestion`

- 改造后,所有技能的 CLI 接口**不**新增 `--skip-*` 类参数
- 改造后,所有技能执行过程中**不**为"过程文档生成判定"触发 `AskUserQuestion`
- 验证手段:`grep` 现有 SKILL.md 的"## 命令行参数解析"小节,行数不变;`grep` "AskUserQuestion" 关键字,本需求相关的条目数不增

### NFR-4:`code-auto` 7 步状态机不变

- `code-auto` 的步骤 0a(git pull) / 步骤 0(读版本) / 步骤 0b(设置标记) / 步骤 1-6(子技能) / 步骤 7(报告) **不**变
- 子技能调用表(需求路径 / BUG 路径)**不**变
- 验证手段:`code-auto/SKILL.md` 的 "## 子技能调用表" 字节级保留(仅"备注"列加 1 行)

### NFR-5:历史档案字节级保留

- V0.0.2 / V0.0.3 既有 15 个需求(REQ-00020~REQ-00034)的过程文档**不**被本需求修改
- 验证手段:`git diff` 仅包含本需求新增 / 改写的文件;既有过程文档(`require/REQ-00020~34/` 等)**0 改动**
- 实施手段:本需求的 `code-it` 任务清单中**不**包含"重写历史过程文档"类任务

### NFR-6:`code-init` 不在改造范围

- `code-init` 是项目级一次性引导,产出的 `INIT-REPORT.md` + `EXISTING-NNN/RESULT.md` 是其核心产物
- `code-init` 不在 9 个主流程技能之列,沿用既有"必须生成"的约定
- 验证手段:`code-init/SKILL.md` 的"## 工作流程"**0 改动**

### NFR-7:`dashboard-conventions.md` 字段约定不扩展

- 看板的解析锚点(`^## .*$` 定位 + 表格行 `^\| .* \|$` 匹配)**不**变
- 看板的强约束区段(需求清单 / 任务清单 / 缺陷清单 / 评审发现汇总 / 派生任务记录 / 变更记录)**不**改名 / 不删
- "变更记录"区段仍**必须存在**,只是某些场景下"无新追加"(整段可为空表格行)
- 验证手段:`dashboard-conventions.md` **0 改动**

### NFR-8:模板不动(零规范变更)

- `requirements.md` / `design.md` / `plan.md` / `task-plan.md` / `fix-plan.md` / `RESULT.md` / `REVIEW-REPORT.md` 等既有模板**不**改章节结构
- 仅**新增** `process-doc-decisions.md` 模板(FR-4)
- 验证手段:`git diff plugins/code-skills/skills/*/templates/` 仅含新增文件,不含修改文件

### NFR-9:`code-rule` 维护的项目级规范不强制覆盖

- `./assistants/rules/` 下的项目级规范**不**参与过程文档生成判定

## 7. 过程文档适配性评审(8.13 评审维度)

### 7.1 评审维度定义

`code-check` 评审 12 维度(8.1-8.12)外**新增** 8.13:

**8.13 过程文档适配性**(本需求 REQ-00035 FR-7 校验点 1 新增,2026-06-15 起生效):
- 检查本需求下所有过程文档**是否都满足 NFR-1(零空白文件)**
- 检查 `process-doc-decisions.md` 中"不生成"的过程文档**判定是否合理**:
  - AI 判定"不生成"的理由是否符合 §6 的判定准则
  - 实际产出内容**不**支持生成该文档(即"真不需要"而非"省 token")
- **不通过** → 派生"建议改"任务(`触发/来源=审查改修`,关联任务=被修正原任务);**不**标"必须改"(因 AI 自适应判定本身有合理自由度)
- 实施:`Glob` `require/.../process-doc-decisions.md` → 解析"不生成"列表 → 对每条查 §6 准则 + 读相关上游文档验证"真不需要"

### 7.2 既有 12 维度不受影响

- 8.10 详设完整性 / 8.11 概设越界检测 / 8.12 行数比例 — 全部沿用
- 8.1-8.9 — 全部沿用

## 11. 边界与异常

### 11.1 E-1:`process-doc-decisions.md` 写入失败

- 触发:`Write` 工具返回失败(权限/磁盘满)
- 行为:屏幕输出 `⚠ process-doc-decisions.md 写入失败(<原因>),判定结果保留在内存,继续执行后续步骤`
- 不阻断

### 11.2 E-2:AI 判定犹豫

- 触发:对某类过程文档"是否生成"无明确结论
- 行为:**倾向生成**(因空白文件最易踩坑,详见 NFR-1)
- 不触发 `AskUserQuestion`(NFR-3 强约束)

### 11.3 E-3:`code-auto` 中子技能判定失败

- 触发:`code-auto` 调子技能时,子技能的过程文档判定失败
- 行为:子技能**仍产出**全部过程文档(降级到改造前行为);屏幕输出 `⚠ code-auto 上下文:子技能 <name> 过程文档判定失败,降级为全产出`
- 不阻断

### 11.4 E-4:历史需求重新评审

- 触发:用户对 V0.0.2 / V0.0.3 既有需求调 `code-check`
- 行为:既有过程文档**不**重新评估(沿用历史);**不**生成 `process-doc-decisions.md`
- 验证:NFR-5 强约束
