# REQ-00030 · 优化 /code-design 与 /code-plan 职责分离

- 需求编码:REQ-00030
- 所属版本:V0.0.3
- 需求状态:已完成
- 创建:2026-06-12
- 最近更新:2026-06-12 14:11

## 1. 需求概述

当前 `/code-design` 输出的概要设计已包含字段类型、约束、索引、错误码、示例、禁用工具列表等本应在**详细设计**阶段才出现的细节;而 `/code-plan` 在很多需求上沦为"把 design 直接翻译成任务",没有补充实质内容。本需求要求对两个技能做**职责分离的硬切分**:**`code-design` 只做概要设计**、`code-plan` 才开始做详细设计。同时收敛"扩展性"选项的询问时机:**小需求不询问**;**触发条件满足**(新增三方依赖 / 跨 ≥3 模块 / 对接多套实现)时才在概设阶段标 `--extensible`,且扩展性相关设计**只**在详设阶段展开、编码阶段才拆"架构骨架"前置任务。

## 2. 背景与目标

### 2.1 背景

- **数据**:V0.0.3 实际落地的 10 个 REQ 中,`design/.../RESULT.md` 行数普遍 ≥ `plan/.../RESULT.md` 行数;极端例(REQ-00025)plan 是 design 的 2.0 倍,反例(REQ-00023)design 单文件 425 行、plan 仅 334 行
- **结构**:`code-design/SKILL.md` 步骤 9A-11A 要求"接口字段、错误码、数据结构字段、索引、自检规范"全部在概设完成;`code-plan/SKILL.md` 步骤 7A 仅说"按概要设计逐模块展开",无强约束必须补做
- **历史依据**:REQ-00020(同 V0.0.2)曾把"按 7 维度自适应问路"下沉到 `code-plan`,但未触达 design 阶段的输出深度

### 2.2 目标(可衡量)

| 目标 | 衡量指标 |
| --- | --- |
| G-1 `code-design` 不再做详设细节 | 落地后 N 个新需求的 `design/.../RESULT.md` 平均行数较 N-1 减 ≥ 30%;新 design 的 §7 / §8 / §9 章节不再出现"字段类型/约束/错误码/伪代码"等内容 |
| G-2 `code-plan` 必须补做详设 | 落地后 N 个新需求的 `plan/.../RESULT.md` 行数普遍 ≥ 同 REQ 的 design 行数;`code-check` 评审清单新增"详设完整性"校验点 |
| G-3 小需求不强制问扩展性 | 小需求(1 任务,0.5-2 天)的 `code-design` 启动不再触发 `AskUserQuestion`,直接采纳 `--balanced` 默认 |
| G-4 扩展性询问严格按触发条件 | 概设阶段仅在"新增三方依赖 OR 跨 ≥3 模块 OR 对接多套实现"时才展示 `--extensible` 选项;扩展性相关字段类型/接口契约/架构骨架**不**在概设展开 |
| G-5 编码阶段才拆架构骨架任务 | 当且仅当扩展性=高(且 `code-plan` 步骤 10A 触发条件满足)时,`PLAN.md` 才插入"架构骨架"作为首个任务;小需求或低扩展性需求**不**插入 |

### 2.3 非目标

- **不**追溯修订已落地的 9 个 REQ(REQ-00021 ~ REQ-00029)的 design / plan
- **不**改 `code-design` / `code-plan` 之外的其他 8 个 `code-*` 技能
- **不**改 `./assistants/rules/` 下任何项目级规范(若需要改规范,通过 `code-rule` 走单独 REQ)
- **不**改任务编号体系、看板字段、frontmatter 字节级保留规则
- **不**新增"扩展性评估"专章;沿用既有"## 设计目标"小节 + 步骤 0b 问路

## 3. 用户角色与场景

- **用户角色**:本仓库维护者(本需求的发出方)
- **典型场景**:
  - 场景 1:**调 code-design 时** —— 用户希望"小需求时不被反复打断,大需求时才被问扩展性"
  - 场景 2:**调 code-plan 时** —— 用户希望"plan 一定要补出 design 漏掉的字段/接口/算法,而非翻译"
  - 场景 3:**评审时** —— `code-check` 应能识别"design 越界做详设"或"plan 漏做详设"两种异常
  - 场景 4:**未来新需求** —— 同样应用本需求的修订,不再发生"design 比 plan 还厚"的反常

## 4. 功能需求

### FR-1:`code-design` 步骤 0b"是否问扩展性"判定规则(收敛)

- **触发**:`code-design` 步骤 0b 在"自适应问题数"决策时,先评估"是否需要扩展性选项"
- **判定逻辑**(必须三个条件**全部不**满足才视为"小需求,不问扩展性"):
  ```
  触发扩展性 = 满足任一:
    1. 需求"待评估三方依赖"清单非空(基于 FR-3.AC-3.1 待评估三方依赖清单)
    2. 模块拆分预评估涉及模块数 ≥ 3(基于 FR-3.AC-3.2)
    3. 上游需求 FR 含"多实现 / 抽象层 / 可替换"语义(关键词匹配)
  ```
- **不触发时**:`AskUserQuestion` **不**展示 `--extensible` / `--minimal`,直接采纳 `--balanced` 默认;`design/.../RESULT.md` 顶部"## 设计目标"小节"整体设计目标"列写 `--balanced`
- **触发时**:沿用既有 1-2 问自适应(小 1 / 中 2 / 大 2)
- **code-auto 上下文**:`AskUserQuestion` 仍走 0 问默认路径(沿用 REQ-00020);但 `--balanced` 默认值替换为**根据触发条件计算**的结果(若触发 → `--extensible`;若不触发 → `--balanced`)

### FR-2:`code-design` 步骤 0b 自适应问路收紧

- **自适应问题数**(修订后,沿用 REQ-00020 + 本需求收敛):
  - **小需求**(1 任务,0.5-2 天工作量,**且** FR-1 判定不触发扩展性):**0** 个 `AskUserQuestion`,直接采纳 `--balanced` 默认
  - **中等需求**(2-5 任务,或 FR-1 触发扩展性):1 个 `AskUserQuestion`(只问整体设计目标: `--minimal` / `--extensible` / `--balanced`)
  - **大需求**(≥6 任务,或需架构骨架):2 个 `AskUserQuestion`(整体 + 功能性 1 维度)
- **架构维度**(扩展性 / 健壮性 / 可维护性 / 封装性 / 可复用性 / 可读性)由 `code-plan` 步骤 0b 沿用既有 7 维度问路(沿用 REQ-00020),**不**在 `code-design` 问
- **职责分离声明**(保留):`code-design` 概设阶段不评估"未来怎么扩",避免 AI 倾向过度设计

### FR-3:`code-design` 步骤 9A / 10A / 11A 输出深度收窄(避免越界)

| 步骤 | 既有要求 | 本需求修订 |

| 步骤 10A 接口与数据结构 | §8 接口概要:列出 API 端点 / 方法签名 / 数据格式 / 错误码 / 鉴权 / 版本策略;§9 数据结构:字段名 / 类型 / 约束 / 关系 / 存储选型 / 索引 | §8 接口概要**只**列:接口名 / 形式(REST/gRPC/事件/函数/SDK 方法) / 状态(新增/扩展/修改) / 一句话职责;§9 数据结构**只**列:实体名 / 状态 / 关键字段(≤ 5 个,无类型无约束) / 一句话关系;完整 schema / 字段类型 / 约束 / 索引 / 错误码 / 鉴权 / 迁移脚本**全部下沉**到 `code-plan` |
| 步骤 11A 三方依赖评估 | 逐项评估:名称/版本/来源/用途/许可/活跃度/体积/安全/替代 | **保留**(本步骤是概设阶段的关键产出);但**仅**列:依赖名 / 一句话用途 / 必要性高/中/低 / 替代评估(项目内已有/外部无替代);**不**展开许可/活跃度/体积/安全态势等深评估,这些由 `code-plan` 在 `dependencies.md` 中展开 |

- **关键不变量**:`code-design` 阶段**不**产出伪代码、**不**产出完整接口 schema、**不**产出字段类型/约束/索引
- **过程文档**:`code-design` 过程文档(`module-breakdown.md` / `interface-specs.md` / `data-changes.md`)**不**生成(本需求后,这些文档由 `code-plan` 产出)

### FR-5:`code-plan` 步骤 7A"补做"职责强化

- **既有描述**:"按概要设计逐模块展开,产出可直接编码的细节"
- **修订为**:"按概要设计逐模块展开,产出**完整可直接编码的细节**:每个模块的内部类/函数签名、调用顺序、并发模型、状态归属、错误处理范式;每个接口的完整入参/出参 schema、错误码、示例、版本策略、鉴权、限流、幂等、链路追踪字段;每个数据结构的字段类型/约束/索引/迁移脚本/存储选型;每个关键算法的伪代码与边界条件。**任何概要设计未覆盖的细节必须在详设阶段补全,不得以下游实施时再补为由跳过**"
- **过程文档强制产出**:`module-details.md` / `interface-specs.md` / `data-changes.md` / `risk-analysis.md` 由"可选"改为"必选"
- **`RESULT.md` §4-§10 章节必填**:`templates/plan.md` §4 模块详细化 / §5 算法与逻辑 / §6 数据结构完整变更 / §7 接口细节 / §8 异常处理 / §9 安全要求 / §10 状态机 / §11 性能与资源 / §12 测试要点 全部由"建议填"改为"必填";若某章节无内容,显式写"本需求不涉及"
- **`PLAN.md` 任务总览"涉及文件"列**:每条任务的"涉及文件"必须能引用 `RESULT.md §4-§10` 某个章节或某个过程文档,不能仅列裸路径

### FR-6:`code-plan` 步骤 10A"架构骨架"前置任务触发条件收紧

- **既有触发条件**(沿用 REQ-00014):
  1. 较高扩展性 / 较高可维护性 / 较高可配置性 + 较高复杂度
  2. 涉及未来可能的不同实现方式
  3. 对接不同三方组件
- **本需求新增收紧条件**(与既有"或"关系):
  4. **满足 FR-1 的扩展性触发条件**(新增三方依赖 OR 跨 ≥3 模块 OR 对接多套实现)
  5. **整体设计目标 = `--extensible`**(由 `code-design` 步骤 0b 判定并写入"## 设计目标"小节)
- **不触发**:小需求 / 整体 `--balanced` / 整体 `--minimal` → **不**插入"架构骨架"作为首个任务;任务 1 直接进入功能点拆分
- **触发时**:`TASK-REQ-...-00001` 内容 = "架构设计 + 架构骨架 = 抽象层 + 接口契约 + 扩展点(不实现具体业务)"(沿用 REQ-00014)

### FR-7:`code-check` 评审清单新增"详设完整性"校验点

- **新增校验点**:`plan/.../RESULT.md` 中每条任务,必须能在 `plan/.../RESULT.md §4-§10` 找到对应章节;若某任务的"涉及文件"未在 §4-§10 任何章节引用 → 派生"详设缺失"任务(类型=`重构` / 触发/来源=`设计变更` / 关联任务=被修正原任务)
- **新增校验点**:`design/.../RESULT.md` §7 / §8 / §9 不应出现"完整字段类型 / 完整错误码 / 完整约束 / 完整索引"等详设深度内容;若出现 → 派生"概设越界"任务(类型=`重构` / 触发/来源=`设计变更` / 关联任务=被修正原任务)
- **新增校验点**:`design/.../RESULT.md` 行数 vs `plan/.../RESULT.md` 行数:若 `design > plan * 1.2` → 标黄警告"概设/详设比例异常,建议复核"
- **`code-check` 不感知新规则的兼容性**:`code-check` 自身的 SKILL.md 步骤需追加这 3 个校验点;**不**修改其他 7 个 `code-*` 技能

### FR-8:扩展性维度的"三阶段下沉"职责分离

| 设计阶段 | 扩展性相关产出 | 文档位置 |

## 10. 验收标准

### AC-1:`code-design` 步骤 0b 自适应问路(FR-1 + FR-2)
- **AC-1.1** 对一个 1 任务的 REQ(FR 数 ≤ 3,无三方依赖)调 `code-design`,**不**触发 `AskUserQuestion`,`design/.../RESULT.md` 顶部"## 设计目标"小节"整体设计目标"= `--balanced`
- **AC-1.2** 对一个 2-5 任务的 REQ 调 `code-design`,触发 1 个 `AskUserQuestion`(只问整体设计目标)
- **AC-1.3** 对一个 ≥6 任务的 REQ 调 `code-design`,触发 2 个 `AskUserQuestion`(整体 + 功能性)
- **AC-1.4** 对一个含"待评估三方依赖"清单的 REQ 调 `code-design`,**无论**任务数,均展示 `--extensible` 选项

### AC-2:`code-design` 步骤 9A / 10A / 11A 输出深度(FR-3)
- **AC-2.1** 调 `code-design` 处理一个新 REQ,生成的 `design/.../RESULT.md` §7.2 模块表**只**含 5 列(模块名 / 路径 / 状态 / 职责 / 依赖)
- **AC-2.2** §8 接口概要**只**列 4 项 / 接口(接口名 / 形式 / 状态 / 一句话职责)
- **AC-2.3** §9 数据结构**只**列 4 项 / 实体(实体名 / 状态 / 关键字段 ≤ 5 个 / 一句话关系)
- **AC-2.4** §10 三方依赖每行**只**列 4 项(依赖名 / 一句话用途 / 必要性 / 替代评估)
- **AC-2.5** 整体行数(不含模板注释)较既有 10 个 REQ 平均值减 ≥ 30%

### AC-3:`code-design` 模板修订(FR-4)
- **AC-3.1** `plugins/code-skills/skills/code-design/templates/design.md` §7 / §8 / §9 章节按修订后结构生成
- **AC-3.2** §7.5 / §8.5 / §9.5 "## 设计边界"小节存在,明确"由 code-plan 展开"的范围
- **AC-3.3** 模板顶部注释含 `> 本模板只覆盖概要设计...` 一行

### AC-4:`code-plan` 补做职责强化(FR-5)
- **AC-4.1** 调 `code-plan` 处理一个新 REQ,生成的 `plan/.../RESULT.md` §4 / §5 / §6 / §7 / §8 / §9 / §10 / §11 / §12 全部存在(章节不缺失)
- **AC-4.2** §4 模块详细化**至少**有 1 个模块的完整类/函数签名(示例: `UserService.login(token: string) -> Session` 而非只列模块名)
- **AC-4.3** §5 算法与逻辑**至少**有 1 个算法的伪代码块
- **AC-4.4** §6 数据结构每个实体的字段表**至少**含 5 个字段(类型 + 约束)
- **AC-4.5** §7 接口细节**至少**有 1 个接口的完整入参/出参 JSON schema
- **AC-4.6** 4 份必选过程文档(`module-details.md` / `interface-specs.md` / `data-changes.md` / `risk-analysis.md`)均存在且非空

### AC-5:`code-plan` 架构骨架触发条件(FR-6)
- **AC-5.1** 对一个 1 任务 + `--balanced` 的 REQ 调 `code-plan`,`PLAN.md` 任务总览**不**含"架构骨架"作为首个任务;首个任务直接是功能点
- **AC-5.2** 对一个含"新增三方依赖"的 REQ 调 `code-plan`,`PLAN.md` 任务总览第一个任务是"架构骨架"任务
- **AC-5.3** 对一个 `--extensible` + 跨 ≥3 模块的 REQ 调 `code-plan`,`PLAN.md` 任务总览第一个任务是"架构骨架"任务

### AC-6:`code-check` 评审清单新增(FR-7)
- **AC-6.1** 调 `code-check` 评审一个新 REQ,若该 REQ 的 `plan/.../RESULT.md` 某条任务"涉及文件"未在 §4-§10 找到对应章节,派生"详设缺失"任务
- **AC-6.2** 调 `code-check` 评审一个新 REQ,若该 REQ 的 `design/.../RESULT.md` §7 / §8 / §9 出现"完整字段类型 / 完整错误码",派生"概设越界"任务
- **AC-6.3** 调 `code-check` 评审一个新 REQ,若 `design` 行数 > `plan` 行数 × 1.2,REVIEW-REPORT.md 标黄警告

### AC-7:扩展性三阶段下沉(FR-8)
