# 需求提示词文档 — 优化 /code-fix 技能及整个缺陷修复流程的状态推进

- 需求编码:REQ-00037
- 所属版本:V0.0.3
- 创建时间:2026-06-22
- 最近更新:2026-06-22 09:02
- 当前版本:v1
- 责任人:用户

## 1. 需求概述

优化 `/code-fix` 技能及整个缺陷修复流程(`/code-fix` → `/code-plan` → `/code-it` → `/code-check`)中**缺陷状态推进**的职责分工,目标是**消除**"修复过程中需反复调用 `/code-fix` 来推进缺陷状态"的痛点:

- `/code-fix` 仅负责**登记缺陷**(初始状态 `待处理`)与维护"前 5 段状态"(`报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞`),**不再**参与"修复规划中及之后"的任何状态推进。
- `/code-plan` 完成修复规划后,自动将缺陷状态推进为 `待开发`。
- `/code-it` 完成开发后,自动将缺陷状态推进为 `待审查`(单任务 BUG 完成时直接;多任务 BUG 在第 1 任务开始时推 `开发中`,全部完成后推 `待审查`)。
- `/code-check` 完成评审后,自动将缺陷状态推进为 `已完成`。
- 整个流程 `/code-fix` → `/code-plan` → `/code-it` → `/code-check` 串行执行,**不需要**中途复跑 `/code-fix`。

## 2. 背景与目标

### 2.1 背景

V0.0.3 当前缺陷修复的推荐流程(沿用 REQ-00027 的"纯登记型 + 下游接力"分工)虽然**职责**已分清,但落地时**用户体验**仍有明显缺陷 —— 用户需在 4 个技能之间反复穿插"复跑 `/code-fix` 来推进缺陷状态":

| 步骤 | 当前调用 | 用户实际做了什么 |
| --- | --- | --- |
| 1 | `/code-fix "..."` | 登记缺陷 |
| 2 | `/code-fix BUG-NNN` | 推进状态:`报告` → `调查中` → `修复规划中` |
| 3 | `/code-plan BUG-NNN` | 产出 `fix-plan.md` |
| 4 | `/code-fix BUG-NNN` | 推进状态:`修复规划中` → `修复编码中` |
| 5 | `/code-it BUG-NNN` | 实施修复 |
| 6 | `/code-fix BUG-NNN` | 推进状态:`修复编码中` → `已修复-待验证` |
| 7 | `/code-check BUG-NNN` | 评审 |
| 8 | `/code-fix BUG-NNN` | 推进状态:`已修复-待验证` → `已修复-已验证` → `已关闭` |

步骤 2 / 4 / 6 / 8 这 4 次"复跑 `/code-fix`"**不**为用户提供增量价值,只是"校验上游技能已就位 + 同步看板"。

### 2.2 业务目标

把"缺陷状态推进"这件事**完全内化**到下游技能的工作流中,做到:
- 用户只需在**起点**(`/code-fix` 登记)和**终点**(`/code-check` 完成)之间,顺次调用 3 个下游技能,**不**需要再为"推进状态"操心。
- 看板上的"缺陷清单"区段的状态字段,任何时刻都反映"真实进度"(`待处理 / 待开发 / 开发中 / 待审查 / 已完成`),不依赖用户主动同步。

### 2.3 非目标

- **不**改 `/code-fix` 的"前 5 段状态"职责 —— 用户仍可调 `/code-fix BUG-NNN` 把状态从 `报告` 推进到 `调查中`(或其他 4 个前段状态)。
- **不**改 `/code-plan` / `/code-it` / `/code-check` 的核心工作流 —— 仅追加"缺陷状态回写"动作。
- **不**改 BUG 编号格式 —— 沿用 `^BUG-\d{5}$` 5 位纯数字。
- **不**做历史 BUG 的状态迁移 —— BUG-00001 / 02 / 03 等历史样本继续用既有 8 状态,不强制迁移。

## 3. 用户角色与场景

### 3.1 用户角色

| 角色 | 关注点 |
| --- | --- |
| 缺陷报告人 | 调 `/code-fix` 快速登记 BUG,无需了解后续修复流程细节 |
| 修复实施人(AI 协作者) | 顺次调 `/code-plan` / `/code-it` / `/code-check`,缺陷状态**自动推进** |
| 项目维护者 | 通过 `/code-dashboard` 看板看 BUG 全生命周期,状态字段可信 |

### 3.2 典型场景

#### 场景 A:单任务 BUG 修复(典型)

```
1. /code-fix "用户报告:登录页密码框不显示" → 产出 fix/BUG-00004/RESULT.md,状态=待处理
2. /code-plan BUG-00004 → 产出 fix-plan.md,自动状态 → 待开发
3. /code-it BUG-00004 → 实施修复,自动状态 → 待审查(单任务时直接跳)
4. /code-check BUG-00004 → 评审,自动状态 → 已完成
```

#### 场景 B:多任务 BUG 修复

```
1. /code-fix "..." → 待处理
2. /code-plan BUG-00005 → 待开发
3. /code-it TASK-BUG-00005-00001 → 启动第 1 个任务,自动状态 → 开发中
4. /code-it TASK-BUG-00005-00002 → 第 2 个任务(第 1 个已完成时启动),状态维持 开发中
5. /code-it TASK-BUG-00005-00003 → 全部任务完成,自动状态 → 待审查
6. /code-check BUG-00005 → 已完成
```

#### 场景 C:用户在 BUG 推进到 `待开发` 后想回退

- **场景 C.1**:用户发现 `code-plan` 规划有误,想回退到 `待处理` 让 `code-plan` 重新规划。
- **当前设计**:用户需调 `/code-fix BUG-NNN`,但本轮需求**已不再**让 `/code-fix` 推进"待开发"等后段状态。
- **影响**:用户需**手工**改 `fix/<BUG-NNN>/RESULT.md` 的状态字段(此为降级路径;主流程不提供)。
- **替代方案**(本轮**不**实现,留作 follow-up):增加 `code-fix BUG-NNN --revert <目标态>` 参数,允许把状态回退到任意前段(留作 Q-1)。

## 4. 功能需求 (FR)

### FR-1:`/code-fix` 仅登记,初始状态 `待处理`

- **描述**:用户调 `/code-fix "用户报告:..."` 登记缺陷时,新建的 `fix/<BUG-NNN>/RESULT.md` 状态字段为 `待处理`(而非 V0.0.3 当前的 `报告`)。
- **范围**:`plugins/code-skills/skills/code-fix/SKILL.md`
- **替换规则**:
  - 步骤 6 "新建分支"写 `fix/<BUG-NNN>/RESULT.md` 时,文档头"状态"字段写 `待处理`(V0.0.3 当前为 `报告`)。
  - 步骤 6 "新建分支"在变更记录区追加:`YYYY-MM-DD HH:mm 缺陷登记 code-fix 创建缺陷 <缺陷编号>(严重度 <P?>,状态=待处理) <缺陷编号>`。
  - 步骤 6 "新建分支"同步 `fix/RESULT.md` 缺陷清单时,本条"状态"列写 `待处理`。
  - 步骤 6 "新建分支"同步版本看板"缺陷清单"时,本条"状态"列写 `待处理`。
- **不变量**:`待处理` 与既有 `报告` 字面共存;`code-fix` 复跑时可仍把状态从 `待处理` 推进到 `调查中`(沿用前段状态推进路径)。

### FR-2:`/code-plan` 完成修复规划后,自动推进缺陷状态 → `待开发`

- **描述**:用户调 `/code-plan BUG-NNN` 完成 9 份文档产出后,自动把缺陷状态从 `待处理` 推进到 `待开发`(覆盖既有"修复规划中"字面);同时同步 `fix/RESULT.md` 与版本看板。
- **范围**:`plugins/code-skills/skills/code-plan/SKILL.md` BUG 路径步骤 27A 末尾
- **替换规则**:
  - 步骤 27A 用 `Edit` 更新 `fix/<缺陷编号>/RESULT.md` 文档头"当前状态"字段:`待处理` → `待开发`
  - 步骤 27A 在变更记录追加:`YYYY-MM-DD HH:mm 状态推进 <缺陷编号> 状态"待处理"→"待开发" <缺陷编号>`
  - 步骤 27A 同步 `fix/RESULT.md` 总览表中本条状态 → `待开发`
  - 步骤 28A 同步版本看板"缺陷清单"区段本条状态 → `待开发`
  - 步骤 28A 在变更记录追加:`YYYY-MM-DD HH:mm 缺陷状态 <缺陷编号> 状态"待处理"→"待开发" <缺陷编号>`
- **与既有"修复规划中"字面的关系**:`待开发` 与 `修复规划中` **字面共存**;本轮**不强制迁移**历史 BUG 的 `修复规划中` 字面;`code-plan` 推进时**只**写 `待开发`(从 `待处理` 推进);若用户**复跑** `code-plan BUG-NNN`(增量更新),检查当前状态:若当前为 `修复规划中`(旧字面),维持 `修复规划中`;若当前为 `待处理`,推进为 `待开发`。
- **新登记 BUG 的状态推进路径**:`待处理` → `待开发`;**升级提示**:从 `报告` / `调查中` 状态推进到 `待开发` 的迁移路径(老 BUG 用老路径) → 由 `code-plan` 检测当前状态字面,若是 `报告 / 调查中 / 修复规划中` 等老字面,维持原字面,**不**强迁。

### FR-3:`/code-it` 启动第 1 个 BUG 任务时,自动推进缺陷状态 → `开发中`

- **描述**:用户调 `/code-it TASK-BUG-NNNNN-NNNNN` 时,若本任务是该 BUG 的**第 1 个**(按 `PLAN.md` 任务总览行序,**首条**任务),且缺陷当前状态为 `待开发`,自动把缺陷状态推进为 `开发中`。
- **范围**:`plugins/code-skills/skills/code-it/SKILL.md` 缺陷分支步骤 21
- **替换规则**:
  - 步骤 21 在读 `fix/<缺陷编号>/RESULT.md` 当前状态后,新增判定:
    - 读 `fix/<缺陷编号>/PLAN.md` 任务总览,按文件行序找到第 1 条任务编号
    - 若本任务编号 == 第 1 条任务编号 **且** 当前状态为 `待开发` → 状态推进为 `开发中`,同步 `fix/<BUG-NNN>/RESULT.md` + `fix/RESULT.md` + 看板
    - 若当前状态**已**为 `开发中`(之前已推过),跳过(幂等)
    - 若本任务**非**第 1 条任务(已在开发中,被后续任务调用)→ 不再推进状态
    - 若当前状态为 `待审查` / `已完成` 等终态 → 主动询问用户是否要重做(沿用既有"特别处理")
  - 步骤 21 用 `Edit` 更新 `fix/<缺陷编号>/RESULT.md` 文档头"当前状态"字段:`待开发` → `开发中`(只在上述判定通过时)
  - 步骤 21 在变更记录追加:`YYYY-MM-DD HH:mm 状态推进 <缺陷编号> 状态"待开发"→"开发中" <缺陷编号>`
  - 同步 `fix/RESULT.md` 总览 + 版本看板"缺陷清单"区段(沿用步骤 21 既有结构)
- **多任务 BUG 的关键语义**:`code-it` 在跑**第 1 个**任务时把状态推到 `开发中`;后续任务推进时,状态**维持** `开发中`(不反复推进)。

### FR-4:`/code-it` 完成所有 BUG 任务后,自动推进缺陷状态 → `待审查`

- **描述**:用户调 `/code-it` 完成某 BUG 的**全部**任务后(单任务 BUG 时即本任务;多任务 BUG 时为最后 1 个任务),自动把缺陷状态从 `开发中` 推进为 `待审查`(单任务 BUG 也从 `待开发` 直接到 `待审查`,跳过 `开发中`)。
- **范围**:`plugins/code-skills/skills/code-it/SKILL.md` 缺陷分支步骤 24
- **替换规则**:
  - 步骤 24 在更新本任务 `已完成` 状态后,新增判定:
    - 读 `fix/<缺陷编号>/PLAN.md` 任务总览,统计开发状态 ∈ {`已完成`, `已取消`} 的任务数
    - 若已完成+已取消任务数 == 总任务数(全部任务都已结束) → 缺陷状态推进:
      - 单任务 BUG(总任务数 = 1):当前状态 = `待开发` → 推进为 `待审查`(直接跳,跳过 `开发中`)
      - 多任务 BUG(总任务数 ≥ 2):当前状态 = `开发中` → 推进为 `待审查`
    - 若仍有任务**未完成**(开发状态 ∈ {`待开始`, `进行中`, `阻塞`})→ 不推进状态,正常结束本任务
  - 步骤 24 用 `Edit` 更新 `fix/<缺陷编号>/RESULT.md` 文档头"当前状态"字段
  - 步骤 24 在变更记录追加:`YYYY-MM-DD HH:mm 状态推进 <缺陷编号> 状态"<旧>"→"待审查" <缺陷编号>`
  - 同步 `fix/RESULT.md` 总览 + 版本看板"缺陷清单"区段
- **幂等**:若当前状态**已**为 `待审查`(之前已推过)→ 跳过,不再写。

### FR-5:`/code-check` 完成评审后,自动推进缺陷状态 → `已完成`

- **描述**:用户调 `/code-check BUG-NNN` 完成缺陷评审(产出 `REVIEW-REPORT.md`)后,自动把缺陷状态从 `待审查` 推进为 `已完成`。
- **范围**:`plugins/code-skills/skills/code-check/SKILL.md` BUG 路径(若存在);若 `code-check` 当前未原生支持 BUG-NNN 入参,扩展之
- **替换规则**:
  - 步骤(评审完成)在写 `review/<BUG-NNN>/REVIEW-REPORT.md` 后,新增"状态回写"动作:
    - 读 `fix/<BUG-NNN>/RESULT.md` 当前状态
    - 状态:`待审查` → `已完成`(若当前已为 `已完成`,跳过)
    - 同步 `fix/<BUG-NNN>/RESULT.md` 文档头 + 变更记录 + `fix/RESULT.md` 总览 + 版本看板"缺陷清单"
  - 步骤在 `code-check` 已有"单需求评审"流程**之外**新增"单缺陷评审"分支(简版):
    - 仅评审 `fix/<BUG-NNN>/PLAN.md` 的任务 + `code/<TASK-BUG-...>/RESULT.md` 的改修正文
    - 产出 `review/<BUG-NNN>/REVIEW-REPORT.md`(沿用既有模板)
    - 评审完成后做"状态回写"
- **与"已修复-已验证 + 已关闭"既有字面的关系**:`已完成` 字面**合并**了"已修复-已验证" + "已关闭"两个老字面;本轮**不强制迁移**历史 BUG 的老字面;`code-check` 推进时**只**写 `已完成`(从 `待审查` 推进);若用户复跑 `code-check` 评审历史 BUG,当前状态若为 `已修复-待验证`,仍维持 `已修复-待验证`(不强迁)。

### FR-6:`/code-fix` 不再推进"修复规划中及之后"状态

- **描述**:本轮需求落地后,`/code-fix` 的状态推进范围**收敛**为:
  - 前 5 段:`报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞`(沿用 REQ-00027)
  - 入口:`待处理`(本轮 FR-1 新增,登记时直接落到 `待处理`)
- **范围**:`plugins/code-skills/skills/code-fix/SKILL.md` 步骤 4 状态推进表
- **替换规则**:
  - 步骤 4 状态推进表的"候选目标状态"列改为:
    | 当前状态 | 候选目标状态(本轮收敛后) |
    | --- | --- |
    | (新建) | 待处理 / 报告 / 调查中(注:`待处理` 为本轮新增,登记时直接落该状态) |
    | 待处理 | 报告 / 调查中 / 已关闭-非缺陷 / 已取消 |
    | 报告 | 调查中 / 已关闭-非缺陷 / 已取消 |
    | 调查中 | 已关闭-非缺陷 / 阻塞 / 已取消 |
    | 已关闭-非缺陷 | (终态) |
    | 已取消 | (终态) |
    | 阻塞 | 解除阻塞(回到报告中合适状态) / 已取消 |
    | 待开发 | (本技能**不**推进;由 `code-plan` 推进) |
    | 开发中 | (本技能**不**推进;由 `code-it` 推进) |
    | 待审查 | (本技能**不**推进;由 `code-check` 推进) |
    | 已完成 | (终态) |
  - 步骤 5 注脚改为:`本技能不推进"待开发 / 开发中 / 待审查"等状态,这些由 code-plan / code-it / code-check 自动推进;若需把状态回退到前段,请直接编辑 fix/<BUG-NNN>/RESULT.md(降级路径)`
  - 步骤 9 "引导下一步" 表中"待开发 / 开发中 / 待审查 / 已完成"行的"下一步建议"改为:"(由 code-plan / code-it / code-check 推进,本技能不参与)"
- **不变量**:`/code-fix` 仍是"纯登记型"(沿用 REQ-00027);**只是**不再"复跑校验/推进"修复流程中的状态。

### FR-7:状态字面共存,5 新状态与 8 老状态不归一化

- **描述**:本轮新增的 5 状态(`待处理 / 待开发 / 开发中 / 待审查 / 已完成`)与既有 8 状态(`报告 / 调查中 / 修复规划中 / 修复编码中 / 已修复-待验证 / 已修复-已验证 / 已关闭 / 已关闭-非缺陷 / 已取消 / 阻塞`)字面共存;**不**强制迁移历史 BUG 的老字面;`code-dashboard` 的看板/统计不归一化。
- **范围**:`plugins/code-skills/skills/code-fix/SKILL.md` + `plugins/code-skills/skills/code-dashboard/SKILL.md`(同步扩展 5 新状态的查询分类)
- **替换规则**:
  - `code-fix/SKILL.md` 步骤 4 状态推进表的"当前状态"列**同时**支持老字面与新字面(老的可以推进到老的,新的可以推进到新的;不允许跨字面归一化)
  - `code-dashboard/SKILL.md` 步骤 4 段 2 "5 类状态"分类规则扩展:增加"待开发 / 开发中 / 待审查"归入"待修复"分类;"待处理"也归入"待修复"分类;"已完成"归入"已修复"分类(沿用既有"P0 待修复 / P1 待修复"区分)
- **约束**:`code-dashboard` 的"PD-2 锁定"沿用 —— 不归一化状态字面,新老并存。

### FR-8:典型完整流程的"## 衔接 / 不要做的事" 段落同步更新

## 5. 非功能需求 / 约束 (NFR)

### NFR-1:状态推进幂等

- 任一技能在推进缺陷状态时,若当前状态**已**为目标状态,**跳过**(幂等)。
- 例:`code-plan BUG-NNN` 推进 `待处理 → 待开发` 时,若当前已是 `待开发`,跳过推进动作,不再写 `fix/RESULT.md` 与看板。

### NFR-2:状态推进失败回退

- 任一技能在写 `fix/<BUG-NNN>/RESULT.md` 文档头"状态"字段后,若同步 `fix/RESULT.md` 或看板失败:
  - 写回旧状态(尽力回退)
  - 屏显 `⚠ 状态推进部分失败:fix/<BUG-NNN>/RESULT.md 已写,但 fix/RESULT.md / 看板同步失败,请手动处理`
  - 不阻断当前技能的后续流程(`code-plan` 仍可继续产出 9 份文档;`code-it` 仍可继续实施修复)

### NFR-3:历史 BUG 状态保留

- 既有 `BUG-00001 / 02 / 03` 等历史样本的 `fix/<BUG-NNN>/RESULT.md` 中的状态字面(`报告 / 修复编码中 / 已修复-待验证 等`)**不强制迁移**。
- 本轮需求**只**影响"未来新登记的 BUG"的状态推进路径;历史的 BUG 状态由既有 `code-fix` 复跑"前 5 段状态"时按原字面维护。

### NFR-4:不修改 BUG 编号格式

- 沿用 `^BUG-\d{5}$` 5 位纯数字;不引入 `BUG-NNNNN-NNNNN` 嵌套式编号给 BUG 本体。
- BUG 任务仍用 `TASK-BUG-NNNNN-NNNNN`(沿用既有)。

### NFR-5:不修改 4 个技能的核心工作流

- `code-plan` / `code-it` / `code-check` 的核心工作流(算法、读输入源、写文件等)**不修改**;仅在"完成本任务/本缺陷"后的"收尾同步"动作中追加"缺陷状态回写"。
- 例:`code-it` 的步骤 24 "撰写 RESULT.md + 推进本任务开发状态" 之后,追加 1 个 "状态回写" 子步骤。

### NFR-6:不修改 frontmatter 与既有"## 工作流程" 小节结构

- 沿用既有各 SKILL.md 的不变量(字节级保留 frontmatter;不修改既有"## 工作流程" 大节结构;新规则追加到锚点位置)。

### NFR-7:看板"缺陷清单"区段同步

- 4 个技能(`code-fix / code-plan / code-it / code-check`)在推进缺陷状态时,**都**同步 `./assistants/<版本号>/RESULT.md` 的"缺陷清单"区段(沿用既有"code-fix 步骤 8 同步看板"模式)。
- 不在看板新增字段或列(沿用既有 7 列:`缺陷编号 / 严重度 / 标题 / 状态 / 报告时间 / 修复时间 / 修复人 / 关联任务 / 修复提交`)。
- "状态"列字面**直接**写新字面(`待处理 / 待开发 / 开发中 / 待审查 / 已完成`),**不归一化**(沿用 PD-2 锁定)。

### NFR-8:不修改 `code-dashboard` 的"5 类状态占比"分类骨架

- `code-dashboard/SKILL.md` 步骤 4 段 2 的"5 类状态"分类骨架(待概要设计 / 待详细设计 / 待代码开发 / 待单元测试 / 待代码评审)**不修改**;仅扩展"待修复 / 已修复"统计中**缺陷**的分类规则(沿用 NFR-7)。

## 9. 边界与异常

### E-1:BUG 当前状态不在推进路径中

- **场景**:`code-plan` 推进时,BUG 当前状态为 `已修复-待验证`(老字面,已由 `code-it` 推过)。
- **行为**:`code-plan` 检测当前状态字面,若是 `报告 / 调查中 / 修复规划中` 等老字面,维持原字面;若是 `待处理`,推进为 `待开发`;**不**强迁。

### E-2:多任务 BUG 中,用户跳过中间任务直接做最后 1 个

- **场景**:用户先调 `code-it TASK-BUG-00005-00003`(第 3 个任务),未先做 `TASK-BUG-00005-00001`(第 1 个)。
- **行为**:`code-it` 步骤 0a 前置任务守卫拦截(沿用 `code-it/SKILL.md` 既有);任务**不执行**,提示用户先做前置任务。
- **状态影响**:BUG 状态保持 `待开发`(未触发"开发中" 推进,因为第 1 个任务未启动)。

### E-3:`code-check` 评审发现"必须改"派生任务时

- **场景**:`code-check` 对 BUG 评审产出"必须改"项,派生 `TASK-BUG-NNNNN-NNNNN` 任务交给 `code-it` 做改修。
- **行为**:BUG 状态**不**回退到 `开发中`(本轮决策);保持 `待审查`(因为尚未通过)。`code-it` 改修完成后,BUG 状态**维持** `待审查`,等用户再调一次 `code-check` 重新评审,通过后 → `已完成`。
- **替代设计**(本轮**不**采用,留作 Q-3):派生的改修任务完成后,自动重新评审,不需要用户手动再调一次 `code-check`。

### E-4:`code-fix` 复跑 `BUG-NNN` 时,BUG 处于 `待开发 / 开发中 / 待审查` 中段

- **行为**:`code-fix` 检测当前状态,若是这 3 字面,**不**做状态推进;屏显 `⚠ 该缺陷当前状态为 <状态>,本技能不推进该状态;请继续调 code-plan / code-it / code-check`。

### E-5:历史 BUG(`报告 / 修复编码中` 等老字面)被新流程推过时

- **行为**:`code-plan` / `code-it` / `code-check` 检测到当前状态为老字面,**不**做强迁;维持老字面继续推进老路径(沿用既有路径);若用户希望把老 BUG 迁到新路径,**不提供**自动迁移工具(留作 Q-1)。

### E-6:`code-check` 当前未原生支持 BUG-NNN 入参

- **现状**(V0.0.3 截至本轮):`code-check` 既有 `REVIEW-REPORT.md` 等模板是按 REQ 路径设计的;BUG 路径需要扩展。
- **行为**:本轮 FR-5 中扩展 `code-check` 接受 `BUG-NNNNN` 入参;新增"单缺陷评审"分支(沿用既有"单需求评审"骨架)。
- **风险**:扩展可能引入 `code-check` 的回归 —— 缓解:`code-check` REQ 路径**字节级不变**(FR-5 限定仅追加 BUG 路径,不动既有逻辑)。
