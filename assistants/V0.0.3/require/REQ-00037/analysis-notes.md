# 分析笔记 — REQ-00037

## 当前痛点(用户原始诉求)

`code-fix → code-plan → code-it → code-check` 是 V0.0.3 缺陷修复的推荐流程,但**现状**下用户需多次复跑 `code-fix <BUG-NNN>` 来"校验/刷新"缺陷状态:

```
1. code-fix "..." → 登记,状态=报告
2. code-fix BUG-00001 → 推进状态到调查中 → 修复规划中
3. code-plan BUG-00001 → 产出 fix/<BUG-NNN>/PLAN.md(注意:现状是 fix-plan.md,后续 code-plan 升级为 RESULT+PLAN.md)
4. code-fix BUG-00001 → 推进状态到修复编码中
5. code-it BUG-00001 → 实施修复
6. code-fix BUG-00001 → 推进状态到已修复-待验证
7. code-check BUG-00001 → 评审
8. code-fix BUG-00001 → 推进状态到已修复-已验证
```

**问题**:步骤 2 / 4 / 6 / 8 重复调用 `code-fix`,但 `code-fix` 此时不提供任何增量价值(它只是"校验 + 同步看板")。

## 用户期望的新流程

```
1. code-fix "..." → 登记,状态=待处理
2. code-plan BUG-00001 → 分析 + 计划,完成后状态自动 → 待开发
3. code-it BUG-00001 (× N 任务):
   - 第 1 个任务开始:状态 → 开发中
   - 全部任务完成(且只有 1 个时直接):状态 → 待审查
4. code-check BUG-00001 → 评审,完成后状态 → 已完成
```

## 关键决策

### 决策 D-1:新增 `待处理` / `待开发` / `开发中` / `待审查` / `已完成` 5 个状态,与既有 8 状态共存

- **理由**:用户原话明确给出了 5 个新状态(`待处理` / `待开发` / `开发中` / `待审查` / `已完成`)。这些状态与既有 8 状态(`报告 / 调查中 / 修复规划中 / 修复编码中 / 已修复-待验证 / 已修复-已验证 / 已关闭 / 已关闭-非缺陷 / 已取消 / 阻塞`)语义有重叠:
  - `待处理` ≈ `报告`
  - `待开发` ≈ `修复规划中`
  - `开发中` ≈ `修复编码中`
  - `待审查` ≈ `已修复-待验证`
  - `已完成` ≈ `已修复-已验证` + `已关闭` 合并
- **选择**:不替换,而是**新增** 5 个状态,与既有 8 状态共存;`code-fix` 在登记时只产出"待处理";"code-plan"/"code-it"/"code-check" 推进时只产出"待开发 / 开发中 / 待审查 / 已完成"。
- **理由(展开)**:
  - 用户的"待处理"语义比"报告"更精确(后者历史含义偏向"刚被登记,还未被分配";前者强调"在队列里等着被处理")
  - 用户的"开发中"语义比"修复编码中"更短(后者历史偏长,描述性强)
  - 用户的"已完成"是单字面 = 闭合流程,**不**区分"已修复-已验证"与"已关闭"(本轮合并)
  - 历史 BUG(BUG-00001 / 02 / 03)若已用旧状态,**不强制迁移**(NFR-3 沿用)

### 决策 D-2:`code-fix` 仍保留 5 个"前段"状态(报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞),但不参与"修复规划中及之后"

- **理由**:用户原话没有否定 `code-fix` 的"前段状态"职责(用户没说要让 `code-fix` 不管"已关闭-非缺陷 / 已取消 / 阻塞"等);本需求**只**优化"修复规划中及之后"的 5 段。
- **影响**:`code-fix` 仍是"纯登记型"(沿用 REQ-00027),但其"复跑校验状态推进"的行为被本轮彻底去掉 —— `code-fix` 复跑时**只**刷新"前 5 个状态"(`报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞`),**不**动"待开发 / 开发中 / 待审查 / 已完成"。

### 决策 D-3:`code-plan` 完成 → 状态 `待开发`;`code-it` 启动第一任务 → 状态 `开发中`;`code-it` 全部完成 → 状态 `待审查`;`code-check` 完成 → 状态 `已完成`

- **依据**:用户原话逐字还原。
- **关键边界**:只有 1 个任务的 BUG,`code-it` 完成时**直接**跳到 `待审查`(中间无 `开发中`),因为"开发中"语义对单任务 BUG 没有意义(还没来得及进 `开发中` 就完了)。

### 决策 D-4:既有 8 状态与新增 5 状态在看板/统计/查询时**不归一化**

- **理由**:`code-dashboard` 的"PD-2 锁定"已明确:不归一化 `已完成(需求分析)` 到 `已完成`;同理,本轮新增的 5 状态与既有 8 状态**字面共存**,各自走各自的推进路径。
- **影响**:`code-dashboard` 的"待修复 / 已修复"统计需要同步扩展(分类判定增加新状态)。

### 决策 D-5:BUG 状态机总览(本轮需求产出)

```
[code-fix 登记] 待处理
   ↓ code-fix 复跑
 报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞
   ↓ code-plan 完成
 待开发
   ↓ code-it 启动第 1 任务
 开发中
   ↓ code-it 全部完成(单任务时直接跳)
 待审查
   ↓ code-check 完成
 已完成
```

## 不变量

- **不**改变 `code-fix` 的"前 5 段状态"职责(报告 / 调查中 / 已关闭-非缺陷 / 已取消 / 阻塞)—— 这些仍是 `code-fix` 推进的范围
- **不**改变 `code-plan` / `code-it` / `code-check` 的核心工作流 —— 仅追加"状态回写"动作
- **不**改 BUG 编号格式 —— 沿用 `^BUG-\d{5}$` 5 位纯数字
- **不**改 `code-dashboard` 的"5 类状态占比"分类 —— 但需要在看板"缺陷清单"区段扩展字段定义(状态字面扩展)
