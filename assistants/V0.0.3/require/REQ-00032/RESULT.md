# 需求提示词文档 — 优化 /code-require 登记结束后报告(输出下一步建议)

- 需求编码:REQ-00032
- 所属版本:V0.0.3
- 文档创建时间:2026-06-12 16:05
- 最近更新:2026-06-12 16:10
- 文档状态:已完成
- 上游:无(用户口头/文本输入)
- 遵循规范:`./assistants/rules/` 下 7 个文件(全沿用)
- 涉及技能:`code-require`(本需求唯一改造对象)

## 1. 需求概述

优化 `code-require` 技能,在技能登记结束后的**屏幕日志**中(非 RESULT.md 文档),输出一段"下一步建议":
- **微小需求**(AI 自主判定) → 建议用户**直接使用 `/code-auto` 技能完成开发任务**(跳过独立的 `/code-design` 概设环节,直接进入自动流水线)
- **其他需求** → 建议用户**先进行 `/code-design` 概设**,再继续后续任务(沿用既有主流程)

## 2. 背景与目标

### 2.1 背景

当前 `code-require` 完成需求分析后(步骤 10A / 10B),仅汇报"新增/修改/待澄清/版本看板同步",**未**给出明确的下一步操作建议。用户在需求登记结束后,需要自行决定:
- 是直接调 `/code-auto`(适合需求明确、改动范围小的场景)
- 还是先调 `/code-design` 走完整流水线(适合需求复杂、需要多轮设计评审的场景)

这种"完成后无引导"的体验,导致:
- 微小需求(比如改一个常量、加一个字段)被强制走完整 design + plan + it + check 流水线,**过度工程化**
- 复杂需求被用户误判为"简单改动"而直接跳到 code-it,**设计不足**

### 2.2 目标

- 在 `code-require` 完成时,主动给出"下一步建议",减少用户决策成本
- 建议**仅 2 类**(微小 / 其他),简单清晰,避免推荐过载
- 屏幕日志展示,**不**修改 RESULT.md 文档结构(零文档变更)
- 与 REQ-00031 元技能改规则一致(单元测试外移),**不**涉及 `/code-unit`

## 3. 用户角色与场景

### 3.1 角色

- **项目开发者**:使用 code-require 登记需求后,期望快速进入下一步
- **AI 协作流程的用户**:通过 code-require → code-design → code-plan → code-it 全流程

### 3.2 场景

| 场景 | 当前体验 | 改造后体验 |
| --- | --- | --- |
| 用户登记"加一个日志字段"等微小需求 | 完成后无引导,需用户自行决定调用 `/code-auto` | 完成时屏幕显示"→ 下一步:建议直接 `/code-auto` 完成开发(本需求判定为微小)" |
| 用户登记"重构认证模块"等复杂需求 | 完成后无引导,需用户自行决定调用 `/code-design` | 完成时屏幕显示"→ 下一步:建议先 `/code-design` 概设(本需求判定为非微小)" |

## 4. 功能需求(FR)

### FR-1:AI 自主判定需求粒度(微小 / 其他)

- **触发位置**:`code-require` 步骤 5A(首次分析) / 步骤 5B(增量更新) 末尾
- **判定时机**:AI 在提取候选需求(步骤 5A)或识别材料变更(步骤 5B)后,**主动**判断本需求是否属于"微小需求",记为内部变量 `isTiny: boolean`
- **判定启发式**(非强制阈值,AI 综合判断):
  - **推荐**判据(同时满足 → 微小):
    1. 需求材料数量 ≤ 1(单个文本/单条消息)
    2. FR(功能需求)数量 ≤ 2
    3. AC(验收标准)数量 ≤ 5
  - **不强制**:AI 可基于上下文(如"该需求在材料中明显跨多个模块")调整判定
  - **不允许出现第三种状态**:只允许 `isTiny = true` 或 `isTiny = false`
- **禁止**:
  - 引入"中等"等其他粒度
  - 弹出 `AskUserQuestion` 让用户手动选(用户已拒绝 Q-1 选项 D)
- **失败降级**:判定失败 → 默认 `isTiny = false`(按"其他需求"走,触发 `/code-design` 建议)

### FR-2:不修改 RESULT.md 文档结构

- **范围**:本需求**只**追加屏幕日志,**不**修改 `require/<REQ>/RESULT.md` 文档
- **理由**:用户 Q-2 选择"只追加到屏幕日志",最小化变更
- **禁止**:
  - 追加 "## 下一步建议" 章节到 `RESULT.md`
  - 在 `RESULT.md` 文档头/FR 编号中新增 `isTiny` 字段
  - 修改 `templates/requirements.md` 模板(NFR-2 零规范变更)
- **理由**:`isTiny` 是 code-require 内部判定变量,**不**是 RESULT.md 文档属性

### FR-3:屏幕日志输出"下一步建议"

- **触发位置**:`code-require` 步骤 10A(首次分析)/ 步骤 10B(增量更新) 末尾,在既有"向用户汇报"段后追加
- **格式契约**(NFR-3 沿用,字符数约束):
  - **FR-3.1**(微小路径):
    ```
    → 下一步建议:本需求判定为"微小需求",建议直接执行 `/code-auto` 完成开发任务
      提示:code-auto 会自动跳过独立的概设/详设步骤,直接进入编码+评审流水线
    ```
  - **FR-3.2**(其他路径):
    ```
    → 下一步建议:本需求判定为"非微小需求",建议先执行 `/code-design` 概设
      提示:概设完成后,code-plan → code-it → code-check 仍按既有主流程推进
    ```
- **触发条件**:
  - **FR-3.1 触发**:`isTiny = true`(FR-1 判定)
  - **FR-3.2 触发**:`isTiny = false`(FR-1 判定)
- **二选一**:**不**同时输出 2 条建议(用户 Q-3 锁定"仅 2 条建议"中**只**选 1 条)
- **不**重复输出:`isTiny` 已在屏幕日志判定后,**不**在后续步骤 9B / 10B 中再次输出建议(避免噪音)
- **保持既有"向用户汇报"段不变**(NFR-2 零规范变更)

### FR-4:不修改其他 8 个 code-* 技能 SKILL.md

- **不**修改:
  - `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` / `code-fix` / `code-init` / `code-version` / `code-rule` / `code-auto` 的 SKILL.md(INV-4 字节级保留)
  - 既有 12 个 REQ 的 RESULT.md(INV-7)
  - 7 个项目级规范(INV-5)
  - 4 个 README/marketplace/plugin/CLAUDE(INV-6)
- **范围限定**:本需求**只**修改 `code-require/SKILL.md` 中的 1 个或 2 个段(步骤 10A / 步骤 10B 末尾)

## 5. 非功能需求 / 约束(NFR)

- **NFR-1** 零文档变更:`require/<REQ>/RESULT.md` 文档**不**追加"## 下一步建议"章节(FR-2 锁定)
- **NFR-2** 零规范变更:`templates/requirements.md` **不**修改
- **NFR-3** 屏幕日志字符数:每条建议 ≤ 2 行,每行 ≤ 80 字(中英混排,沿用 REQ-00013 Q-3)
- **NFR-4** 幂等性:多次执行 `code-require` 后,屏幕日志建议**不**累积(每次只输出一组建议)
- **NFR-5** 不引入新依赖:0 新增三方依赖
- **NFR-6** INV 字节级保留:沿用 REQ-00030 的 INV-1~INV-10 校验点(本需求不新增 INV,本需求"屏幕日志新增"是**纯追加**,不涉及修改既有段落)
- **NFR-7** 标题解析:屏幕日志"下一步建议"**不**涉及标题解析(无 `REQ-NNNNN · 标题` 格式)
- **NFR-8** 与 REQ-00031 元技能改协同:`code-auto` 步骤 4.b 沿用 FR-6 恒等跳过;`code-require` 不收集"是否需要单测"偏好(Q-4 锁定 A)
- **NFR-9** 与 `code-dashboard` 引导 N 的差异:本需求建议**只**含 2 条,不沿用 `code-dashboard` 的"5 类优先级"格式;但内部使用"微小" / "其他" 2 分类(高/中/低 优先级不适用本需求)

## 7. 交互逻辑(状态机、关键流程)

### 7.1 微小需求判定状态机

```
[材料收集] → [材料数+FR+AC 综合评估] → {isTiny=true} → 输出 /code-auto 建议
                                          ↓
                                       {isTiny=false} → 输出 /code-design 建议
                                          ↓
                                       {判定失败} → 退化 isTiny=false → /code-design
```

```
    → (新增)按 FR-3 输出"下一步建议"1-2 行
  完成
```

## 9. 边界与异常

- **E-1**(材料数为 0):视为"无材料,无法判定" → `isTiny = false` → 走 `/code-design` 建议
- **E-2**(FR 数为 0):视为"无 FR,无需求" → 沿用 code-require 既有"中止"逻辑(本需求不介入)
- **E-3**(code-require 重入):本需求判定**不**累积(沿用 NFR-4)
- **E-4**(增量更新 / 步骤 5B):同样触发 FR-1 判定,输出 FR-3 建议
- **E-5**(判定失败):`isTiny = false` 退化 → 走 `/code-design` 建议
- **E-6**(用户在建议前中断):屏幕日志建议**不**输出(沿用 code-require 既有"中止"逻辑)
- **E-7**(本仓库无项目级规范):沿用 code-require 既有 AskUserQuestion(本需求不介入)

## 10. 验收标准(AC)

### AC-1:微小需求判定

- **AC-1.1** 当材料数 = 0 → `isTiny = false`
- **AC-1.2** 当材料数 ≤ 1 + FR ≤ 2 + AC ≤ 5(同时满足) → `isTiny = true`(推荐判据)
- **AC-1.3** 当 FR 数 ≥ 3 → `isTiny = false`
- **AC-1.4** 当 AC 数 ≥ 6 → `isTiny = false`

### AC-2:屏幕日志输出

- **AC-2.1** 微小路径(`isTiny = true`)→ 输出 FR-3.1 建议 2 行
- **AC-2.2** 其他路径(`isTiny = false`)→ 输出 FR-3.2 建议 2 行
- **AC-2.3** 每次只输出一组建议(**不**同时输出 2 条)
- **AC-2.4** 不修改 `RESULT.md` 文档结构
- **AC-2.5** 不在屏幕日志使用"本需求"等指代词(沿用 FR-2.AC-2.4 强约束,改为"本需求" + "已分析需求"等明确指代)

### AC-3:零变更校验

- **AC-3.1** `code-require/SKILL.md` frontmatter L1-3 字节级保留(INV-1)
- **AC-3.2** `code-require/SKILL.md` 既有"## 工作流程"小节 0 改(INV-2 字节级保留;本需求在 步骤 10A / 步骤 10B 末尾**纯追加**)
- **AC-3.3** 既有 7 个项目级规范 0 改(INV-5)
- **AC-3.4** 4 个 README/marketplace/plugin/CLAUDE 0 改(INV-6)
- **AC-3.5** 既有 12 个 REQ 的 RESULT.md 0 改(INV-7)
- **AC-3.6** 其他 9 个 `code-*` 技能 SKILL.md 0 改(INV-4)

### AC-4:与既有规则协同

- **AC-4.1** 沿用 REQ-00031 元技能改规则:`code-auto` 步骤 4.b 恒等跳过(不主动调 code-unit)
- **AC-4.2** 不收集"本项目是否需要单测"偏好(Q-4 锁定 A)
- **AC-4.3** 不修改 `templates/requirements.md` 模板(NFR-2)
