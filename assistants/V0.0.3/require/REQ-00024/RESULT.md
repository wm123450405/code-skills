## 1. 需求概述

**为谁**:`code-skills` 用户(本项目自身使用者)+ AI 协作者。

**解决什么问题**:当前 `/code-auto` 使用 `from REQ-NNNNN` 关键字区分"全流程模式"与"续跑模式",用户必须记忆关键字语法;关键字解析与"无关键字时整串视为需求内容"的兜底规则耦合,易产生歧义(如 `from` 作为自然语言动词出现在需求中)。

**带来什么价值**:通过**文件系统路径**自动判定输入语义 — 看 `./assistants/<版本号>/require/<输入>/` 或 `fix/<输入>/` 文件夹是否存在。无关键字语法,无歧义,无学习成本。

---

## 2. 背景与目标

- **背景**:
  - `code-auto` 现有 2 种模式:模式 A(全流程,无关键字) + 模式 B(续跑,`from REQ-NNNNN` 关键字)
  - 模式识别依赖正则 `^from REQ-\d{5}(\s+(.*))?$` 匹配首个 token
  - 退出码 5:模式 B 校验失败(`require/REQ-NNNNN/RESULT.md` 不存在)
  - 用户反馈:关键字 `from` 与自然语言冲突(如需求"从 X 导入 Y"中"从"易被误判)
- **业务目标**:
  - 移除 `from` 关键字依赖,改用**文件/文件夹存在性**作为唯一判定依据
  - 默认输入内容 = 需求编号(最常见场景)
  - 路径感知自动分流:`require/<输入>/` 存在 → 需求;`fix/<输入>/` 存在 → 缺陷;都不存在 → 视为需求内容
  - 0 破坏性变更:既有 `RESULT.md` 仍可直接续跑(无需手工重命名/移动)
- **本次目标**:本轮迭代交付 `code-auto/SKILL.md` 模式识别逻辑替换,沿用既有 6 步状态机结构。

## 4. 功能需求(FR)

### FR-1:移除 `from` 关键字解析

- **描述**:`code-auto` 不再识别 `from REQ-NNNNN` 关键字;模式识别完全由"路径感知"承担。
- **入口**:`/code-auto` 启动 → 步骤 1 模式识别
- **前置条件**:无
- **主流程**:
  1. 接收用户输入(单一字符串或空格分隔的 token 列表)
  2. 拼接 token 为单一字符串,去除首尾空白
  3. **不再**匹配 `^from REQ-\d{5}(\s+(.*))?$` 正则
  4. 跳到 FR-2 路径感知判定
- **分支/异常**:
  - 用户输入含 `from` 字符串(原"模式 B 续跑"用法)→ **不**特殊处理;整串直接进入 FR-2 路径感知;若 `require/<整串>/` 目录存在 → 当作需求编号;否则 → 视为需求内容
- **数据变化**:无
- **来源**:用户原始描述第 1-2 段

### FR-2:路径感知模式识别(默认输入 = 需求编号)

- **描述**:`code-auto` 接收用户输入后,按以下顺序检查文件/文件夹存在性,自动判定输入语义。
- **入口**:`/code-auto` 步骤 1(替代原"正则匹配"逻辑)
- **前置条件**:无(在任何模式识别之前,先读 `.current-version` 确认工作空间)
- **主流程**(4 条判定链):
  1. **检查 `require/<input>/` 文件夹**:`Bash: test -d "./assistants/<version>/require/<input>"`
     - **存在** → 进入 FR-3 需求编号模式(继续判定 RESULT.md)
     - **不存在** → 进入第 2 步
  2. **检查 `fix/<input>/` 文件夹**:`Bash: test -d "./assistants/<version>/fix/<input>"`
     - **存在** → 进入 FR-4 缺陷编号模式
     - **不存在** → 进入第 3 步
  3. **既不是需求编号也不是缺陷编号** → 视为需求内容(进入 FR-5 全流程模式)
- **分支/异常**:
  - 输入含空格 → 整串拼接后参与路径判定(原"模式 A 多 token"行为不变)
  - 输入为空 → 提示用法 + 退出码 4(沿用既有"缺参数"行为)
- **数据变化**:无
- **来源**:用户原始描述第 1-2 段

### FR-3:需求编号模式(已存在/未存在 RESULT.md 二级判定)

- **描述**:`require/<input>/` 存在 → 视为需求编号,继续判定 RESULT.md 是否存在,二选一进入"概要设计"或"需求设计"。
- **入口**:FR-2 第 1 步命中
- **前置条件**:`require/<input>/` 文件夹存在
- **主流程**:
  1. **检查 `require/<input>/RESULT.md`**:`Bash: test -f "./assistants/<version>/require/<input>/RESULT.md"`
  2. **存在** → 跳过需求分析(`code-require` 不调),直接进入**概要设计阶段**(即 `code-design` 步骤)
  3. **不存在** → 视为该需求**尚未完成需求设计**;进入**需求设计阶段**(即 `code-require` 步骤)
- **分支/异常**:
  - `require/<input>/` 存在但为空(无任何文件)→ 视为"尚未完成需求设计" → 同不存在路径 → 进入需求设计
  - `require/<input>/` 存在且有其他材料(如 design.png / meeting.md)但**无** RESULT.md → 同"未完成需求设计"
- **数据变化**:无
- **来源**:用户原始描述第 2-3 段(条件 1+2)

### FR-4:缺陷编号模式(直接进入缺陷修复详设)

- **描述**:`fix/<input>/` 存在 → 视为缺陷编号,直接进入缺陷修复详设流程。
- **入口**:FR-2 第 2 步命中
- **前置条件**:`fix/<input>/` 文件夹存在
- **主流程**:
  1. `fix/<input>/` 存在 → 视为缺陷编号
  2. 跳过 `code-require`(需求分析)
  3. 直接进入**缺陷修复详设**流程(即 `code-plan` 步骤读取 `fix/<input>/RESULT.md` 与 `fix/<input>/PLAN.md`,产出修复方案)
- **分支/异常**:
  - `fix/<input>/` 存在但缺 `RESULT.md` / `PLAN.md` → 沿用 `code-plan` 缺陷分支既有的"缺文件"错误处理(无需 `code-auto` 额外处理)
- **数据变化**:无
- **来源**:用户原始描述第 4 段

### FR-5:需求内容模式(全流程,从需求分析开始)

- **描述**:`require/<input>/` 与 `fix/<input>/` 都不存在 → 视为需求内容,进入全流程(从 `code-require` 开始)。
- **入口**:FR-2 第 3 步命中
- **前置条件**:两个目录都不存在
- **主流程**:
  1. 整串视为需求内容(沿用既有"模式 A"行为)
  2. 调 `code-require` 录入需求(分配新需求编号,创建 `require/<新编号>/` 目录,写 RESULT.md)
  3. 进入既有 6 步状态机(code-design → code-plan → ...)
- **分支/异常**:
  - 输入是空字符串 → 沿用"缺参数"行为:提示用法 + 退出码 4
  - 输入很长(>1KB) → 视为需求内容写入(模型层无长度限制)
- **数据变化**:分配新需求编号并创建 `require/<新编号>/` 目录
- **来源**:用户原始描述第 5 段

## 7. 交互逻辑

### 交互 1:`/code-auto <input>` 路径感知判定

- **触发**:用户调 `/code-auto <input>` 命令
- **流程**:
  1. 步骤 0a 拉取最新代码(沿用既有)
  2. 步骤 0 读 `.current-version`,失败退出码 3(沿用既有)
  3. **步骤 1 路径感知判定**(本需求新增):
     - 检查 `./assistants/<version>/require/<input>/` 目录
     - 检查 `./assistants/<version>/fix/<input>/` 目录
  4. 进入对应模式(FR-3 / FR-4 / FR-5)
- **状态机**:
  ```mermaid
  stateDiagram-v2
    [*] --> PathDetect: 用户调 /code-auto <input>
    PathDetect --> ReqPathExists: require/<input>/ 存在
    PathDetect --> FixPathExists: require/<input>/ 不存在 && fix/<input>/ 存在
    PathDetect --> TreatAsReqContent: 都不存在
    ReqPathExists --> ReqSkipRequire: RESULT.md 存在
    ReqPathExists --> ReqRunRequire: RESULT.md 不存在
    ReqSkipRequire --> Step2Design: 跳到概要设计
    ReqRunRequire --> Step2Design: 走全流程
    FixPathExists --> FixSkipRequire: 跳到缺陷修复详设
    TreatAsReqContent --> Step1Require: 视为需求内容(分配新编号)
  ```
- **关键规则**:
  - 路径检测是**只读**操作,无副作用
  - 路径判定不缓存,每次 `code-auto` 调用重新检测(避免并发修改导致误判)
- **空状态/加载状态/错误状态**:
  - 缺 `.current-version` → 沿用既有"未检测到激活版本"提示
  - `require/<input>/` 与 `fix/<input>/` 存在但不是目录(E-19)→ 屏显警告后视为需求内容

### 交互 2:屏幕日志格式(FR-6 展开)

## 9. 边界与异常

- **极端输入**:
  - 输入字符串长度 > 1KB → 视为需求内容(沿用既有"模式 A 多 token"行为,无长度限制)
  - 输入含特殊字符(空格 / 反斜杠 / 路径分隔符)→ 整串直接参与路径检测(若整串作为目录名包含特殊字符,大多数 OS 会拒绝创建;但**只读检测**不会失败)
  - 输入含中文 / Unicode → 路径检测按字符串字面值匹配(OS 文件系统层处理)
- **文件系统异常**:
  - `.current-version` 不存在 → 退出码 3(沿用既有)
  - `./assistants/<version>/` 不存在 → 沿用既有"无版本工作空间"提示(退出码 3)
  - `<input>` 包含路径分隔符(如 `../` `a/b`)→ 路径检测按字面值(`test -d` 失败)→ 视为需求内容
  - `<input>` 是空字符串 → 沿用既有"缺参数"提示(退出码 4)
- **并发场景**:
