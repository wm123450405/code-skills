# 分析笔记 — REQ-00040

## 核心问题(从用户原始输入提取)

> 优化 `/code-fix` 技能,在登记缺陷时,分析是否需要启动程序复现缺陷过程。若当前项目允许启动程序时应当启动程序并按照缺陷描述复现缺陷发生过程,以此可获得更多的日志、截图或交互数据来作证缺陷,方便后续分析并解决缺陷。复现过程中的日志、截图、交互数据都要被登记到缺陷信息中。

**关键语义抽取**:
1. **触发时机**:登记缺陷时(`code-fix` 步骤 1 / 步骤 6)
2. **分析**:是否需要启动程序(自动判定,失败降级)
3. **复现动作**:按缺陷描述复现,获取 3 类证据(日志 / 截图 / 交互数据)
4. **登记位置**:缺陷信息中(即 `fix/<BUG-NNN>/RESULT.md`)

**关键约束**:
- "若当前项目允许启动程序" → 探测而非强启动
- "应当启动程序并按照缺陷描述复现" → 启动 + 按描述执行
- "日志、截图、交互数据" → 3 类产物的明确划分
- "都要被登记到缺陷信息中" → 产物必登,不可丢弃

## 改造点设计

### 1. `code-fix/SKILL.md` 步骤 0 末尾追加"项目可启动性探测"子节

- 位置:步骤 0 末尾(在"4. 验证目录存在"后)
- 作用:探测 build/start/test 等命令是否可用,记录到上下文
- 算法:
  1. 检查 `package.json` 的 `scripts.start` / `scripts.build` / `scripts.test`
  2. 检查 `pyproject.toml` / `setup.py` / `requirements.txt` 是否存在
  3. 检查 `Makefile` / `docker-compose.yml` 是否存在
  4. 检查 `Cargo.toml` / `go.mod` / `pom.xml` 等(其他语言)
  5. 输出 `canStart: bool` + `startCommand: string | null`
- **不**触发 `AskUserQuestion`(自动判定);若 0 个探测命中 → 仍可登记,只是 `canStart = false`

### 2. `code-fix/SKILL.md` 步骤 6 末尾追加"复现产物登记"动作

- 位置:步骤 6 末尾(在"写变更记录" 之后)
- 触发条件:`canStart = true` **且** 用户在步骤 1 / 5 提供了复现步骤
- 动作:
  1. 启动项目(按 `startCommand`)
  2. 按用户提供的复现步骤执行
  3. 收集 3 类产物:
     - **日志**:进程 stdout / stderr → `fix/<BUG-NNN>/reproduce/run.log`
     - **截图**:UI 缺陷(若 `startCommand` 启动的是 web/desktop)→ `fix/<BUG-NNN>/reproduce/screenshot-{N}.png` (若 N=0 跳过)
     - **交互数据**:API 缺陷(若 `startCommand` 启动的是 web/api)→ `fix/<BUG-NNN>/reproduce/interaction-{N}.{json|txt}` (若 N=0 跳过)
  4. 写入 `fix/<BUG-NNN>/RESULT.md` 新增"## 复现产物登记" 区段
- 失败降级:
  - 启动失败 / 进程崩溃 / 超时(默认 60s)→ 屏显 `⚠` + 把失败原因写入"## 复现产物登记" 区段(`实际行为:` 子项)+ 继续登记(NFR-4)
  - 复现未触发 bug → 把"未触发"写入区段 + 继续登记

### 3. `code-fix/templates/bug.md` 新增"## 复现产物登记" 区段

- 位置:在"## 缺陷描述" 段后(在"## 根因分析" 段前)
- 字段:
  - **复现方式**:`程序复现` / `文本复现` / `未复现`
  - **启动命令**:`<command>`(若启动成功)
  - **运行时间**:YYYY-MM-DD HH:mm~HH:mm
  - **产物清单**(表格):
    | 产物类型 | 文件路径 | 大小 | 用途 |
    | --- | --- | --- | --- |
    | 日志 | reproduce/run.log | <N> KB | 进程 stdout/stderr 记录 |
    | 截图 | reproduce/screenshot-1.png | <N> KB | 步骤 N 后 UI 状态 |
    | 交互数据 | reproduce/interaction-1.json | <N> KB | 步骤 N 后 API 响应 |
  - **实际行为**:<复现过程中观察到的现象,含错误信息/堆栈/UI 异常>
  - **复现结论**:`已复现` / `未复现` / `复现失败(<原因>)`

### 4. 与既有"## 缺陷描述" 段的协同

- 既有"## 缺陷描述" 段的"### 复现步骤(若有)" 子节:沿用,**不**重写
- 新增"## 复现产物登记" 段:产物证据,引用"### 复现步骤" 段的步骤
- 协同关系:复现步骤 = 用户描述"应该怎么做";复现产物 = 实际跑出来的证据

## 与 `code-fix` 纯登记型边界

- **严守**:本需求**不**让 code-fix 实施代码修改
- 复现动作 ≠ 代码修改:复现是"运行已有代码 + 收集证据",**不**触发 `code-it` / `code-plan`
- 启动命令探测**不**触发 `AskUserQuestion`(NFR-3):纯本地命令探测,无需用户介入

## 复杂度评估

- **新增产物**:`fix/<BUG-NNN>/reproduce/` 子目录(可能含 run.log / screenshot-N.png / interaction-N.json)
- **改造 SKILL.md**:`code-fix/SKILL.md` 步骤 0 + 步骤 6 末尾追加,既有结构字节级保留
- **改造模板**:`bug.md` 新增"## 复现产物登记" 区段
- **新触发约束**:NFR-3(不触发 `AskUserQuestion`)+ NFR-5(历史 BUG 不追加新区段)
- **不触发 `dashboard-conventions §规则 1`**:本需求**不**新增看板列(产物放 `fix/<BUG-NNN>/` 子目录,**不**进 `fix/RESULT.md` / 看板)
- **不触发 `skill-conventions §规则 2` 强制三同步**:改造仅在 code-fix SKILL.md + bug.md,**不**改 CLAUDE.md / 其他技能 SKILL.md
- **不涉及技术选型**:`canStart` 探测是本地命令检查,**不**触发 §615 硬约束(RE界33 锁)

## 风险点

- **R-1**:`code-fix` 复跑已有 BUG(NOT 新建)→ **不**触发复现动作(本需求只在"新建分支"触发)
- **R-2**:用户报告缺陷时**未**给复现步骤 → 跳过复现动作,登记为 `复现方式 = 文本复现`
- **R-3**:启动命令运行 60s 未结束 → 屏显 `⚠ 启动超时` + 终止进程 + 降级为文本复现
- **R-4**:复现成功但程序产生大量日志(> 10MB)→ 截断日志到前 1MB + 屏显 `⚠ 日志过大,已截断`
- **R-5**:产物子目录与既有"## 关联项" / "## 修复日志" 段的引用关系 → 沿用既有"## 关联项" 段,产物路径以相对路径登记
