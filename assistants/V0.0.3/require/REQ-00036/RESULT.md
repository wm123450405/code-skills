# REQ-00036 — 清理技能文件中的开发痕迹

- 所属版本:V0.0.3
- 状态:已锁定

## 1. 需求概述

本仓库 `code-skills` 是一套用于走完完整软件开发周期的 Claude Code 技能集合。在 V0.0.1 → V0.0.2 → V0.0.3 的迭代过程中,各技能的 SKILL.md 和 templates 沉淀了大量"开发痕迹"——具体开发某条 REQ / BUG 时产生的编号引用、决策锁定段落、已退场功能的历史说明等。这些痕迹对**使用本技能开发其他项目**毫无价值,反而会干扰阅读与维护。本需求要做的是:**对所有 `code-*` 技能的 SKILL.md + 同目录 templates/ 做一次"去开发化"清理,只保留技能的实际工作流。**

## 2. 背景与目标

- **背景**:`code-skills` 仓库自身也是用本套技能迭代开发的,SKILL.md 在迭代过程中被反复追加段落,带上了「本需求 REQ-NNNNN 新增」「本需求 REQ-NNNNN FR-X 收窄」「本需求 YYYY-MM-DD 起生效」等标注;另外,部分被后续需求替换或退场的旧实现(如 `code-unit` 技能、`fix-plan.md` 过程文档)以「原 ... 退场前表述」「沿用原 `code-unit` 步骤 0a」等方式散落在各 SKILL.md。
- **业务目标**:让本仓库作为一套**可独立分发的技能库**时,SKILL.md 读起来只有"该技能做什么 / 怎么做",没有任何"这个技能在什么需求下被加了什么 / 改了什么"的回溯痕迹。
- **本次目标**:本轮清理覆盖**所有** `code-*` 技能的 `SKILL.md` 和同目录 `templates/`;不动 `checklists/` / `guidelines/` 目录(本次先不动,留作后续);不动 `assistants/` 工作空间。

## 3. 用户角色与场景

- **仓库维护者**:把本仓库发布到 marketplace / 单插件 install 时,SKILL.md 是用户唯一会读到的文档,需是干净的「功能文档」而不是「开发日志」。
- **下游技能使用者**:通过 `/code-xxx` 调技能时,屏显文本与文档不应被大量 REQ-NNNNN / 决策 Q-N 编号干扰。
- **AI agent**:`code-auto` 串行驱动子技能时,解析 SKILL.md 的步骤编号、子节标题不应误读"本需求 REQ-00035 新增"这类段尾标记。

## 4. 功能需求(FR)

### FR-1:清理「本需求 REQ-NNNNN」「本需求 BUG-NNNN」类段落尾注
- **描述**:在所有 SKILL.md 和 templates 中,删除「(本需求 REQ-NNNNN FR-X 新增)」「(本需求 REQ-NNNNN 新增,YYYY-MM-DD 起生效)」「(本需求 BUG-NNNN 新增)」等出现在段尾括号内的需求/缺陷编号引用;删除整段以「(本需求 REQ-NNNNN X-Y 锁定)」作为唯一新增长度的句尾注释。
- **入口**:在所有 14 个技能目录下执行(`code-answer` / `code-auto` / `code-check` / `code-dashboard` / `code-design` / `code-fix` / `code-init` / `code-it` / `code-merge` / `code-plan` / `code-publish` / `code-require` / `code-rule` / `code-version`)。
- **前置条件**:无。
- **主流程**:
  1. 扫描所有 `plugins/code-skills/skills/*/SKILL.md` 中形如 `(本需求 REQ-\d{5,}( ...)?)\s*[,、)]\s*$` 的尾注。
  2. 扫描所有 `plugins/code-skills/skills/*/templates/*` 中的同类尾注。
  3. 删除或重写,只保留段落的实际内容(不删整个段落,只删括号里那串「本需求 X 新增」字样)。
- **分支/异常**:
  - 段落若**整个段落的唯一价值**就是这条 REQ 编号引用(例如「## 标题解析(REQ-00013 新增)」),将整段合并 / 删除(由 code-it 自行判断,见 FR-1.AC-1.2)。
- **数据变化**:对应文件被改写,行数减少。
- **来源**:(用户 ARGS;V0.0.3 需求 REQ-00036)

### FR-2:清理「原 code-unit 退场」「fix-plan.md 退场」类历史变更注释
- **描述**:删除或简化所有「沿用原 `code-unit` 步骤 X」「原 `fix-plan.md` 退场,沿用 E-9 边界」「原 code-design 步骤 0b 的 4 问题 → 1 问题 → 0 问题收敛过程」等回溯性表述。改为简洁的**当前事实**陈述(例如「`code-it` 步骤 8a 项目可测性守卫」即可,不需要加「沿用原 code-unit 步骤 0a 字节级」)。
- **入口**:同上,所有 14 个技能目录的 SKILL.md + templates/。
- **前置条件**:无。
- **主流程**:
  1. 扫描「原 `code-unit`」「原 `fix-plan.md`」「原 `code-design` 步骤 0b」「本需求后**不**再使用」等关键词。
  2. 删除「原 ...」或「沿用原 ...」前缀,直接陈述**当前**事实。
  3. 若段落全部价值在解释"退场前是什么",且**当前**事实已经在别处说清,直接删段。
- **分支/异常**:
  - 「既有 7 个 PLAN 不追溯重拆」「既有 15 需求 NFR-5 字节级保留」等**真实**说明"历史边界不能动"的合规性条款,需**保留**——这是**功能约束**而非开发痕迹。
- **数据变化**:同上。
- **来源**:(用户 ARGS;V0.0.3 REQ-00036)

### FR-3:清理「Q-1 锁定 A」「Q-P7 锁定」类开发决策记录
- **描述**:SKILL.md 中的「(Q-1 锁定 A:无上限)」「(Q-3 锁定 A)」「(Q-2 采纳默认)」「(Q-P7 锁定)」等决策锚点编号,本质是当时与用户/产品对齐过程的 Q&A 索引,对后续维护无价值。
- **入口**:同上。
- **前置条件**:无。
- **主流程**:
  1. 扫描「Q-\d+ 锁定」「Q-P\d+ 锁定」「Q-\d+ 采纳」「Q-\d+ 隐含答复」类引用。
  2. 删除这些编号引用(如果整句只是「(Q-1 锁定 A)」)则整段可删;如果 Q-N 编号在同一句内提供了必要语义(例如「Q-1 = 锁定 A,无上限」本身是有信息量的),转写为**直接陈述**「(无上限)」「(默认)」即可,不保留 Q-N 编号。
- **分支/异常**:
  - 「(沿用 V0.0.0 NFR-X)」「(沿用 V0.0.1 NFR-Y)」这类**版本级规范继承引用**是**跨版本约束**,应**保留**(因为它告诉读者"这个规则来自哪一版规范")。
- **数据变化**:同上。
- **来源**:(用户 ARGS;V0.0.3 REQ-00036)

### FR-4:清理「(YYYY-MM-DD 起生效)」「(2026-06-15 起生效)」类生效时间标记
- **描述**:在 SKILL.md 中,段尾或小节尾的「(YYYY-MM-DD 起生效)」「(本需求 X YYYY-MM-DD 起生效)」属于"该规则从哪天开始生效"的回溯记录,清理后技能中只描述"当前规则是什么",不再标生效日。
- **入口**:同上。
- **前置条件**:无。
- **主流程**:
  1. 扫描「YYYY-MM-DD 起生效」类字面。
  2. 若段尾括号内只有这串,删除整串;若括号内还含其他信息,只删日期部分。
- **分支/异常**:
  - 「版本时间线」类内容(若任何 SKILL.md 里有"自 V0.0.0 引入 / V0.0.1 改写"这种**版本化叙述**段落)整段删除,见 FR-2 的「真实历史边界」例外。
- **数据变化**:同上。
- **来源**:(用户 ARGS;V0.0.3 REQ-00036)

### FR-5:清理指向 fix/plan/review 工作空间产物的开发性引用
- **描述**:SKILL.md 中「`fix/<BUG-NNN>/RESULT.md`」「`<版本号>/plan/RESULT.md` 写入责任划分」等**目录级**引用是功能契约,**保留**;但「`fix-work-log.md` / `fix-compile-and-run.md` / `fix-test-results.md` / `deviations.md`」等**已退场**的过程文档名引用需要清理。
- **入口**:同上。
- **前置条件**:无。
- **主流程**:
  1. 区分两类引用:
     - 目录级(`<版本号>/require/`、`<版本号>/plan/`、`<版本号>/code/`、`<版本号>/fix/`)→ **保留**(功能契约)。
     - 已退场的具体文件名(`fix-plan.md` / `fix-work-log.md` / `fix-compile-and-run.md` / `fix-test-results.md` 等)→ 删除或替换为现行名(任务路径用 `work-log.md` / `compile-and-run.md` / `test-results.md` / `deviations.md`)。
  2. 历史档(已生成的旧文件名)**不**强制改名,只更新 SKILL.md 的描述。
- **分支/异常**:
  - `code-fix` 技能同时管缺陷登记和缺陷路径,它的 `fix/` 目录语义**不是退场**——`code-fix` 自身**仍然**是有效技能,该技能 SKILL.md 中 `fix/RESULT.md` 等**保留**。
- **数据变化**:SKILL.md 中具体文件名引用被替换或删除;`assistants/<版本>/fix/<BUG-NNN>/fix-*.md` 等历史文件**不动**。
- **来源**:(用户 ARGS;V0.0.3 REQ-00036)

## 5. 非功能需求 / 约束(NFR)

- **NFR-1 — 清理幂等**:清理操作是**一次性快照动作**;若被回滚后再跑一次,结果应一致(无残留,且不会再次清理掉正常的功能描述)。
- **NFR-2 — 字节级保留 frontmatter**:每个 SKILL.md 的 YAML 前置元数据(`name` / `description`)**不**修改、**不**删除。
- **NFR-3 — 不修改章节结构**:本轮**不**重排 SKILL.md 的 `## 步骤 N` / `## 不要做的事` 等大节;只删除段尾括号 / 简化回溯注释。如果某条 FR-1 的尾注删除后,留下一个空段头(例如「### 步骤 X(REQ-NNNNN 新增)」括号内只这一串),**降级**为去掉括号的「### 步骤 X」即可,标题层级**不变**。
- **NFR-4 — 不破坏跨技能契约**:`code-plan` 引用 `code-design` 产物的章节号、`code-it` 引用 `RESULT.md` 的字段名、`code-require` 引用 `templates/requirements.md` 的章节号、所有占位符映射(FR/NFR/AC 占位符表)——这些**不**因清理而失效。
- **NFR-5 — 保留生效日 / 决策约束的语义等价物**:若某段落的"YYYY-MM-DD 起生效"是该规则**唯一**生效条件(例如"项目可测性守卫自 2026-06-15 起只在 V0.0.3 之后的需求生效"),不删除——但**不强制**;在 NFR-3 范围内 code-it 可自行判定去留(默认:删除,因为 V0.0.3 已是当前版本)。
- **NFR-6 — 占位符保留**:`templates/*.md` 里的 `REQ-00001` / `BUG-00001` / `TASK-REQ-00001-00001` 等示例**保留**(它们是模板占位符,不是开发痕迹);但「(本需求 REQ-NNNNN FR-X 收窄至 5 列)」「(沿用本需求 REQ-NNNNN ...)」类**对占位符的注释**应清理(FR-1 已覆盖)。
- **NFR-7 — 变更可审查**:清理后 `git diff` 应**只**显示删除行(无新增行 / 无修改行);如需新增占位说明(例如把"原 code-unit"替换为"`code-it` 项目可测性守卫"),允许**修改**行。
- **NFR-8 — 一次提交**:清理后由 `code-it` 步骤 N 兜底提交;commit message 形如 `chore(code-it): REQ-00036 清理 SKILL.md + templates/ 中的开发痕迹(N 文件 M 行)`。
- **NFR-9 — 不影响当前已激活版本流程**:V0.0.3 当前正在被本仓库自身开发使用,清理动作**不**应破坏 V0.0.3 下任何工作的进行(意味着本轮清理后的 SKILL.md 应仍可被 `code-require` / `code-design` / `code-plan` / `code-it` / `code-check` 正确解析)。
- **NFR-10 — 可回退**:本轮清理是一次性动作,但若发现清理过头,可由 `git revert` 整次提交回退。

## 9. 边界与异常

- **E-1**:某些 SKILL.md 在「本需求 REQ-NNNNN 新增」字样被删后,该段仅剩一句没有价值的话(例如「该规则由 4 维度收敛为 1 维度,本需求 REQ-00030 进一步收敛为 0 问」)→ 删除整段。
- **E-2**:某些「(沿用 V0.0.0 NFR-X)」实指**规范名**(如 `encoding-conventions §规则 1`)→ 保留,因为是跨技能契约,不是开发痕迹。
- **E-3**:code-it 在清理后若发现某 SKILL.md 整体被清空或大段残缺(说明判定失误)→ 回退到该文件原状,在 `analysis-notes.md` 标注"已跳过,原因:整体回溯痕迹无法剥离"。
