### 问题 1:行数计算范围

- **问题**:应采用哪种"实际代码逻辑行数"定义?
- **选项**:
 - A. 只计算非空非注释行(推荐)— 去掉注释 / 空行 / 格式化换行
 - B. 默认逻辑行 + 可选原始行(用户可用 `--loc=full` 切换)
 - C. 代码+注释合并计(不推荐)— 与用户原始描述语义不符
- **用户回答**:A. 只计算非空非注释行(推荐)
- **影响**:RESULT.md §FR-1(逻辑行计算函数)需明确定义"非空非注释行" = 去掉 `^\\s*$` + 注释行 + 纯格式化换行
- **细化**:
 - **注释行**:`//` / `/* */` / `#` / `<!-- -->` / `--`(SQL)等行级注释
 - **多行注释**:`/* */` / `""" """`(Python docstring) 等块级注释按行展开,每行都计
 - **空行**:全空白字符 / 仅含换行符的行
 - **格式化换行**:用户描述的"格式化换行"指 prettier / black / gofmt 等自动格式化产生的换行 — 这类换行在去掉空行后**通常**就是非空非注释行(因为它们有代码)
 - **退化**:无 tokei / cloc 时,回退到启发式(去空行 + 去注释行)

### 问题 2:语言覆盖

- **问题**:本需求需支持哪些语言的"逻辑行"统计?
- **选项**:
 - A. 5 主流语言(Python / JS / TS / Go / Java)— 避免冷门语言引入新依赖
 - B. 全语言(依赖 tokei / cloc)— 覆盖仓库可能的所有语言
 - C. 纯解析器路线(Python tokenize / JS acorn)— 不需要外部依赖
- **用户回答**:B. 全语言(依赖 tokei / cloc)
- **影响**:RESULT.md §FR-2(工具集成)需明确依赖 tokei(cloc 备选)+ 检测机制
- **细化**:
 - **首选工具**:`tokei`(Rust 编写,支持 200+ 语言,输出 `Code` 列即"逻辑行")
 - **备选工具**:`cloc`(Perl 编写,支持 50+ 语言,输出 `code` 列)
 - **检测机制**:
 1. `Bash: tokei --version` → 命中 → 使用 tokei
 2. 否则 `Bash: cloc --version` → 命中 → 使用 cloc
 3. 都不存在 → 屏幕输出 `⚠ 建议安装 tokei/cloc 以获得准确逻辑行统计;本次回退到启发式`,使用启发式算法
 - **tokei/cloc 安装**:本技能**不**负责安装,仅检测;若不存在,屏显 `⚠` 警告 + 回退到启发式

### 问题 3:code-it 位置

- **问题**:code-it 是否也要加"逻辑行计算"动作?
- **选项**:
 - A. code-it 实施后追加逻辑行 metadata(推荐)— 改造步骤 8 末尾
 - B. code-it 不动,只调 code-check — 零代码接触
 - C. 两处都加 — 编码主动遵守 + 评审验证
- **用户回答**:A. code-it 实施后追加逻辑行 metadata(推荐)
- **影响**:RESULT.md §FR-3(code-it 改造)需在步骤 8 实施开发末尾追加"逻辑行 metadata 收集"子步骤
- **细化**:
 - **code-it 步骤 8 末尾追加子步骤**:`calcLogicLoc` 子步骤
 - 读 `git diff --numstat` 输出(本任务变更文件列表 + 新增/修改行数)
 - 对每个变更文件调 tokei/cloc(沿用 FR-2 检测机制)
 - 计算"逻辑行新增" = 新增代码行 - 空行 - 注释行
 - 计算"逻辑行总规模" = 修改后文件总逻辑行
 - 写入 `code/<TASK-...>/RESULT.md` 新增"## 逻辑行统计"小节
 - **code-check 步骤 8 评审**:
 - 读 `code/<TASK-...>/RESULT.md` 的"## 逻辑行统计"小节
 - 判定"单文件逻辑行新增 > 阈值"或"单文件逻辑行总规模 > 阈值"→ 派生"代码行数超标"发现
 - 阈值默认值:单文件逻辑行总规模 ≤ 500 行(沿用既有"代码行数"经验值,可由用户配置)
