# 分析笔记 — REQ-00028

更新时间:2026-06-10

## 候选需求点提取(从用户 args 中抽取)

### 功能点
- **FR-1**:用户调用 `/code-answer` 技能,描述一个"功能/需求"作为查询对象
- **FR-2**:技能在版本工作空间内定位"功能/需求"对应的需求清单:
  - 优先 `code-init` 登记的基线需求(`require/EXISTING-NNN/RESULT.md`)
  - 其次 `code-require` 登记的开发需求(`require/REQ-NNNN/RESULT.md`)
- **FR-3**:扫描**所有历史需求变迁**(包括跨版本)以理解"该功能"在整个生命周期中的演变
- **FR-4**:若需求已登记且详细 → 直接基于需求文档回答
- **FR-5**:若需求未登记或登记不详细 → 扫描相关源代码补足功能逻辑
- **FR-6**:回答以**报告**形式呈现,内容**仅限**:
  - 回答本身(主要功能 + 逻辑)
  - 参考引用(指向具体的需求文件 `RESULT.md` 行号 / 源代码文件 `path:line`)
- **FR-7**:**过程不产生任何结果文件、不产生任何过程文件** — 报告仅输出到屏幕

### 约束
- **NFR-1**(无副作用):技能执行不创建、不修改 `./assistants/<版本号>/` 下任何文件
- **NFR-2**(无过程文件):不创建 `materials-index.md` / `analysis-notes.md` 等过程文件
- **NFR-3**(只读):所有工具调用均为只读(`Read` / `Glob` / `Grep`),禁用 `Write` / `Edit`
- **NFR-4**(可重复):同一查询多次执行,产出结果幂等
- **NFR-5**(跨版本):扫描需求时不限于当前激活版本,涵盖所有历史版本(基线 + 各开发版本)

### 边界
- **E-1**:用户输入的查询对象无对应需求 → 屏幕提示"未找到对应需求",建议用户提供功能关键字或代码路径
- **E-2**:用户输入的查询对象模糊 → 屏幕返回候选需求清单,请求用户进一步明确
- **E-3**:扫描源代码时找不到相关文件 → 报告注明"未找到对应源代码",并列出已扫描的目录
- **E-4**:用户未提供查询对象 → 反问用户提供(简短示例)
- **E-5**:大仓库扫描成本过高 → 采用关键词检索 + 限定目录(先精确匹配路径,再扩大范围),避免全量扫描
