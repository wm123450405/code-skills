# REQ-00028 — 新增 code-answer 技能(只读功能查询)

- 需求编码:REQ-00028
- 所属版本:V0.0.3
- 状态:已锁定
- 责任人:用户
- 创建:2026-06-10
- 最近更新:2026-06-10
- 当前版本:v1

## 1. 需求概述

为 `code-skills` 仓库新增一个**只读型**技能 `/code-answer`,用于在用户询问"某个功能 / 某条需求"时,自动检索项目内**所有历史需求**(`code-init` 登记的基线 + `code-require` 登记的开发需求),结合**历史需求变迁**(`RESULT.md` §13 变更记录),给出"该功能的主要功能 + 关键逻辑"回答;若需求未登记或登记不详细,自动扫描项目源代码补足;最终以**屏显报告**形式呈现,**不**产生任何结果文件、**不**产生任何过程文件,完全无副作用。

## 2. 背景与目标

- **背景**:`code-skills` 仓库目前 10 个 `code-*` 技能都是"动作型"(写入工作空间),缺少一个"查询型"技能 — 用户在长会话中可能需要回看"某条需求在历史上是怎么定义的"、"某个功能在哪个版本里被改过",但目前只能手动 `Read` 多份 `RESULT.md`,效率低且容易遗漏跨版本变迁。
- **业务目标**:让"功能逻辑查询"成为一线命令,可随时调用,响应式给出"需求定义 + 演进 + 引用"。
- **本次目标**:
  1. 新增 `plugins/code-skills/skills/code-answer/SKILL.md`,定义该技能的目标、输入、输出、工具集、工作流、边界。
  2. 严格沿用 `code-dashboard` 的"只读 + 屏显"范式,**不**写任何文件。
  3. 与 `code-require` / `code-init` / `code-dashboard` 形成清晰边界,不抢职责。

## 3. 用户角色与场景

- **角色 1:开发者(本仓库的协作者)**
  - **场景 1.1**:在长会话中忘记某条需求的细节 → 调 `/code-answer "code-publish 的发布前置检查"`,秒出回答 + 引用。
  - **场景 1.2**:跨版本演进摸底 → 调 `/code-answer "code-dashboard 的总览模式"`,看到该功能在 V0.0.2 → V0.0.3 之间的所有变更记录。
  - **场景 1.3**:需求未登记时 → 调 `/code-answer "用户登录功能"`,扫描源代码补足。

- **角色 2:新成员(包括 AI Agent)**
  - **场景 2.1**:快速理解项目"做过的功能" → 调 `/code-answer "<功能关键字>"`,获取基线 + 历次演进。
  - **场景 2.2**:验证某条需求的存在性 → 调 `/code-answer "REQ-00025"`,直接定位到对应 RESULT.md。

## 4. 功能需求(FR)

### FR-1:接收用户查询
- **描述**:用户通过 `/code-answer <查询>` 调用本技能,`<查询>` 是一个**自然语言关键字或功能描述**(如 `"code-dashboard 总览模式"` / `"用户登录"` / `"REQ-00025"`)。
- **入口**:Claude Code 命令行 `/code-answer <查询>`。
- **前置条件**:无前置(本技能不要求存在激活版本 — 见 FR-2 的回退策略)。
- **主流程**:
  1. 解析用户输入,提取查询字符串。
  2. 若字符串匹配 `^REQ-\d{5}$` / `^EXISTING-\d{3}$` → 走"精确编号"路径(FR-3a)。
  3. 否则 → 走"关键字模糊匹配"路径(FR-3b)。
- **分支/异常**:
  - 查询为空 → 边界 E-1:屏显用法 + 退出。
- **数据变化**:**无**(本技能不写任何文件)。
- **来源**:用户 args 描述 §"用于用户询问当前的功能逻辑时使用"。

### FR-2:版本上下文检测(可选)
- **描述**:本技能**不强制**要求存在激活版本 — 优先尝试读取 `./assistants/.current-version`,读取成功则同时扫描"当前激活版本"以提供更聚焦的回答;读取失败(`code-version` 尚未调过)时,自动降级为"全版本扫描"。
- **入口**:步骤 0 强制前置,沿用既有技能契约。
- **前置条件**:无。
- **主流程**:
  1. `Read "./assistants/.current-version"`。
  2. 文件不存在 → 屏显 `⚠ 未检测到激活版本,自动全版本扫描`,不视为错误,继续。
  3. 文件存在 → 读取内容,记为 `<激活版本>`,作为"优先匹配版本"。
- **数据变化**:**无**。
- **来源**:用户澄清记录 2026-06-10 §问题 1 答复"全版本"。

### FR-3a:精确编号匹配(REQ-NNNNN / EXISTING-NNN)
- **描述**:若查询字符串匹配 `^REQ-\d{5}$` 或 `^EXISTING-\d{3}$`,**全版本**直接定位 `require/<查询>/RESULT.md`。
- **主流程**:
  1. `Glob "./assistants/*/require/<查询>/RESULT.md"`。
  2. 命中 0 份 → 边界 E-2:`✗ 未找到对应需求 <查询>` + 列出本仓库已知的需求编号前 10 条作为提示。
  3. 命中 1 份 → `Read` 该文件,提取 §1 需求概述 + §4 功能需求 + §13 变更记录。
  4. 命中 ≥ 2 份(同编号跨版本)→ 全部 `Read`,按版本号升序列出"该需求在每个版本中的定义" — 重点呈现**差异**(后续版本对前序的修订/废弃)。
- **数据变化**:**无**。
- **来源**:用户 args §"主要阅读 `/code-init` 创建的需求清单、`/code-require` 技能登记的需求功能"。

### FR-3b:关键字模糊匹配
- **描述**:若查询字符串不匹配编号格式,按关键字在**全版本** `require/*/RESULT.md` 中模糊搜索。
- **主流程**:
  1. `Glob "./assistants/*/require/*/RESULT.md"` 列出全部需求清单。
  2. 对每份文件 `Grep` 查询关键字(支持中英文、忽略大小写),按"标题命中 > 概述命中 > 功能需求命中"权重打分排序。
  3. 取前 5 条命中(命中数为 0 时走边界 E-3)。
  4. 对每条命中 `Read` 并提取关键信息(需求概述 + 主要 FR + 变更记录摘要)。
  5. 用户可在屏显报告中看到"匹配 N 条,按相关度排序"提示。
- **分支/异常**:
  - 命中数 = 0 → 边界 E-3:屏显"未找到匹配需求,将尝试扫描源代码"(自动转入 FR-4)。
  - 命中数 > 5 → 屏显仅展示前 5 条,并附"另有 M 条相关需求,可通过更精确的关键字缩小范围"。
- **数据变化**:**无**。
- **来源**:用户 args §"结合所有历史需求变迁,回答当前需求的主要功能和逻辑"。

### FR-4:源代码补足(仅在需求信息不足时触发)
- **描述**:当 FR-3 命中 0 条 **或** 命中条目的 §4 功能需求描述过简时,自动扫描项目源代码补足功能逻辑。
- **入口**:FR-3 触发后(命中 0 条 / 命中但描述过简)。
- **前置条件**:FR-3 已执行。
- **主流程**:
  1. 尝试 Glob 常见源码目录(用户澄清记录 2026-06-10 §问题 2 答复"`src/` `lib/` `app/` `pkg/` `cmd/` `internal/` `packages/` `modules/` `core/`")。
  2. 全部 0 命中 → 边界 E-4:屏显"未找到任何源码目录(项目可能无代码或使用非标准布局),请提供代码路径",退出。
  3. 命中 ≥ 1 个 → 在命中目录下 `Grep -i "<查询关键字>" --type <ext>` 跨文件检索(ext 列表: `.ts` `.js` `.py` `.go` `.java` `.rs` `.rb` `.php` `.cs` `.cpp` `.h` `.swift` `.kt` `.vue` `.tsx` `.jsx` `.svelte`,按文件实际存在性动态调整)。
  4. 对命中的代码段提取前后 5-10 行上下文,**仅**作为"功能逻辑"补充呈现,引用格式 `path:line`。
  5. **不**做修改建议 / 不做代码评审 — 严格遵守"只读"契约。
- **数据变化**:**无**。
- **来源**:用户 args §"若需求没有登记或登记的不够详细,再扫描相关源代码获取功能逻辑"。

### FR-5:报告生成
- **描述**:将 FR-3 / FR-4 收集到的信息整理为屏显报告,严格遵循"**仅回答本身 + 参考引用**"格式。
- **主流程**:报告结构(本需求内置,不可省略任何区段):
  ```
  === code-answer 报告 ===
  查询:<用户的查询字符串>
  匹配来源:<需求清单 N 条 / 源代码 M 处 / 二者皆有>
  
  ## 主要功能
  <2-5 句话,概括该功能做什么>
  
  ## 关键逻辑
  <按编号列出 1-5 条核心逻辑点,每条 ≤ 3 行>
  
  ## 历史变迁(如适用)
  <按时间倒序列出变更记录,来源为 RESULT.md §13>
  
  ## 参考引用
  - 需求:<需求编号> §<章节> — `./assistants/<版本号>/require/<REQ>/RESULT.md:L<行号>`
  - 代码:`<相对路径>:<行号>` — <一句话说明该段代码的作用>
  
  === 报告结束 ===
  ```
- **分支/异常**:
  - 仅命中需求 / 仅命中代码 → 对应参考引用区段填实际内容,缺失区段显示"(无)"。
  - 完全无命中 → 走边界 E-5:屏显简短说明 + 引导用户提供更精确的查询。
- **数据变化**:**无**(报告仅屏显,无文件产出)。
- **来源**:用户 args §"所有回答最后以报告的方式呈现,报告内容仅限回答本身和参考引用"。

### FR-6:无副作用保证
- **描述**:本技能**严禁**调用 `Write` / `Edit` / `Bash` / `WebFetch` / `WebSearch` / `Task` / `Agent`。
- **校验**:执行结束(无论成功/失败)后,理论上 `git status` 保持调用前状态。
- **数据变化**:**绝对无**。
- **来源**:用户 args §"过程不产生任何结果文件、不产生任何过程文件"。

### FR-7:幂等性
- **描述**:同一查询多次执行,产出**完全一致**的屏显报告(顺序、内容、引用均一致)。
- **校验**:FR-5 报告结构确定性 + Glob / Grep 顺序稳定 + 不依赖时间戳。
- **来源**:用户澄清记录 2026-06-10 §问题 3 答复"仅屏显输出"。

## 5. 非功能需求 / 约束(NFR)

- **可观测性**:无运行时指标(本技能无副作用,无可观测对象)。
- **兼容性**:
  - 与 `code-dashboard` 工具集契约一致(只 `Read` / `Glob` / `Grep`)。
  - 不修改其他 11 个 `code-*` 技能 SKILL.md 的 frontmatter。
  - 不修改 `marketplace.json` / `plugin.json`(走 Claude Code 技能自动发现协议)。
- **可维护性**:
  - SKILL.md 结构沿用既有 10 个技能的章节布局(目标 / 适用场景 / 不适用 / 工作目录约定 / 输入 / 输出 / 工具使用约定 / 工作流程 / 边界与异常 / 衔接 / 不要做的事)。
  - FR / NFR / AC 编号与本仓库既有风格保持一致。
- **可重复性**(NFR-4 沿用):屏显报告总行数无硬约束,但建议 ≤ 80 行(超出时分页或截断 + "..." 提示)。

## 6. 页面与界面(不适用)

本技能无 UI 页面 — 输出仅为命令行屏显报告。

## 7. 交互逻辑

### 交互 1:标准查询流程

- **触发**:用户调 `/code-answer <查询>`
- **流程**:
  1. 解析查询字符串
  2. 步骤 0:版本上下文检测(FR-2)
  3. 步骤 1:扫描需求清单(FR-3a 或 FR-3b)
  4. 步骤 2:补足源代码(FR-4,条件性)
  5. 步骤 3:生成报告(FR-5)
  6. 步骤 4:屏显输出,退出
- **状态机**:
  ```mermaid
  stateDiagram-v2
    [*] --> ParseQuery
    ParseQuery --> ExactMatch: 匹配编号格式
    ParseQuery --> FuzzyMatch: 不匹配编号
    ExactMatch --> Found: 命中 ≥ 1
    ExactMatch --> NotFound: 命中 0
    FuzzyMatch --> Found: 命中 ≥ 1
    FuzzyMatch --> NotFound: 命中 0
    Found --> ScanCode: 描述过简(可选)
    Found --> Generate: 描述充分
    NotFound --> ScanCode
    ScanCode --> Generate: 命中
    ScanCode --> Error: 未找到源码目录
    Generate --> [*]
    Error --> [*]
  ```

## 8. 数据与状态

- **输入数据**:仅用户查询字符串 + 已存在的需求清单 / 源代码(只读)。
- **产出数据**:**无**(屏显输出不落盘)。
- **内部状态**:**无**(无持久化状态、无缓存)。
- **会话级临时变量**:扫描结果数组(运行结束即释放,符合"无副作用")。

## 9. 边界与异常

- **E-1**:查询字符串为空 → 屏显用法示例 + 退出
  ```
  用法: /code-answer <查询>
  示例:
    /code-answer REQ-00025
    /code-answer "code-dashboard 总览模式"
    /code-answer "用户登录"
  ```
- **E-2**:精确编号查询未命中 → 屏显 `✗ 未找到对应需求 <查询>`,列出已知前 10 条需求编号作为提示。
- **E-3**:关键字模糊匹配 0 命中 → 屏显 `⚠ 未找到匹配需求,将尝试扫描源代码`,自动转入 FR-4。
- **E-4**:源码目录全不命中 → 屏显 `✗ 未找到源码目录(项目可能无代码或使用非标准布局)`,列出已尝试的目录清单,提示用户提供路径。
- **E-5**:需求 + 代码全无命中 → 屏显简短说明 + 建议用户提供更精确的查询。
- **E-6**:`./assistants/.current-version` 不存在(项目未初始化)→ 屏显 `⚠ 未检测到激活版本,自动全版本扫描`,不视为错误(本技能不强依赖版本)。
- **E-7**:命中需求但 §4 内容极简(如只有标题无 FR)→ 自动转入 FR-4 补足。
- **E-8**:`Grep` 检索耗时过长(> 5 秒,实际很少触发)→ 截断扫描,屏显 `⚠ 扫描超时,展示前 N 条结果`,避免长时间挂起。
- **E-9**:内部未预期异常 → 屏显 `✗ 内部错误: <msg>`,退出(无副作用可污染)。

## 10. 验收标准(AC)

### AC-1:基础查询(精确编号)
- **对应需求**:FR-1, FR-3a
- **验证方式**:手工
- **步骤**:
  1. 执行 `/code-answer REQ-00025`
  2. 屏显报告应包含"## 主要功能" / "## 参考引用"区段
  3. 引用区段应包含 `./assistants/V0.0.2/require/REQ-00025/RESULT.md` 形式的路径
- **预期结果**:正确呈现 REQ-00025 的功能定义 + 引用路径
- **优先级**:高

### AC-2:关键字模糊匹配
- **对应需求**:FR-1, FR-3b
- **验证方式**:手工
- **步骤**:
  1. 执行 `/code-answer "code-dashboard"`
  2. 屏显应列出与 dashboard 相关的需求(预计至少命中 V0.0.2 的 REQ-00023)
  3. 报告应按"标题命中 > 概述命中 > FR 命中"权重排序
- **预期结果**:相关需求按相关度排序展示,前 5 条
- **优先级**:高

### AC-3:跨版本变迁展示
- **对应需求**:FR-3a, FR-5
- **验证方式**:手工
- **步骤**:
  1. 执行 `/code-answer REQ-00023`(该需求在 V0.0.2 定义,V0.0.3 演进)
  2. "## 历史变迁"区段应展示从 V0.0.2 → V0.0.3 的变更
- **预期结果**:跨版本演进信息可见
- **优先级**:中

### AC-4:无副作用验证
- **对应需求**:FR-6
- **验证方式**:自动化(检查 `git status`)
- **步骤**:
  1. 执行 `git status`(记录初始状态)
  2. 执行 `/code-answer <任意查询>`
  3. 再次执行 `git status`
- **预期结果**:两次 `git status` 输出完全一致(无新增 / 修改 / 删除任何文件)
- **优先级**:高

### AC-5:源代码补足
- **对应需求**:FR-4
- **验证方式**:手工(在含源码的项目中)
- **步骤**:
  1. 在一个含 src/ 的项目中执行 `/code-answer "<某功能关键字>"`
  2. 需求清单 0 命中时,自动扫描 src/ 下文件
  3. 报告应包含 "## 参考引用 - 代码" 子区段,格式 `path:line`
- **预期结果**:代码引用准确,前后文 5-10 行
- **优先级**:中

### AC-6:屏显报告契约
- **对应需求**:FR-5
- **验证方式**:手工
- **步骤**:
  1. 任意查询执行成功
  2. 检查报告结构是否包含:头部(查询 / 匹配来源) + ## 主要功能 + ## 关键逻辑 + ## 历史变迁(可选) + ## 参考引用 + 尾部(报告结束)
- **预期结果**:所有必要区段齐全
- **优先级**:高

### AC-7:无版本时降级
- **对应需求**:FR-2
- **验证方式**:手工
- **步骤**:
  1. 删除 `./assistants/.current-version`(临时)
  2. 执行 `/code-answer <查询>`
  3. 屏显应出现 `⚠ 未检测到激活版本,自动全版本扫描` 提示
  4. 报告仍正常产出
- **预期结果**:降级路径正常工作,无错误
- **优先级**:中
