## 缺陷标题

`/code-require` 仍出现技术选型问路,违反 REQ-00033 字面硬约束

## 缺陷描述

### 用户原始报告(verbatim)

> 技能 `/code-require` 还是会出现技术选型的选择的问题,应该只能在 `/code-design` 阶段出现选择技术选型相关提问。`/code-require` 不应该分析或记录任何与技术选型相关的内容,只分析需求本身的功能,技术选型完全放到 `/code-design` 技能中去完成。

### 复现步骤

1. 用户在 V0.0.3 调用 `/code-require` 提交需求
2. 需求材料中涉及"实现方式 / 选型"相关隐含信息(例:涉及某个特定 monorepo 工具的子包识别 / 涉及 monorepo vs single-package 的选择)
3. `code-require` 步骤 5A(提取候选需求)或步骤 7A(与用户澄清)被触发
4. **实际行为**:`code-require` 弹出与"技术选型 / 实现方式 / 框架 / 库"相关的问路选项(如"选择哪个 monorepo 工具?" / "选择 monorepo 还是单包?" / "用 Jest 还是 Vitest?")
5. **期望行为**:`code-require` **不**触发技术选型相关问路,只分析需求本身的功能(FR / NFR / AC / 页面 / 交互 / 数据 / 边界),技术选型完全放到 `/code-design` 阶段

### 期望行为 vs 实际行为

| 维度 | 期望行为 | 实际行为 |
| --- | --- | --- |
| 步骤 5A 提取 FR | 仅提取功能点 | 偶尔提取"实现方式"相关 FR(例:"支持 monorepo 子包识别" 应归属 `code-design` 阶段) |
| 步骤 7A 问路 | 仅问"功能 / 交互 / 数据 / 边界"类问题 | 偶尔弹出"技术选型 / 框架 / 库"类选项 |
| 步骤 8A NFR | 仅记录功能级非功能需求(性能 / 安全 / 合规) | 偶尔记录"技术选型"相关 NFR(例:"采用 pnpm-workspace.yaml" 应归属 `code-design`) |
| 步骤 1 收集需求 | 记录需求本身 | 偶尔拦截/记录用户携带的"技术选型偏好" |

## 根因分析

> 本节由 `code-it` 调查后补充(本技能只登记,根因分析留作 `code-plan` 阶段)。

### 初步根因(本技能维护)

`code-require` 步骤 615 已有"不涉及技术选型 / 技术栈 / 技术方案的确定"硬约束(沿用 REQ-00033 字面),但步骤 7A 的"能给出 2-4 个具体选项 → 使用 `AskUserQuestion` 工具" 字面未与"不弹技术选型问路"字面绑定,导致步骤 7A 在模糊场景下会**间接引导** AI 自动生成"实现方式"类问路选项。

**关键差距**:
- 步骤 615 字面"不涉及"是**约束性**字面(用户授权偏离)
- 步骤 7A 字面"2-4 个具体选项"是**操作性**字面(如何问)
- 两个字面**未交叉** → 步骤 7A 在生成"如何实现"类问路时,无明确拦截机制
