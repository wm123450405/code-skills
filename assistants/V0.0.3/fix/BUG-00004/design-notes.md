## 1. 关键设计问题与决策

### 1.1 步骤 8.7 命名与位置

- **问题**:`code-it` 步骤 8 末尾有 8.5(按需写单测)+ 8.6(逻辑行统计)。新步骤放哪?
- **选项 A**:步骤 8.7(数字递增)
- **选项 B**:在步骤 0 之前(像 `code-require` / `code-design` 那样)
- **选项 C**:不新增步骤,合并到步骤 8(实施开发)
- **决策**:**A**(步骤 8.7)
- **理由**:
 1. 步骤 0 之前没有"taskType" / "changedFiles" / "testable" 信息,无法判定 `decisions`
 2. 步骤 8 末尾(8.6 之后)信息完备,可立即物化 `decisions`
 3. 步骤 8.5/8.6 已经建立了"步骤 8 子步骤"模式(数字递增),沿用一致

### 1.2 `decisions` 字典的存储形式

- **问题**:`decisions` 是运行时变量还是持久化?
- **选项 A**:运行时变量(SKILL.md 步骤 8.7 物化,在内存中,后续步骤引用)
- **选项 B**:持久化(写入 `code/<任务>/process-doc-decisions.md`,作为跨步骤"单一事实来源")
- **决策**:**A + B** 双轨
- **理由**:
 1. **A**(运行时变量)是必要的 — 步骤 9/10/11 守卫需要立即查询,不能 IO 读文件
 2. **B**(持久化)是必要的 — `process-doc-decisions.md` 是评审要查的文档,必须留档
 3. 两者**不矛盾**:步骤 8.7 在物化运行时 `decisions` 同时,也立即 `Write process-doc-decisions.md`(若需要)

### 1.3 步骤 9/10/11 守卫粒度

- **问题**:守卫加在"步骤开头"还是"步骤内部每个动作"?
- **选项 A**:步骤开头守卫(整步跳过)
- **选项 B**:动作级守卫(检测命令 / 执行 / 记录 三段各自守卫)
- **决策**:**A**(步骤开头)
- **理由**:
 1. 步骤 9/10/11 的 3 个动作(检测/执行/记录)**强耦合** — 没命令可执行就没结果可记录
 2. 步骤开头守卫简洁,后续动作**不**需要重复判定
 3. 字节级保留既有 3 段(检测/执行/记录)逻辑,只在步骤开头加 if-else

### 1.4 RESULT.md 模板改造方式

- **问题**:`templates/RESULT.md` 第 8 段"过程文档清单"如何改造?
- **选项 A**:整体重写第 8 段(变成动态表格)
- **选项 B**:在第 8 段追加"决策依据"子表
- **决策**:**B**(追加子表)
- **理由**:
 1. 既有 RESULT.md 9 段字节级保留(NFR-2 既有字段 0 改)
 2. 追加子表既反映 `decisions` 又保留原清单,字节级保留约束与新需求兼顾

### 1.5 其他 6 个技能是否修复

- **问题**:`code-require` / `code-design` / `code-check` / `code-plan` / `code-fix` / `code-init` / `code-rule` 也有"判定表 ↔ 工作流"脱节问题,是否一并修复?
- **选项 A**:一并修复(7 个技能都改)
- **选项 B**:**仅修复 `code-it`**(因为它是日常工作中唯一会触发该问题的技能)+ 6 个技能旁路验证 + 记录结论
- **决策**:**B**
- **理由**:
 1. 6 个技能的判定表几乎都是"始终生成",**实际不会**触发过度生成(已由本轮端到端排查确认)
 2. 一并修复会大幅扩大改动面(7 × 2 = 14 个 SKILL.md 子步骤改造),违反"最小变更路径"
 3. 留作未来需求(若日后某 6 个技能因条件分支触发问题,再单独修复)
 4. TASK-BUG-00004-00004 旁路验证**记录**了结论,作为 BUG-00004 修复完整性的旁证

## 2. 候选方案否决记录

### 2.1 否决方案:在 `code-it/SKILL.md` 步骤 0 之前加"过程文档判定"步骤(类似 `code-require` / `code-design`)

- **否决理由**:
 1. 步骤 0 之前**没有** `taskType` / `changedFiles` / `testable` 信息,无法判定 `decisions`
 2. 必须放在步骤 8 末尾(8.6 之后),信息完备
 3. 6 个技能在步骤 0 之前的"判定"实际是**空操作**(因为信息不足,所有决策都"待定"),效果等同于"不判定"
 4. 这是 6 个技能**当前**没出现问题的根本原因 — 不是因为它们"接入了",而是因为它们的判定表几乎都是"始终生成"

### 2.2 否决方案:改造 `code-it` 全部 6 类过程文档(不仅 `compile-and-run.md` / `test-results.md`)

- **否决理由**:
 1. 本需求**只**针对 BUG-00004 暴露的"过度生成"问题
 2. `work-log.md` / `deviations.md` 始终生成(沿用既有)
 3. `unit-test-results.md` 已有步骤 8a 守卫判定,无需本需求再改
 4. 看板"变更记录"判定由步骤 15 既有逻辑决定,无需本需求再改
 5. **范围最小化** — 只改 2 个真正出问题的(`compile-and-run.md` / `test-results.md`)

### 2.3 否决方案:把 `process-doc-decisions.md` 改成"无论是否触发都生成"

- **否决理由**:
 1. 既有"只有不生成判定时才生成"的规则简洁清晰
 2. 全部"生成"时生成 `process-doc-decisions.md` 会产生无意义的空文件
 3. 违反"过程文档自适应判定"原始设计意图
