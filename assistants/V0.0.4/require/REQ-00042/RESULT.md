# 需求提示词文档 — REQ-00042 · 代码产出中禁止包含追踪编号

> 上游:`./assistants/V0.0.4/require/REQ-00042/`(材料源自用户对话)
> 遵循规范:`./assistants/rules/`(若存在)

## 1. 需求概述

优化 `/code-it` 技能,使其在编写代码、代码注释或代码说明文档时,**不得**包含需求编号(REQ-NNNNN)、缺陷编号(BUG-NNNNN)、任务编号(TASK-REQ-NNNNN-NNNNN / TASK-BUG-NNNNN-NNNNN)以及任务信息(任务标题、任务状态、任务依赖关系等)。**允许**包含需求的梗概描述(如"实现用户登录功能"而非"实现 REQ-00042")和缺陷的梗概描述(如"修复空指针异常"而非"修复 BUG-00001")。

核心原则:**代码应自文档化**——注释解释"做什么"和"为什么",而不是引用内部追踪系统的编号。脱离 `assistants/` 工作空间后,代码中的追踪编号将变得无意义。

(源自:用户对话输入)

## 2. 背景与目标

### 背景
- `/code-it` 是本技能库中**唯一**被允许修改项目源码的技能
- 当前 `/code-it` 在实施开发时,AI 可能在代码注释中引用需求/缺陷/任务编号作为追溯标记(如 `// REQ-00042: 实现用户登录` 或 `// TASK-REQ-00042-00001`)
- 这种引用在 `assistants/` 工作空间内可追溯,但代码一旦脱离该上下文(如被复制到其他项目、开源、或 `assistants/` 目录被清理),编号就完全失去意义
- 项目代码的使用者(其他开发者、代码审查者)不应需要查阅内部追踪系统才能理解代码意图

### 目标
1. `/code-it` 产出的所有代码(含注释)中不出现任何 REQ-NNNNN / BUG-NNNNN / TASK-* 格式的编号
2. `/code-it` 产出的代码注释使用人类可读的功能描述替代追踪编号
3. `/code-it` 的编码原则中明确记录此约束
4. 不影响 commit message 格式(commit message 可继续包含编号,由 `commit-conventions.md` 独立规范)

## 3. 用户角色与场景

- **代码阅读者**:阅读 `/code-it` 产出的代码时,注释直接说明功能意图,无需查阅 `assistants/` 目录
- **代码维护者**:修改代码时,注释提供足够的上下文,不依赖外部追踪系统
- **AI 协作者(code-it 调用者)**:在实施开发时,自动将需求/缺陷的编号替换为梗概描述写入注释
- **技能维护者**:在 `code-it` 的编码原则中维护此规则,确保所有 AI 调用者遵循

## 4. 功能需求(FR)

### FR-1: 代码中禁止出现追踪编号
- **适用范围**:`/code-it` 产出的所有源代码文件(含 `.ts` / `.js` / `.py` / `.go` / `.java` / `.rs` / `.md` 等任何编程语言或文档格式)
- **禁止内容**:
  - 需求编号字面:`REQ-NNNNN`(含 5 位纯数字及扩展格式)
  - 缺陷编号字面:`BUG-NNNNN`(含 5 位纯数字及扩展格式)
  - 任务编号字面:`TASK-REQ-NNNNN-NNNNN` / `TASK-BUG-NNNNN-NNNNN`(含嵌套式及扩展格式)
  - 以上编号的变体出现在代码注释、文档字符串、README 等任何代码周边文本中
- **允许内容**:
  - 需求的功能梗概(如"实现用户登录验证"替代"实现 REQ-00042")
  - 缺陷的问题梗概(如"修复空指针:用户名为空时崩溃"替代"修复 BUG-00001")
  - 纯技术性注释(如"O(n) 时间复杂度"、"TODO: 后续优化缓存策略")
- **不适用范围**:
  - commit message(由 `commit-conventions.md` 独立规范)
  - `./assistants/` 目录下的工作产物(保留编号作为追溯)
  - `code-it` 技能自身的内部文件(SKILL.md / references/ / templates/ 等)

(源自:用户需求描述)

### FR-2: 代码注释中禁止出现任务信息
- **禁止内容**:
  - 任务标题(如"TASK-REQ-00042-00001: 实现用户登录接口")
  - 任务状态(如"开发状态:已完成")
  - 任务依赖关系(如"依赖 TASK-REQ-00042-00002")
  - 任务触发/来源(如"触发来源:需求新增")
  - 任何其他任务元数据
- **替代方案**:使用功能描述(如"用户登录接口 —— 验证用户名密码并返回 JWT token")
- **例外**:代码中允许出现通用的 TODO / FIXME / HACK 标记,只要它们不包含追踪编号

(源自:用户需求描述)

## 5. 非功能需求 / 约束(NFR)

### NFR-1: 向后兼容
- 已存在的代码(含编号的注释)不受影响,不由本需求强制修改
- 仅约束 `/code-it` 未来的代码产出

### NFR-2: 不影响 commit message 格式
- commit message 格式由 `commit-conventions.md` 独立规范
- commit message 可继续包含需求/缺陷/任务编号(如 `chore(code-it): REQ-00042 代码产出中禁止包含追踪编号`)

### NFR-3: 不修改其他技能
- 仅修改 `/code-it` 技能及其关联文件
- 不修改 `code-design` / `code-plan` / `code-check` / `code-require` / `code-fix` 等其他技能

### NFR-4: 不修改 rules/ 目录
- 本需求不新增或修改 `./assistants/rules/` 下的规范文件
- 若后续需要将"代码注释不引用编号"提升为项目级编码规范,应通过 `code-rule` 独立操作

### NFR-5: 不修改 SKILL.md 流程结构
- 仅修改 `references/common.md` 和 `templates/RESULT.md` 的内容
- 不修改 `SKILL.md` 的步骤编号、步骤标题、决策点

## 9. 边界与异常

- **E-1: commit message 中的编号**:commit message 不受本需求约束,可继续包含编号
- **E-2: `./assistants/` 工作产物**:RESULT.md / work-log.md / deviations.md 等过程文档可继续包含编号,这些是追溯所需
- **E-3: 既有代码**:已存在的代码中含编号的注释不强制修改,仅约束新产出
- **E-4: 第三方平台编号**:若项目使用了第三方平台前缀(如 JIRA-),同样适用禁止规则
- **E-5: 代码中的 URL 引用**:若代码中包含指向 issue tracker 的 URL(如 `https://github.com/.../issues/123`),不受本需求约束(URL 是自包含的)
