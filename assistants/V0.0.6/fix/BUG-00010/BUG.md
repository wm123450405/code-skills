# 缺陷登记 — BUG-00010

## 缺陷标题

`/code req` 与 `/code fix` 的 CHECK 阶段对"建议改/可选改"项的自动应用规则不明确

## 缺陷描述(用户原始输入)

> `/code` 技能的 `req` 和 `fix` 模式下在 `CHECK` 阶段时若检查出建议修改或可选修改项时,应当提示用户是否修改,若用户选择修改则应当进行修正,用户也可以选择不修改。若使用了 `--auto` 模式则无需用户确认直接进行修改,所有的检查中的问题无论大小都自动被完全修正。

## 触发条件

1. 用户在 `/code req "<需求描述>"` 或 `/code fix "<缺陷描述>"` 流程中
2. CHECK 阶段产出"建议改"或"可选改"项(已存在)
3. 当前规则:`fix` 子命令 CHECK 阶段描述为"对'必须改'自动修复,对'建议改'询问用户(非 --auto)";但 `req` 子命令的 CHECK 阶段描述未明确"必须改"的处理路径,且"建议改"行为仅写"对'建议改'询问用户(非 --auto)"
4. **"可选改"项在两份 SKILL.md 中都未明确处理**
5. **`--auto` 模式下"建议改/可选改"是否自动应用未明确**

## 现状描述

| 项 | 当前 SKILL.md 描述 | 期望行为 |
| --- | --- | --- |
| 必须改(必须改) | fix: "对'必须改'自动修复" / req: 未明确 | fix 已有;req 应自动修复(评审-编码循环) |
| 建议改(建议改) | 两者:"对'建议改'询问用户(非 --auto)" | 默认模式:逐项 AskUserQuestion;用户选修改则应用,选跳过则保留 |
| 可选改(可选) | 两者:**未明确** | 默认模式:逐项 AskUserQuestion;`--auto`:自动应用所有项 |
| `--auto` 模式 | req: "所有 AskUserQuestion 自动选推荐项(第一项)";fix: 同 | 当前只覆盖"选择第一项",未明确"建议改/可选改"的处理深度 |

## 可能成因

### 根因 1:`references/req/check.md` 未明确"建议改/可选改"的处理规则

`references/req/check.md` 是 req/fix 共用的 CHECK 阶段细节文档,可能没有显式描述三档分类(必须改/建议改/可选)的处理路径。

### 根因 2:SKILL.md 中"建议改"行为描述模糊

SKILL.md 的 req/fix 步骤 5 CHECK 阶段描述只说"对'建议改'询问用户",但未规定:
- 用户选择"修改"后,如何回到 CODING 阶段去修正
- 用户选择"不修改"后,如何继续往下走

### 根因 3:`--auto` 模式的语义不清

`--auto` 在 SKILL.md 的全局说明是"所有 AskUserQuestion 自动选第一项(推荐项)",但用户期望是"无论大小,所有问题全部自动应用修复"。两者冲突:
- 当前 `AskUserQuestion` 设计下,选第一项是"修改"
- 用户期望 `--auto` 不弹任何 AskUserQuestion,直接修复所有项

## 影响范围

| 受影响对象 | 影响 |
| --- | --- |
| `/code req` 流程 | 用户行为不可预期(询问弹窗时不知道是否要应用) |
| `/code fix` 流程 | 同上 |
| 已有 BUG-00009 修复产物 | 不受影响 |
| 与 `--confirm` 的关系 | `--confirm` 是阶段边界确认,与本缺陷无关 |

## 严重程度

**P2** — 不阻塞主流程,但用户体验不一致,与"评审-编码循环"的强约束不对齐。

## 与既有规范/缺陷的关系

- 与 BUG-00009(ver 兜底提交)同属"自动应用规则不明确"类缺陷
- 与 req/fix 的评审-编码循环(必须改)机制相关;本缺陷补充"建议改/可选改"两档的行为定义

## 修复方向(待 DESIGN 阶段确认)

候选方案:
- **方案 A**(推荐):在 `references/req/check.md` 与 SKILL.md 的 CHECK 阶段描述中,明确"必须改/建议改/可选改"三档的两态行为:
  - 默认模式:必须改 → 自动修复;建议改 → 逐项 AskUserQuestion(每项独立弹窗,选修改则走 CODING 修复;选跳过则保留);可选改 → 同建议改
  - `--auto` 模式:全部自动应用,不再弹窗
- **方案 B**:仅在 SKILL.md 描述中补充;不动 references
- **方案 C**:方案 A + 在 references/req/check.md 中加 `handleCheckFindings(findings, mode)` 函数伪代码

## 用户期望行为(原始需求拆解)

| 场景 | 期望行为 |
| --- | --- |
| 默认模式 + 必须改 | 走评审-编码循环,自动修复(已有) |
| 默认模式 + 建议改 | 弹 AskUserQuestion,问"是否修改";用户选修改则修复,选不修改则跳过 |
| 默认模式 + 可选改 | 同建议改 |
| `--auto` 模式 + 必须改 | 自动修复(已有) |
| `--auto` 模式 + 建议改 | 自动修复(本次新增) |
| `--auto` 模式 + 可选改 | 自动修复(本次新增) |

## 复现路径

无具体复现路径;这是规则定义缺陷,任何执行 req/fix CHECK 阶段时都会遇到。

## 复现材料

- 用户输入:`/code fix "<本次描述>"`
- 相关文件:`plugins cache .../SKILL.md` 与 `references/req/check.md`
