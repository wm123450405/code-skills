## 1. 任务概述

在 `references/ver/common.md` 中,把"从 RESULT.md 状态列读取"的两处算法(§4.1 发布检查 / §8.6 高优先级缺陷统计)改为统一调用契约层 `deriveItemStatus()`(定义在 `references/_shared/contracts.md` §3)。`deriveItemStatus` 的实现细节在契约层,本任务只做调用点改造。

## 2. 改动内容

### 2.1 文件顶部新增 §0 派生函数引用

- 提示本文件所有"派生状态"逻辑统一通过 `deriveItemStatus()` 实现
- 引用契约层 `references/_shared/contracts.md` §3
- 标注 RESULT.md 不再作为动态状态数据源(FR-1)

### 2.2 §4.1 `preflightCheck()` 算法重写(原第 614-639 行)

**改动前**:读 dashboard → parseSection → 判断 `req.状态.startswith("已完成")` / `defect.状态 != "已修复"`。

**改动后**:Glob 各需求/缺陷目录 → 对每个目录调 `deriveItemStatus(id)` → 判断 `not status.completed`。

**关键改动点**:
- 删除 `dashboard = Read(...)` 和 `parseSection(...)` 调用
- 改用 `Glob("assistants/${version}/req/REQ-*/")` 和 `Glob("assistants/${version}/fix/BUG-*/")`
- 新增辅助函数 `extractReqId()` / `extractBugId()` / `readReqMeta()` / `readBugMeta()` 的隐式依赖(契约层 §3 已定义 `ItemStatus` 结构)

### 2.3 §8.6 `countHighPriorityBugs()` 算法重写(原第 1147-1162 行)

**改动前**:遍历 bugRows → 取"优先级"/"状态"字段 → 跳过已完成的、统计 P0/P1。

**改动后**:Glob 缺陷目录 → 对每个缺陷读 BUG.md 的优先级 + 调 `deriveItemStatus()` → 仅统计 `status.completed=false` 的。

**关键改动点**:
- 函数签名增加 `version: string` 参数(原 `bugRows` 参数仅用于回退展示)
- 删除 `getField(bug, "状态")` 和 `status in ["已完成", "已修复-已验证", "已关闭"]` 判断
- 改用 `deriveItemStatus()` 的 `completed` 字段(契约层 §3 的判定公式)

## 3. 关键决策

- **决策 1**:`deriveItemStatus()` 不在本文件内重复定义,只在 §0 引用契约层
  - 理由:NFR-1 硬约束"单一事实来源";契约层是唯一权威,本文件内重复定义会导致漂移
  - 权衡:读者需要打开两个文件才能完整理解算法;换取契约一致性
- **决策 2**:不改 §8.5 状态归类 / §8.7 建议生成
  - 理由:这两处已经走 `parseLastStage(req.content)`,从 PROCESS.md 读取,本来就不依赖 RESULT 状态列
  - 验证:`grep "req\.状态\|defect\.状态" references/ver/common.md` 仅命中 §4.1/§8.6 之外的位置,均为其他无关内容
