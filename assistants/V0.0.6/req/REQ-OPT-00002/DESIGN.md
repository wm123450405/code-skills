## 1. 设计概述

本 REQ 的设计核心是**把"负向禁止"转化为"正向必须"**:对 95 条"不要做"事项做 3 分类处置(类 1 移除 / 类 2 转必须 / 类 3 保留),同时做全局精简单调。设计分三层:

1. **分类层**:每个 SKILL.md 扫描 → 逐条归类(类 1 / 2 / 3)
2. **改造层**:类 1 项移除 + 在原强制流程处加必须性;类 2 项改写为"必须做"句式,新设"必须做事项清单";类 3 项精简到 ≤ 3 条/文件
3. **精简单调层**:全技能消除繁复描述、移除空白段、保留原意

## 3. 接口设计

### 3.1 "必须做事项清单"段格式

**位置**:每个 SKILL.md 内,在"## 不要做的事"段**之前**

**格式**:
```markdown
## 必须做事项清单

> 以下事项为强制要求,违反即视为本技能执行失败。

- [事项 1]:<句式必须为正向,带"必须"或"强制">
- [事项 2]:<同上>
- [事项 N]:<同上>
```

**示例**(req/SKILL.md 的"必须做"清单片段):
```markdown
## 必须做事项清单

- 进入技能时**必须**完整执行 §0 不变式验证
- 阶段开始/完成**必须**追加 `PROCESS.md` 一行(详见 common.md §3)
- 修改既有规则文件时**必须**用 Edit 在末尾追加;新建分类**必须**用 Write 首次
- 修改 `RESULT.md` 时**必须**只动本技能负责的区段,不得越界
- 代码注释**必须**用功能描述,不得出现 REQ-NNNNN / BUG-NNNNN / TASK-* 追踪编号
- 阶段内澄清问题每轮**必须**聚焦 1-3 个最阻塞的点
- 新建/读取文件前**必须**先 Read 现有内容(避免覆盖)
- 阶段失败时**必须**追加 `PROCESS.md` 失败行,不得静默吞错
```

### 3.2 95 条事项分类模板

| 类 | 判定标准 | 处置 | 落地位置 |
| --- | --- | --- | --- |
| **类 1** | 事项内容已在某处强制(§0 不变式 / 阶段执行器 / 启动纪律自检表 / 契约层 / 子命令不变量) | 从"不要做"清单移除;在原强制处加"必须"/"强制"字样 | 原强制处(如 §0 不变式、common.md 阶段执行器) |
| **类 2** | 可改写为正向"必须做"句式(不是"必须不做") | 改写句式,合并到"必须做事项清单" | 该 SKILL.md 的"## 必须做事项清单"段 |
| **类 3** | 既不在流程中,也无法直接转"必须做" | 保留在"## 不要做的事"清单,但合并/精简到 ≤ 3 条/文件 | 该 SKILL.md 的"## 不要做的事"段 |

### 3.3 类 2 改写规则(强制)

| 原"不要做"句式 | 转"必须做"句式 | 示例 |
| --- | --- | --- |
| 不要 X | X 前必须 Y | "不要修改 CWD" → "修改 CWD 源码前必须确认 PROCESS.md 阶段 = CODING" |
| 不要 X | 必须(强制)做 Y 而非 X | "不要用 Write 覆盖" → "修改既有文件必须先用 Read 看现状" |
| 不要 X | X 时必须遵守 Y | "不要臆造内容" → "导出 FAQ 时必须只复制源文档原文" |
| 不要在没有 A 的情况下 X | 做 X 前必须先有 A | "不要在没有 current-version 时继续" → "执行前必须读取 ./assistants/.current-version,不存在则提示先调 /code ver" |
| 不要做"X-Y"复合操作 | "X-Y" 是错误路径,必须按正确路径 | "不要回滚已 commit 的状态" → "冲突未解决时必须返回非 0,不得回滚已 commit" |

**反例(不允许)**:"必须不做 X" / "必须禁止 X" / "严禁 X" 这类仍带负向的句式 → **禁止**。

### 3.4 精简单调原则

| 项 | 操作 |
| --- | --- |
| 同一规则出现在 2 处以上 | 删除冗余,只在 1 处保留(优先级:§0 不变式 > 契约层 > 必须做清单 > 阶段执行器 > 不要做清单) |
| 段标题存在但内容为空或仅 1-2 行无信息量 | 删除该段 |
| "## 衔接" / "## 上游/下游" 等元信息段(若内容已在 §0 或契约层说明) | 删除 |
| 长链重复强调("严禁 / 不要 / 必须 / 强制"在同一段 3 次以上) | 整合为 1-2 次 |
| 表格行内容重复 | 合并行 |
| 引用块内重复主表内容 | 删除引用块,保留主表 |

## 5. 关键流程

### 5.1 整治流程(每个 SKILL.md 重复)

```
1. Read 整个 SKILL.md
2. 用 grep / 人工识别 "## 不要做的事" 段所有条目
3. 逐条归类(类 1 / 类 2 / 类 3),写入 TASK-N 附录
4. 类 1 项:在原强制流程处加 "必须" / "强制" 字样;从 "不要做" 清单删除
5. 类 2 项:改写为 "必须做" 句式;加入 "## 必须做事项清单" 段
6. 类 3 项:精简到 ≤ 3 条/文件(同义合并)
7. 精简单调:消除繁复描述、移除空白段(参见 §3.4)
8. 验证:AC-2 / AC-3 / AC-4 / AC-5
```

### 5.2 精简判定流程

```
对每个 SKILL.md / reference 文件:
  1. 扫描段落,识别空段 / 1 行内容段 / 重复强调段
  2. 判定是否在 NFR-1 / NFR-3 范围内(不改 frontmatter / 不增长行数)
  3. 若可删:删除;若需保留:精简到 1-2 行
  4. 复查 NFR-3(行数不增长)
```

## 6. 方案选型

### 决策 D-1:"必须做事项清单"段位置
- **选择**:放在 "## 不要做的事" 段**之前**
- **备选**:
  - A. 放在 frontmatter 之后 §0 之前 — 优点:最先看到;缺点:与 §0 不变式视觉混淆
  - B. 放在文末 — 缺点:不符合"先看必须,再看不要"的心智模型
- **理由**:opt.md §4.6 建议"补齐完成条件(可勾选)";先看"必须做什么",再看"不要做什么"是更清晰的心智模型
- **权衡**:在 "不要做" 之前会让"必须做"段先被读到,但对老读者(习惯先看不要做)略不习惯

### 决策 D-2:类 3 项保留上限
- **选择**:每个 SKILL.md ≤ 3 条
- **备选**:
  - A. ≤ 5 条 — 较宽松,但仍偏多
  - B. ≤ 1 条 — 最严格,但可能丢失关键约束
- **理由**:opt.md §4.7 提到"用正向流程补,而不是禁止清单";3 条是经验值(过多会失去焦点,过少会漏关键约束)

### 决策 D-3:精简单调的"原意保留"判定
- **选择**:用 git diff 评审整治前后的 SKILL.md,逐段对比,确保核心约束未丢失
- **备选**:
  - A. 仅文字评审(可能漏判)
  - B. 用 grep 检索关键约束词("禁止 / 必须 / 严禁"等)出现次数是否保留
- **理由**:diff 是最直接的"原意保留"验证
- **权衡**:评审成本略高(每文件都要 diff),但精度高
