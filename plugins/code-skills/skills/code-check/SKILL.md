# code-check — 代码检查(版本感知)

## 目标
在代码合并前**系统化发现缺陷**,把可执行、可追溯、可派工的评审意见转交给 `code-it` 处理,使代码质量与设计/需求/规范保持一致。

## 适用场景
- 任何进入主干/共享分支的 PR/MR
- 重大重构、关键路径变更、安全敏感改动
- 整个需求的代码全部就绪后做整体评审
- 单个任务的增量评审(可重跑,基于新增的 code/RESULT.md)

## 不适用
- 当前**没有激活的版本工作空间**(请先调 `code-version`)
- 需求尚未在 PLAN.md 中拆分任务
- 任务的开发状态全部为 `待开始` 或 `已取消`
- 单元测试结果尚未确定(测试状态不在 `已运行-通过` / `不适用` / `已运行-失败` 等已落地状态)
- 紧急线上修复(走 hotfix 流程,不在本流水线)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
```

    ├── plan/

    │   └── <需求编号>/PLAN.md              # 上游任务计划(只读,但本技能会追加新任务)
    ├── code/
    │   └── <任务编码>/                     # code-it 产出(只读,作为评审上下文)
    │       └── RESULT.md
    ├── test/
    │   └── <任务编码>/                     # code-it 自含的 unit-test-results.md(沿用本需求 REQ-00034 FR-2/FR-3,
    │       └── RESULT.md
    └── review/
        ├── <需求编号>/                     # 本技能整体评审产物,可写
```

- **本技能不修改** `./assistants/rules/`、`<版本号>/require/`、`<版本号>/design/`、`<版本号>/plan/RESULT.md`、`<版本号>/code/`、`<版本号>/test/` 下的任何内容;`<版本号>/test/` 目录在 `code-unit` 退场后**仅保留** V0.0.2 / V0.0.3 既有历史档案(字节级保留,沿用本需求 REQ-00034 NFR-2)

- **本技能会创建** `<版本号>/review/<需求编号>/` 与 `<版本号>/review/<新任务编码>/` 下的文档

## 输入
- **需求编码**(必填)
  - **默认**格式:5 位纯数字 `^REQ-\d{5}$`(本仓库主动产出)
  - **接收**可放宽:`^REQ-[A-Za-z0-9.\-_]+$`(前缀 `REQ-` + 后缀 1+ 位字母数字/`.`/`-`/`_`;沿用 `encoding-conventions §规则 1` 接收端宽松正则,新规则 REQ-00025)
- **任务编码**(派生任务评审时,`review/<任务编码>/` 路径使用)
  - **生成端**沿用 5 位纯数字格式:`^TASK-(REQ|BUG)-\d{5}-\d{5}$`(默认)
  - **接收端**放宽为:`^TASK-(REQ|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$`(后缀自由)
- **上游需求**:`./assistants/<版本号>/require/<需求编号>/RESULT.md`(必须存在)
- **上游概要设计**:`./assistants/<版本号>/design/<需求编号>/RESULT.md`(必须存在,作为设计基准)
- **上游详细设计**:`./assistants/<版本号>/plan/<需求编号>/RESULT.md`(必须存在,作为详细基准)
- **上游任务计划**:`./assistants/<版本号>/plan/<需求编号>/PLAN.md`(必须存在,作为评审范围)
- **代码改修正文**:`./assistants/<版本号>/code/<任务编码>/RESULT.md`(每个任务,作为评审对象)
- **测试总结(可选,仅历史档案)**:`./assistants/<版本号>/test/<任务编码>/RESULT.md`(V0.0.2 / V0.0.3 既有 `code-unit` 产出,字节级保留,沿用本需求 REQ-00034 NFR-2;新任务**不**产此文件,由 `code-it` 步骤 8.5 自含的 `code/<任务>/unit-test-results.md` 替代)
- **项目级规范**:`./assistants/rules/` 下所有文件(建议存在)
- **当前项目代码**:CWD 下的源文件(实际评审对象)

## 输出
主产出物:
- `./assistants/<版本号>/review/<需求编号>/REVIEW-REPORT.md` —— 整体评审报告
- `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 中追加的"审查改修"类任务
- 每个新任务的 `./assistants/<版本号>/review/<新任务编码>/RESULT.md` —— 改修要求(给 `code-it` 直接消费)

辅助过程文档:
- `<版本号>/review/<需求编号>/work-log.md`
- `<版本号>/review/<需求编号>/review-checklist-applied.md`
- `<版本号>/review/<需求编号>/findings-no-task.md`
- 每个新任务的 `<版本号>/review/<新任务编码>/work-log.md`

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:

## 过程文档自适应判定

> 本节是 `code-check` 启动时执行"过程文档生成判定"的依据。AI 在执行过程中按本节的"不涉及/不适用"判定准则自适应决定本轮是否生成对应过程文档(评审级)。
> 详见 `templates/process-doc-decisions.md`(决策记录文件模板);本节与之配套。

### 适用过程文档清单(评审级)

| 过程文档 | 判定准则 |
| --- | --- |
| `work-log.md` | **始终生成**(评审过程记录) |
| `review-checklist-applied.md` | **始终生成**(评审清单应用记录) |
| `findings-no-task.md` | **有"建议改"或"可选"级别发现 → 生成;0 条 → 不生成** |
| 看板"变更记录" | **本轮有追加 → 生成;无变更 / 幂等空跑 → 不追加** |

### 执行流程

1. 步骤 0(版本检测)完成后,执行本步骤
2. 读上游输入(PLAN.md / code/<任务>/RESULT.md 等)
3. 对每类过程文档,按"适用过程文档清单(评审级)"判定
4. 收集决策结果 `decisions: [{type, decision, reason}]`
5. 若 `decisions` 存在"不生成" → `Write process-doc-decisions.md`
6. 若 `decisions` 全为"生成" → **不写** `process-doc-decisions.md`
7. 后续步骤按决策结果执行

### 边界与异常

- **E-1**:`process-doc-decisions.md` 写入失败 → 屏显警告,不阻断
- **E-2**:AI 判定犹豫 → 倾向**生成**
- **E-3**:`code-auto` 上下文 → 本步骤仍执行

- **不**修改 frontmatter(L1-3 字节级保留)
- **不**修改既有"## 工作流程"小节
- **不**修改"## 不要做的事"小节
- **不**触发 `AskUserQuestion`(NFR-3 强约束)
- **不**新增 CLI 参数(NFR-3 强约束)

## 标题解析(REQ-00013 新增)

> 适用对象:所有用户可见的屏幕输出位置(启动 / 完成 / 派生任务 / 评审发现 / 错误)
> 依据规范:FR-8.AC-8.1~8.4(评审启动 / 派生任务标题截断 / 评审发现报告 / 评审完成) + NFR-10 派生任务标题 ≤ 30 字 + NFR-3

**工具函数**(伪代码):

```ts
function truncateTitle(title: string, maxLen: number = 30): string {
  if ([...title].length <= maxLen) return title
  return [...title].slice(0, maxLen).join('') + '...'
}

function formatReqTitle(reqNum: string, title: string): string {
  return `${reqNum} · ${truncateTitle(title)}`
}

function formatTaskTitle(taskNum: string, title: string): string {
  return `${taskNum} · ${truncateTitle(title)}`
}
```

**标题解析入口**:

```ts
function parseResultTitle(filePath: string): string {
  const content = require('fs').readFileSync(filePath, 'utf-8')
  const match = content.match(/^# 需求提示词文档 — (.+)$/m)
  return match ? match[1] : ''
}

function parsePlanTaskTitle(planPath: string, taskNum: string): string {
  const content = require('fs').readFileSync(planPath, 'utf-8')
  const lines = content.split('\n').filter(l => l.startsWith('|') && l.includes(taskNum))
  for (const line of lines) {
    const cols = line.split('|').map(c => c.trim())
    if (cols[1] === taskNum && cols[5]) return cols[5]
  }
  return ''
}
```

**派生任务"标题"列截断**(D-5 选定 A,NFR-10 强约束):

```ts

```

- 兜底:使用本技能内置的 `checklists/review-checklist.md`
- 写入 `review-checklist-applied.md`,本次评审实际应用了哪些检查项

### 步骤 8 — 逐任务评审
对每个待评审任务,按以下顺序检查:

**8.1 正确性** —— 代码是否真的实现了任务所声明的功能
- 对应需求中的 FR / AC
- 对应详细设计中的关键逻辑
- 边界条件(空、零、负、极值、超长、并发、超时)是否处理
- 异常路径是否覆盖
- **不通过** → 标"必须改"

**8.2 规范遵循** —— 是否符合 `./assistants/rules/`
- 命名规范(变量/函数/类/文件/常量)
- 目录结构与模块边界
- 错误处理范式
- 日志/可观测性规范
- API 风格(REST/gRPC/事件/错误码)
- 数据建模(字段命名/类型/约束/索引)
- 安全规范(鉴权/加密/防注入/敏感数据)
- 性能规范(缓存/批量/异步)
- 测试规范(覆盖率门槛)
- **不通过** → 视条款的强制程度标"必须改"或"建议改"

**8.3 详细设计符合度** —— 是否与 `plan/RESULT.md` 一致
- 函数/类签名是否一致
- 数据结构字段是否一致
- 接口入参/出参/错误码是否一致
- 状态机迁移是否正确
- 偏离了设计但有合理理由 → 记录为"用户授权的偏离"(若 code/RESULT.md 已有则接受,否则询问)
- **不一致且无授权** → 标"必须改"

**8.4 安全** —— 是否引入安全风险
- 输入校验是否充分
- 鉴权/授权是否到位
- 敏感数据处理(密码/token/手机号)
- SQL/NoSQL 注入、命令注入、反序列化漏洞、XSS
- 依赖已知 CVE
- **不通过** → 标"必须改"

**8.5 性能** —— 是否引入明显性能问题
- 算法复杂度是否合理
- 是否有 N+1 查询、循环里的 IO、同步阻塞
- 资源使用(内存/连接/锁)是否合理
- **不通过** → 视影响标"必须改"或"建议改"

**8.6 可维护性** —— 是否便于他人理解和扩展
- 命名是否自解释
- 函数是否单一职责
- 是否有不必要的复杂度、过度抽象
- 注释是否解释"为什么"
- 魔数/硬编码
- **不通过** → 标"建议改"

**8.7 测试质量** —— 测试是否真正起到守护作用
- 是否有正常路径测试
- 边界/异常是否有测试
- 测试是否独立、可读
- mock 是否滥用
- 是否有未覆盖的关键路径
- **不通过** → 视覆盖度标"必须改"或"建议改"

**8.8 一致性** —— 与项目既有代码风格是否一致
- 命名、目录、文件组织
- 错误处理风格
- 日志格式
- **不通过** → 标"建议改"

**8.9 与上下游任务的一致性**
- 是否破坏其他任务的接口契约
- 是否有未声明的副作用
- **不通过** → 标"必须改"

**8.10 详设完整性**
- `plan/.../RESULT.md` 中每条任务的"涉及文件"必须能在 §4-§10 找对应章节
- 若任务的"涉及文件"未在 §4-§10 任何章节引用 → 派生"重构"类型任务(`触发/来源=设计变更`,关联任务=被修正原任务)
- **不通过** → 标"必须改"
- 实施:解析 `plan/<REQ>/PLAN.md` 任务总览表,提取每条任务的"涉及文件";解析 `plan/<REQ>/RESULT.md` §4-§10,提取所有"<文件路径>"引用;对每条任务的涉及文件,校验是否在 §4-§10 任一章节被引用

**8.11 概设越界检测**
- `design/.../RESULT.md` §7 / §8 / §9 不应出现"完整字段类型 / 完整错误码 / 完整约束 / 完整索引"等详设深度内容
- 触发的正则模式:
  - `\|\s*\w+\s*\|\s*(string|number|integer|boolean|datetime|UUID)\s*\|` —— 表格行含类型
  - `\b(错误码|error[_ ]?code)\s*[:=]?\s*\d{4,}` —— "错误码: 40001" 模式
  - `\b(鉴权|auth|authentication)\s*[:=]` —— "鉴权:" 模式
  - `\b(索引|index)\s*[:=]?\s*\w+\s*\(` —— "索引: <name>(" 模式
  - `\b(迁移脚本|migration)\s*[:=]` —— "迁移脚本:" 模式
- 命中任意一条 → 派生"重构"类型任务(`触发/来源=设计变更`,关联任务=被修正原任务)
- **不通过** → 标"必须改"

**8.12 行数比例警告**
- `design/.../RESULT.md` 行数 / `plan/.../RESULT.md` 行数 **> 1.2** → 标黄警告(`design > plan * 1.2` 通常意味着概设"过深")
- 阈值 = 1.2
- **不阻断** `code-check` —— 警告而非错误,供 `code-check` 评审人员参考
- 警告文案:`⚠ 概设/详设行数比例异常(<ratio>),建议复核概设是否过深`

**8.13 过程文档适配性**
- 检查本需求下所有过程文档**是否都满足 NFR-1(零空白文件)**
- 检查 `process-doc-decisions.md` 中"不生成"的过程文档**判定是否合理**:
  - AI 判定"不生成"的理由是否符合 §6 准则
  - 实际产出内容**不**支持生成该文档(即"真不需要"而非"省 token")
- **不通过** → 派生"建议改"任务(`触发/来源=审查改修`,关联任务=被修正原任务);**不**标"必须改"(因 AI 自适应判定本身有合理自由度)
- 实施:`Glob` `require/.../process-doc-decisions.md` → 解析"不生成"列表 → 对每条查 §6 准则 + 读相关上游文档验证"真不需要"
- 仅适用于 REQ-00035+ 的需求(V0.0.3 既有 15 需求 NFR-5 字节级保留,**不**触发 8.13)

每条发现记入"评审笔记",包含:

- 类别(正确性/规范/设计/安全/性能/可维护性/测试/一致性/接口)

## 整版本模式 — 评审范围与适用场景(本需求新增)

### 适用场景(3 类)

- **R-1**:版本发布前一次性"扫雷"——把当前版本所有"已完成"需求跑一遍,聚合所有发现和派生任务
- **R-2**:在切换新版本前,确认当前版本所有需求的"必须改"列表为空
- **R-3**:在 CI / 批处理场景下,把"整版本评审"作为版本验收的最后一步

### 评审范围

- **输入**:本版本 `./assistants/<版本号>/RESULT.md` 看板的"需求清单"区段
- **筛规则**:状态 = `已完成`(不归一化 `已完成(需求分析)` 到 `已完成`,PD-2 锁定;沿用 `code-dashboard` 行为)
- **输出**:每个被评审需求 1 份 `review/REQ-NNNNN/REVIEW-REPORT.md`(沿用模式 1 模板,INV-2 字节级一致)+ 1 份聚合 `./assistants/<版本号>/REVIEW.md`(5 章节,NFR-6 路径固定)

### 与模式 1 关系

- **模式 1**(`/code-check REQ-NNNNN`):单需求评审,既有,**字节级不变**(INV-1/4/11/12)
- **模式 2**(`/code-check` 无参):整版本评审,本需求新增,**完全复用模式 1 步骤 4-15** 单需求评审逻辑
- **互覆盖**:模式 2 包含模式 1 的所有产物(单需求 REVIEW-REPORT.md),并额外写 1 份聚合 REVIEW.md

### 不触发的区段

- **看板 "评审发现汇总" 区段**:整版本模式**不**写入此区段(沿用模式 1 单需求评审时由 code-check 写入);聚合文件 `REVIEW.md` 已涵盖
- **看板 "派生任务记录" 区段**:整版本模式**不**写入此区段(派生任务仍追加到各 `PLAN.md` 的"任务总览"表,沿用模式 1 既有路径)
- **新增区段**:0 触发(`dashboard-conventions §规则 1` 字段约定不扩展,INV-7 锁定)

### 待澄清未决项(留作 follow-up)

- **Q-8.1**:用 `code-rule` 沉淀 `review-conventions.md`(整版本模式的检查清单)— **不阻塞**,留作 follow-up
- **Q-8.2**:把 `code-check` 加入 REQ-00005 的"首步拉取+末步提交"改写范围 — **不阻塞**,留作 follow-up
- **Q-8.3**:在 `code-check/templates/` 新增 `REVIEW-ALL.md` 模板 — **本设计倾向不新增**(D-1.B);若 T-001 实施后实际行数 > 150,触发 R-8 重新评估
