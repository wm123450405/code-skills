# RESULT.md 模板 — 审查改修要求(给 code-it 消费)

> 本文件是 `code-check` 技能**为每个新派生的"审查改修"任务**产出的 `RESULT.md`。
> 保存到 `./assistants/<版本号>/review/<新任务编码>/RESULT.md`。
>
> **版本感知**:本文件存放在版本工作空间内,文档头应明确记录版本号。
>
> **关键不同**:这个 `RESULT.md` 不是给人类阅读的总结,而是给 **`code-it` 直接消费的工作输入**。
> 当 `code-it` 遇到类型=`修改` 且 触发/来源=`审查改修` 的任务时,**不读** `plan/RESULT.md`(详细设计),
> 而是**只读这份文档**作为本任务的全部输入。

---

## 文档头
```
# <新任务编码> — <一句话标题>

- 任务编码:<REQ-YYYY-NNNN-XXX>
- 需求编码:<REQ-YYYY-NNNN>
- 所属版本:<版本号>(如 v1.0.0)
- 任务类型:修改 / 重构 / 修复 / 新增
- **触发/来源:审查改修**  ← 关键字段,code-it 据此选择不同的输入源
- 严重程度:必须改 / 建议改
- 触发时间:YYYY-MM-DD HH:mm
- 触发者:code-check(<评审者>)
- 关联原任务:<T-YYY, T-ZZZ>(被审查发现问题的任务)
- 关联评审报告:./assistants/<版本号>/review/<需求编号>/REVIEW-REPORT.md §X
- 状态:待开始
- 责任人:<code-it 执行者>
- 当前版本:v1
```

## 1. 任务概述
用 2-5 句话概括:**本任务要修正什么(由 code-check 发现的问题),为什么需要修正,修正后达到什么效果**。

## 2. 问题清单

每条问题独立编号,包含位置、类别、描述、建议改修方案:

- **依据**:

  - 关联原任务:T-YYY 的 code/RESULT.md §X
- **现象**:
  ```

  ```

## 4. 不需要做的(避免越界)
> 重要:明确列出"**本任务不做的事**",避免 code-it 越界。

- 不重构与本任务无关的代码(即使看起来不顺眼)
- 不修改接口签名(除非 F-X 显式要求)
- 不修改测试代码(除非 F-X 显式要求)
- 不动 `plan/RESULT.md`(那是上游设计,本任务是实现层)
- 不动 `code/<原任务>/RESULT.md`(那是历史档案)
- **不要扩展任务范围**:即使在改修过程中发现其他问题,记到 `deviations.md`,不擅自修复
