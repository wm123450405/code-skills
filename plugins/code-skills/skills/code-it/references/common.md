# 通用流程细节 — code-it

> 本文件为 code-it 技能提供语言无关的通用流程细节。始终加载。

## §1 步骤 0a — 前置任务守卫
- 按 PLAN.md 任务总览文件行序判定"当前任务的前置任务是否全部完成"
- 前置任务 = 当前任务之前的所有任务
- 未完成 → 中止 + 打印推荐命令
- PLAN.md 不存在/解析失败 → 守卫通过(软失败退化)

## §2 步骤 0 — 版本上下文检测
1. Read .current-version
2. 不存在 → 中止

## §3 步骤 5 — 读取上游文档
- 触发/来源 = 审查改修 → 读 review/<TASK>/RESULT.md
- 其他触发/来源 → 读 plan/<REQ>/RESULT.md(详细设计)
- 必要时读 design/<REQ>/RESULT.md 和 require/<REQ>/RESULT.md

## §4 步骤 7 — 处理任务前置依赖与状态
- 待开始 → 推进为进行中,更新 PLAN.md
- 进行中/阻塞 → 继续
- 已完成 → 询问是否重做
- 已取消 → 错误

## §5 步骤 8 — 实施开发
### 通用编码原则
- 贴合项目现有风格
- 命名自解释,避免缩写
- 边界条件显式处理,不吞异常
- 关键路径埋点
- 代码内注释解释"为什么"
- 不引入未经评审的新依赖
- 每次只做本任务范围的事

### 审查改修任务特殊规则
- 按 review/RESULT.md 的"应当改修的文件清单"展开
- 严格遵守"不需要做的"部分,不越界
- 发现其他问题 → 记到 deviations.md,不擅自修复

## §6 步骤 8a — 项目可测性守卫
7 项检查(仅检查模块目录):
1. package.json(含 scripts.test)
2. pyproject.toml(含测试配置)
3. Cargo.toml
4. go.mod
5. pom.xml
6. build.gradle[.kts]
7. test/ 目录
至少 1 个模块命中 → testable = True

## §7 步骤 8.5 — 按需写单测
3 类自动判定:
- 文档类型 → 不写单测,写占位
- 函数级代码类 → 写单测 + 跑通
- 配置/类型定义 → 不写单测,写占位

## §8 步骤 8.6 — 逻辑行统计
- 检测工具:tokei > cloc > heuristic
- 变更文件:git diff --name-only HEAD~1
- 非 git 仓库 → 跳过

## §9 步骤 8.7 — 过程文档自适应判定
物化 decisions 字典,作为步骤 9/10/11/13/16 的输入

## §10 步骤 9 — 编译验证
1. 检测构建命令
2. Bash 执行构建
3. 记录到 compile-and-run.md
4. 失败 → 进入步骤 12 错误修复循环

## §11 步骤 10 — 启动运行验证
检测 → 执行 → 记录 → 失败处理

## §12 步骤 11 — 测试(若适用)
检测测试命令 → 执行 → 记录 test-results.md → 失败处理

## §13 步骤 12 — 错误修复循环
```
loop:
  if 编译/启动/测试 有错误:
    - 代码 bug → 修代码 → 重跑
    - 设计缺陷 → 记入 deviations.md,停下询问
    - 环境/依赖问题 → 尝试解决,记入 work-log.md
  else: 跳出循环
最多连续失败 5 次后必须停下询问用户
```

## §14 步骤 13-16 — 收尾

- 步骤 14.5:推进看板开发状态
- 步骤 15:同步版本看板
- 步骤 16:完善过程文档与汇报
