# code-it — 开发编码(版本感知)

## 目标
按 `PLAN.md` 中的一条具体任务,产出**可工作、可编译、可启动运行**的代码。保持与详细设计/概要设计/需求的一致性,不偏离任务边界。

## 适用场景
- 任何按 `PLAN.md` 单条任务执行的编码工作
- 重构或特性开发的具体落地
- Bug 修复的具体落地
- `code-review` 派生的"审查改修"任务的执行

## 不适用
- 当前**没有激活的版本工作空间**(请先调 `code-version`)
- 任务尚未在 `PLAN.md` 中存在
- 跨多任务的批量改动(应拆分为多次本技能调用)
- 紧急线上修复(走 hotfix 流程)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
本技能的目录粒度是**任务**:
```
./assistants/
├── rules/                                  # 项目级规范(跨版本,只读)
└── <版本号>/
    ├── RESULT.md                           # 版本看板(本技能追加"任务清单" / "缺陷清单" / "变更记录")
    ├── require/<需求编号>/RESULT.md        # 上游需求(只读)
    ├── design/<需求编号>/RESULT.md         # 上游概要设计(只读)
    ├── plan/
    │   ├── <需求编号>/RESULT.md            # 上游详细设计(只读)
    │   └── <需求编号>/PLAN.md              # 上游任务计划(只读,但本技能会更新本任务的开发状态字段)
    ├── review/<任务编码>/                  # 触发/来源=审查改修 时输入源(只读)
    │   └── RESULT.md
    └── code/
        └── <任务编码>/                     # 本技能产出,可写
            ├── RESULT.md                   # 改修总结
            ├── work-log.md
            ├── compile-and-run.md
            ├── deviations.md
            └── test-results.md (若适用)
```

- 路径以**当前工作目录(CWD)**为基准
- **本技能不修改** `./assistants/rules/`、`<版本号>/require/<编号>/`、`<版本号>/design/<编号>/`、`<版本号>/plan/<编号>/RESULT.md`、`<版本号>/review/<任务编号>/` 下的任何内容
- **本技能会修改** `<版本号>/plan/<需求编号>/PLAN.md` 中本任务的开发状态字段(只改这一处)
- **本技能会修改** CWD 下的项目源码(这是唯一允许的生产代码改动场景)

## 输入
- **任务编码**(必填):格式 `<需求编号>-<任务序号>`
- **上游详细设计**:`./assistants/<版本号>/plan/<需求编号>/RESULT.md`(必须存在;**触发/来源=审查改修 时不读**)
- **上游任务计划**:`./assistants/<版本号>/plan/<需求编号>/PLAN.md`(必须存在,且包含本任务)
- **审查改修输入**(仅当 `触发/来源=审查改修`):`./assistants/<版本号>/review/<任务编码>/RESULT.md`
- **关联原任务**(仅当 `触发/来源=审查改修`):`./assistants/<版本号>/code/<关联原任务编码>/RESULT.md`
- **项目级规范**:`./assistants/rules/` 下所有文件(建议存在)
- **当前项目代码**:CWD 下的源文件、构建配置、运行配置

## 输出
主产出物:
- `./assistants/<版本号>/code/<任务编码>/RESULT.md` —— 改修总结
- CWD 下的实际代码变更(diff/提交)
- `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 中本任务开发状态推进(待开始/进行中 → 已完成)

辅助过程文档(同任务目录下):
- `work-log.md`
- `compile-and-run.md`
- `deviations.md`
- `test-results.md`

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:
- 任务开发状态推进 → "任务清单"中本任务行更新
- 编译/运行命令执行 → "执行的开发命令记录"追加
- 发现缺陷 → "缺陷清单"追加
- 任意阶段 → "变更记录"追加

## 工具使用约定
- 改代码:`Edit` / `Write` —— 对 CWD 下源码
- 读设计/规范/计划:`Read`
- 探项目代码:`Glob` / `Grep` / `Read`
- **编译/运行/测试:`Bash`** —— 频繁使用,见步骤 9–12
- 跟踪变更:用 `work-log.md` 持续追加,或用 `git status` / `git diff`(若项目是 git 仓库)
- 与用户澄清:优先 `AskUserQuestion`;只在无法自行判断时用

## 修改文件前必须重读最新内容(强制)

为支持对同一文件的**并发开发**(多人/多任务/多会话可能同时编辑同一文件的不同位置),本技能在每次准备修改某个文件的指定位置时,**必须**遵循以下流程:

1. **重读**:先 `Read` 该文件的**最新**完整内容
   - 不复用本会话之前读过的内容
   - 不依赖上游 `code-design` / `code-plan` 中的引用(那些只是定位锚点,不等于文件当前状态)
   - 若该文件为新建(尚未存在),跳过此步
2. **重新定位**:基于最新内容,确定上游给出的"结构化锚点"(段落/章节/类型/方法/函数/成员/分支/循环)在当前文件中**具体落在哪一段文本**
   - 若结构化锚点已不存在(如上游设计后,其他任务已删除该方法)→ 停下,询问用户
   - 若结构化锚点存在但匹配到多个位置 → 用 `Read` 上下文(类前后、方法签名、注释)消歧;若仍无法唯一定位,停下,询问用户
3. **明确改动范围**:在 `Edit` / `Write` 调用前,本会话内部明确:
   - `old_string` 的具体内容(从最新文件中复制,不要从记忆中复用)
   - `new_string` 与 `old_string` 的差异
4. **执行改动 + 校验**:
   - 执行 `Edit` / `Write`
   - 必要时 `Read` 改后的相关段落(尤其是连续多步改动同一文件时),确认改动落到了预期位置,未误改其他位置

**为什么**:同一文件可能被其他任务、其他会话、其他协作者修改;若直接按"记忆中的旧行号 / 旧文本片段"使用 `Edit`,极易:
- 匹配到错误位置导致误改
- 因 `old_string` 不再唯一而 `Edit` 失败
- 与他人的并发改动产生冲突而不知

上游 `code-design` / `code-plan` 已约定使用**结构化语义**而非行号来描述"改哪里",正是因为只有"按结构定位 + 实时读取"才能稳定应对并发场景。本技能是该约定的**执行端**,必须落实"每次修改前先 Read 最新内容"。

**禁止**:
- 跳过 `Read` 直接按本会话早期读过的内容做 `Edit`
- 把上游文件中的代码片段当作 `old_string` 使用(那是设计稿,不是当前文件)
- 一次 `Edit` 失败后,在不重读的情况下反复重试(应先重读,再决定调整 `old_string` 还是停下询问)

## 标题解析(REQ-00013 新增)

> 适用对象:所有用户可见的屏幕输出位置(启动 / 完成 / 中止 / 错误 / 中止报告)
> 依据规范:FR-6.AC-6.2(`code-it` 中止报告含"REQ-NNNNN · 标题" + "TASK-... · 标题")+ NFR-3 字符数 ≤ 30 + REQ-00010 守卫中止报告升级

**工具函数**(伪代码):

```ts
function truncateTitle(title: string, maxLen: number = 30): string {
  if ([...title].length <= maxLen) return title
  return [...title].slice(0, maxLen).join('') + '...'
}

function formatTaskTitle(taskNum: string, title: string): string {
  return `${taskNum} · ${truncateTitle(title)}`
}

function formatReqTitle(reqNum: string, title: string): string {
  return `${reqNum} · ${truncateTitle(title)}`
}
```

**标题解析入口**:

```ts
function parsePlanTaskTitle(planPath: string, taskNum: string): string {
  const content = require('fs').readFileSync(planPath, 'utf-8')
  const lines = content.split('\n').filter(l => l.startsWith('|') && l.includes(taskNum))
  for (const line of lines) {
    const cols = line.split('|').map(c => c.trim())
    if (cols[1] === taskNum && cols[5]) return cols[5]
  }
  return ''  // E-3 退化
}

function parseResultTitle(filePath: string): string {
  const content = require('fs').readFileSync(filePath, 'utf-8')
  const match = content.match(/^# 需求提示词文档 — (.+)$/m)
  return match ? match[1] : ''
}
```

**屏幕输出格式契约**:

| 场景 | 格式 |
| --- | --- |
| 启动 | `正在处理: TASK-... · <任务标题>` |
| 完成 | `已完成: TASK-... · <任务标题>` |
| 中止(REQ-00010 守卫) | `⛔ code-it 中止(存在未完成的前置任务)\n正在处理: REQ-NNNNN · <需求标题>(任务 TASK-... · <任务标题>)` |
| 错误 | `✗ 错误: TASK-... · <任务标题>(<错误信息>)` |

**REQ-00010 守卫中止报告模板升级**(FR-6.AC-6.2 强约束):

```
⛔ code-it 中止(存在未完成的前置任务)

正在处理: REQ-NNNNN · <需求标题>(任务 TASK-... · <任务标题>)

前置任务状态:
  ✓ TASK-... · <任务标题>(开发状态=已完成)
  ✗ TASK-... · <任务标题>(开发状态=待开始)← 未完成
  ✗ TASK-... · <任务标题>(当前任务)

推荐执行 /code-it TASK-... · <任务标题> 完成后,再执行 /code-it TASK-... · <任务标题>
```

**边界与异常**:
- E-2:标题 > 30 字符 → `truncateTitle` 自动截断
- E-3:标题字段缺失 → 退化"编号+(无标题)"
- E-4:历史任务(REQ-00001~00003 旧格式)→ 透传旧格式 + 拼接标题(同 `code-dashboard` NFR-3)

**约束**:
- **不**使用"本任务"等指代词(FR-6.AC-6.4 强约束)
- **不**修改 frontmatter(L1-3 字节级保留,NFR-7)
- **不**修改既有章节(锚点 = "## 工具使用约定" 段后 + "## 工作流程" 前,本节为纯追加)
- **不**修改 `code-it` 末尾兜底提交逻辑(沿用 REQ-00005 既有 + REQ-00017 P-1 推进看板)

## 步骤 0a — 前置任务守卫(REQ-00010 新增)

> 适用对象:本技能在"步骤 0 版本上下文检测"之前,判定"当前任务的前置任务是否全部完成"
> 依据规范:FR-1(新增守卫)+ FR-2(中止 + 打印推荐命令)+ NFR-3(零规范变更 — 沿用 `PLAN.md` 任务总览文件行序,零字段新增)+ NFR-6(`PLAN.md` 缺失守卫通过退化)+ NFR-4(与 `code-auto` 退出码兼容)
> 触发条件:**仅**在步骤 1 判定为任务分支(任务编码匹配 `^TASK-REQ-\d{5}-\d{5}$` 5+5 位嵌套式)时触发;缺陷分支(`^TASK-BUG-\d{5}-\d{5}$`)守卫不触达
> 复用工具函数(沿用 §"标题解析(REQ-00013 新增)"小节,**不**重写):`truncateTitle` / `formatTaskTitle` / `formatReqTitle` / `parsePlanTaskTitle` / `parseResultTitle`

### 步骤 0a.1 目标

在 `code-it` 启动时(步骤 0 之前)按 `plan/REQ-NNNNN/PLAN.md` 任务总览区段的**文件行序**判定"当前任务的前置任务是否全部完成",未完成则中止 + 引导用户先执行前置任务;`PLAN.md` 不存在或解析失败时守卫**通过**(软失败退化),不阻断 `code-it`。

### 步骤 0a.2 触发条件

| 条件 | 守卫触达? |
| --- | --- |
| 输入匹配 `^TASK-REQ-\d{5}-\d{5}$`(任务分支,新格式) | **是** |
| 输入匹配 `^TASK-BUG-\d{5}-\d{5}$`(缺陷分支) | 否(走 §"缺陷分支" 17-25) |
| 输入匹配 `^(REQ\|BUG)-\d{5}-\d{5}$`(旧格式,透传) | **是** |
| 输入不匹配上述任一格式 | 否(由原 §"步骤 1" 报错) |
| 无 `./assistants/.current-version` | 否(由原 §"步骤 0" 报错) |

### 步骤 0a.3 算法

```
function preTaskGuard(taskNum, version):
  // 1. 反推所属需求(taskNum 第 3 段为 reqNum 数字,5+5 位嵌套式)
  //    格式: TASK-REQ-NNNNN-NNNNN → reqNum = NNNNN
  if not taskNum.startsWith('TASK-REQ-'): return  // 缺陷分支守卫不触达

  // 2. 读 PLAN.md(NFR-6 退化)
  planPath = `./assistants/${version}/plan/${reqNum}/PLAN.md`
  if not fileExists(planPath):
    log('⚠ code-it 前置任务守卫:PLAN.md 不存在,守卫通过(退化)')
    return

  // 3. 解析任务总览区段(沿用 dashboard-conventions §规则 1 锚点)
  //    锚点: ^## 任务总览$
  //    行匹配: ^\| .* \|$  (表格行)
  //    列锚点: 第 1 列 = 任务编号 / 第 5 列 = 标题 / 第 6 列 = 开发状态
  tasks = parsePlanOverview(planPath)  // 按文件行序

  // 4. 找到当前任务位置
  idx = tasks.findIndex(t => t.num === taskNum)
  if idx == -1: return  // 任务不在 PLAN.md — 由原步骤 2 报错

  // 5. 前置任务 = 当前任务之前的所有任务(按文件行序)
  preTasks = tasks.slice(0, idx)
  if preTasks.length == 0:
    log('✓ code-it 前置任务守卫通过(无前置任务)')
    return

  // 6. 判定每个前置任务的"开发状态"
  //    "未完成"定义(沿用 Q-2 锁定 A):开发状态 ∈ {待开始, 进行中, 阻塞}
  //    "已完成"定义:开发状态 = 已完成
  unfinished = preTasks.filter(t => t.devStatus != '已完成')
  if unfinished.length == 0:
    log('✓ code-it 前置任务守卫通过(全部前置任务已完成)')
    return

  // 7. 守卫不通过 → 打印中止报告 + 推荐命令 + exit 1
  logAbortReport(reqNum, taskNum, tasks, unfinished)
  exit(1)
```

### 步骤 0a.4 通过条件

全部前置任务开发状态 = `已完成` **或** 无前置任务(当前任务在任务总览中位于最前) **或** `PLAN.md` 不存在 / 解析失败(软失败退化)。

通过时屏幕输出(按场景):

```
✓ code-it 前置任务守卫通过(无前置任务)
  任务:<任务编码> · <任务标题>(位于 PLAN.md 任务总览最前)
```

```
✓ code-it 前置任务守卫通过(全部前置任务已完成)
  任务:<任务编码> · <任务标题>(前置 N 个,全部已完成)
```

```
⚠ code-it 前置任务守卫:PLAN.md 不存在,守卫通过(退化)
  任务:<任务编码> · <任务标题>(无 PLAN.md,无法判定前置)
```

### 步骤 0a.5 不通过处理

任一前置任务开发状态 ≠ `已完成` → 打印"⛔ code-it 中止(存在未完成的前置任务)" + 详细未完成清单 + 推荐执行命令 + `exit 1`(NFR-4 `code-auto` 据此中断)。

屏幕输出模板(沿用 §"标题解析" REQ-00010 升级版,FR-6.AC-6.2 强约束):

**关键约束**(FR-2.AC-2.5~2.7):

- 用户复制即可执行

### 步骤 0a.6 退化条件(NFR-6 / 软失败)

| 场景 | 处理 |

### 步骤 0a.7 边界与异常(本小节子节,本需求 REQ-00020 M-4 归并 E-1 / E-4 / E-8 / E-9 为职责归属表)

#### E-2:守卫不通过(存在未完成前置)
- 打印中止报告(§"步骤 0a.5" 模板)+ 推荐命令(指向第一个未完成前置)+ `exit 1`

#### E-3:`PLAN.md` 不存在(NFR-6 退化)
- 视为"无 PLAN.md,无法判定前置",守卫**通过**
- 屏幕输出 `⚠ code-it 前置任务守卫:PLAN.md 不存在,守卫通过(退化)`

#### E-5:`PLAN.md` 任务总览区段格式错乱(无锚点 / 表格行解析失败)
- 视为"无法判定",守卫**通过**(软失败)
- 屏幕输出 `⚠ code-it 前置任务守卫:PLAN.md 任务总览区段格式错乱,守卫通过(退化)`

#### E-6:`code-auto` 调 `code-it` 时守卫不通过
- 退出码 1 → `code-auto` 中断(NFR-4)
- 既有 `code-auto` 任务循环"按 PLAN.md 任务总览行序"调 `code-it` → 守卫**总是通过**(前置任务必已完成)
- 本需求守卫作为"双保险"(REQ-00007 协同 NFR-4)

#### E-7:多个未完成前置
- 打印**所有**未完成前置(屏幕输出多个 ✗ 行)
- 推荐命令**只**指向**第一个**未完成前置(沿用 `--minimal` 锁定,不展开为多命令)

- 沿用 §"标题解析" E-3 退化:屏幕输出"任务编码+(无标题)"(无 `parsePlanTaskTitle` / `parseResultTitle` 结果时)
- 不阻断守卫判定(判定仅依赖"开发状态"列,**不**依赖"标题"列)

### 步骤 0a.8 性能

- **总耗时**:**< 1 秒**(NFR-8 强约束)
  - 单 `PLAN.md` 解析:Read 一次(典型 5-50 KB,<< 1 秒)
  - 任务总览表格行扫描:O(n),n = 任务数(典型 1-20)
  - 任务比较:O(n)
  - **实测**:**<< 1 秒**(实际 0.1 秒级)

### 步骤 0a.9 与既有"步骤 7 处理任务前置依赖与状态"的协同

- **既有"步骤 7"**(原行为,不修改):检查 `PLAN.md` 任务行"前置任务"**字段**(`关联任务` 列),显式字段判定
- **本守卫"步骤 0a"**(新增):按 `PLAN.md` **文件行序**判定,**不**依赖任何字段
- **职责正交**:两种机制覆盖不同场景
  - 步骤 0a:兜底"PLAN.md 内所有任务按登记顺序串行"(FR-1 强约束)
  - 步骤 7:兜底"PLAN.md 内任务有显式依赖字段"(如 BUG 任务的"关联任务",可选字段)
- **并存**:不修改 §"步骤 7" 既有内容(FR-3.AC-3.1 + INV-2/3)
- **冲突处理**:步骤 0a 先判定(更严格);若步骤 0a 通过,步骤 7 仍按原"显式字段"逻辑判定

### 步骤 0a.10 与 `code-auto` 的协同(NFR-4 双保险)

- `code-auto` 任务循环"按 PLAN.md 任务总览行序"调 `code-it`(REQ-00007 FR-4.AC-4.3)
- 步骤 0a 守卫**总是通过**(`code-auto` 按序调 → 前置任务必已完成)
- 用户手动乱序调 `code-it` → 步骤 0a 守卫拦截
- 守卫不通过 → 退出码 1 → `code-auto` 中断(沿用 FR-7.AC-7.1)

## 工作流程

### 步骤 0 — 版本上下文检测(强制前置)
1. 读取 `./assistants/.current-version`
2. **文件不存在** → 立即停下,告知用户先调 `code-version`
3. 读取内容,记为 `<版本号>`,后续所有路径用 `assistants/<版本号>/...`
4. 验证 `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 存在(任务编码解析后)

### 步骤 1 — 收集并解析输入 ID
1. 若用户未提供,主动询问
2. **判定输入类型**:
   - 匹配 `^REQ-\d{4}-\d{4}-\d{3}$` 或用户约定的任务编码格式 → **任务分支**(走步骤 2-16)
   - 匹配 `^BUG-\d{3}$` → **缺陷分支**(走步骤 17-25)
3. **任务分支解析**:
   - 需求编码 = 任务编码去掉前缀 `TASK-REQ-` 与末尾 `-<任务序号>`,如 `TASK-REQ-00001-00001` → `REQ-00001`
   - 任务序号 = 末尾五位数字,如 `00001`
4. **缺陷分支解析**:
   - 输入直接作为 `<缺陷编号>`,无需拆分

### 步骤 2 — 校验任务存在
1. 读取 `./assistants/<版本号>/plan/<需求编号>/PLAN.md`
2. 在"任务总览"中查找本任务编号
3. **未找到** → 错误:
   > 错误:`PLAN.md` 中未找到任务 `<任务编码>`。请先运行 `code-plan` 拆分任务,或在 `PLAN.md` 中补充本任务。
4. **找到** → 记录该任务的当前状态、类型、关联任务、前置任务

### 步骤 3 — 定位 / 创建工作目录
1. 检查 `./assistants/<版本号>/code/<任务编码>/` 是否存在
2. 不存在 → `mkdir -p` 创建
3. 存在 → 复用(意味着之前可能重跑过,会基于上次的过程文档继续)

### 步骤 4 — 读取项目级规范
1. `Glob "./assistants/rules/**/*"`
2. 不存在/为空 → `AskUserQuestion` 询问是否继续(同 `code-design` 步骤 3 的处理)
3. 存在 → 逐个 `Read`,归类与摘要写入 `work-log.md` 起始区

### 步骤 5 — 读取上游文档
**关键**:根据任务的**触发/来源**字段,选择不同的输入源。

依次:
1. `Read "./assistants/<版本号>/plan/<需求编号>/PLAN.md"` —— 找到本任务,记录**触发/来源**字段
2. 根据触发/来源决定:
   - **`触发/来源 = 审查改修`**(由 `code-review` 派生):
     - `Read "./assistants/<版本号>/review/<任务编码>/RESULT.md"` —— **本任务的全部输入**(改修要求、文件清单、改修方案)
     - `Read "./assistants/<版本号>/code/<关联原任务>/RESULT.md"` —— 关联原任务的改修正文(理解上下文)
     - **不读** `./assistants/<版本号>/plan/<需求编号>/RESULT.md`(那是上游设计,本任务是修补,不是新设计)
   - **其他触发/来源**:
     - `Read "./assistants/<版本号>/plan/<需求编号>/RESULT.md"` —— 详细设计(主要依据)
     - 必要时:`Read "./assistants/<版本号>/design/<需求编号>/RESULT.md"` 与 `./assistants/<版本号>/require/<需求编号>/RESULT.md` 作为上下文
3. 把本任务涉及的关键信息摘录到 `work-log.md`:
   - 普通任务:PLAN.md §3 任务详情 + 详细设计对应章节
   - 审查改修任务:review/RESULT.md 的"问题清单"和"应当改修的文件"
4. **预检规范 vs 任务冲突**:
   - 普通任务:同 `code-design` 步骤 4
   - 审查改修任务:review/RESULT.md 的"关联依据"已明确引用的规范/设计条款,记录到 work-log.md

### 步骤 6 — 探索项目代码(核对实现风格)
- 项目类型与构建/运行命令检测(详见步骤 9/10)
- 既有相似功能的命名、错误处理、日志风格
- 既有测试用例的写法与覆盖度
- 任务涉及模块的当前状态

写入 `work-log.md` 的"项目现状"部分。

### 步骤 7 — 处理任务前置依赖与状态
1. **检查前置任务**:
   - 若本任务有"前置任务",确认这些任务在 `PLAN.md` 中状态为"已完成"或"已取消(可绕过)"
   - 若有前置任务未完成,主动询问用户:
     > 前置任务 <T-XXX> 当前状态为"<状态>",未完成。是否继续?
     > - A. 继续(我会自己处理前置任务的依赖,记录在 work-log.md)
     > - B. 暂停,先去完成前置任务
2. **处理当前任务状态**:
   - `待开始` → 推进为 `进行中`,更新 `PLAN.md`,在变更记录中记录
   - `进行中` / `阻塞` → 继续(可能是中断后回来)
   - `已完成` → 主动询问用户:
     > 任务 <任务编码> 已标记为"已完成"。是否要重做?
     > - A. 重做(我会在 `deviations.md` 记录重新实现的原因,并把本任务关联到新任务)
     > - B. 取消(跳过)
   - `已取消` → 错误:
     > 任务 <任务编码> 已标记为"已取消"。如需重启,请先在 `PLAN.md` 中处理。

### 步骤 8 — 实施开发
**关键**:根据任务的**触发/来源**字段,采用不同的工作模式。

**普通任务**(`触发/来源` ≠ `审查改修`):
按 `PLAN.md` 中本任务的"关键变更"展开:
- 涉及文件/模块
- 接口/签名(若新增)
- 数据结构(若新增)
- 关键逻辑
- 边界与异常
- 验证手段

**审查改修任务**(`触发/来源` = `审查改修`,由 `code-review` 派生):
按 `<版本号>/review/<任务编码>/RESULT.md` 的"应当改修的文件清单"展开:
- 对每个文件,按"当前状态 → 应改为 → 理由"精确修改
- **严格遵守** `review/RESULT.md` 的"不需要做的"部分,**不越界**
- 改修过程中发现其他问题 → 记到 `deviations.md`,**不擅自修复**
- 改修完成后,跑 review/RESULT.md 的"验证手段"中列出的测试
- 关联原任务的测试**必须仍通过**(无回归)

**通用编码原则**(两种任务都适用):
- 贴合项目现有风格(命名、错误处理、日志)
- 命名自解释,避免缩写
- 边界条件显式处理,不吞异常
- 关键路径埋点,日志不要噪音
- 代码内注释解释"为什么",而非"是什么"
- **不引入未经评审的新依赖**(`dependencies.md` 在 code-design 阶段已确定)
- **每次只做本任务范围的事**;发现需要跨任务改动,记入 `deviations.md` 并停下询问用户

**过程记录**:
- 持续追加到 `work-log.md`(每完成一个文件、每跑一次命令、每遇到一个问题都记)
- 重要的临时决策、踩坑、查到的关键信息都记
- **审查改修任务**额外记录:每条 review 发现(F-1, F-2, ...)的修复情况

### 步骤 9 — 编译验证
1. **检测构建命令**
2. **执行构建**:`Bash` 执行
3. **记录到 `compile-and-run.md`**
4. **失败处理** → 进入步骤 12 错误修复循环
5. **同步到版本看板**:在"执行的开发命令记录"追加本条命令及结果

### 步骤 10 — 启动运行验证
1. **检测运行命令**
2. **执行**:`Bash` 运行,设置超时
3. **验证启动成功**
4. **记录到 `compile-and-run.md`**
5. **同步到版本看板**
6. **失败处理** → 进入步骤 12

### 步骤 11 — 测试(若适用)
1. **检测测试命令**
2. **执行**:`Bash` 运行
3. **记录到 `test-results.md`**
4. **同步到版本看板**
5. **失败处理** → 进入步骤 12

### 步骤 12 — 错误修复循环
**关键原则:必须迭代修复直到编译通过、启动正常、测试通过**。

```
loop:
  if 编译/启动/测试 有错误:
    解析错误信息
    定位错误位置(行号/堆栈)
    判断错误类型:
      - 代码 bug(本任务范围内可修)→ 修代码 → 重跑
      - 设计缺陷(详细设计不可行)→ 记入 deviations.md,停下,询问用户
      - 环境/依赖问题(非代码 bug)→ 尝试解决(装依赖、配环境变量等),记入 work-log.md
      - 上游错误(PLAN.md / 详细设计 / 规范有错)→ 停下,询问用户
    重新跑对应验证步骤
  else:
    跳出循环
```

**最多连续失败 N 次后必须停下询问用户**:
- N 推荐设为 5
- 每次失败都要在 `work-log.md` 记录已尝试的修复与结果
- 停下时主动呈现错误信息、已尝试的修复、推断的根因、建议的下一步

**禁止**:
- 跳过验证直接进入步骤 13
- 在错误未消除的情况下标记任务为"已完成"
- 用 `--no-verify` / `--force` / 注释掉失败代码等方式绕过错误

### 步骤 13 — 撰写 RESULT.md(改修总结)
按 `templates/RESULT.md` 的结构生成,内容:
1. 任务信息(编码、标题、类型、来源 PLAN.md)
2. 改修内容总览(改了哪些文件、新增了哪些、删除了哪些)
3. 详细改动(每个文件的变更要点)
4. 关键决策与权衡(开发过程中做的选择)
5. 偏离设计/规范的地方(交叉引用 `deviations.md`)
6. 验证结果(编译、启动、测试)
7. 已知问题/未完成项(留给后续任务或 PR 评审)
8. 关联任务与提交(若项目是 git 仓库,记录提交哈希)

### 步骤 14 — 更新 PLAN.md(标记完成)
回到 `./assistants/<版本号>/plan/<需求编号>/PLAN.md`,只修改本任务相关字段:
- 状态:`进行中` → `已完成`
- 完成时间:YYYY-MM-DD HH:mm
- 完成人:<执行者>
- 提交哈希(若项目是 git 仓库):<git sha>
- 在 PLAN.md 的"变更记录"中追加:
  ```
  YYYY-MM-DD HH:mm  状态更新  T-XXX 状态"进行中"→"已完成",提交 <hash>
  ```
- 视情况小幅更新版本号(如 v1.1 → v1.2),但**不重写其他任务**

### 步骤 14.5 — 推进看板开发状态(REQ-00017 新增,2026-06-05 起生效)

**职责**:把本任务的看板"开发状态"列从"待开发"推进到"已完成"(本步骤**已经**是步骤 15 的核心职责,本节为**显式契约**)。

**与步骤 15 的关系**:步骤 15 "同步版本看板"中第 2 款"找到本任务行,更新开发状态:待开始/进行中 → 已完成" 即为本步骤的实施细节。本节只是把这条职责**显式地**与 `/code-plan` 拆任务约束对齐,确保不再有"更新看板"派生任务(详 `/code-plan` SKILL.md §"步骤 10A 任务拆分" → "#### 拆任务约束(REQ-00017 强约束)")。

**算法**(沿用步骤 15 既有实现,本节不重写):
1. `Read "./assistants/<版本号>/RESULT.md"`,定位"任务清单"区段(沿用 `dashboard-conventions §规则 1` 锚点 `^## 任务清单$`)
2. 找到本任务行(任务编码双格式匹配 `^TASK-(REQ|BUG)-\d{5}-\d{5}$` 或 `^(REQ|BUG)-\d{5}-\d{5}$`)
3. 把"开发状态"列从 `待开始` / `进行中` 推进到 `已完成`
4. 把"完成时间"列填入 `YYYY-MM-DD HH:mm`
5. 写回看板文件
6. 在"变更记录"区段追加(沿用既有"任务完成"事件类型,**不**新增枚举值):
   ```
   YYYY-MM-DD HH:mm  任务完成  <任务编码> <任务标题>(开发状态:已完成)  <任务编码>
   ```

**幂等性**:本任务"开发状态"已是"已完成"时,**不重复写**(步骤 15 既有逻辑)。

**失败处理**:解析失败 / 写回失败 → 步骤 15 既有"在'变更记录'追加开发状态更新"逻辑会捕获;`/code-it` 整体仍视为完成。

**边界**:
- 看板"任务清单"区段缺失 → 步骤 15 第 1 款解析失败会捕获
- 本任务行缺失 → 步骤 15 第 2 款解析失败会捕获
- 看板本任务行"开发状态"非"待开始"也非"进行中" → 步骤 15 第 2 款按既有逻辑处理

**与既有步骤的关系**:
- 步骤 14 路径**不变**(本步骤紧接其后,**不**改变其语义)
- 步骤 15 "同步版本看板"的实施细节即为本步骤
- 本节只是把步骤 15 的部分职责**显式**与 `/code-plan` 拆任务约束对齐,作为职责衔接的"接口契约"

### 步骤 15 — 同步版本看板(强制)

   - `提交哈希`:填入(若有)

## 衔接
- **下游**:
  - 任务分支:`code-unit` 与 `code-review` 在本任务完成后对本次变更展开;后续任务的 `code-it` 会以本任务的产出为依赖
  - 缺陷分支:用户调 `code-fix <缺陷编号>` 推进状态;可选 `code-review` / `code-unit`
- **上游**:
  - 任务分支:`code-version`(必须);`code-plan` 的 `PLAN.md` / `RESULT.md`;项目级规范
  - 缺陷分支:`code-version`(必须);`code-fix` 的 `fix/<缺陷编号>/RESULT.md`;`code-plan` 的 `fix-plan.md`;项目级规范
- **横向**:
  - 任务分支:`./assistants/<版本号>/code/<其他任务编码>/RESULT.md` 形成本次需求的所有任务执行档案
  - 缺陷分支:与同版本其他 `fix/<其他缺陷编号>/` 互不干扰

## 不要做的事
- 不要在没有 `./assistants/.current-version` 的情况下继续执行
- 不读 `PLAN.md` / `fix-plan.md` / `RESULT.md` / `rules/` 就直接动手
- 在编译/启动/测试有错误的情况下标记任务完成
- 用绕过手段(注释掉失败代码、`--no-verify`、`--force` 等)规避错误
- 一次性抛出所有澄清问题
- 跨任务做未授权的改动
- 修改 `./assistants/rules/` / `<版本号>/require/<编号>/` / `<版本号>/design/<编号>/` / `<版本号>/plan/<编号>/RESULT.md` 下的任何内容
- 修改 `PLAN.md` 中除本任务状态/完成字段/变更记录之外的任何内容
- 把代码改动保存到 `assistants/code/` 目录(那是文档目录,不是源码目录)
- **任务分支(审查改修)的特殊禁止**:
  - **不读** `<版本号>/plan/<需求编号>/RESULT.md`(那是上游设计,本任务是修补)
  - **不扩展任务范围** —— 即使改修中发现新问题,记到 `deviations.md`,不擅自修复
  - **不修改** `<版本号>/review/<任务编码>/RESULT.md` 中"不需要做的"部分
  - **不破坏** 关联原任务的现有功能(其测试必须仍通过)
- **缺陷分支的特殊禁止**:
  - **不在没有 `fix-plan.md` 的情况下继续** —— 必须先 `code-plan <缺陷编号>`
  - **不修改 `fix-plan.md`** —— 那是上游;若发现方案不可行,停下,询问用户(可能要回 `code-plan` 重新规划)
  - **不修改 `fix/<缺陷编号>/RESULT.md` 中"缺陷描述" / "根因分析"等稳定章节** —— 只更新状态字段、修复日志、变更记录
  - **不修改其他缺陷的 `fix/<其他缺陷编号>/`** —— 各缺陷独立
  - **不把缺陷修复的过程文档写到 `code/<任务>/`** —— 那是任务路径的目录;缺陷用 `fix/<缺陷>/` + `fix-` 前缀
  - **不直接关闭缺陷** —— 完成实施后状态是 `已修复-待验证`,验证通过后由 `code-fix` 推进到 `已修复-已验证`,再由 `code-fix` 关闭
- 修改 `./assistants/<版本号>/RESULT.md` 中非本技能负责的区段(本技能负责:任务清单 / 缺陷清单 / 执行的开发命令记录 / 变更记录)
