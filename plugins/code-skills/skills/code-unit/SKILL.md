# code-unit — 单元测试(版本感知)

## 目标
为 `code-it` 产出的代码补齐/编写单元测试,固化"代码应该如何被使用",降低回归风险。把 `PLAN.md` 中任务的**测试状态**推进到合格状态。

## 可改测试代码边界

本技能被允许**仅**修改 CWD 下的测试文件(如 `tests/` / `__tests__/` / `*.test.*` / `*.spec.*`)。**不得**修改 `plugins/code-skills/skills/*/SKILL.md` 或其他生产代码文件;生产代码改动由 `code-it` 唯一实施。本技能产出 `test/<TASK-...>/RESULT.md` 测试记录。

## 适用场景
- `code-it` 写完代码后,补齐/编写单元测试
- 既有代码的回归测试补齐
- 重构时确保行为不变的测试守护
- 修复 bug 时为防回归写测试
- `code-it` 阶段没写测试(或只写了正常路径),需要补全

## 不适用
- 当前**没有激活的版本工作空间**(请先调 `code-version`)
- 任务尚未在 `PLAN.md` 中存在
- 任务开发状态为 `已取消`
- 集成测试、E2E 性能测试(本技能专注**单元测试**)
- 紧急线上修复(走 hotfix 流程,不在本流水线)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
本技能的目录粒度是**任务**(与 `code-it` 同级但路径不同)——测试工作与开发工作分离存放:
```
./assistants/
├── rules/                                  # 项目级规范(跨版本,只读)
└── <版本号>/
    ├── RESULT.md                           # 版本看板(本技能追加"任务清单" / "缺陷清单" / "变更记录")
    ├── require/<需求编号>/RESULT.md        # 上游需求(只读)
    ├── design/<需求编号>/RESULT.md         # 上游概要设计(只读)
    ├── plan/
    │   ├── <需求编号>/RESULT.md            # 上游详细设计(只读)
    │   └── <需求编号>/PLAN.md              # 上游任务计划(只读,但本技能会更新本任务的"测试状态"字段)
    ├── code/
    │   └── <任务编码>/                     # code-it 产出(只读,作为上下文)
    │       ├── RESULT.md
    │       └── ...
    └── test/
        └── <任务编码>/                     # 本技能产出,可写
            ├── RESULT.md                   # 测试改修总结
            ├── work-log.md
            ├── test-results.md
            ├── coverage-analysis.md
            └── deviations.md
```

- 路径以**当前工作目录(CWD)**为基准
- **本技能不修改** `./assistants/rules/`、`<版本号>/require/<编号>/`、`<版本号>/design/<编号>/`、`<版本号>/plan/<编号>/RESULT.md`、`<版本号>/code/<任务编号>/` 下的任何内容
- **本技能会修改** `<版本号>/plan/<需求编号>/PLAN.md` 中本任务的**测试状态**字段及"单元测试状态"细节
- **本技能会修改** CWD 下的项目测试代码

## 输入
- **任务编码**(必填):格式 `<需求编号>-<任务序号>`
  - **生成端**沿用 5 位纯数字格式:`^TASK-(REQ|BUG)-\d{5}-\d{5}$`(默认,本仓库主动产出)
  - **接收端**放宽为:`^TASK-(REQ|BUG)-[A-Za-z0-9.\-_]+-[A-Za-z0-9.\-_]+$`(后缀自由;沿用 `encoding-conventions §规则 1` 接收端宽松正则,新规则 REQ-00025)
- **上游详细设计**:`./assistants/<版本号>/plan/<需求编号>/RESULT.md`(必须存在)
- **上游任务计划**:`./assistants/<版本号>/plan/<需求编号>/PLAN.md`(必须存在,且包含本任务)
- **代码改修正文**(可选但强烈建议):`./assistants/<版本号>/code/<任务编号>/RESULT.md`(来自 `code-it`)
- **项目级规范**:`./assistants/rules/` 下所有文件(建议存在)
- **当前项目代码与既有测试**:CWD 下的源码与测试

## 输出
主产出物:
- `./assistants/<版本号>/test/<任务编码>/RESULT.md` —— 测试改修总结
- CWD 下的实际测试代码新增/修改
- `./assistants/<版本号>/plan/<需求编号>/PLAN.md` 中本任务**测试状态**推进

辅助过程文档(同任务目录下):
- `work-log.md`
- `test-results.md`
- `coverage-analysis.md`
- `deviations.md`

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:
- 测试状态推进 → "任务清单"中本任务行更新
- 发现代码 bug → "缺陷清单"追加
- 任意阶段 → "变更记录"追加

## 工具使用约定
- 改测试代码:`Edit` / `Write` —— 对 CWD 下测试文件
- 读设计/规范/计划/代码摘要:`Read`
- 探项目代码与既有测试:`Glob` / `Grep` / `Read`
- **运行测试:`Bash`** —— 频繁使用,见步骤 10
- 跟踪变更:用 `work-log.md` 持续追加,或用 `git status` / `git diff`(若项目是 git 仓库)
- 与用户澄清:优先 `AskUserQuestion`;只在无法自行判断时用

## 标题解析(REQ-00013 新增)

> 适用对象:所有用户可见的屏幕输出位置(启动 / 完成 / 守卫跳过 / 错误)
> 依据规范:FR-7.AC-7.2(守卫跳过报告含"编号+标题")+ NFR-3 字符数 ≤ 30 + REQ-0009 守卫跳过

**工具函数**(伪代码):

```ts
function truncateTitle(title: string, maxLen: number = 30): string {
  if ([...title].length <= maxLen) return title
  return [...title].slice(0, maxLen).join('') + '...'
}

function formatTaskTitle(taskNum: string, title: string): string {
  return `${taskNum} · ${truncateTitle(title)}`
}
```

**标题解析入口**:

```ts
function parsePlanTaskTitle(planPath: string, taskNum: string): string {
  const content = require('fs').readFileSync(planPath, 'utf-8')
  const lines = content.split('\n').filter(l => l.startsWith('|') && l.includes(taskNum))
  for (const line of lines) {
    const cols = line.split('|').map(c => c.trim())
    if (cols[1] === taskNum && cols[5]) return cols[5]
  }
  return ''  // E-3 退化
}
```

**屏幕输出格式契约**:

| 场景 | 格式 |
| --- | --- |
| 启动 | `正在处理: TASK-... · <任务标题>` |
| 完成 | `已运行-通过: TASK-... · <任务标题>` / `已运行-失败: TASK-... · <任务标题>` |
| 守卫跳过(REQ-0009) | `⏭ code-unit 跳过: TASK-... · <任务标题>(项目不可测)` |
| 错误 | `✗ 错误: TASK-... · <任务标题>(<错误信息>)` |

**REQ-0009 守卫跳过屏幕输出模板升级**(FR-7.AC-7.2 强约束):

```
⏭ code-unit 跳过: TASK-... · <任务标题>(项目不可测)

原因:项目根无 7 项构建/测试文件/目录(任一):
  - package.json(含 scripts.test)
  - pyproject.toml(含测试配置)
  - Cargo.toml
  - go.mod
  - pom.xml
  - build.gradle(.kts)
  - test/ 目录

后续:测试状态 = 不适用(沿用 V0.0.1 既有枚举)
```

**边界与异常**:
- E-2:标题 > 30 字符 → `truncateTitle` 自动截断
- E-3:标题字段缺失 → 退化"编号+(无标题)"
- E-4:历史任务(旧格式)→ 透传 + 拼接标题

**约束**:
- **不**使用"本任务"等指代词
- **不**修改 frontmatter(L1-3 字节级保留,NFR-7)
- **不**修改"步骤 0a 守卫"既有逻辑(沿用 REQ-0009 既有 7 项检查)
- **不**修改 `## 边界与异常` 章节(沿用 REQ-0009 E-1 ~ E-8 既有)

## 工作流程

### 步骤 0a — 项目可测性检查(守卫,REQ-0009 新增)

> 本步骤是 `code-unit` 在"步骤 0 版本上下文检测"**之前**的**守卫**。**仅**当项目根存在 7 项构建/测试文件/目录(任一)时,才视为"项目可测",进入"步骤 0"+"步骤 1-16"原流程;否则视为"项目不可测",**跳过单测过程**,看板"任务清单"区段本任务"测试状态"列写入 `不适用`,**不**写 `test/<任务编码>/RESULT.md`,退出码 = 0(NFR-6 让 `code-auto` 继续)。

#### 步骤 0a.1 守卫检查项清单(7 项)

> 仅检查项目根,**不**递归子目录(避免误判)。命中任一 → 可测,全部不命中 → 不可测。

| 序号 | 文件/目录 | 类型 | 验证方式 | 备注 |
| --- | --- | --- | --- | --- |
| 1 | `package.json`(含 `scripts.test`) | Node.js / 前端 | `Glob` + `Read` 验证 `scripts.test` 字段非空字符串 | 必须含 `scripts.test`(避免"有 package.json 但无测试命令"误判) |
| 2 | `pyproject.toml`(含测试配置) | Python | `Glob` + `Read` 验证 `[tool.pytest]` / `[tool.pytest.ini_options]` / `[tool.tox]` / `[tool.nox]` 任一存在 | 4 种测试配置入口任一即可 |
| 3 | `Cargo.toml` | Rust | `Glob` 存在性检查 | 存在即视为可测 |
| 4 | `go.mod` | Go | `Glob` 存在性检查 | 存在即视为可测 |
| 5 | `pom.xml` | Java(Maven) | `Glob` 存在性检查 | 存在即视为可测 |
| 6 | `build.gradle` / `build.gradle.kts` | Java(Gradle) / Android | `Glob` 检查任一存在 | 任一存在即视为可测 |
| 7 | `test/` 目录 | 通用 | `Glob` 目录存在性检查 | 存在即视为有测试结构 |

#### 步骤 0a.2 守卫判定逻辑

```
1. 顺序执行 7 项检查(复杂度递减:复杂验证优先)
   ├─ 检查 1(package.json):Glob → 存在 → Read 验证 scripts.test
   ├─ 检查 2(pyproject.toml):Glob → 存在 → Read 验证测试配置
   ├─ 检查 3-6:仅 Glob 存在性
   └─ 检查 7:Glob 目录存在性
2. 命中任一 → testable = True → 屏幕输出"✓ 守卫通过" → 进入"步骤 0"
3. 全部不命中 → testable = False → 屏幕输出"⏭ 守卫不通过" → 进入"步骤 0a.3 跳过流程"
```

#### 步骤 0a.3 守卫不通过 → 跳过流程

```
1. 屏幕输出"⏭ code-unit 跳过(项目不可测)"(FR-7.AC-7.1)
2. 输出守卫检查详情(具体哪些文件命中/未命中)
3. **不**调用任何 `Bash`(FR-2.AC-2.4)
4. **不**写 `test/<任务编码>/RESULT.md`(FR-3.AC-3.1, Q-3 锁定 A)
5. **不**写 `work-log.md` / `test-results.md` / `coverage-analysis.md` / `deviations.md`(FR-3.AC-3.2)
6. 写看板"任务清单"区段:本任务行"测试状态"列 = `不适用`(FR-2.AC-2.1, Q-2 锁定 A 沿用 V0.0.1 既有枚举)
7. exit 0(NFR-6 让 `code-auto` 继续)
```

#### 步骤 0a.4 屏幕报告格式

**守卫通过**:
```
✓ code-unit 守卫通过(项目可测)进入正常流程

任务:<任务编码>
守卫检查:
  - package.json:✓ (含 scripts.test)
  - pyproject.toml:✗
  - Cargo.toml:✗
  - go.mod:✗
  - pom.xml:✗
  - build.gradle:✗
  - test/ 目录:✗

```

**守卫不通过**(FR-7.AC-7.1):
```
⏭ code-unit 跳过(项目不可测)

守卫检查:
  - package.json:✗
  - pyproject.toml:✗
  - Cargo.toml:✗
  - go.mod:✗
  - pom.xml:✗
  - build.gradle:✗
  - test/ 目录:✗

看板"任务清单"区段:测试状态 → 不适用
```

#### 步骤 0a.5 退出码契约

| 场景 | 退出码 | `code-auto` 响应 |
| --- | --- | --- |
| 守卫不通过 + 跳过(FR-2.AC-2.2, NFR-6) | 0 | 继续下一个任务 |
| 守卫通过 + 测试通过 | 0 | 继续下一个任务 |
| 守卫通过 + 测试失败 | ≠ 0 | 中断 |
| 守卫通过 + 崩溃 | ≠ 0 | 中断 |
| 任务编码不存在 / 无 `.current-version` | ≠ 0 | 中断(原行为) |

### 步骤 0 — 版本上下文检测(强制前置)
1. 读取 `./assistants/.current-version`
2. **文件不存在** → 立即停下,告知用户先调 `code-version`
3. 读取内容,记为 `<版本号>`,后续所有路径用 `assistants/<版本号>/...`

### 步骤 1 — 收集并解析任务编码
1. 若用户未提供任务编码,主动询问
2. **校验格式**:`^REQ-\d{4}-\d{4}-\d{3}$`(或其他用户约定的格式)
3. **解析**:
   - 需求编码 = 任务编码去掉前缀 `TASK-REQ-` 与末尾 `-<任务序号>`,如 `TASK-REQ-00001-00001` → `REQ-00001`
   - 任务序号 = 末尾五位数字,如 `00001`

### 步骤 2 — 校验任务存在
1. 读取 `./assistants/<版本号>/plan/<需求编号>/PLAN.md`
2. 在"任务总览"中查找本任务编号,记录其**开发状态**与**测试状态**
3. **未找到** → 错误:
   > 错误:`PLAN.md` 中未找到任务 `<任务编码>`。请先运行 `code-plan` 拆分任务,或在 `PLAN.md` 中补充本任务。

### 步骤 3 — 检查开发状态
- **开发状态 = `已取消`** → 错误,拒绝继续
- **开发状态 ∈ {`待开始`, `进行中`, `阻塞`}`** → 主动询问:
  > 任务 <任务编码> 开发状态为"<状态>",代码可能尚未实现。仍要继续编写单元测试吗?
  > - A. 继续(TDD 模式 / 我已自行确认代码可用)
  > - B. 暂停,先去完成开发
- **开发状态 = `已完成`** → 继续

### 步骤 4 — 检查测试状态,决定行为
| 当前测试状态 | 本技能行为 |
| --- | --- |
| `未编写` | 从头编写测试;成功后 → `已运行-通过` / `已运行-失败` |
| `已编写` | 补充遗漏测试场景 + 跑通全部;成功后 → `已运行-通过` / `已运行-失败` |
| `已运行-通过` | 询问:重新跑一次确认(回归验证)/ 补强覆盖 / 跳过 |
| `已运行-失败` | 修复失败的测试(可能是测试 bug 或代码 bug);成功后 → `已编写` → 跑通 → `已运行-通过` |
| `不适用` | 询问:业务变化需要重新评估 / 维持 `不适用`(须填理由) |
| `阻塞` | 询问:阻塞原因 / 是否已解除 / 转入其他分支 |

### 步骤 5 — 定位 / 创建工作目录
1. 检查 `./assistants/<版本号>/test/<任务编码>/` 是否存在
2. 不存在 → `mkdir -p` 创建
3. 存在 → 复用(可能之前重跑过,继续在 `work-log.md` 上追加)

### 步骤 6 — 读取项目级规范
1. `Glob "./assistants/rules/**/*"`
2. 不存在/为空 → `AskUserQuestion` 询问是否继续
3. 存在 → 逐个 `Read`,归类与摘要写入 `work-log.md` 起始区(关注"测试规范")

### 步骤 7 — 读取上游文档
依次:
1. `Read "./assistants/<版本号>/plan/<需求编号>/RESULT.md"` —— 详细设计(主要依据,关注 §12 测试要点 / §5 算法 / §6 数据结构)
2. `Read "./assistants/<版本号>/plan/<需求编号>/PLAN.md"` 中本任务的"单元测试状态"部分
3. `Read "./assistants/<版本号>/code/<任务编号>/RESULT.md"`(若存在)—— 实际改了什么、关键决策
4. 关键摘录写入 `work-log.md`

### 步骤 8 — 探索项目代码与既有测试
- 检测测试框架(Jest / Mocha / Pytest / Go test / JUnit / Vitest / ...):
  - 看 `package.json` 的 devDependencies
  - 看 `pytest.ini` / `pyproject.toml` 的 `[tool.pytest]`
  - 看 `go.mod` 是否包含 testing
  - 看 `pom.xml` / `build.gradle` 的 test 依赖
  - 看文件命名风格(`*.test.*` / `*_test.*` / `*Test.*` / `*.spec.*`)
- 检测测试命令与覆盖率命令
- 识别既有测试的命名、结构、风格
- 识别本任务涉及模块的既有测试覆盖度

写入 `work-log.md` 的"项目现状"部分。

### 步骤 9 — 推导测试点并编写测试

**测试点推导**:
- 从详细设计的"§5 算法与逻辑"提取每个算法的输入/输出
- 从"§6 数据结构完整变更"提取每条字段约束的验证
- 从"§7 接口细节"提取每个接口的入参/出参/错误码的覆盖
- 从"§8 异常处理"提取每个异常路径
- 从"§9 安全要求"提取鉴权/校验/敏感数据
- 从"§10 状态机"提取每个状态迁移
- 从"PLAN.md 任务详情的'边界与异常'"提取边界用例
- 从"PLAN.md 任务详情的'验证手段'"提取作者预期

**测试维度**(完整覆盖):
- 正常路径
- 边界条件(空、零、负、极值、超长、Unicode)
- 异常路径(参数错误、依赖不可用、超时、并发)
- 状态迁移(每个迁移 + 非法迁移)
- 鉴权/授权(若涉及)
- 性能/资源(若 §11 性能与资源有具体目标)

**测试编写原则**:
- 单一职责:一个测试一个断言意图(允许少量相关断言)
- 不测实现细节,测**行为**
- Mock 外部依赖(数据库、网络、文件系统、第三方服务),**不 Mock 被测对象**
- AAA 结构:Arrange → Act → Assert
- 命名可读:`test_<单元>_<场景>_<期望>`(或项目约定的风格)
- 独立性:测试间不共享状态,顺序无关
- 失败要可读:断言失败时能立刻定位问题
- 测试是文档:阅读测试即可理解被测代码的契约

**过程记录**:
- 持续追加到 `work-log.md`(每写一个测试、每跑一次命令、每改一次代码都记)
- 重要的临时决策(为什么这样 Mock、为什么这个边界值)记入

### 步骤 10 — 运行单元测试(含相关联模块的测试)
1. **检测测试命令**
2. **运行范围**:
   - **本任务的测试** + **与本任务共享模块的测试**(防回归)
   - 范围确定方式:
     - `git diff --name-only HEAD~1` 拿到本次变更的文件列表(若 git 仓库)
     - 找到所有依赖这些文件的测试文件
     - 或:`pytest <本任务测试文件> <相关模块测试目录>` / `go test ./<相关包>/...` / `npx jest <本任务> <相关模块>`
3. **执行测试**:`Bash`
4. **记录到 `test-results.md`**
5. **同步到版本看板**:本任务的命令追加到"执行的开发命令记录"
6. **失败处理** → 进入步骤 11 错误修复循环

### 步骤 11 — 错误修复循环
**关键原则:必须迭代修复直到全部通过,或明确标记"不适用"**。

```
loop:
  if 测试有失败:
    解析错误信息
    定位失败用例
    判断失败原因:
      - 测试代码 bug(本测试写错了)→ 修测试 → 重跑
      - 被测代码 bug(实际代码逻辑错)→ 不修代码!记录到 deviations.md,停下,询问用户
      - 环境/依赖问题(测试环境异常)→ 尝试解决,记入 work-log.md
      - 测试设计问题(场景本身不成立)→ 改测试,记入 deviations.md
    重新跑测试
  else:
    跳出循环
```

**最多连续失败 N 次后必须停下询问用户**:
- N 推荐设为 5
- 每次失败都要在 `work-log.md` 记录已尝试的修复与结果

**重要边界**:
- **不修复被测代码的 bug** —— 这是 `code-it` 的工作,本技能只记录
- **不删除失败用例** —— 让失败显式,留给 `code-it` 修复
- **禁用** `--testPathIgnorePatterns` / 跳过失败用例 / 注释掉测试 等绕过手段

### 步骤 12 — 覆盖率分析
1. **运行带覆盖率的测试**
2. **记录到 `coverage-analysis.md`**
3. **评估是否需要补测试**:
   - 若覆盖率低于项目规范(在 `./assistants/rules/` 中),或
   - 关键路径未覆盖(状态机迁移、安全边界、性能目标)
   - 则**回步骤 9 补测试** → 步骤 10 重跑
4. **豁免**:对简单 getter/纯转发/外部依赖代码,记录豁免理由

### 步骤 13 — 撰写 RESULT.md(测试改修总结)
按 `templates/RESULT.md` 的结构生成,内容:
1. 任务信息
2. 测试范围总览(本任务测试 + 相关模块测试)

## 衔接
- **上游**:`code-version`(必须);`code-it` 的 `RESULT.md`(代码改修正文);`code-plan` 的 `PLAN.md` / `RESULT.md`;项目级规范
- **下游**:
  - `code-check` 消费本任务的测试代码与 `RESULT.md`
  - 后续任务的 `code-unit` 会复用本任务的测试模式
- **横向**:`./assistants/<版本号>/test/<其他任务编码>/RESULT.md` 形成本次需求的所有测试工作档案
- **与 code-it 的协作**:
  - 若步骤 11 发现代码 bug → 写进 `deviations.md`,**不修代码**;通知用户回到 `code-it` 修
  - 修完 bug 后,可重跑 `code-unit` 验证

## 不要做的事
- 不要在没有 `./assistants/.current-version` 的情况下继续执行
- 不读 `PLAN.md` / `RESULT.md` / `code/RESULT.md` / `rules/` 就直接动手
- 在测试有失败的情况下把测试状态标为"已运行-通过"
- 用 `--testPathIgnorePatterns` / 跳过失败用例 / 注释测试 等手段绕过失败
- **修改被测代码** —— 这是 `code-it` 的工作
- 修改 `./assistants/rules/` / `<版本号>/require/<编号>/` / `<版本号>/design/<编号>/` / `<版本号>/plan/<编号>/RESULT.md` / `<版本号>/code/<任务编号>/` 下的任何内容
- 修改 `PLAN.md` 中除本任务**测试相关字段**之外的任何内容(尤其是开发相关字段)
- 把测试代码改动保存到 `assistants/test/` 目录(那是文档目录,测试代码应在 CWD 下的项目测试目录)
- 修改 `./assistants/<版本号>/RESULT.md` 中非本技能负责的区段(本技能负责:任务清单 / 缺陷清单 / 执行的开发命令记录 / 变更记录)
