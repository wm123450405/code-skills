# code-answer — 只读功能查询

## 目标

为 `code-skills` 用户提供一个**只读型**"功能逻辑查询"入口:在用户调 `/code-answer <查询>` 时,自动检索项目内**所有历史需求**(`code-init` 登记的 `EXISTING-NNN` 基线 + `code-require` 登记的 `REQ-NNNNN` 开发需求),结合**历史需求变迁**(`RESULT.md` §13 变更记录),给出"该功能的主要功能 + 关键逻辑"回答;若需求未登记或登记不详细,自动扫描项目源代码补足功能逻辑;最终以**屏显报告**形式呈现,**不**产生任何结果文件、**不**产生任何过程文件,完全无副作用。

## 适用场景

- **场景 1 — 回看需求细节**:在长会话中忘记某条需求的细节 → 调 `/code-answer "code-publish 的发布前置检查"`,秒出回答 + 引用
- **场景 2 — 跨版本演进摸底**:想知道某条需求在 V0.0.1 → V0.0.2 → V0.0.3 之间是怎么改的 → 调 `/code-answer REQ-00023`
- **场景 3 — 需求未登记时的源代码补足**:用户问"用户登录功能",但项目没登记此需求 → 调 `/code-answer "用户登录"`,扫描源码补足
- **场景 4 — 新成员快速理解项目**:调 `/code-answer "<功能关键字>"`,获取基线 + 历次演进
- **场景 5 — 验证需求存在性**:调 `/code-answer REQ-00025`,直接定位到对应 `RESULT.md`

## 不适用

- 想"修改"任何工作空间文件(本技能**严禁** `Write` / `Edit`)
- 想"修改版本"或"切换版本" → 用 `code-version`
- 想"添加 / 修改需求" → 用 `code-require`
- 想"修改编码规范" → 用 `code-rule`
- 想"查看版本进度看板" → 用 `code-dashboard`(本技能与 `code-dashboard` 正交,前者查功能定义,后者查状态)
- 紧急线上修复(走 hotfix 流程)

## 工作目录约定(强制)

```
./assistants/
├── .current-version              # 当前激活版本标记(本技能读,缺失时降级全版本扫描)
├── rules/                        # 项目级规范(本技能不读)
└── <版本号>/
    └── require/                  # 扫描目标(只读)
        ├── EXISTING-NNN/         # code-init 创建的基线需求
        │   └── RESULT.md
        └── REQ-NNNNN/            # code-require 创建的开发需求
            └── RESULT.md
```

- 路径以**当前工作目录(CWD)**为基准
- 本技能**只读**:`./assistants/.current-version` + `./assistants/*/require/*/RESULT.md` + CWD 下常见源码目录
- 本技能**不写**任何文件(NFR-1 强约束;FR-6 严守;FR-7 幂等)
- 本技能**不修改**任何其他 11 个 `code-*` 技能 SKILL.md 的 frontmatter(NFR-2 严守)
- 本技能**不修改** `./assistants/rules/` 下任何文件

## 工作流程

### 步骤 0 — 版本上下文检测(可选)

1. `Read "./assistants/.current-version"`
2. **文件不存在** → 屏显 `⚠ 未检测到激活版本,自动全版本扫描`(FR-2 降级,不视为错误,继续)
3. **文件存在** → 读取内容,记为 `<激活版本>`,作为"优先匹配版本"在报告中标注
4. **指向不存在版本**(目录缺失)→ 屏显 `⚠ 版本 <X> 工作空间不存在,自动全版本扫描`(不阻断)

### 步骤 1 — 解析查询字符串

1. 若输入为空 → 屏显用法 + 退出(边界 E-1)
2. 提取查询字符串
3. 判定路径:
   - 匹配 `^REQ-\d{5}$` / `^EXISTING-\d{3}$` → 精确编号路径(步骤 2a)
   - 其他 → 关键字模糊路径(步骤 2b)

### 步骤 2a — 精确编号匹配(FR-3a)

1. `Glob "./assistants/*/require/<查询>/RESULT.md"`
2. 命中 0 份 → 边界 E-2:`✗ 未找到对应需求 <查询>` + 列出本仓库已知的需求编号前 10 条作为提示
3. 命中 1 份 → `Read` 该文件,提取 §1 需求概述 + §4 功能需求 + §13 变更记录
4. 命中 ≥ 2 份(同编号跨版本)→ 全部 `Read`,按版本号升序列出"该需求在每个版本中的定义",重点呈现**差异**

### 步骤 2b — 关键字模糊匹配(FR-3b)

1. `Glob "./assistants/*/require/*/RESULT.md"` 列出全版本需求清单
2. 对每份文件 `Grep` 查询关键字(支持中英文、忽略大小写)
3. 按"标题命中 > 概述命中 > FR 命中"权重打分排序
4. 取前 5 条命中(命中数 = 0 → 走边界 E-3,自动转入步骤 3 源代码补足)
5. 对每条命中 `Read` 并提取关键信息(需求概述 + 主要 FR + 变更记录摘要)

### 步骤 3 — 源代码补足(FR-4,条件性)

> **触发条件**:步骤 2a/b 命中 0 条 **或** 命中条目的 §4 功能需求描述过简(< 3 条 FR)

1. 尝试 Glob 常见源码目录:`src/` `lib/` `app/` `pkg/` `cmd/` `internal/` `packages/` `modules/` `core/`
2. 全部 0 命中 → 边界 E-4:屏显 `✗ 未找到源码目录(项目可能无代码或使用非标准布局)`,列出已尝试目录清单 + 提示用户提供路径
3. 命中 ≥ 1 个 → 在命中目录下 `Grep -i "<查询关键字>"` 跨文件检索(ext 列表:`.ts` `.js` `.py` `.go` `.java` `.rs` `.rb` `.php` `.cs` `.cpp` `.h` `.swift` `.kt` `.vue` `.tsx` `.jsx` `.svelte`,按文件实际存在性动态调整)
4. 对命中的代码段提取前后 5-10 行上下文,**仅**作为"功能逻辑"补充呈现,引用格式 `path:line`
5. **不**做修改建议 / 不做代码评审 — 严格遵守"只读"契约

### 步骤 4 — 拼装报告(FR-5)

按"输出"小节定义的 5 段结构(头部 + 主要功能 + 关键逻辑 + 历史变迁 + 参考引用)拼装:
- 仅命中需求 / 仅命中代码 → 对应参考引用区段填实际内容,缺失区段显示"(无)"
- 完全无命中 → 边界 E-5:屏显简短说明 + 引导用户提供更精确的查询
- 屏显总行数建议 ≤ 80 行

### 步骤 5 — 屏显输出 + 退出

1. 屏显报告(按"输出"小节格式)
2. 退出(无状态保留,NFR-1 幂等)
3. **不**写 `auto-report.md` / **不**写任何归档文件

## 附录 A:关键字匹配评分算法(FR-3b 实现细节)

```
function scoreFile(filePath, query):
  // 权重:标题命中 = 10 / 概述命中 = 5 / FR 命中 = 2
  score = 0
  title = parseTitle(filePath)         // 第 1 行 # 标题
  overview = readSection(filePath, "需求概述")  // §1
  frs = readSection(filePath, "功能需求")  // §4
  if contains(title, query): score += 10
  if contains(overview, query): score += 5
  for fr in frs:
    if contains(fr, query): score += 2
  return score
```

- 权重按"用户最关心的内容"排序:标题最直接 → 概述最易扫读 → FR 最具体
- 命中数 > 5 → 仅返回前 5(按分数降序;同分时按文件路径字典序)
- 同分退化:按 `assistants/<版本号>/require/<REQ>/RESULT.md` 路径字典序(确定性,保证幂等)

## 附录 B:屏显报告确定性保证(FR-7 幂等性)

- **顺序稳定**:`Glob` 路径字典序 + `Grep` 文件遍历顺序由文件系统决定(可重复)
- **内容确定**:不依赖时间戳 / 不调用外部 API / 不读取运行期状态
- **不写文件**:相同输入 + 相同仓库状态 → 相同输出
- **多次执行验证**:同一查询执行 2 次,输出 byte-for-byte 一致(可作为 AC 验收)
