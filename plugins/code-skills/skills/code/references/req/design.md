# 软件设计阶段 — `/code req`

> 本文件为 `/code req` 子命令的 DESIGN 阶段提供详细流程。在进入 DESIGN 阶段时加载。

## 目标

在 REQUIRE.md 的基础上,结合当前项目实际状况(语言/框架/既有模块/接口),给出可被评审、可被 PLAN 阶段进一步拆解的系统级架构方案,产出 `DESIGN.md`。

## 前置条件

- `REQUIRE.md` 必须存在且 REQUIRE 阶段已完成
- 项目级规范 `./assistants/rules/` 已读取

## 输入

- `req/<REQ-NNNNN>/REQUIRE.md`(上游需求)
- 项目级规范:`./assistants/rules/` 下所有文件
- 当前项目代码:CWD 下的源文件、配置文件

## 输出

主产出物:`req/<REQ-NNNNN>/DESIGN.md`
辅助产物:`req/<REQ-NNNNN>/LOG.md`(可选,非必要不记录)

## 工作流程

### 步骤 0 — 设计目标确认(与用户确认)

> 在读取需求之前,先与用户确认设计目标。非 `--auto` 模式触发 AskUserQuestion。

#### 扩展性触发判定

```
触发扩展性确认 = 满足任一:
1. 需求中"待评估三方依赖"清单非空
2. 模块拆分预评估涉及模块数 ≥ 3
3. 上游需求 FR 含"多实现/抽象层/可替换/多套实现"语义
```

#### 自适应问题数

| 需求规模 | 扩展性触发 | AskUserQuestion 数 |
| --- | --- | --- |
| 小需求(1 任务) | 否 | 0 个 |
| 小需求(1 任务) | 是 | 1 个 |
| 中等需求(2-5 任务) | — | 1 个 |
| 大需求(≥6 任务) | — | 2 个 |

#### 确认内容

**设计目标**(3 选 1):
```
整体设计目标:
A. --minimal(最小改动,能跑就行)
B. --extensible(预留扩展点,考虑未来变更)
C. --balanced(两者兼顾,推荐)
```

**维度优先级**(仅当问题数 ≥ 1):
```
当前需求最看重哪个维度?
A. 功能性(功能完整,正确工作)(推荐)
B. 扩展性(易于增加新功能)
C. 可维护性(代码清晰,易于修改)
```

#### 屏显模板

```
=== code req DESIGN 设计目标确认 ===
设计目标: <--minimal/--extensible/--balanced>
维度优先级: <功能性/扩展性/可维护性>(<高/中/低>)
已回写至 DESIGN.md "## 设计目标" 小节
```

### 步骤 1 — 读取需求

1. `Read "req/<REQ-NNNNN>/REQUIRE.md"`
2. 提取 FR / NFR / AC / 页面与交互 / 关联需求
3. 预检规范 vs 需求冲突

### 步骤 2 — 探索项目现状

1. 探测项目语言/框架/构建系统
2. `Glob` 探索项目结构
3. `Grep` 搜索关键模块/接口名
4. 识别可复用的既有模块和接口

### 步骤 3 — 架构方案构思

1. 确定模块边界与职责
2. 设计模块间接口
3. 确定数据结构与流转
4. 评估第三方依赖

### 步骤 4 — 模块拆分

对每个模块,明确:

| 维度 | 说明 |
| --- | --- |
| 模块名 | 唯一标识 |
| 职责 | 一句话描述模块做什么 |
| 接口 | 对外暴露的 API/函数签名 |
| 依赖 | 依赖的其他模块/三方库 |
| 备注 | 特殊注意事项 |

> 模块拆分粒度:够 PLAN 阶段拆分为独立任务即可,不展开到具体函数实现。

### 步骤 5 — 接口设计

对每个关键接口,明确:

```
接口名: <名称>
用途: <一句话描述>
入参: <参数列表,含类型与约束>
出参: <返回值,含类型与含义>
异常: <可能的错误与处理方式>
安全: <认证/授权/校验要求>
```

### 步骤 6 — 数据结构设计

对每个关键数据结构,明确:

```
结构名: <名称>
用途: <一句话描述>
字段列表:
  - <字段名>: <类型> — <说明>
约束: <唯一性/非空/长度等>
生命周期: <创建/更新/销毁时机>
```

### 步骤 7 — 关键流程设计

对每个关键业务流程,用伪代码或流程图描述:

```
流程: <流程名>
触发条件: <何时触发>
步骤:
  1. <步骤 1>
  2. <步骤 2>
  ...
异常路径:
  - <异常 1>: <处理方式>
```

### 步骤 8 — 方案选型

对每个需要决策的技术点,记录:

```
决策: <决策编号>
问题: <需要决策什么>
选项:
  - A: <方案 A> — 优点/缺点
  - B: <方案 B> — 优点/缺点
选择: <选项> 
理由: <为什么选这个>
权衡: <放弃什么,获得什么>
```

### 步骤 9 — 用户确认(强制)

> **⚠️ 本步骤不可跳过。** 在撰写 DESIGN.md 前,必须与用户确认设计方案的三个关键方面。非 `--auto` 模式下每项确认必须触发 `AskUserQuestion`。
> 
> **最小确认要求**:小需求(1 任务)至少执行 9b(方案选型确认);中等需求(2-5 任务)至少执行 9b+9c;大需求(≥6 任务)执行全部 9a+9b+9c+9d。

#### 9a. 扩展性确认

确认设计方案的可扩展性:

- 是否需要预留扩展点?(如插件机制、配置项、回调钩子)
- 是否考虑未来变更?(如多语言支持、多平台适配、数据格式扩展)
- 模块拆分粒度是否合适?(过细 → 过度工程;过粗 → 职责不清)

提问示例:
```
设计扩展性确认:当前方案是否需要预留扩展点?
选项:
A. 不需要,当前方案够用(推荐)
B. 需要预留配置扩展(如 config 文件)
C. 需要预留回调钩子(如 middleware)
D. 需要完整的插件机制
```

#### 9b. 方案选型确认

对每个存在多方案的技术决策点,向用户确认:

- 列出方案选项,说明优缺点
- 用户选择后记录决策理由
- 记录权衡(放弃什么,获得什么)

提问示例:
```
数据存储方案:当前涉及 3 个选项
A. 本地文件存储 — 简单/无依赖/不适合并发
B. SQLite — 轻量/支持查询/需要 C 库
C. PostgreSQL — 完整/高并发/运维成本高(推荐)
```

#### 9c. 改修方案确认

确认每个模块的具体修改方案:

- 模块修改范围:新增/修改/删除哪些文件
- 接口变更影响:是否破坏现有接口、是否需要版本化
- 数据结构变更兼容性:是否需要迁移脚本、是否向后兼容
- 第三方依赖变更:新增/升级/移除哪些依赖

提问示例:
```
接口变更确认:修改用户模块接口
A. 新增接口,保持旧接口兼容(推荐)
B. 修改旧接口,做版本化(v1→v2)
C. 直接替换旧接口(不兼容)
```

#### 9d. 危险操作确认(强制)

> 当设计方案涉及以下**危险操作**时,本步骤**不可跳过**,必须触发 `AskUserQuestion`。

**危险操作判定标准**(满足任一即触发):

1. **移除现有行为**:删除/禁用现有功能、命令行参数、API 接口
2. **变更默认行为**:修改默认值、默认流程、默认输出格式
3. **标记过时**:将现有功能标记为 deprecated
4. **影响其他 `/code` 子命令**:修改其他 `/code` 子命令依赖的接口/契约/文件格式

**处理流程**:

1. 识别危险操作 → 明确列出受影响的功能/文件/技能
2. 给出 2-3 个具体选项 → 使用 `AskUserQuestion`
3. 用户确认后 → 在 DESIGN.md 决策中标注"用户确认"
4. 不可自行选择"移除/变更"方案

**提问示例**:
```
⚠️ 危险操作确认:决策 2 涉及变更默认行为

当前默认行为:每个阶段完成后弹出 AskUserQuestion(继续/暂停/取消)
设计方案选择:移除默认 AskUserQuestion,改为自动流转

此变更影响所有使用 `/code req` / `/code fix` 的交互流程:
A. 同意移除,改为自动流转(需用户明确确认)
B. 保留默认 AskUserQuestion,仅 --confirm 增强
C. 保留默认 AskUserQuestion,同时支持 --confirm 增强

请选择或说明期望方案。
```

> **注意**:危险操作确认**不可**被 `--auto` 模式跳过。`--auto` 模式下检测到危险操作时,应中止并提示用户手动确认。

### 步骤 10 — 检索关联设计

1. `Glob "req/*/DESIGN.md"` 列出同版本所有设计
2. 检查接口/模块/数据重用机会
3. 在 DESIGN.md 中记录关联设计

### 步骤 11 — 撰写 DESIGN.md

按 `templates/req/DESIGN.md` 结构生成:

```
# 软件设计 — <REQ-NNNNN> · <标题>

## 1. 设计概述
## 2. 模块拆分
## 3. 接口设计
## 4. 数据设计
## 5. 关键流程
## 6. 方案选型
## 7. 规范合规
## 8. 变更记录
```

### 撰写原则

- 够 PLAN 阶段拆任务即可,不展开到伪代码级别
- 每个接口/数据结构有明确的入参/出参/异常
- 方案选型有明确的理由和权衡
- 标注与规范的冲突点(若有)
- 用户确认的决策标注"用户确认"(来源:确认环节)

## 阶段完成确认

> 三态确认模型:阶段边界确认由 `--confirm`/`--auto`/默认 控制。阶段内内容确认(扩展性/方案选型/改修方案/危险操作)仅在 `--auto` 模式下跳过。详见 common.md §4、§7、§11。

### 用户确认

步骤 9 中每个确认环节使用 `AskUserQuestion`:

### --confirm 模式

阶段完成后弹出增强确认:
```
=== code req --confirm: DESIGN 阶段完成 ===
软件设计完成: <N> 模块 / <M> 接口 / <K> 决策

产出物文件:
  - req/<REQ>/DESIGN.md

你可以手动修改上述文件,完成后选择:
A. 继续(重新读取产出物,获取最新修改,进入 PLAN 阶段)
B. 中止(保存当前进度,退出)
```

### --auto 模式

自动继续,屏幕输出 `[code req --auto] DESIGN 完成,自动继续`。阶段内确认自动选推荐项,记录为"自动选择"。

### 默认模式(无 flag)

阶段边界自动继续,无输出。阶段内确认正常触发 `AskUserQuestion`。

## 确认环节约束

- 每轮确认最多 1-3 个问题,不一次问完所有问题
- `--confirm` 模式:阶段边界增强确认,阶段内确认正常触发 `AskUserQuestion`
- `--auto` 模式:自动选推荐项(第一项),记录为"自动选择"
- 默认模式:阶段边界自动继续,阶段内确认正常触发 `AskUserQuestion`
- 所有确认结果记录在 DESIGN.md 的对应章节(标注"用户确认"或"自动选择")
