# 缺陷登记阶段 — `/code fix`

> 本文件为 `/code fix` 子命令的 INIT 阶段提供详细流程。在进入 INIT 阶段时加载。

## 目标

将用户输入的缺陷描述(自然语言/日志/截图等)转化为结构化的 `BUG.md`,作为后续 DESIGN/PLAN/CODING/CHECK 阶段的共同基线。

## 与 REQUIRE 阶段的差异

| 方面 | REQUIRE 阶段(`/code req`) | INIT 阶段(`/code fix`) |
| --- | --- | --- |
| 产出 | REQUIRE.md(FR/NFR/AC) | BUG.md(缺陷描述/触发条件/可能成因/影响范围) |
| 关注点 | 功能点(系统必须做什么) | 缺陷(系统哪里出了问题) |
| 要素 | FR/NFR/AC/用户角色/交互流程 | 缺陷描述/触发条件/可能成因/影响范围/严重程度 |
| 模板 | templates/req/REQUIRE.md | templates/fix/BUG.md |

## 输入

- 用户输入的缺陷描述(自然语言,必填)
- 用户提供的补充材料(复现步骤/日志/截图/堆栈等,可选)
- 项目级规范:`./assistants/rules/` 下所有文件

## 输出

主产出物:`fix/<BUG-NNNNN>/BUG.md`
辅助产物:`fix/<BUG-NNNNN>/LOG.md`(可选,非必要不记录)

## 缺陷编号分配

### 新建缺陷

```
function allocateBugNum(versionPath):
  existing = Glob("fix/BUG-*/")
  if existing.length == 0:
    return "BUG-00001"
  nums = existing.map(extractNumber).sort()
  return "BUG-" + pad(nums[nums.length - 1] + 1, 5)
```

### 续跑缺陷

- 用户传入 `BUG-NNNNN` → 直接使用该编号
- 验证 `fix/<BUG-NNNNN>/` 目录存在

## 工作流程

### 步骤 1 — 创建目录与 PROCESS.md

1. 检查 `fix/<BUG-NNNNN>/` 是否存在
   - 不存在 → `mkdir -p`,创建 `PROCESS.md` 初始行
   - 存在 → 从 PROCESS.md 恢复

2. PROCESS.md 初始行:
```
| YYYY-MM-DD HH:mm | INIT | 开始 | 创建缺陷 <BUG-NNNNN> |
```

### 步骤 2 — 收集缺陷材料

1. 若用户输入为自然语言描述 → 直接作为主缺陷材料
2. `Glob "fix/<BUG-NNNNN>/**/*"` 列出补充材料
3. 对每个文件尝试读取:
   - 文本/图片直接读取
   - 不可读格式 → 在 LOG.md 中标注,提示用户提供文字摘要

### 步骤 3 — 提取缺陷要素

从材料中提取:

| 要素 | 说明 | BUG.md 章节 |
| --- | --- | --- |
| 缺陷描述 | 用户原始报告 + 期望行为 vs 实际行为 | 缺陷描述 |
| 触发条件 | 必要条件/环境因素/发生频率 | 触发条件 |
| 可能成因 | 根因假设/涉及文件 | 可能成因 |
| 影响范围 | 影响用户/模块/严重程度/后果 | 影响范围 |

### 步骤 4 — 严重程度判定

按以下标准判定严重程度:

| 级别 | 标准 | 示例 |
| --- | --- | --- |
| P0 | 系统不可用/数据丢失/安全漏洞 | 服务崩溃、数据损坏、权限绕过 |
| P1 | 核心功能不可用/严重影响 | 登录失败、支付中断、主流程阻塞 |
| P2 | 功能部分不可用/有绕过方案 | 非核心页面报错、特定条件下异常 |
| P3 | 体验问题/优化建议 | 文案错误、样式偏差、性能缓慢 |

- 默认:`P2`
- 若用户未明确指定,按上述标准自动判定,在 `AskUserQuestion` 中确认

### 步骤 5 — 与用户澄清(非 --auto)

对模糊、歧义、缺失的缺陷信息,向用户确认:

- 能给出 2-4 个具体选项 → 使用 `AskUserQuestion`
- 需要开放式描述 → 自然语言提问

常见澄清问题:
- 严重程度确认(P0/P1/P2/P3)
- 复现步骤是否完整
- 影响范围确认
- 期望行为确认

> **不过滤技术选型类问题**:本阶段确定缺陷范围,修复方案/技术选型归 DESIGN 阶段。

### 步骤 6 — 撰写 BUG.md

按 `templates/fix/BUG.md` 结构生成:

```
# 缺陷分析 — <BUG-NNNNN> · <标题>

> 所属版本:<版本号>
> 创建时间:YYYY-MM-DD HH:mm

## 缺陷描述
### 用户原始报告
### 复现步骤
### 期望行为 vs 实际行为

## 触发条件
## 可能成因
## 影响范围
## 变更记录
```

### 撰写原则

- **先保留原文**:撰写「缺陷描述」时,将用户输入的缺陷描述一字不差地放入引用块(`>`),不可改写或概括;若用户分多次补充,每段用单独的引用块
- **详尽**:每一条缺陷信息都展开清晰描述,便于后续修复
- **不臆造**:无材料佐证的细节写进"待澄清"或明确标注"假设"
- **可追溯**:每条信息标注来源(用户报告/日志/截图/堆栈)
- **不涉及技术选型**:本阶段只确定缺陷范围,修复方案/技术选型/框架选择归 DESIGN 阶段

### 步骤 7 — 同步版本看板

在 `RESULT.md` 缺陷清单追加:

```
| BUG-NNNNN | <标题> | [PROCESS.md](fix/BUG-NNNNN/PROCESS.md) |
```

## 技术选型过滤

> 本阶段不涉及技术选型。以下关键词命中的内容**不写入** BUG.md,留待 DESIGN 阶段处理:
> `{技术选型, 实现方式, 框架, 库, 工具, 数据库, ORM, 消息队列, 缓存, 架构风格, 部署形态}`

## 阶段完成确认

> 三态确认模型:阶段边界确认由 `--confirm`/`--auto`/默认 控制。阶段内内容确认(缺陷澄清)仅在 `--auto` 模式下跳过。详见 `../req/common.md`「阶段执行器」「交互确认」「--confirm 模式」。

### --confirm 模式

阶段完成后弹出增强确认:
```
=== code fix --confirm: INIT 阶段完成 ===
缺陷登记完成: <标题>(严重度: P<N>)

产出物文件:
  - fix/<BUG>/BUG.md

你可以手动修改上述文件,完成后选择:
A. 继续(重新读取产出物,获取最新修改,进入 DESIGN 阶段)
B. 中止(保存当前进度,退出)
```

### --auto 模式

自动继续,屏幕输出 `[code fix --auto] INIT 完成,自动继续`。阶段内澄清问题自动选推荐项。

### 默认模式(无 flag)

阶段边界自动继续,无输出。阶段内澄清问题正常触发 `AskUserQuestion`。

## 暂停处理

- 选 B(中止):追加 PROCESS.md `| <时间> | INIT | 中止 | 用户在 --confirm 确认环节中止 |`,退出
- 下次调用时从 PROCESS.md 恢复,从 INIT 阶段继续
