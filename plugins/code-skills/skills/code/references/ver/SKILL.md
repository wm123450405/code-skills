# `/code ver` — 版本管理与开发看板

> 工作流细节见 [`common.md`](common.md)。

## 目标

3 模式 1 看板:
- **看板模式**(无参): 1 行 ASCII + 状态分布 + P0/P1 + ≤5 条建议
- **初始化**(无 assistants/ 或无 .current-version): 扫描代码 → 写 INIT-REPORT → 登记 EXISTING-NNN 基线
- **切换**(带版本号): 4 情形处理 + CWD 描述文件 `version` 同步
- **发布**(`--publish`): 前置检查 → 基线识别 → 生成 DEPLOY/UPDATE/FAQ.md

## 工作目录

```
./assistants/
├── rules/                  跨版本共享,本技能只读
├── .current-version        本技能读写
└── <版本号>/
    ├── RESULT.md           看板(本技能首次创建,req/fix 后续追加)
    ├── req/REQ-NNNNN/      req 产出
    └── fix/BUG-NNNNN/      fix 产出
```

## 输入

| 形态 | 行为 |
| --- | --- |
| 无参 + 已初始化 | 看板模式(只读) |
| 无参 + 未初始化 | 初始化场景(1A-8A) |
| `<版本号>` | 切换/创建(1B-6B) |
| `--publish` | 发布检查(1C-5C) |

版本号禁用: `/` `\` `:` `*` `?` `"` `<` `>` `|` `空格`。

## 看板输出契约

固定 4 段,正常 ≤6 行 / 退化 ≤4 行:

```
段 1:  [███████████░] 92%           # 固定 12 字符,█/░;公式见 common.md §8.4
段 2:  状态: 待编码 0 / 待审查 0     # 仅展示 N>0 的状态;PROCESS.md 缺失视为待需求分析
段 3:  P0 待修复: █ 2 | P1 待修复: ░ 0  # █=P0 有,▓=P1 有,░=无;不展示 P2/P3 与标题
段 4:  > /code fix BUG-00001 [高] (依据: P0 待修复 2 个)   # ≤5 条,严格 /code 语法
```

退化: `— / 无需求无缺陷,无需进度` + `状态: (无)`。

## 切换(1B-6B)四种情形

| 情形 | 条件 | 动作 |
| --- | --- | --- |
| A | 目标不存在 + 无激活版本 | 创建并切换 |
| B | 目标不存在 + 有激活版本 | 询问: 先发版 / 直接切换 / 取消 |
| C | 目标存在 + 与激活不同 | 询问: 切换 / 不切换 / 取消 |
| D | 目标存在 + 与激活相同 | 询问: 确认 / 重建看板 / 取消 |

切换时若旧版本有活跃内容,默认引导先发版。完成后 CWD 描述文件 `version` 字段同步(package.json / pom.xml / Cargo.toml / pyproject.toml;go.mod 无版本号字段跳过)。

## 工具

| 模式 | 可用 |
| --- | --- |
| 看板 | `Read` `Glob` `Grep`(只读,禁用 Write/Edit/Bash) |
| 其他 | `Read` `Glob` `Bash` `Write` `Edit` |

## 不要做的事

- 看板模式下不调 Write/Edit/Bash(只读)
- 不修改 `./assistants/rules/`
- 不归一化状态字面(严格按字面匹配)
- 不把旧格式任务编号 `^(REQ|BUG)-(\d+)-(\d+)$` 改写为新格式 `TASK-...`(双格式兼容,旧字面透传)
- 不在版本号用路径分隔符
- 不删除其他版本的目录
