# RESULT.md 模板 — 需求提示词文档

> 本文件是 `code-require` 技能产出的 `RESULT.md` 章节结构与填写规范。
> 模型在生成 `RESULT.md` 时应严格按本结构组织,确保后续 `code-design` / `code-plan` 等技能可直接消费。
>
> **版本感知**:本文件存放在 `./assistants/<版本号>/require/<需求编码>/RESULT.md`,文档头应明确记录版本号。

---

## 文档头
```
# <需求编码> — <需求标题>

- 需求编码:<REQ-YYYY-NNNN>
- 所属版本:<版本号>(如 v1.0.0)
- 状态:草稿 / 已澄清 / 已锁定
- 责任人:<用户/产品>
- 创建:<YYYY-MM-DD>
- 最近更新:<YYYY-MM-DD HH:mm>
- 当前版本:v<n>
```

## 1. 需求概述
用 2-5 句话概括:**为谁,解决什么问题,带来什么价值**。避免细节,这是电梯演讲版本。

## 2. 背景与目标
- **背景**:为什么提出这个需求?(业务问题/用户痛点/机会窗口)
- **业务目标**:完成后要达到的可衡量效果
- **本次目标**:本次迭代要交付的范围

## 3. 用户角色与场景
- 列出涉及的角色(用户/管理员/内部员工/外部系统等)
- 每个角色列举 1-3 个关键场景,使用用户故事格式:
  - 作为 [角色],我想要 [动作],以便 [价值]
- 场景附:**前置条件**、**主流程**、**异常/备选流程**

## 4. 功能需求(FR)

每条 FR 单独成段,使用统一编号:

```
### FR-1:<一句话标题>
- **描述**:详细描述该功能做什么、怎么做
- **入口**:用户从哪个页面/入口进入此功能
- **前置条件**:触发该功能需要满足的条件
- **主流程**:步骤化描述
- **分支/异常**:各种条件下的处理
- **数据变化**:对数据库/缓存/状态的影响
- **来源**:(<材料名> §<章节>)
```

数量无上限,务必把材料中提到的每个独立功能点拆为一条 FR。

## 5. 非功能需求 / 约束(NFR)
按类别列出:
- **性能**:响应时延、吞吐量、容量上限
- **可用性**:SLA、容错、降级策略
- **安全**:鉴权、加密、审计、防注入、防越权
- **可观测性**:关键指标、日志字段、告警阈值
- **兼容性**:对老客户端/老接口/老数据的影响
- **合规**:GDPR/等保/行业规范
- **国际化**:文案、时区、货币
- **可维护性**:代码组织、模块边界

## 6. 页面与界面

按页面逐一描述,不要笼统说"有一个列表页":

```
### 页面 1:<页面名>
- **路径**:`/xxx/yyy`
- **入口**:从哪些页面/操作可进入
- **布局**:整体结构(顶栏/侧栏/主区/底栏),关键区域尺寸
- **组件清单**:
  - <组件名>:<位置/尺寸/交互/关键状态>(空/加载/正常/错误/禁用)
- **样式要点**:配色、字体、间距、圆角等关键视觉规范
- **响应式**:桌面/平板/手机的关键差异
- **来源**:(<设计稿/材料名>)
```

## 7. 交互逻辑

对复杂交互给出**状态机**或**流程图**(Mermaid 文字描述或 ASCII):

```
### 交互 1:<交互名>
- **触发**:
- **流程**:
  1. 用户做 A
  2. 系统展示 B
  3. ...
- **状态机**(可选,使用 mermaid):
  ```mermaid
  stateDiagram-v2
    [*] --> Idle
    Idle --> Loading: 点击提交
    Loading --> Success: 接口成功
    Loading --> Error: 接口失败
    Error --> Idle: 点击重试
  ```
- **关键规则**:并发/超时/重试/幂等
- **空状态/加载状态/错误状态**:各自的展示
```

## 8. 数据与状态
- **核心实体**:实体名、关键字段、字段类型、约束
- **实体关系**:实体间的关联(1:1 / 1:N / N:M)
- **状态生命周期**:有状态的对象(如订单/工单)的状态机
- **数据来源**:哪些字段来自用户输入、哪些来自外部系统
- **数据保留**:留存期限、清理策略

## 9. 边界与异常
列出所有**非常规路径**:
- 极端输入(空、超长、特殊字符、Unicode)
- 网络异常(超时、断网、半开连接)
- 并发场景(同一用户多端登录、重复提交)
- 权限缺失(未登录、角色不足)
- 第三方依赖不可用
- 数据不一致(脏数据、并发写入冲突)
- 时间/时区相关(跨时区、夏令时、闰秒)

## 10. 验收标准(AC)

每条 AC 必须**可独立验证**:

```
### AC-1:<标题>
- **对应需求**:FR-1, FR-2
- **验证方式**:手工 / 自动化测试 / 接口测试
- **步骤**:
  1. ...
  2. ...
- **预期结果**:可观察的客观结果
- **优先级**:高/中/低
```

## 13. 变更记录
```
| 时间 | 版本 | 变更类型 | 变更摘要 | 变更人 |
| --- | --- | --- | --- | --- |
| 2026-06-03 10:30 | v1 | 初始创建 | 完成首次需求澄清 | <用户> |

```

变更类型:
- **新增**:全新内容(对应新增材料)
- **修改**:基于材料变更修订
- **删除**:材料已删除,对应需求被取消
- **澄清**:基于与用户的问答补充说明
