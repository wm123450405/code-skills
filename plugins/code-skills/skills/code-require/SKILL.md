# code-require — 需求分析(版本感知)

## 目标
把零散、异构的需求材料(文档/设计稿/视频/录音/聊天记录等)合并为一份**详尽、可执行、可追溯**的需求提示词文档 `RESULT.md`,作为后续 `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` 的共同基线。

## 适用场景
- 新功能、新模块、新产品的首次需求澄清
- 现有功能的重大变更
- 跨模块/跨系统的需求对齐
- 用户追加新材料,需要补全或修正已有 `RESULT.md`

## 不适用
- 已知明确的、只涉及单一文件/单一函数的修改
- 紧急线上修复(走 hotfix 流程)
- 当前**没有激活的版本工作空间**(请先调 `code-version`)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
本技能的目录粒度是**需求**:
```
./assistants/<版本号>/require/<需求编码>/
├── <用户放置的原始材料...>     # 用户在调用本技能前/后放入
├── RESULT.md                    # 本技能产出,最终需求提示词文档
├── materials-index.md           # 过程文档:材料登记
├── clarifications.md            # 过程文档:与用户的问答记录
├── related-requirements.md      # 过程文档:关联需求
└── analysis-notes.md            # 过程文档:自由形式的分析笔记(可选)
```

**版本看板**:`./assistants/<版本号>/RESULT.md`(由 `code-version` 初始化,本技能只追加"需求清单" / "变更记录"区段)。

- 路径以**当前工作目录(CWD)**为基准
- `./assistants/rules/` 是跨版本共享的项目级规范(只读)
- 用户在调用本技能前应把所有需求材料放入 `<版本号>/require/<需求编码>/` 目录;若目录不存在,本技能会创建并提示用户放置
- 多个项目各自维护各自的 `./assistants/`,互不污染

## 输入
- **需求编码**(必填):用户口头或文本指定,例如 `REQ-00001`
- **需求材料**:由用户预先放在 `./assistants/<版本号>/require/<需求编码>/` 目录下的:
  - 需求文档(.md / .docx / .pdf)
  - 设计稿(.png / .jpg / .figma 链接)
  - 演示视频(.mp4 / 链接)
  - 会议记录、聊天记录、邮件(.md / .txt / .docx)
  - 沟通录音(.mp3 / .wav / .m4a)
  - 任何其他参考资料

## 输出
主产出物:`./assistants/<版本号>/require/<需求编码>/RESULT.md`

辅助过程文档(同目录下):
- `materials-index.md`
- `clarifications.md`
- `related-requirements.md`
- `analysis-notes.md`(可选)

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:
- 完成需求分析(首次/增量)→ 在"需求清单"追加或更新本条记录
- 任意阶段 → 在"变更记录"追加条目

## 工具使用约定
- 读激活版本:`Read "./assistants/.current-version"`(若不存在,立即提示用户调 `code-version`)
- 文本/文档:`Read` 工具直接读取
- 图片:`Read` 工具直接展示(模型可观察)
- 音视频 / 无法直接读取的格式:在 `materials-index.md` 中登记,**主动询问用户**让其用文字概括关键信息
- 整目录浏览:`Glob` 列出所有文件,`Grep` 跨文件检索关键词
- 跨需求关联:`Glob` + `Read` 扫描 `./assistants/<版本号>/require/` 下其他 `<需求编码>/RESULT.md`
- 与用户澄清:**优先使用 `AskUserQuestion`** 工具给出结构化选项;无法结构化时使用自然语言提问

## 标题解析(REQ-00013 新增)

> 适用对象:所有用户可见的屏幕输出位置(启动 / 完成 / 中止 / 错误 / 报告)
> 依据规范:Q-1 锁定"从已有内容派生,不新增字段"+ Q-2 锁定"`REQ-00001 · 标题`"(中点 `·`)+ Q-3 锁定"字符数 ≤ 30"+ FR-3 + NFR-3

**工具函数**(伪代码):

```ts
function truncateTitle(title: string, maxLen: number = 30): string {
  if ([...title].length <= maxLen) return title
  return [...title].slice(0, maxLen).join('') + '...'
}

function formatReqTitle(reqNum: string, title: string): string {
  return `${reqNum} · ${truncateTitle(title)}`
}
```

**标题解析入口**:

```ts
function parseResultTitle(filePath: string): string {
  const content = require('fs').readFileSync(filePath, 'utf-8')
  const match = content.match(/^# 需求提示词文档 — (.+)$/m)
  return match ? match[1] : ''  // E-3 退化:返回空字符串
}
```

**屏幕输出格式契约**:

| 场景 | 格式 |
| --- | --- |
| 启动 | `正在处理: REQ-NNNNN · <需求标题>` |
| 完成 | `完成: REQ-NNNNN · <需求标题>` |
| 中止 | `⛔ code-require 中止: REQ-NNNNN · <需求标题>(<原因>)` |
| 错误 | `✗ 错误: REQ-NNNNN · <需求标题>(<错误信息>)` |

**边界与异常**:
- E-2:标题 > 30 字符 → `truncateTitle` 自动截断到 30 字 + `...`
- E-3:标题字段缺失(理论不可能)→ 退化:屏幕输出"编号+(无标题)"
- E-9:多次执行 `code-require` → 标题覆盖(NFR-4 幂等)

**约束**:
- **不**使用"本需求"等指代词 — 替换为"编号+标题"(FR-2.AC-2.4 强约束)
- **不**修改 frontmatter(L1-3 字节级保留,NFR-7)
- **不**修改既有章节(锚点 = "## 工具使用约定"段后 + "## 工作流程"前,本节为纯追加)
- **不**修改 `require/.../RESULT.md` 模板(NFR-2 零规范变更)

## 命令行参数解析(本需求 REQ-00021 新增,FR-1)

> 本小节是 `code-require` 的**可选参数**解析约定。本参数**不**影响主流程,无参时按原行为执行。

### 支持的参数

| 参数 | 性质 | 模板产出物 | 备注 |
| --- | --- | --- | --- |
| `--result <模板文件>` | 可选 | `REQUIRE.<ext>`(后缀 = 模板后缀) | 沿用主产出物同目录 |

### 解析规则(FR-1)

1. 技能启动时(步骤 0 之前)解析用户输入,识别 `--result <路径>`
2. 校验模板文件存在性:不存在 → 屏显 `⚠ 模板文件不存在:<路径>`,**跳过**模板填充(不阻断主流程)
3. 记录参数到 `analysis-notes.md` "## 命令行参数解析"节
4. 无参时:原行为执行,3 技能只产出 `RESULT.md`,**不**产出 `REQUIRE.<ext>`

### 边界与异常

- **E-1**:`--result` 缺值(只写 `--result` 不写路径)→ 屏显 `⚠ --result 缺模板文件路径`,跳过
- **E-2**:`--result` 路径含通配符(`*.docx`)→ 屏显 `⚠ 模板路径不支持通配符`,跳过
- **E-3**:`--result` 路径 `../` 跳出工作空间 → 屏显 `⚠ 模板路径不安全`,跳过
- **E-4**:`code-auto` 上下文 → `code-auto` **不**传 `--result`(沿用 REQ-00007 Q-4)

## 工作流程

### 步骤 0a — 拉取最新代码(强制前置,新增)
所有其他 `code-*` 技能的第一步都是这一步。位于既有"步骤 0"之前。
1. 探测 `git` 可用性 → 执行 `Bash: git --version`,失败 → 中断 + 提示"未检测到 git,本需求需要 git,请先安装 git 后重试"(E-12)
2. 执行 `Bash: git pull`(默认 upstream / tracking 分支)
3. 拉取失败 3 种情况分类处理(Q-2 锁定 A:中断 + 报错退出):
   - **冲突**(退出码非 0 + stderr 含 "CONFLICT" / "Merge conflict" / "unmerged")→ 打印 `✗ git pull 失败:存在未解决的冲突,冲突文件:<列出>`,提示"请手动解决冲突后,重跑该技能"(E-2)
   - **网络**(退出码非 0 + stderr 含 "Could not resolve host" / "Connection" / "timeout")→ 打印 `✗ git pull 失败:网络/remote 不可达,远程仓库:<remote-url>`,提示"请检查网络/代理后重试"(E-3)
   - **凭据**(退出码非 0 + stderr 含 "Permission denied" / "Authentication failed" / "403")→ 打印 `✗ git pull 失败:权限/凭据`,提示"请检查 git 凭据(SSH key / token)后重试"(E-4)
   - **其他错误** → 透传 stderr,中断退出
4. 拉取成功(包含 no-op / "Already up to date"):
   - **立即**执行 `Read "./assistants/.current-version"`,记为"拉取后版本"(NFR-8 强约束)
   - 进入步骤 0b

### 步骤 0b — 工作版本号对齐检查(`code-require` 专属,新增)
**仅 `code-require` 执行此步骤。`code-design` / `code-plan` 跳过。**
1. 读取"用户意图版本":
   - 读取 `Read "./assistants/.current-version"` 的"拉取前版本"(步骤 0a 之前的快照 — 若本会话内已读取则复用,否则新读)
   - 解析用户原始输入(口头/文本)中的版本号,若有显式指定则取用户指定
   - 默认:用户意图 = 拉取前版本
2. 对比"用户意图版本"与"拉取后版本"(步骤 0a 读取):
   - **一致** → 直接进入既有"步骤 0 — 版本上下文检测"
   - **不一致** → 进入步骤 3(询问)
3. 询问(FR-2.AC-2.3):
   ```
   ✗ 检测到上下文版本不一致
     本地(拉取后):<拉取后版本>
     用户意图:<用户意图版本>
     需求 <需求编码> 在 <用户意图版本> 中可能不存在,但在 <拉取后版本> 中已存在
   ```
   - 弹 `AskUserQuestion`,3 选 1:
     - **A. 切到 <拉取后版本> 后继续(推荐)**
     - **B. 在 <用户意图版本> 中重新创建 <需求编码>**
     - **C. 取消**
4. 选项处理:
   - **选 A** → 调独立技能 `code-version <拉取后版本>`,**不传其他参数**
     - 切换成功 → 打印"已切到 <拉取后版本>,请重跑 `code-require`",**退出** `code-require`,无文件变更(FR-2.AC-2.4)
     - 切换失败 → 透传 `code-version` 错误,提示"请手动运行 `code-version <拉取后版本>` 后重试"
   - **选 B** → 在当前 `<用户意图版本>` 继续,直接进入既有"步骤 0 — 版本上下文检测"(FR-2.AC-2.5)
   - **选 C** → 退出 `code-require`,无任何文件变更(FR-2.AC-2.6)

### 步骤 0b.0 — 调用上下文检测(BUG-00001 新增,2026-06-06)

> 本步骤是 BUG-00001 修复方案 A3(脏标记文件)的子技能侧 — 在 `code-require` 触发 `AskUserQuestion` **前**先检测 `code-auto` 上下文。

- **检测机制**:
  ```
  Bash: test -f ./assistants/.code-auto-running && echo "DETECTED" || echo "NOT_DETECTED"
  ```
- **24 小时超时判断**:同 `code-design` 步骤 0b.0
- **决策**:
  - **检测到 `code-auto` 上下文** → 步骤 0b 的 `AskUserQuestion` 自动选 A(切到拉取后版本后继续),屏幕输出 `⚠ code-auto 上下文:跳过版本不一致问路,自动选 A(切到 <拉取后版本> 后继续)`
  - **未检测到 `code-auto` 上下文** → 正常进入步骤 0b 询问用户
  - **检测失败** → 屏幕输出 `⚠ 无法检测 code-auto 上下文,默认按用户手动调子技能处理(触发 AskUserQuestion)` + 正常进入步骤 0b
- **D-5 修订说明**:同 `code-design` / `code-plan` 步骤 0b.0
- **约束**:**不**修改 frontmatter;**不**修改既有"## 工作流程"小节;**不**修改既有"步骤 0b"小节

### 步骤 0 — 版本上下文检测(强制前置)
**所有其他 `code-*` 技能的第一步都是这一步**。
1. 读取 `./assistants/.current-version`
2. **文件不存在** → 立即停下,告知用户:
   > 错误:未找到 `./assistants/.current-version`,无激活的版本工作空间。
   > 请先调用 `code-version` 技能创建或切换到一个版本(例如 `code-version` 输入 `v1.0.0`)。
3. 文件存在 → 读取内容,记为 `<版本号>`,后续所有路径用 `assistants/<版本号>/...`
4. 验证 `./assistants/<版本号>/` 目录存在,否则提示用户(版本可能已被人工删除)

> 本步骤是幂等的:每次技能启动都重读,因为用户可能在两次调用之间切了版本。

### 步骤 1 — 收集需求编码
- 若用户未提供,主动询问并等待回复
- 编码格式不强求,记录用户实际使用的字符串即可(建议 `REQ-YYYY-NNNN`)

### 步骤 2 — 定位 / 创建工作目录
1. 检查 `./assistants/<版本号>/require/<需求编码>/` 是否存在
2. 不存在 → `mkdir -p` 创建,然后**停下来**告知用户:
   - 目录已创建(并展示完整路径)
   - 请把所有需求材料放入该目录
   - 放入后回复"已就绪"或追加更多材料
3. 已存在但为空 → 同样提示用户放入材料
4. 已存在且有材料 → 直接进入步骤 3

### 步骤 3 — 登记所有材料
1. `Glob "./assistants/<版本号>/require/<需求编码>/**/*"` 列出全部文件
2. 对每个文件尝试读取:
   - 文本/图片直接读取
   - 音视频 / 二进制不可读 → 在 `materials-index.md` 中标注"待用户提供文字摘要"
3. 生成 / 更新 `materials-index.md`,字段:
   - `文件路径`
   - `类型`(文档/图片/视频/音频/其他)
   - `用途`(根据内容判断)
   - `读取状态`(已读 / 部分摘要 / 待用户提供)
   - `关键摘要`(2-5 行核心信息)

### 步骤 4 — 判断 RESULT.md 是否存在
- **不存在** → 走"**首次分析**"分支(步骤 5A–9A)
- **存在** → 走"**增量更新**"分支(步骤 5B–9B)

### 步骤 5A — 首次分析:提取候选需求
- 通读所有材料,提取:
  - 功能点(FR)
  - 约束(NFR):性能/安全/兼容/合规/可观测性等
  - 页面与界面元素
  - 交互流程与状态机
  - 数据字段与实体
  - 边界条件与异常路径
- 在 `analysis-notes.md` 中记录原始提取(便于回溯)

### 步骤 6A — 检索关联需求
1. `Glob "./assistants/<版本号>/require/*/RESULT.md"` 列出同版本内其他需求的 RESULT.md
2. (可选,跨版本) `Glob "./assistants/*/require/*/RESULT.md"` 列出全部版本
3. `Grep` 关键词(本需求中出现的功能/模块/接口名)做粗筛
4. 对疑似关联的需求,`Read` 其 `RESULT.md` 摘要
5. 在 `related-requirements.md` 中记录:
   - 关联需求编码
   - 关联点(共享模块/接口/数据/约束)
   - 对本需求的影响(需要兼容 / 需要扩展 / 互斥)
   - **关联需求所在版本**(若跨版本)

### 步骤 7A — 与用户澄清不确定项
对材料中**模糊、有歧义、缺失**的功能或交互,向用户确认:
- 能给出 2-4 个具体选项 → 使用 `AskUserQuestion` 工具
- 需要开放式描述 → 用自然语言提问
- **每轮问答**追加到 `clarifications.md`,字段:
  - `时间`
  - `问题`
  - `选项/上下文`
  - `用户回答`
  - `影响的 RESULT.md 章节`

> **不要一次问完所有问题**。每轮聚焦 1-3 个最阻塞的点,得到答复后先更新对应章节,再迭代出下一批问题。

### 步骤 8A — 撰写 RESULT.md
按 `templates/requirements.md` 的章节结构生成完整 `RESULT.md`:
1. 需求概述
2. 背景与目标
3. 用户角色与场景
4. 功能需求(FR-1, FR-2, ...)
5. 非功能需求 / 约束(NFR)
6. 页面与界面(列出页面/区块/关键样式)
7. 交互逻辑(状态机、关键流程)
8. 数据与状态(实体、字段、生命周期)
9. 边界与异常
10. 验收标准(AC)
11. 关联需求(引用 6A 的结果)
12. 待澄清 / 未决项(本轮无法澄清的,留作后续)
13. 变更记录(初始条目)

要求:
- **详尽**:任何一句需求都展开为可独立验证的描述
- **不臆造**:无材料佐证的细节写进"待澄清"或明确标注"假设"
- **可追溯**:每条 FR/AC 标注来源材料(如:`(源自:产品 PRD v2, §3.1)`)

### 步骤 9A — 同步版本看板(强制)
**所有 `code-*` 技能在收尾前都必须执行此步骤。**
1. `Read "./assistants/<版本号>/RESULT.md"`,定位"需求清单"区段
2. 检查本条需求是否已存在:
   - **不存在** → 在"需求清单"表格追加一行,状态填 `已完成`,FR/NFR/AC 数量填入
   - **已存在** → 状态按需更新(如 `进行中` → `已完成`),"FR/NFR/AC 数量"列刷新
3. 在"变更记录"区段追加:
   ```
   YYYY-MM-DD HH:mm  需求新增  <需求编码> 需求分析完成(共 N 条 FR / M 条 NFR / K 条 AC)  <需求编码>
   ```
4. 视情况小幅更新文档头"最近更新"时间

### 步骤 10A — 完善过程文档
- 收尾 `materials-index.md` / `clarifications.md` / `related-requirements.md` / `analysis-notes.md`
- 向用户汇报:本次新增了哪些 FR/AC、哪些被列为待澄清、关联了哪些需求、版本看板的更新点

### 步骤 N — 末尾兜底提交(强制收尾,新增)
**所有其他步骤执行完毕后的最后一步,覆盖"过程文件 + 结果文件"。**

> 注:本步骤对**首次分析(步骤 7A–9A / 10A)** 和**增量更新(步骤 7B–10B)** 两条分支都执行,均作为收尾。

1. **收集 dirty 文件**:
   - 执行 `Bash: git status --porcelain`
   - 输出为空 → 打印 `无文件变更,跳过末尾提交`,**跳过步骤 2-4,直接退出**(NFR-4 幂等 / AC-3.5)
   - 输出非空 → 进入步骤 2

2. **暂存文件**:
   - 执行 `Bash: git add <所有 dirty 文件>`(从步骤 1 的输出中提取)
   - 失败 → 透传 stderr,中断退出

3. **生成 commit message 预览**(NFR-6 沿用 V0.0.1 实践):
   ```
   chore(code-require): <需求编码> <需求标题>

   需求分析完成:<N> FR / <M> NFR / <K> AC
   过程文件 3 + 结果文件 1 = <N> 文件待提交
   ```
   - `<需求标题>` 从本技能本次执行的需求中提取(若是首次分析,取本轮 RESULT.md 的标题;若是增量更新,取上游 RESULT.md 的标题)
   - 数字 N / M / K 从本轮 RESULT.md 的统计中提取

4. **弹 3 选 1 确认**(Q-3 锁定 A / FR-3.AC-3.4):
   - **A. 确认提交(推荐)** → 进入步骤 5
   - **B. 修改 commit message** → 重新生成预览(可由用户编辑)→ 返回步骤 4
   - **C. 取消** → 跳过 commit,打印"已取消提交,文件保持暂存/工作区状态",**退出**(E-9)

5. **执行 commit**:
   - 执行 `Bash: git commit -m "<message>"`
   - 成功 → 打印 `commit 完成,hash: <hash>`,退出
   - 失败(退出码非 0,pre-commit hook / 其他)→ 打印 stderr,**不重试**(E-10),提示"末尾提交失败,文件已暂存,请手动处理后执行 git commit",退出

### 步骤 5B — 增量更新:识别材料变更
1. 读取 `materials-index.md`(上一轮登记)
2. 对比当前 `Glob` 结果,标记:
   - **新增**材料
   - **修改**材料(若可读,重新读;若不可读,询问用户是否变了)
   - **删除**材料(文件不在了)
3. 更新 `materials-index.md` 反映最新状态

### 步骤 6B — 重读现有 RESULT.md

- 保留"变更记录"章节,准备追加新条目

### 步骤 7B — 局部更新 RESULT.md

## 模板填充步骤(本需求 REQ-00021 新增,FR-2)

> 本小节是 `code-require` 在主产出物 `RESULT.md` 完成后,根据 `--result` 参数执行**可选**的模板填充步骤。
> 本步骤**不**是必需步骤;无 `--result` 时跳过。

### 触发条件

- 用户调 `code-require <REQ> --result <模板文件>` 时触发
- **不**触发条件:无 `--result` 参数 / `code-auto` 上下文 / 模板文件不存在 / 模板为二进制格式

### 执行流程(FR-2)

1. **读取模板**:`Read <模板文件路径>`
   - 可文本化格式(.md / .html / .txt / .json / .xml / .csv / .yaml 等)→ 完整读取
   - 二进制格式(.docx / .xlsx / .pdf / .pptx)→ 屏显 `⚠ 模板格式二进制,无法读取占位符,跳过填充`,**不**尝试读取(避免乱码)
2. **占位符提取**:扫描 `{{...}}` 风格占位符
3. **结构化数据准备**:从 `RESULT.md` 提取 8 类数据(REQ_ID / REQ_TITLE / 需求概述 / FR_LIST / NFR_LIST / AC_LIST / 关联需求 / 待澄清)
4. **占位符替换**:按本需求内置的 15 个占位符映射表替换
5. **写出文件**:`<原主产出物同目录>/REQUIRE.<ext>`(后缀 = 模板后缀)
6. **屏显**:
   ```
   === code-require 模板填充 ===
     模板:<路径>
     输出:<路径>(<N> 个占位符已替换)
   ```
7. **过程文档**:在 `analysis-notes.md` 追加"模板填充结果"节(模板路径 / 输出路径 / 占位符填充数 / 失败占位符列表)

### 边界与异常(本节子节)

- **E-5**:模板无占位符 → 把 `RESULT.md` 完整内容追加到模板
- **E-6**:占位符未识别 → 已识别的替换,未识别的保留原样
- **E-7**:二进制格式模板 → 屏显 `⚠ 跳过`,不报错
- **E-8**:模板文件 > 1MB → 屏显 `⚠ 过大`,继续
- **E-9**:模板路径 `../` 跳出工作空间 → 屏显 `⚠ 模板路径不安全`,跳过
- **E-10**:模板路径含通配符 → 屏显 `⚠ 模板路径不支持通配符`,跳过
- **E-11**:多次执行(`--result` 模板相同)→ 输出文件**覆盖**(幂等)

- **不**修改 `RESULT.md`(主产出物)

- **不**修改"## 不要做的事"小节
