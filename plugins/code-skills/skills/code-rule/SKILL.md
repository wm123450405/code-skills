# code-rule — 编码规范管理

## 目标
让用户用最自然的语言描述一条编码规范,本技能负责:
- **澄清**:识别模糊点,用 `AskUserQuestion` 主动追问
- **归类**:把规范归入已知的规范分类(命名/错误处理/安全/性能/...)
- **结构化**:把口语化描述转成结构化条款(含示例/例外/适用范围)
- **落地**:写入 `./assistants/rules/<分类>.md`,供后续所有 `code-*` 技能消费

**定位**:`code-rule` 是"规范基建"工具,所有其他 `code-*` 技能(`code-design` 等)都把 `./assistants/rules/` 视为**只读**的强约束输入。

## 适用场景
- 启动新项目,需要建立首批编码规范
- 在项目进行中追加新的规范(命名/错误处理/安全/性能/...)
- 发现现有规范有缺口,需要扩展某分类下的条款
- 团队 review 后形成新的规范条目,统一沉淀

## 不适用
- 修改 `./assistants/rules/` 下既有的具体条款内容(本技能只**追加/扩展**,删除/重构请直接用编辑器处理或后续扩展本技能)
- 给单个项目版本打"临时补丁规范"(规范是跨版本共享的,不存在版本级规范)
- 给某一类规范(命名/错误处理)做**整体重写**(本技能只增量)

## 工作目录约定(强制)

```
./assistants/
├── rules/                              # 项目级规范(本技能产出/扩展),跨所有版本共享
│   ├── framework-conventions.md       # C-1 框架规范(本技能可能新建,条件性)
│   ├── dependency-conventions.md     # C-2 三方依赖规范(本技能可能新建,条件性)
│   ├── naming-conventions.md         # C-3 语言与命名规范(本技能可能新建,默认)
│   ├── directory-conventions.md      # C-4 目录与模块规范(本技能可能新建,默认;替代 module-conventions)
│   ├── coding-style.md               # C-5 代码书写规范(本技能可能新建,默认)
│   ├── commit-conventions.md         # C-6 提交与合并规范(本技能可能新建,条件性)
│   ├── dashboard-conventions.md      # 保留:看板
│   ├── doc-conventions.md            # 保留:文档
│   ├── marketplace-protocol.md       # 保留:marketplace 协议
│   ├── skill-conventions.md          # 保留:技能
│   ├── module-conventions.md         # 弃用:已迁移到 directory-conventions.md(DEPRECATED)
│   ├── encoding-conventions.md       # 编码格式权威源(由 code-it 在 REQ-00002 中创建)
│   ├── migration-mapping.md          # 旧→新编码迁移映射(由 code-it 在 REQ-00002 中创建)
│   └── <其他分类>.md
├── .current-version                # 与本技能无关
└── <版本号>/                        # 与本技能无关
```

- 路径以**当前工作目录(CWD)**为基准
- 本技能**只**操作 `./assistants/` 下的 `rules/` 目录(以及 `rules/` 的父目录 `assistants/` 若不存在则创建)
- 本技能**不**读写 `./assistants/.current-version`
- 本技能**不**读写 `./assistants/<版本号>/` 下任何内容
- 本技能**不**修改其他 `code-*` 技能负责的文件
- 其他 `code-*` 技能(`code-design` 等)会读取 `rules/` 作为只读输入,**不**通过本技能调用,而是直接读

## 输入
- **规范描述**(必填):用户的自然语言,可一段可一句
  - 例:"函数命名用 camelCase,不要用下划线"
  - 例:"所有错误必须用自定义异常,不能用 errno"
  - 例:"数据库操作必须走 ORM,禁止裸 SQL"
- **可选补充**:用户可一次给多条(用换行/编号/自然分段),本技能会逐条处理

## 输出
主产出物:`./assistants/rules/<分类>.md` 文件
- 若该分类文件**不存在** → 新建,基于 `templates/rule.md` 模板填充
- 若该分类文件**已存在** → 在文件中**追加**新的"规则 N"小节(不重写既有内容)

**不修改**:
- `./assistants/.current-version`
- `./assistants/<版本号>/` 下任何文件
- `./assistants/rules/` 下的其他分类文件

## 工具使用约定
- 读分类文件:`Glob "./assistants/rules/**/*"` 列出已有规范,`Read` 单个文件
- 探 CWD 根:`Glob "./assistants/rules/*"` + `Bash: ls ./assistants/` 了解现状
- 建目录:`Bash: mkdir -p "./assistants/rules/"`
- 写规范文件:`Write`(首次) / `Edit`(追加)
- 与用户澄清:**优先使用 `AskUserQuestion`** 给出结构化选项;确实无法结构化时使用自然语言提问

## 工作流程

### 步骤 0 — 不需要版本上下文
**与所有其他 `code-*` 技能不同,本技能不需要读取 `./assistants/.current-version`。**
- 规范是跨版本共享的,与具体版本无关
- 即使没有创建过任何版本(`.current-version` 不存在、`<版本号>/` 不存在),本技能仍可正常执行
- 本技能执行过程中**不创建**版本工作空间(那是 `code-version` 的职责)

### 步骤 1 — 探查 `./assistants/` 现状
1. `Bash: ls -la "./assistants/" 2>/dev/null`(若目录不存在,`ls` 会报错,记为"目录不存在")
2. `Glob "./assistants/rules/*"` 列出已有规范文件(若 `rules/` 不存在,结果为空)
3. `Read "./assistants/.current-version"`(本技能不写它,但读一下方便汇报"是否已有激活版本")
4. 把发现整理成内部状态:
   - `assistants_exists`:布尔
   - `rules_dir_exists`:布尔
   - `existing_rule_files`:列表(如 `["architecture.md", "naming.md"]`)
   - `current_version`:字符串或 null

### 步骤 2 — 兜底创建目录
按需创建(不覆盖已有内容):
- 若 `./assistants/` 不存在 → `Bash: mkdir -p "./assistants/rules/"`
- 若 `./assistants/rules/` 不存在但 `./assistants/` 存在 → `Bash: mkdir -p "./assistants/rules/"`

**绝不**创建 `.current-version` 或 `<版本号>/`,那是其他技能的事。

### 步骤 3 — 收集规范描述
- 若用户在调用本技能时已经给出了规范描述(在第一轮消息里) → 直接使用,记为 `<原始描述>`
- 若用户没有给 → 用自然语言主动询问:
  > 请用一两句话描述你想添加的编码规范(例:"函数命名统一用 camelCase"、"数据库操作必须走 ORM"等)。
  > 可以一次描述多条,每条一行。

### 步骤 4 — 拆分 + 类型识别 + 初步归类(plan 微调后扩展)

> **plan 阶段微调**(Q-PLAN-2):原 design M-1"类型识别引擎"作为独立子流程,变更为本步骤的子段 4.2(模块命名 M-1 保留便于追踪)。

#### 4.1 拆分
1. 把 `<原始描述>` 按换行/编号/自然段拆成 N 条独立规则,记为 `Rule[1..N]`
   - 若 N=0 → 回到步骤 3
   - 若 N>1 → 询问用户"是否逐条处理(本轮)还是一次性处理"

#### 4.2 类型识别(自动 + 显式)
对每条规则,识别其属于 **Type A / B / C** 中的哪一类:

1. **关键词扫描**(关键词表):
   - **Type A**:`规则` / `规范` / `约定` / `命名` / `提交` / `代码风格` / `依赖` / `框架` / `目录` / `模块` / `package` / `architecture` / `naming` / `commit` / `dependency`
   - **Type B**:`CLAUDE.md` / `AI 工作` / `AI 约定` / `AI 写` / `AI 读` / `AI 应` / `AI 协作者`
   - **Type C**:`模板` / `template` / `templates/` / 具体模板文件名(如 `plan.md` / `requirements.md`)

2. **置信度评估**:
   - 1 个类型 + 1+ 关键词 + 目标文件明确 → **高**(直接进入对应类型子流程)
   - 1 个类型 + 0 关键词 + 语义提示模糊 → **中**(`AskUserQuestion` 确认)
   - 多类型 / 0 类型 / 0 关键词 → **低**(`AskUserQuestion` 列出候选)

3. **显式覆盖**:用户可用 `"对 X 文件加 Y"` 形式显式指定(`X = CLAUDE.md` → Type B,`X = templates/*` → Type C,`X = 其他` → Type A)

#### 4.3 初步归类(Type A 分支)
仅 Type A 进入此子段(B/C 路由到 §Type B / §Type C 子流程):

1. 对每条规则,基于关键词/语义给出一个**候选分类**(可能多个):
   - **C-1 框架规范**(条件性):`框架` / `架构` / `框架选型` / `framework` / `architecture`
   - **C-2 三方依赖规范**(条件性):`依赖` / `第三方` / `库` / `包` / `dependency` / `third-party` / `library` / `package`
   - **C-3 语言与命名规范**(默认):`命名` / `camelCase` / `snake_case` / `PascalCase` / `前缀` / `后缀` / `naming`
   - **C-4 目录与模块规范**(默认):`目录` / `模块` / `包结构` / `文件组织` / `directory` / `module` / `package`
   - **C-5 代码书写规范**(默认):`代码风格` / `注释` / `错误处理` / `异常` / `性能` / `安全` / `编码规范` / `coding-style` / `comment` / `error-handling`
   - **C-6 提交与合并规范**(条件性):`提交` / `commit` / `合并` / `merge` / `PR` / `分支` / `branch`
   - **保留:看板**:`看板` / `仪表盘` / `状态机` / `dashboard`
   - **保留:文档**:`README` / `中英` / `doc`
   - **保留:marketplace**:`marketplace` / `plugin.json`
   - **保留:技能**:`SKILL.md` / `frontmatter`
   - **保留:编码格式**:`编码格式` / `encoding` / `TASK 编码` / `BUG 编码`(→ encoding-conventions.md,新)
   - **保留:迁移映射**:`迁移` / `migration` / `编码追溯`(→ migration-mapping.md,新)
   - **其他**:无法归入上述 → 用户输入的新分类名

2. 把每条规则的候选分类展示给用户,用 `AskUserQuestion` 让用户**确认或调整**分类(可一次问多条)
   - 这一步**不能省**:分类决定文件归属,错了会污染其他分类
   - **条件性分类**(C-1/C-2/C-6)需先追问"现在需要 / 未来占位 / 跳过"
3. 把每条规则的候选分类展示给用户,用 `AskUserQuestion` 让用户**确认或调整**分类(可一次问多条)
   - 这一步**不能省**:分类决定文件归属,错了会污染其他分类

### 步骤 5 — 澄清规则细节(每条规则独立)
对每条规则,基于规范描述和分类,识别**未覆盖的字段**,用 `AskUserQuestion` 追问。每次聚焦 1-3 个最阻塞的字段。

需澄清的字段(按优先级):
1. **强制级别**(必问,决定 code-check 是否阻断)
   - 必须(Mandatory):违反会被 code-check 标记为 P0,阻断合并
   - 推荐(Recommended):默认遵守,偏离需在 PR 中说明理由
   - 参考(Reference):仅作建议,不强制
2. **适用范围**(必问,避免一刀切)
   - 全语言/全项目 / 特定语言(Python/JS/Go/...) / 特定模块 / 仅新增代码
3. **正反示例**(尽量问,没有示例的规则难以落地)
   - 可以让用户口头描述,或直接给一段示例代码
4. **例外场景**(可选)
   - 哪些情况下可以破例?例:"性能关键路径可豁免"
5. **关联规范**(可选)
   - 是否与已有规范文件中的某条规则相关?如果有,需引用

**每轮问答**记录到内部会话状态,稍后写入文件的"规则 N"小节。

> **不要一次问完所有字段**。前 1-2 个问题(强制级别 + 适用范围)必问,得到答复后先生成结构化规则,再迭代问示例/例外。

### 步骤 6 — 探测目标文件现状
对每条规则的目标分类文件:
1. 检查 `./assistants/rules/<分类>.md` 是否存在
2. 存在 → `Read` 整个文件,理解现有内容,识别:
   - 文件已有的"规则 N"小节(避免编号冲突)
   - 是否已有**语义相同或冲突**的规则(若有,询问用户:跳过 / 覆盖 / 共存)
3. 不存在 → 准备基于 `templates/rule.md` 模板新建

### 步骤 7 — 写文件
**情形 A:目标文件不存在**
1. `Write "./assistants/rules/<分类>.md"`,内容基于 `templates/rule.md` 模板
2. 模板头部填入:
   - 分类中文名 + 英文名
   - 适用范围(项目级共享)
   - 强制级别默认(留空或由用户在步骤 5 指定)
3. 在"规则 1"小节写入本条规则的结构化内容

**情形 B:目标文件已存在**
1. 用 `Edit` 工具,**在文件末尾**追加新的"规则 N"小节
2. 不修改文件中任何已有内容
3. N = 现有"规则 N"编号 + 1

**规则小节的标准结构**:
```markdown
## 规则 N:<规则简称>

### 条款
<用一句或一段清晰的话描述规则>

### 强制级别
- 必须 / 推荐 / 参考

### 适用范围
- <语言/模块/范围>

### 正面示例
\```
<好的代码或做法>
\```

### 反面示例
\```
<坏的代码或做法>
\```

### 例外
- <什么情况下可破例,无则写"无">

### 关联规范
- ./assistants/rules/<其他分类>.md §规则 M
- 无 / 待补充
```

### 步骤 8 — 汇报
对每条规则,向用户汇报:
- **分类**:<分类中文名>(对应文件 `<分类>.md`)
- **规则简称**:`<规则简称>`
- **强制级别**:`<必须/推荐/参考>`
- **文件位置**:`./assistants/rules/<分类>.md §规则 N`
- **新增 / 追加**到已有文件
- **下游影响**:
  - `code-design` 在生成概要设计时,会读这条规范作为"接口/模块/数据"决策的约束
  - `code-check` 在评审时,会按强制级别决定是否标记为缺陷

### 步骤 9 — 多规则与迭代
- 若用户一次给了 N>1 条规则,处理完一条后,继续处理下一条,直到全部完成
- 处理完全部后,向用户询问:
  > 已处理 N 条规则。
  > - 选项 A:继续追加更多规范
  > - 选项 B:查看 `./assistants/rules/` 现有规范清单
  > - 选项 C:结束(建议接下来调 `code-version` 创建版本工作空间)

## 已有规范识别与冲突处理

为避免规则污染,处理每条规则前必须:

### 1. 重复检测
- 在 `./assistants/rules/<分类>.md` 中 `Grep` 规则简称关键词
- 若已有语义完全相同的规则 → 询问用户:
  - 跳过(规则已存在)
  - 补充到现有规则(用 `Edit` 在该规则小节追加子条款)
  - 覆盖现有规则(用 `Edit` 替换该规则小节)
  - 作为新规则追加(并存)

### 2. 冲突检测
- 已有规则 A 说"必须用 camelCase",新规则说"必须用 snake_case" → **冲突**
- 询问用户:
  - 撤销新规则
  - 覆盖旧规则(用 `Edit` 替换旧规则小节)
  - 保留两者并标注冲突(用 `Edit` 在两边都加 "(与 §规则 M 冲突,待裁决)")

### 3. 跨分类引用
- 新规则与**其他分类**文件中的某条规则相关(如命名规范引用了模块规范) → 在"关联规范"字段填写

## Type A 子流程:占位模式 / 引导模式

> 适用于**条件性分类**(C-1 / C-2 / C-6)和**默认分类**(C-3 / C-4 / C-5)的特殊处理。

### 占位模式(FR-3,条件性分类)

**触发条件**:
- 用户描述命中 C-1 / C-2 / C-6(条件性分类)
- 用户选择"未来占位"

**算法**:
```
若目标分类文件不存在:
  用 templates/rule.md 的最小骨架创建文件:
    - 分类标题(中文 + 英文)
    - "本规范由 code-rule 维护"声明
    - "## 适用场景"占位
    - "## 强制级别约定"说明
    - "## 规则 1: (待添加)"占位
否则:
  走"已有规范识别与冲突处理"
```

**用户选择分支**:
- "现在需要" → 走 §引导模式
- "未来占位" → 走 §占位模式
- "跳过" → 不创建任何文件

### 引导模式(FR-4,默认分类)

**触发条件**:
- 用户描述命中 C-3 / C-4 / C-5(默认分类)
- 用户**首次添加**该类规则(目标文件不存在)

**算法**:
```
若目标分类文件不存在:
  提示用户"是否同时补 1-2 条示例规则?"
  若用户同意:
    走"新建"标准流程(澄清字段 + 写文件)
  若用户跳过:
    走 §占位模式
否则:
  走"已有规范识别与冲突处理"
```

**不变量**:
- INV-2:6 个新分类文件**仅含最小骨架**,无预填规则
- INV-2:即使引导模式,用户**未提供示例**时仍走占位模式

## Type B 子流程(AI 工作指引追加)

> 适用于用户描述含 `CLAUDE.md` / `AI 工作` / `AI 约定` 等关键词,目标是追加 AI 行为指令到 `<本仓库>/CLAUDE.md`。

### 触发与识别
- 关键词命中 + 高置信度 → 直接进入 Type B 子流程
- 用户可显式覆盖:`"对 CLAUDE.md 增加 X"`

### 数据结构(5 字段)

```markdown
### 指引 N:<指引简称>

#### 描述
<一句话或一段说明该指引做什么>

#### 适用场景
<什么情况下 AI 应遵守>

#### 期望行为
<AI 应如何行动,可含"读哪个文件/调哪个技能/避免什么">

#### 来源
- 用户原始描述:"<用户最初说的话>"
- 添加时间:YYYY-MM-DD HH:mm
```

### 插入位置与算法

1. `Read <本仓库>/CLAUDE.md` 全文
2. `Grep "## AI 工作约定(由 code-rule 维护)"` 检测小节是否存在
3. **不存在** → 在文件末尾追加:
   ```markdown
   ## AI 工作约定(由 code-rule 维护)

   > 本小节由 `code-rule` 技能维护;手动修改可能被 `code-rule` 覆盖。
   > 适用对象:Claude Code 在本仓库工作时

   ### 指引 1: (待添加)
   ```
4. **存在** → 在该小节末尾追加新"指引 N"小节(N = 现有编号 + 1)
5. **绝不**修改 CLAUDE.md 其它任何小节(INV-3)

## Type C 子流程(模板内容提示追加)

> 适用于用户描述含 `模板` / `template` / `templates/` / 具体模板文件名,目标是给模板文件追加"内容提示"段。

### 触发与识别
- 关键词命中 + 高置信度 → 直接进入 Type C 子流程
- 用户可显式覆盖:`"对 templates/plan.md 增加 X"`

### 数据结构(4 字段,末尾模式;3 字段,内联模式)

**末尾追加模式**(`## 提示: <主题>`):
```markdown
## 提示: <主题>

- **字段**:<主题>
- **必填**:是 / 否
- **简明说明**:<1-2 句话>
- **错误示例**(可选):<错误填法>
```

**内联模式**(`### 提示: <字段>`):
```markdown
### 提示: <字段>

- **字段**:<字段>
- **必填**:是 / 否
- **简明说明**:<1-2 句话>
```

### 插入位置与算法

1. 从用户描述中提取目标模板路径(若未指定,追问用户)
2. `AskUserQuestion` 询问插入位置(末尾追加 / 内联)
3. **末尾追加**:
   - 检测模板文件末尾是否含 `## 提示` 二级小节
   - **不存在** → 在模板文件末尾追加小节标题 + 1 个空"提示 1"占位
   - **存在** → 在该小节末尾追加新"提示 N"小节
4. **内联**:
   - 用户指定目标二级小节(精确匹配 `## N. <小节名>`)
   - **指定** → 在该小节末尾追加 `### 提示: <字段>` 三级小节
   - **未指定** → 追问用户(不自动推断,避免误插)
5. **绝不**修改模板其他内容(INV-4)

## 过程文档格式

本技能**不**在 `./assistants/rules/` 之外生成过程文档。所有产出**直接是规范文件本身**(结构化、可被其他技能直接消费)。

**例外**:若用户在 `AskUserQuestion` 之外有大量上下文讨论,本技能可在 CWD 下临时创建 `./.rule-session.md`(本技能结束后可保留/可删除),用于本会话内多轮追问的上下文保持。但**不**写入 `./assistants/`。

## 衔接

- **下游(被消费方)**:`code-require` / `code-design` / `code-plan` / `code-it` / `code-unit` / `code-check` 全部会 `Glob "./assistants/rules/**/*"` 读取本技能的产出,作为只读约束
- **上游(本技能的输入)**:用户直接发起的自然语言描述
- **横向**:本技能与 `code-version` 无依赖关系,可任意顺序调用
  - 典型顺序:先 `code-rule` 建立规范 → 再 `code-version` 创建版本 → 再 `code-require` 起步
  - 但也可以先 `code-version` → 再 `code-rule` 补充规范

## 不要做的事
- 不要读取或写入 `./assistants/.current-version`(与本技能无关)
- 不要操作 `./assistants/<版本号>/` 下任何文件
- 不要重写已有的规范文件(本技能只**追加/扩展**,不重写)
- 不要在分类未确认时直接写入文件(必须先步骤 4 的分类确认)
- 不要把"必须"和"推荐"的规则混用同一文件头部声明(每条规则各自标注)
- 不要在没有澄清强制级别的情况下写入文件(强制级别是必填字段)
- 不要在没有 `Read` 现有文件的情况下用 `Write` 覆盖(会丢失历史)
- 不要把音视频/图片等不可直接读取的材料作为规则描述的来源(让用户用文字概括)
- 不要替用户决定分类(必须用 `AskUserQuestion` 让用户确认或调整)
