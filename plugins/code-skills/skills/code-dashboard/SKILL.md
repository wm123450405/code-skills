# code-dashboard — 开发看板(版本感知,只读)

## 目标

在 `code-version` 已建立的工作空间之上,提供一个**只读型**全局看板入口,让用户(或 AI 协作者)在长会话或多次调用之间,能用一行命令看到:

- **当前激活版本**下,需求 / 任务 / 缺陷的整体执行情况
- **最多 5 条"下一步执行建议"**(可执行命令 + 依据 + 优先级)
- 支持**双粒度**:无参数(版本总览)/ 指定 `REQ-NNNNN`(需求粒度)

本技能**不修改**任何文件,所有输出仅打印到屏幕;**不调用** `Write` / `Edit` / `Bash`;**只**调用 `Read` / `Glob` / `Grep`。可随时调用,多次执行幂等。

## 适用场景

- **会话开头总览**:每个工作日 / 每次会话开头,调一次 `/code-dashboard` 看到"还差什么"
- **长会话中的"接下来做什么"**:在多次 `code-*` 调用之间,调一次 `/code-dashboard` 获得下一步建议
- **需求粒度摸底**:想知道某条需求(`/code-dashboard REQ-00001`)的进度 + 其下任务 + 关联缺陷
- **评审前快速摸底**:在做 `code-check` 前,先 `/code-dashboard` 看一下版本整体状态

## 不适用

- 想"修改看板字段 / 添加区段 / 改列名"——`code-dashboard` 是**只读**,看板字段扩展需走 `code-rule`(触发 `dashboard-conventions §规则 1` 3 文件同步)
- 想"切到历史版本 / 查看 V0.0.0 / V0.0.1 的整体状态"——`code-dashboard` 只读**当前激活版本**;切版本走 `code-version`
- 想"自动登记缺陷 / 推进任务状态"——`code-dashboard` **不写**任何文件,无副作用
- 想"跨版本汇总"——本需求**不**实现(留作 v2 follow-up)
- 紧急线上修复(走 hotfix 流程,不在本看板范围)

## 工作目录约定(强制)

```
./assistants/
├── .current-version                # 当前激活版本标记(本技能读)
├── rules/                          # 项目级规范(跨版本共享,本技能只读)
└── <版本号>/                       # 当前激活版本工作空间(本技能只读)
    ├── RESULT.md                   # 主看板
    ├── require/<需求编号>/
    │   └── RESULT.md               # 需求模式:需求详情
    └── plan/<需求编号>/
        └── PLAN.md                 # 需求模式:任务清单
```

- 路径以**当前工作目录(CWD)**为基准
- 本技能**只读**:`.current-version` / `<版本号>/RESULT.md` / `require/.../RESULT.md` / `plan/.../PLAN.md` / `rules/encoding-conventions.md`
- 本技能**不写**任何文件(NFR-7 严守;FR-7 AC-7.1 验证 `git status` clean)
- 本技能**不修改**其他 10 个 `code-*` 技能的 SKILL.md frontmatter(NFR-6 严守)
- 本技能**不修改** `marketplace.json` / `plugin.json`(走 Claude Code 技能自动发现协议)

## 输入

- **0 个参数** → 总览模式(版本粒度)
- **1 个参数 `REQ-NNNNN`**(`^REQ-\d{5}$` 严格匹配)→ 需求模式(需求粒度)
- **1 个参数不匹配 `^REQ-\d{5}$`** → 错误模式(打印用法 + 退出,无副作用)
- **2+ 个参数** → 错误模式(同 1 个参数不匹配)

## 输出

**屏幕输出**(无文件返回):
- 总览模式:4 段(本需求 REQ-00023 改造,沿用 NFR-1 ≤ 12 行)
  1. 总开发进度(单行百分比 + 12 字符 ASCII 比例条)
  2. 各状态数量占比(5 类状态:待概要设计 / 待详细设计 / 待代码开发 / 待单元测试 / 待代码评审)
  3. 高优先级缺陷(P0 / P1 醒目标记,只列数量)
  4. 后续操作建议(每状态 1 条,共 ≤ 5 条)
- 需求模式:5 段
  1. 需求元信息(标题 / 状态 / 概要设计 / 详细设计)
  2. 任务清单(任务编码 / 标题 / 开发状态 / 测试状态)
  3. 关联缺陷(从主看板筛 `关联任务` 含 `REQ-NNNNN-` 的缺陷)
  4. 任务进度(已完成 / 总数 + 比例)
  5. 下一步建议(最多 5 条)

错误模式:用法示例 + 退出(无副作用)

**总览模式 4 段契约样例**(本需求 REQ-00023 改造,沿用 plan/REQ-00023/RESULT.md §6.1):
```
总开发进度
──────────
[████████░░░░] 67%

各状态数量占比
──────────────
待概要设计: 需求 2 / 缺陷 0(占比 50.0%)
待代码开发: 需求 1 / 缺陷 0(占比 25.0%)
待代码评审: 需求 1 / 缺陷 0(占比 25.0%)

高优先级缺陷
───────────
P0 待修复: █ 1
P1 待修复: ░ 0

后续操作建议
────────────
> 建议: /code-design REQ-00023
> 依据: REQ-00023 在"待概要设计"状态
> 优先级: 中
```

**退化**(无需求无缺陷):
- 段 1:显示 `— / 无需求无缺陷,无需进度`(沿用算法 1)
- 段 2:该段退化显示 `(无)`
- 段 3:仍展示骨架 `P0 待修复: ░ 0` / `P1 待修复: ░ 0`(AC-5.3 锁定)
- 段 4:显示"无后续动作" 1 行(沿用既有)

**总行数**:有需求有缺陷 + 5 类状态全部有触发 → ≤ 12 行(NFR-1 + AC-7.1);退化场景 ≤ 6 行

- **读**:`Read` 工具
  - `Read "./assistants/.current-version"`(步骤 0)
  - `Read "./assistants/<版本号>/RESULT.md"`(总览模式 步骤 2;需求模式 步骤 2b 关联缺陷筛)
  - `Read "./assistants/<版本号>/require/<需求编号>/RESULT.md"`(需求模式 步骤 2b)
  - `Read "./assistants/<版本号>/plan/<需求编号>/PLAN.md"`(需求模式 步骤 2b)
- **列目录**:`Glob` 工具
  - `Glob "./assistants/<版本号>/require/REQ-*"`(需求模式 步骤 2a:列本版本所有需求,用于 E-3 错误引导)
- **搜索/解析**:`Grep` 工具
  - `Grep "^## "`(步骤 3:行号锚点定位区段)
  - `Grep "^| .* |$"`(步骤 3:匹配表格行)
- **不调用**:`Write` / `Edit` / `Bash` / `WebFetch` / `WebSearch` / `Task` / `Agent`(NFR-7 严守)
- **不调用**任何 MCP 工具

## 工作流程

### 步骤 0 — 版本上下文检测(强制前置)

1. `Read "./assistants/.current-version"`
2. **文件不存在** → 走边界 E-1:`✗ 未检测到激活的版本工作空间` + 引导 + 退出
3. 读取内容,记为 `<版本号>`(例:`V0.0.2`)
4. **指向不存在版本**(目录缺失)→ 走边界 E-2:`✗ 版本 <X> 工作空间不存在` + 引导 + 退出
5. 成功 → 进入步骤 1

### 步骤 1 — 参数解析(算法 0)

| 输入 | 输出 |
| --- | --- |
| 无参数 | mode=`总览` |
| `REQ-NNNNN`(匹配 `^REQ-\d{5}$`) | mode=`需求`, reqNum=`REQ-NNNNN` |
| 1 参不匹配 / 2+ 参 | 边界 E-4:`✗ 参数格式错误: <arg>` + 用法 + 退出 |

### 步骤 2 — 数据加载

#### 步骤 2a(总览模式)

1. `Read "./assistants/<版本号>/RESULT.md"`(主看板,1 次 IO)
2. 缺失 → 边界 E-5:`✗ 看板文件不存在` + 退出
3. 成功 → 进入步骤 3

#### 步骤 2b(需求模式)

1. `Glob "./assistants/<版本号>/require/REQ-*"` 列出本版本所有需求
2. 验证 `reqNum` 在列表中;**不存在** → 边界 E-3:`✗ 在 <V> 中未找到需求 <REQ>` + 列出本版本所有需求 + 退出
3. **并行** `Read` 3 个文件(同消息块内并发):
   - `<版本号>/RESULT.md`(关联缺陷筛)
   - `<版本号>/require/<REQ>/RESULT.md`(需求详情)
   - `<版本号>/plan/<REQ>/PLAN.md`(任务清单)
4. 任一文件缺失 → 该文件对应字段显示 `?` 占位,继续渲染(不退出)
5. 成功 → 进入步骤 3

### 步骤 3 — 区段解析(算法 1)

单遍扫描主看板文本(若需求模式,从 `require/.../RESULT.md` 解析需求详情):

1. 按行切分文本为 `lines[]`
2. 遍历 `lines`,匹配正则 `^## (.+)$` 收集所有区段标题 + 行号 → `anchors{}`
3. 按"目标区段"提取"行区间":
   - 需求清单(总览)
   - 任务清单(总览)
   - 缺陷清单(总览 + 需求模式筛关联)
4. 在每个区间内,匹配 `^\| .* \|$` 收集表格行(过滤 `^\| ---` 表头分隔行)
5. 按 `|` 切分列 → `RequirementRow[]` / `TaskRow[]` / `BugRow[]`
6. 任务编号列按**算法 4** 解析(`parseTaskId()`)

**L2 退化**:
- 区段缺失(锚点不存在)→ 该区段返回 `[]`(后续显示 `(无)`)
- 表格列错位(列数 ≠ 期望)→ 退化到"原始 markdown 块"原样输出
- 字段值缺失(单元格空)→ 显示 `?` 占位

### 步骤 4 — 聚合 + 渲染

#### 段 1:总开发进度(总览模式,本需求 REQ-00023 新增)

- **公式**(加权平均,沿用 FR-2 + AC-1):
  ```
  total_done = Σ 需求已完步骤数 + Σ 缺陷已完步骤数
  total_expected = (需求数 × 6) + (缺陷数 × 5)
  总进度 = round(total_done / total_expected × 100)%
  ```
- **每条需求贡献**(6 步):需求分析 + 概要设计 + 详细设计 + 编码 + 单元测试 + 代码评审
- **每条缺陷贡献**(5 步):缺陷登记 + 详细设计 + 编码 + 单元测试 + 代码评审
- **不适用的环节按已完成计入**(AC-1.3 锁定;如纯文档任务的"单元测试=不适用",**算**已完成 1 步)
- 显示:
  ```
  总开发进度
  ──────────
  [████████░░░░] 67%
  ```
- **退化**(分母 = 0,即无需求无缺陷,沿用 AC-1.4):显示 `— / 无需求无缺陷,无需进度`
- **比例条**:`算法 5 renderBar(filled, total)`,固定 12 字符 + `█` / `░`(沿用既有)

#### 段 2:各状态数量占比(总览模式,本需求 REQ-00023 改造)

- **5 类状态**(沿用 FR-3 + AC-2 锁定):
  | 状态 | 判定条件(派生自既有看板列,**不**新增字段) |
  | --- | --- |
  | **待概要设计** | 需求清单"概要设计"列= `—` 或空 + 不属于缺陷 |
  | **待详细设计** | 概要设计已完成 + 详细设计未开始(`plan/<REQ>/RESULT.md` 不存在;缺陷:`fix/<BUG>/PLAN.md` 不存在) |
  | **待代码开发** | 详细设计已完成 + 任务清单有任务"开发状态≠已完成" |
  | **待单元测试** | 任务"开发状态=已完成 ∧ 测试状态∈{未编写, 已编写, 已运行-失败}" |
  | **待代码评审** | 所有任务真正可发布 + 评审发现汇总"必须改"未处理 |
- **缺陷路径映射**:缺陷跳过"待概要设计",直接进入"待详细设计"
- **算法 2**:`classifyState(item, allTasks, reviews): DashboardState`(沿用 plan/REQ-00023/RESULT.md §4.2)
  - E-7 边界:同需求同时属"待代码开发"+"待单元测试" → 归入"待代码开发"
- 显示(沿用 FR-4):
  ```
  各状态数量占比
  ──────────────
  待概要设计: 需求 2 / 缺陷 0(占比 50.0%)
  待代码开发: 需求 1 / 缺陷 0(占比 25.0%)
  待代码评审: 需求 1 / 缺陷 0(占比 25.0%)
  ```
- **退化**(0/0 行不展示,AC-3.3):某状态(需求+缺陷)= 0 → 该行不展示
- **算法 3**:`buildBreakdown(items: Item[]): BreakdownRow[]`(沿用 plan/REQ-00023/RESULT.md §4.3)
  - 占比公式:`pct = (reqCount + bugCount) / total × 100%`,**保留 1 位小数**(AC-3.2)
  - 5 行占比加和 ≈ 100%(AC-3.5,允许 0.1% 四舍五入误差)
  - 屏显格式:`待概要设计: 需求 N / 缺陷 M(占比 X.X%)`(AC-3.1)
- **退化**(无需求无缺陷):该段退化显示 `(无)`(沿用既有)

#### 段 3:高优先级缺陷(总览模式,本需求 REQ-00023 改造,沿用既有)

- 沿用既有实现(FR-6 锁定,**不**简化):
- 统计:`bugs.filter(b => b.severity === "P0" && b.status !== "已修复").length` 等
- 显示:
  ```
  高优先级缺陷
  ────────────
  P0 待修复: █ 1
  P1 待修复: ▓ 0
  ```
- **不**展示具体缺陷标题(避免屏显膨胀,AC-5.3 锁定 P0/P1 均为 0 时仍保留骨架 `░ 0`)
- **不**展示 P2 / P3 数量(沿用既有 §"步骤 4 段 3" 段,只突出 P0 / P1)
- 标记:█(U+2588,实心,P0)/ ▓(U+2593,半实心,P1)/ ░(U+2591,空心,退化)

#### 段 4:后续操作建议(总览模式,本需求 REQ-00023 改造)

- **核心规则**(沿用 FR-5 + AC-4):**每类状态 1 条推荐**,共 ≤ 5 条
- **算法 4**:`parseTaskId(raw): {format, type, parentNum, taskNum, displayId} | null`(双正则兼容,新规则 REQ-00025;详 §附录 A)
- **5 类状态映射**(AC-4.5 严格按既有 10 个 `code-*` SKILL.md frontmatter 真实语法):
  | 状态(需求路径) | 命令 | 状态(缺陷路径) | 命令 |
  | --- | --- | --- | --- |
  | 待概要设计 | `/code-design <REQ>` | (缺陷不经过此状态) | — |
  | 待详细设计 | `/code-plan <REQ>` | 待详细设计 | `/code-plan <BUG>` |
  | 待代码开发 | `/code-it TASK-REQ-...-00001`(首个未完成) | 待代码开发 | `/code-it TASK-BUG-...-00001` |
  | 待单元测试 | `/code-unit TASK-REQ-...-00001` | 待单元测试 | `/code-unit TASK-BUG-...-00001` |
  | 待代码评审 | `/code-check <REQ>` | 待代码评审 | `/code-check <REQ>`(缺陷评审关联"原需求") |
- **展示规则**(AC-4.1 ~ AC-4.4):
  - 同状态需求 + 缺陷并存 → **只展示 1 条**(需求路径优先,AC-4.2 锁定)
  - 状态无触发 → **该行不展示**
  - 全部 5 类无触发 → 显示"无后续动作" 1 行
  - 严格 5 行上限(AC-4.3 沿用)
- 显示(每条 3 行):
  ```
  > 建议: /code-design REQ-00023
  > 依据: REQ-00023 在"待概要设计"状态
  > 优先级: 中
  ```

#### 段 5(需求模式):任务清单 + 关联缺陷 + 任务进度

- 任务清单:`PLAN.md` "任务总览" 表格的子集(只含本需求的任务)
- 关联缺陷:从主看板"缺陷清单"筛 `relatedTask.startsWith(reqNum + "-")`
- 任务进度:`已完成 / 总数 + 比例`

### 步骤 5 — 下一步建议生成(算法 3)

#### 总览模式(5 类优先级)

1. **P0 高**:
   - 版本无任何需求 → `建议: 执行 /code-require 添加首个需求`
   - 有 P0 待修复缺陷 → `建议: 执行 /code-fix BUG-NNNNN`
   - 需求"已完成(需求分析)"但"概要设计=未开始" → `建议: 执行 /code-design REQ-NNNNN`
2. **P1 中**:
   - 任务"开发状态=待开始" → `建议: 执行 /code-it TASK-REQ-NNNNN-NNNNN` 或 `建议: 执行 /code-it REQ-NNNNN-NNNNN`(兼容旧格式,见算法 4)
   - 需求"概要设计=已完成 ∧ 详细设计=未开始" → `建议: 执行 /code-plan REQ-NNNNN`
3. **P2 低**:
   - 任务"测试状态=已编写" → `建议: 执行 /code-unit TASK-...`
4. **特殊**:
   - 全版本已完成(所有需求=已完成 ∧ 所有任务=已完成 ∧ 无 P0/P1 待修复缺陷)→ `建议: 执行 /code-version V0.0.x 创建下一个版本`

#### 需求模式

1. 需求"概要设计=未开始" → `建议: 执行 /code-design <REQ>`(高)
2. 需求"详细设计=未开始" → `建议: 执行 /code-plan <REQ>`(中)
3. 任务"开发状态=待开始" → `建议: 执行 /code-it <TASK>`(中)
4. 任务"开发状态=已完成 ∧ 测试状态=未编写" → `建议: 执行 /code-unit <TASK>`(中)
5. 需求全完成 → `该需求已全部完成,无需后续动作`(`—`)

#### 取前 5 条 + 降序(AC-4.3)

按优先级降序(P0 高 > P1 中 > P2 低 > `—`)取前 5 条;无任何建议触发时显示"无后续动作" 段(AC-4.4)。

### 步骤 6 — 屏幕打印(无文件副作用)

- 按"段 1 → 段 2 → 段 3 → 段 4"顺序打印(总览)或"元信息 → 任务清单 → 关联缺陷 → 任务进度 → 建议"顺序打印(需求)
- 退出(无状态保留,NFR-7 幂等)

## 边界与异常

按 3 层退化组织(L1 / L2 / L3):

### L1 启动错误(致命,必须退出)

| 场景 | 触发条件 | 屏幕输出 | 依据 |

## 附录 A:任务编号解析(算法 4,本需求 REQ-00025 双正则兼容)

```
parseTaskId(raw):
  // 宽松正则优先(沿用 encoding-conventions §规则 1 接收端,后缀自由)
  m = match(/^TASK-(REQ|BUG)-([A-Za-z0-9.\-_]+)-([A-Za-z0-9.\-_]+)$/, raw)
  if m:
    return { format: "new", type: m[1], parentNum: m[2], taskNum: m[3], displayId: raw }
  // 旧格式透传(同上游放宽)
  m = match(/^(REQ|BUG)-([A-Za-z0-9.\-_]+)-([A-Za-z0-9.\-_]+)$/, raw)
  if m:
    return { format: "old", type: m[1], parentNum: m[2], taskNum: m[3], displayId: raw }
  // 解析失败
  return null
```

- **新格式**:`TASK-REQ-00001-00001` / `TASK-BUG-00001-00001`(默认 5 位纯数字;沿用 `encoding-conventions §规则 1/3`);**接收端放宽**为 `[A-Za-z0-9.\-_]+`(后缀自由,新规则 REQ-00025)
- **旧格式**:`REQ-00001-00001` / `BUG-00001-00001`(V0.0.1 REQ-00001 起的字面,本技能**只读**不重命名);同样放宽
- 解析失败 → `null` → 调用方按字面显示(NFR-2 L2 退化)

## 附录 B:ASCII 比例条(算法 5)

```
renderBar(filled, total):
  if total === 0:
    return "[" + "░" × 12 + "] 0%"
  pct = round(filled / total × 100)
  blocks = round(pct / 100 × 12)
  return "[" + "█" × blocks + "░" × (12 - blocks) + "] " + pct + "%"
```

- 固定 12 字符(Q-D3 锁定)
- `█`(U+2588)实心 / `░`(U+2591)空心 / `▓`(U+2593)P1 标记
- 主流终端(Windows Terminal / iTerm2 / GNOME Terminal)均支持 BMP 字符
