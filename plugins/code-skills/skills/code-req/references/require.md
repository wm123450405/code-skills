# 需求分析阶段 — code-req

> 本文件为 code-req 技能的 REQUIRE 阶段提供详细流程。在进入 REQUIRE 阶段时加载。

## 目标

将用户输入的需求描述(自然语言/文档/设计稿等)转化为结构化的 `REQUIRE.md`,作为后续 DESIGN/PLAN/CODING/CHECK 阶段的共同基线。

## 输入

- 用户输入的需求描述(自然语言,必填)
- 用户放置在 `req/<REQ-NNNNN>/` 目录下的补充材料(可选)
- 项目级规范:`./assistants/rules/` 下所有文件

## 输出

主产出物:`req/<REQ-NNNNN>/REQUIRE.md`
辅助产物:
- `req/<REQ-NNNNN>/LOG.md`(可选,非必要不记录)
- `req/<REQ-NNNNN>/clarifications.md`(有用户确认时生成,追加式记录问答)

## 需求编号分配

### 新建需求

```
function allocateReqNum(versionPath):
  existing = Glob("req/REQ-*/")
  if existing.length == 0:
    return "REQ-00001"
  nums = existing.map(extractNumber).sort()
  return "REQ-" + pad(nums[nums.length - 1] + 1, 5)
```

### 续跑需求

- 用户传入 `REQ-NNNNN` → 直接使用该编号
- 验证 `req/<REQ-NNNNN>/` 目录存在

## 工作流程

### 步骤 1 — 创建目录与 PROCESS.md

1. 检查 `req/<REQ-NNNNN>/` 是否存在
   - 不存在 → `mkdir -p`,创建 `PROCESS.md` 初始行
   - 存在 → 从 PROCESS.md 恢复

2. PROCESS.md 初始行:
```
| YYYY-MM-DD HH:mm | INIT | 开始 | 创建需求 <REQ-NNNNN> |
```

### 步骤 2 — 收集需求材料

1. 若用户输入为自然语言描述 → 直接作为主需求材料
2. `Glob "req/<REQ-NNNNN>/**/*"` 列出补充材料
3. 对每个文件尝试读取:
   - 文本/图片直接读取
   - 不可读格式 → 在 LOG.md 中标注,提示用户提供文字摘要

### 步骤 3 — 提取需求要素

从材料中提取:

| 要素 | 说明 | REQUIRE.md 章节 |
| --- | --- | --- |
| 功能点(FR) | 系统必须做什么 | §3 功能需求 |
| 约束(NFR) | 性能/安全/兼容/合规/可观测性 | §4 非功能需求 |
| 验收标准(AC) | 可验证的完成条件 | §5 验收标准 |
| 用户角色 | 谁会使用 | §1 需求概述 |
| 交互流程 | 用户操作步骤 | §1 需求概述 |
| 边界条件 | 异常路径/边界值 | §1 需求概述 |

### 步骤 4 — 检索关联需求

1. `Glob "req/*/REQUIRE.md"` 列出同版本所有需求
2. `Grep` 关键词做粗筛
3. 对疑似关联的需求,读取其 REQUIRE.md 摘要
4. 在 `REQUIRE.md §6 关联需求` 中记录关联点

### 步骤 5 — 与用户澄清(增强)

> 对模糊、歧义、缺失的功能或交互,分两轮向用户确认。

#### 5a. 需求细节澄清

对模糊、歧义、缺失的功能点或交互,向用户确认:

- 能给出 2-4 个具体选项 → 使用 `AskUserQuestion`
- 需要开放式描述 → 自然语言提问
- 每轮最多 1-3 个最阻塞的问题
- 追加 `clarifications.md` 记录问答(格式见下方)

**clarifications.md 字段格式**:
```
| 时间 | 问题 | 选项/上下文 | 用户回答 | 影响的 REQUIRE.md 章节 |
| --- | --- | --- | --- | --- |
| 2026-06-30 22:00 | 登录失败是否需要提示具体原因? | A.统一"登录失败" B.区分"密码错/账号不存在/已锁定" | B,区分 | §3 FR-1 错误提示 |
```

> **不要一次问完所有问题**。每轮聚焦 1-3 个最阻塞的点,得到答复后先更新对应章节,再迭代出下一批问题。

#### 5a-1. 技术选型过滤(在弹出 AskUserQuestion 前执行)

> 在生成 AskUserQuestion 候选选项后,先按关键词集过滤技术选型类问题,延迟到 DESIGN 阶段。

**关键词集**(20 个):
```
{技术选型, 实现方式, 框架, 库, 工具, monorepo, single-package, pnpm, npm, yarn, Jest, Vitest, Mocha, React, Vue, Angular, 数据库, ORM, 消息队列, 缓存, Redis, MongoDB}
```

**过滤规则**:
- 命中任一关键词的候选选项 → **不**弹出 AskUserQuestion
- 改为追加到 `clarifications.md` 的 `## 延迟到 DESIGN 阶段(技术选型类)` 区段
- 每条含:`时间` / `原始问题` / `命中的关键词` / `延迟到=DESIGN` 字段
- 未命中关键词的候选选项 → 照常弹出 AskUserQuestion

> **不过滤技术选型类问题**:本阶段确定功能点,技术选型归 DESIGN 阶段。

#### 5b. 边界条件确认

对以下边界条件,向用户确认:

- **异常路径**:当用户输入异常/系统错误时,预期行为是什么?
- **边界值**:数据范围/数量/时间的边界值如何处理?
- **用户角色与权限**:不同角色的权限边界在哪里?
- **与现有功能交互**:新功能与现有功能的交互方式?

每轮确认后追加 `clarifications.md`。

#### 5c. 需求冲突确认(强制)

> 当提取的 FR 之间存在潜在冲突时,本步骤**不可跳过**。必须触发 `AskUserQuestion` 确认关系,不可仅标注"设计推断"延迟到 DESIGN 阶段。

**冲突判定标准**(满足任一即触发):

1. **参数互斥**:两个命令行参数/配置项语义重叠或互斥(如 `--confirm` 与 `--auto`)
2. **行为矛盾**:两个 FR 描述的行为在边界条件下矛盾
3. **接口冲突**:两个 FR 涉及同一接口/文件的互斥修改方向

**处理流程**:

1. 检测到冲突 → 明确列出冲突的 FR 编号和冲突点
2. 给出 2-4 个具体选项 → 使用 `AskUserQuestion`
3. 用户确认后 → 更新 FR 描述,标注"用户确认"(来源:clarifications.md)
4. 不可将冲突类 FR 标注为"设计推断"来源

**提问示例**:
```
需求冲突确认:FR-1 与 FR-6 存在潜在冲突
FR-1: 新增 --confirm 参数(强制确认)
FR-6: --confirm 与 --auto 互斥(设计推断)

这两个参数的关系需要明确:
A. 互斥,同时传入报错退出(推荐)
B. --confirm 优先,忽略 --auto
C. 两者共存,正交控制不同维度

请选择或说明期望行为。
```

> **注意**:需求冲突确认**不**受技术选型过滤影响——即使冲突涉及 `{技术选型, 实现方式}` 等关键词,仍必须在本阶段确认,**不可**延迟到 DESIGN 阶段。

### 步骤 6 — 撰写 REQUIRE.md

按 `templates/REQUIRE.md` 结构生成:

```
# 需求分析 — <REQ-NNNNN> · <标题>

## 0. 用户原始输入
## 1. 需求概述
## 2. 背景与目标
## 3. 功能需求(FR)
## 4. 非功能需求(NFR)
## 5. 验收标准(AC)
## 6. 关联需求
## 7. 变更记录
```

### 撰写原则

- **先保留原文**:撰写 §0 时,将用户输入的需求描述一字不差地放入引用块(`>`),不可改写或概括;若用户分多次补充,每段用单独的引用块
- **详尽**:每一条需求都展开为可独立验证的描述
- **不臆造**:无材料佐证的细节写进"待澄清"或明确标注"假设"
- **可追溯**:每条 FR/AC 标注来源材料
- **不涉及技术选型**:本阶段只确定功能点,技术选型/架构风格/框架选择归 DESIGN 阶段
- **待澄清标注**(新增):对用户确认的项标注"已确认(来源:clarifications.md)",对假设的项标注"假设(待确认)"

### 步骤 7 — 同步版本看板

在 `RESULT.md` 需求清单追加:

```
| REQ-NNNNN | <标题> | [PROCESS.md](req/REQ-NNNNN/PROCESS.md) |
```

## 技术选型过滤

> 本阶段不涉及技术选型。以下关键词命中的内容**不写入** REQUIRE.md,留待 DESIGN 阶段处理:
> `{技术选型, 实现方式, 框架, 库, 工具, 数据库, ORM, 消息队列, 缓存, 架构风格, 部署形态}`

## clarifications.md 记录机制(新增)

> 追加式记录,每次用户确认后追加一行。

**格式**:
```
| 时间 | 问题 | 回答 | 来源 |
| --- | --- | --- | --- |
| 2026-06-30 19:00 | 用户登录失败后锁定策略? | 5次失败锁定30分钟 | 用户确认 |
| 2026-06-30 19:05 | 订单超时时间? | 30分钟未支付自动取消 | 假设(待确认) |
```

**生成判定**:
- 本轮有用户问答 → 生成 clarifications.md
- 本轮无用户问答 → 不生成

## 非 --auto 模式确认

阶段完成后弹出确认:
```
需求分析完成: <N> FR / <M> NFR / <K> AC / <C> 待澄清
选项:
A. 继续 DESIGN 阶段(推荐)
B. 暂停
C. 取消
```

## 确认环节约束(新增)

- 每轮最多 1-3 个最阻塞的问题,不一次问完所有问题
- 非 `--auto` 模式触发 `AskUserQuestion`
- `--auto` 模式自动选推荐项,将未确认项标注为"假设(待确认)"
- 追加 clarifications.md 记录所有问答(含 --auto 模式的自动选择)
