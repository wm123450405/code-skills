# RESULT.md 模板 — 概要设计

> 本文件是 `code-design` 技能产出的 `RESULT.md` 章节结构与填写规范。
> 模型在生成 `RESULT.md` 时应严格按本结构组织,确保后续 `code-plan` 可直接消费。
>
> **版本感知**:本文件存放在 `./assistants/<版本号>/design/<需求编码>/RESULT.md`,文档头应明确记录版本号。

---

## 文档头
```
# <需求编码> — <概要设计标题>

- 需求编码:<REQ-YYYY-NNNN>
- 所属版本:<版本号>(如 v1.0.0)
- 上游:./assistants/<版本号>/require/<需求编码>/RESULT.md (v<n>)
- 遵循规范:./assistants/rules/<规范文件1>, <规范文件2>, ...(共 N 个)
- 状态:草稿 / 已评审 / 已锁定
- 责任人:<设计者>
- 创建:<YYYY-MM-DD>
- 最近更新:<YYYY-MM-DD HH:mm>
- 当前版本:v<n>
```

## 1. 设计概述
用 2-5 句话概括:**本设计要解决什么问题,核心架构思路是什么,新增/复用/修改的范围**。电梯演讲版本。

## 2. 需求回顾
引用上游 `RESULT.md`,**不要全文复制**,只摘取关键约束:
- 关键 FR 列表(对应到本设计的哪些模块)
- 关键 NFR(性能/可用性/安全等目标值)
- 关键 AC(影响设计走向的验收点)
- 需求中的"待澄清"项(可能影响设计)

## 2.5 规范遵循
本节是规范约束的**总账**,后面每节涉及规范时只引用本节条目,不再重复描述。

### 2.5.1 适用的规范文件
| 规范文件 | 类别 | 关键约束 | 本设计对应章节 |
| --- | --- | --- | --- |
| ./assistants/rules/architecture.md | 功能架构 | ... | §5, §6, §7 |
| ./assistants/rules/module-conventions.md | 模块规划 | ... | §7 |
| ./assistants/rules/api-standards.md | 接口定义 | ... | §8 |
| ./assistants/rules/data-modeling.md | 数据结构 | ... | §9 |
| ... | ... | ... | ... |

### 2.5.2 规范自检结论
- **完全合规**的章节:§5, §7, §8, ...
- **经用户授权偏离**的章节:见 §2.5.3
- **待澄清冲突**:见 §2.5.4

### 2.5.3 用户授权的偏离
```
- 章节:§X
- 偏离内容:<具体偏离>
- 依据规范:./assistants/rules/<文件> §<章节>(要求 Y)
- 偏离理由:<技术必要 / 业务必要 / 特殊场景>
- 授权时间:YYYY-MM-DD HH:mm

```

### 2.5.4 待澄清的规范冲突
```
- 冲突:规范要求 X,需求 Y 要求 Z
- 阻塞章节:§X
- 已提供选项 / 用户选择 / 状态
```

## 3. 设计目标与非目标
- **目标**:本设计要满足的架构/质量目标(可衡量)
- **非目标**:本设计不解决的问题(与需求的 Out of Scope 对齐)

## 4. 约束清单
- **硬约束**(不可违反):语言/平台/合规/已有接口/部署环境等
- **软约束**(可权衡):性能预算/团队熟悉度/运维成本/上线时间

## 5. 架构总览
**必须**用图描述,文字说明无法替代:
- **组件图**(Mermaid / ASCII):展示本设计涉及的所有组件及其关系
- **数据流图**:关键请求从进入到返回的流转路径
- **控制流图**:关键状态/分支的流向
- **部署视角**(如适用):运行时各组件如何部署
