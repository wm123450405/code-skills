# assistants/qanda/ — 项目级 Q&A 长期沉淀目录

> 本目录用于**长期沉淀**与发布部署相关的常见问题与解答(Q&A),**跨所有版本共享**(项目级,不随版本归档)。
>
> 本目录中的内容会被 `code-publish` 技能在发布时**自动聚合**到对应版本的 `publish/Q&A.md`,供运维 / 现场支持在发布后查阅。
>
> ⚠ 本目录**当前由人工手动维护**(REO-00006 §。

---

## 用途

本目录是发布部署 Q&A 的**长期沉淀池**。典型内容示例:

- **部署阶段常见问题**:服务启动失败、端口冲突、配置错误
- **升级阶段常见问题**:DB 升级失败、配置冲突、回滚决策
- **初始化阶段常见问题**:DB 建表报错、初始数据缺失
- **首次进入系统常见问题**:默认账号、初始密码(已修改)、首次操作建议
- **安全 / 性能常见问题**:证书配置、负载均衡、调优

不限于以上;**任何与"发布部署"相关的 Q&A** 都可沉淀。

## 文件命名建议

**格式**:`<主题>.md`,kebab-case(小写 + `-` 连接),文件后缀固定 `.md`。

**具体示例**(供参考,**不**强制):

| 文件名 | 适用主题 |
| --- | --- |
| `deploy-faq.md` | 全新部署常见问题 |
| `db-init-faq.md` | 数据库初始化常见问题 |
| `upgrade-faq.md` | 升级部署常见问题 |
| `rollback-faq.md` | 回滚操作常见问题 |
| `first-login-faq.md` | 首次进入系统常见问题 |
| `performance-faq.md` | 性能调优常见问题 |
| `security-faq.md` | 安全配置常见问题 |

**命名约定**:

- **kebab-case**:全小写 + `-` 分隔(与 `code-publish/templates/` 风格一致)
- **主题词简洁**:1~3 个单词;过长文件名不便于阅读
- **加 `-faq` 后缀**:与"用户文档 / 操作手册"区分(本目录只放 Q&A)
- **避免特殊字符**:不用中文文件名(部分 Markdown 渲染器兼容差);不用空格(用 `-`)

## 引用规范

### 被 `code-publish` 聚合

每次调用 `/code-publish` 时,本目录中**除 `README.md` 外**的所有 `.md` 文件会按以下规则被聚合到 `<版本号>/publish/Q&A.md`:

- **排除**:`README.md`(本文件,目录说明)
- **排序**:按文件名字典序(确保幂等)
- **每个文件**占 1 个 `## <主题>(来源:qanda/<文件>)` 章节
- **章节顺序**:N 个聚合章节 + 末尾占位章节(兜底)

### 文件内容格式建议

每个 Q&A .md 文件建议按以下结构组织:

```markdown
# <主题>常见问题

> 本文件由 assistants/qanda/ 沉淀;被 code-publish 聚合到 <版本号>/publish/Q&A.md。

## Q: <问题 1>
A: <答案 1>

## Q: <问题 2>
A: <答案 2>

## Q: <问题 3>
A: <答案 3>
```

**要点**:

- **H1 标题**:与文件名呼应(如 `deploy-faq.md` → `# 部署常见问题`)
- **`## Q: ...` / `A: ...`**:Q&A 形式,易扫描
- **不写"如何贡献"等元说明**:本 README 已说明,Q&A.md 只放问答本身
- **不写"敏感信息"**:默认账号 / 密码等放入 `publish/Q&A.md` 之前应**先**在 Q&A 中脱敏(实际不脱敏,但需在 DEPLOY.md 单独提示"首次登录立即修改")

### 与 `publish/Q&A.md` 的区别

| 维度 | 本目录(`qanda/*.md`) | `publish/Q&A.md`(版本产物) |
| --- | --- | --- |
| 生命周期 | 长期沉淀(跨版本) | 本版本发布时生成 |
| 维护方式 | 人工手动 | 自动聚合(由 `code-publish`) |
| 内容来源 | 项目维护者 | 本目录的"快照" |
| 文件类型 | 单一文件 / 长期维护 | 单一文件 / 每次发布重生成 |
| 受众 | 维护者(向本目录添加内容) | 运维 / 现场支持(发布后查阅) |

## 维护方式

### 当前(v1)— 人工手动整理

**步骤 1**:决定主题
- 项目维护者决定"需要沉淀哪类 Q&A"
- 主题应**有真实需求**:发布部署中**确实会遇到**的问题

**步骤 2**:创建文件
- 在本目录下创建 `<主题>.md`(命名遵循上文"文件命名建议")
- 写入 Q&A 内容(遵循上文"文件内容格式建议")

**步骤 3**:重跑 `code-publish`
- 调 `/code-publish` 或 `/code-publish <版本号>`
- `code-publish` 自动读取本目录的 .md 文件,聚合到 `publish/Q&A.md`
- 在终端报告中查看"聚合自 qanda/ 的 K 个文件"

**步骤 4**:人工审阅
- `code-publish` **不**自动 commit(留 dirty tree)
- 人工审阅 `publish/Q&A.md` 聚合结果
- 确认无误后 `git add` + `git commit`

### 未来(v2)— 可能由独立技能管理

未来可能由独立技能(如 `code-qanda`)管理本目录,提供:
- 自动扫描"git log / 缺陷清单"提取常见问题
- 模板化生成 Q&A 骨架(用户填答案)
- 与 `code-fix` 联动(已修复的缺陷自动沉淀为 Q&A)

**但 v1 不实现**(本需求 v1 不创建 `code-qanda` 技能)。

### 不做的事

- **不要**在本目录放**非 Q&A** 内容(如产品介绍 / 业务说明)— 那应该放在 `docs/` 或 `README.md` 之外的项目文档
- **不要**用本目录作"草稿区" — 草稿应该放本仓库内的 `drafts/` 子目录(本仓库暂无此约定)
- **不要**在本目录放**版本专属**内容(如"V0.0.2 修复的 bug")— 那应该用 git commit message 记录,版本专属 Q&A 写在 `release-notes.md` 中
