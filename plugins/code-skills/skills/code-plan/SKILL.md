# code-plan — 详细设计 & 编码计划(版本感知)

## 目标
把概要设计"系统长什么样"落地为**两件事**:
1. **详细设计**:`RESULT.md` —— 给出可直接编码的算法逻辑、伪代码、完整数据结构变化、接口入参/出参/异常/安全细节
2. **编码计划**:`PLAN.md` —— 把详细设计拆分为可独立执行、可追踪状态的任务,每条任务有唯一编号 `<需求编号>-<任务序号>`

## 适用场景
- 跨多文件/多模块的改动
- 需要多人协作或分阶段交付的任务
- 任何在动手编码前希望降低返工成本的场景
- 开发过程中需要根据进展更新计划与状态

## 不适用
- 当前**没有激活的版本工作空间**(请先调 `code-version`)
- 需求或概要设计尚未就绪
- 已知明确的、只涉及单一文件/单一函数的修改
- 紧急线上修复(走 hotfix 流程)

## 工作目录约定(强制)

**版本工作空间**:`./assistants/<版本号>/`(由 `./assistants/.current-version` 决定)。
本技能的目录粒度是**需求**:
```
./assistants/
├── rules/                                  # 项目级规范,跨版本共享,只读
└── <版本号>/
    ├── RESULT.md                           # 版本看板(本技能追加区段)
    ├── require/<需求编码>/                 # 上游需求,只读
    │   └── RESULT.md
    ├── design/<需求编码>/                  # 上游概要设计,只读
    │   └── RESULT.md
    └── plan/<需求编码>/                    # 本技能产出,可写
        ├── RESULT.md                       # 详细设计
        ├── PLAN.md                         # 编码计划(任务列表 + 状态)
        ├── materials-index.md
        ├── design-notes.md
        ├── module-details.md
        ├── interface-specs.md
        ├── data-changes.md
        ├── risk-analysis.md
        ├── rule-compliance.md
        └── clarifications.md
```

- 路径以**当前工作目录(CWD)**为基准
- **本技能不修改** `rules/`、`<版本号>/require/<编号>/`、`<版本号>/design/<编号>/` 下任何内容,只读
- `RESULT.md` 与 `PLAN.md` **必须**保持一致:PLAN.md 的每条任务都应能追溯到 RESULT.md 中的某个详细设计点

## 输入
- **需求编码**(必填)
- **上游需求**:`./assistants/<版本号>/require/<需求编码>/RESULT.md`(必须存在)
- **上游概要设计**:`./assistants/<版本号>/design/<需求编码>/RESULT.md`(必须存在)
- **项目级规范**:`./assistants/rules/` 下所有文件(建议存在,缺失时本技能会询问)
- **当前项目代码**:CWD 下的源文件(用于核对实现细节、命名、依赖方向)

## 输出
主产出物(同目录下):
- `RESULT.md` —— 详细设计
- `PLAN.md` —— 编码计划

辅助过程文档(同目录下):
- `materials-index.md` / `design-notes.md` / `module-details.md` / `interface-specs.md` / `data-changes.md` / `risk-analysis.md` / `rule-compliance.md` / `clarifications.md`

**版本看板同步**:本技能在以下节点同步 `./assistants/<版本号>/RESULT.md`:
- 完成/更新详细设计与计划 → "详细设计与任务计划汇总"追加或更新
- 拆分新任务(首次) → "任务清单"批量登记
- 追加"修改类"任务 → "任务清单"追加新行,关联任务字段
- 设立/调整里程碑 → "里程碑"区段同步
- 任意阶段 → "变更记录"追加条目

## 工具使用约定
- 读激活版本:`Read "./assistants/.current-version"`
- 读上游:`Read "./assistants/<版本号>/require/<需求编码>/RESULT.md"` 与 `design/...`
- 读规范:`Glob "./assistants/rules/**/*"` 列出,逐一 `Read`
- 探项目代码:`Glob` / `Grep` / `Read`(核对命名、既有函数签名、错误处理风格)
- 写文档:`Write`(首次)/ `Edit`(增量)
- 与用户澄清:优先 `AskUserQuestion`;自然语言兜底

## 工作流程

### 步骤 0 — 版本上下文检测(强制前置)
1. 读取 `./assistants/.current-version`
2. **文件不存在** → 立即停下,告知用户先调 `code-version`
3. 读取内容,记为 `<版本号>`,后续所有路径用 `assistants/<版本号>/...`
4. 验证 `./assistants/<版本号>/require/<需求编码>/RESULT.md` 与 `design/.../RESULT.md` 都存在(否则提示用户先跑上游)

### 步骤 1 — 收集输入 ID 并判定路径

1. 若用户未提供,主动询问
2. **判定输入类型**:
   - 匹配 `^REQ-\d{4}-\d{4}$` 或用户约定的需求编码格式 → **需求分支**(走步骤 2-18A)
   - 匹配 `^BUG-\d{3}$` → **缺陷分支**(走步骤 19-28)
3. **需求分支校验两份上游文件都存在**:
   - `./assistants/<版本号>/require/<需求编码>/RESULT.md` — 不存在 → 提示先跑 `code-require`
   - `./assistants/<版本号>/design/<需求编码>/RESULT.md` — 不存在 → 提示先跑 `code-design`
4. **缺陷分支校验上游文件存在**:
   - `./assistants/<版本号>/fix/<缺陷编号>/RESULT.md` — 不存在 → 提示先跑 `code-fix` 登记缺陷

### 步骤 2 — 定位 / 创建工作目录
- 检查 `./assistants/<版本号>/plan/<需求编码>/` 是否存在
- 不存在 → `mkdir -p` 创建
- 存在 → 直接进入步骤 3

### 步骤 3 — 读取项目级规范
与 `code-design` 步骤 3 同构:
1. `Glob "./assistants/rules/**/*"`
2. 不存在/为空 → 用 `AskUserQuestion` 询问是否继续
3. 存在 → 逐个 `Read`,归类到功能架构/模块规划/接口定义/数据结构/安全/性能/测试/可观测性等
4. 摘要写入 `materials-index.md` 的"项目级规范"部分

### 步骤 4 — 读取上游需求与概要设计
1. `Read "./assistants/<版本号>/require/<需求编码>/RESULT.md"` —— 提取 FR / NFR / AC / 待澄清
2. `Read "./assistants/<版本号>/design/<需求编码>/RESULT.md"` —— 提取模块拆分 / 接口概要 / 数据结构概要 / 规范遵循
3. 两份文档的关键交叉点(每条 FR 对应的设计章节)写入 `materials-index.md`
4. **预检规范 vs 设计冲突**:
   - 概要设计中的技术选型/接口风格/数据模型与规范对比
   - 概要设计中的"用户授权的偏离"与本详细设计是否仍继承
   - 发现的冲突记入 `rule-compliance.md` 的"待澄清冲突"区,稍后步骤 8A 一并询问

### 步骤 5 — 探索项目代码(核对实现细节)
重点不是"探索架构",而是"为详细设计提供实现约束":
- 目标语言/框架的具体规范(类型系统、错误模型、并发原语)
- 既有相似功能的实现风格(命名、错误处理、日志格式)
- 既有测试用例的风格与覆盖度
- 既有工具函数(时间、加密、校验)避免重复造轮子

写入 `materials-index.md` 的"项目现状(实现细节)"部分。

### 步骤 6 — 检查 RESULT.md / PLAN.md 是否存在
四种情形:
- **都不存在** → 首次设计(步骤 7A–18A)
- **RESULT.md 存在,PLAN.md 不存在** → 详细设计已就绪,补做编码计划(步骤 7C–11C)
- **PLAN.md 存在,RESULT.md 不存在** → 计划已就绪,补做详细设计(不常见;步骤 7D)
- **都存在** → 增量更新(步骤 7B–13B)

### 步骤 7A — 首次:详细化设计
按概要设计逐模块展开,产出可直接编码的细节。过程文档分工:
- `module-details.md`:每个模块的内部结构、关键类/函数、调用顺序
- `interface-specs.md`:每个接口的完整入参/出参 schema、错误码、示例、版本/兼容策略
- `data-changes.md`:每个数据结构的字段、类型、约束、索引、迁移脚本
- `risk-analysis.md`:异常路径、安全边界、性能热点、资源限制、回退策略

每条结论标注"依据规范"和"对应概要设计章节"。

### 步骤 8A — 澄清冲突与不确定项
处理两类待澄清:

1. **规范 vs 设计冲突**(从步骤 3/4 收集):
   - 用 `AskUserQuestion` 给出选项
   - 结论写入 `rule-compliance.md` 的"冲突解决记录"
2. **实现细节不确定项**:
   - 关键算法的具体选型(如加密算法、缓存淘汰策略)
   - 错误重试/退避策略
   - 锁/事务的边界
   - 第三方库版本锁定粒度
   - 灰色地带的接口行为

**每轮 1-3 个最阻塞点**;每轮问答追加到 `clarifications.md`,字段:
- 时间 / 问题 / 选项 / 用户回答 / 影响的章节(RESULT.md 与 PLAN.md)

### 步骤 9A — 测试策略推导
为每条详细设计点推导测试要点:
- 单元测试范围
- 集成测试范围
- 端到端测试范围
- 性能/安全测试(如适用)

写入 `risk-analysis.md` 的"测试要点"区。

### 步骤 10A — 任务拆分
**任务拆分原则**:
- 最小可执行单元:每条任务 0.5–2 天内可完成
- 单一职责:每条任务只做一件事
- 可独立验证:每条任务有清晰的"完成定义"
- 顺序友好:从底层依赖/数据结构 → 核心逻辑 → 集成层 → 测试 → 文档

**任务类型**:
- `新增`:全新代码/接口
- `修改`:对已有代码/接口的扩展或行为变更
- `重构`:不改外部行为的内部结构调整
- `修复`:针对已知 bug 的修复
- `测试`:独立提交的测试补齐
- `文档`:独立提交的文档更新

**任务编号**:
- 格式:`<需求编号>-<任务序号>`,如 `REQ-2026-0001-001`、`REQ-2026-0001-002`
- 任务序号自 001 起递增,三位补零
- 首次拆分时,按"应被编码的先后顺序"分配序号
- 任务类型作为前缀标签写在标题里(如 `[新增] 用户登录接口`),不参与编号

**任务双状态字段(每个任务)**:开发状态 与 测试状态
- **开发状态**(必填,初值 = `待开始`):
  - `待开始` / `进行中` / `已完成` / `已取消` / `阻塞` / `待重新评估`
  - 含义:代码本身是否被实现
- **测试状态**(必填,初值 = `未编写`):
  - `未编写` / `已编写` / `已运行-通过` / `已运行-失败` / `不适用` / `阻塞`
  - 含义:单元测试是否被编写/运行/通过

**任务双状态的初始化**:
- **代码类任务**(类型 = 新增/修改/重构/修复/测试):开发状态=`待开始`,测试状态=`未编写`
- **纯文档任务**(类型 = 文档):开发状态=`待开始`,测试状态=`不适用`(在"不适用理由"中注明"纯文档任务")
- 若任务范围明确不含可测代码(如纯配置变更),可在 `code-plan` 或 `code-it` 中讨论后设为 `不适用`

**双状态语义**:**任务"真正可发布" = 开发状态=已完成 ∧ 测试状态∈{已运行-通过, 不适用}**。只有两轴同时满足,任务才算真正完成。

**任务触发/来源字段(每个任务)**:回答"为什么创建本任务"
- 值:`需求新增` / `需求变更` / `需求撤回` / `设计变更` / `规范变更` / `代码变更` / `主动优化` / **`审查改修`** / `缺陷修复` / `性能优化` / `安全加固` / `测试补齐` / `其他`
- **关键差异**:`code-it` 执行时,根据 `触发/来源` 决定从哪里读输入
  - 大多数情况 → `./assistants/<版本号>/plan/<需求编号>/RESULT.md`(详细设计)
  - **`触发/来源=审查改修`** → `./assistants/<版本号>/review/<任务编码>/RESULT.md`(改修要求) ★
  - `触发/来源=需求撤回` → 任务标"已取消",无输入
- **填写规则**:
  - `code-plan` 首次拆分时:为每条新任务标注"是什么触发的"
  - `code-review` 派生新任务时:必须填 `审查改修`,并在"关联任务"指向被修正的原任务
  - 增量更新时:可改,但要写明理由(写到变更记录)

### 步骤 11A — 任务依赖与里程碑
- 明确每条任务的前置任务
- 识别里程碑(可演示/可发布的最小集合)
- 用 Mermaid 绘制任务依赖图

### 步骤 12A — 检索关联编码计划
1. `Glob "./assistants/<版本号>/plan/*/PLAN.md"` 列出同版本其他计划
2. (可选,跨版本) `Glob "./assistants/*/plan/*/PLAN.md"`
3. 重点关注共享模块/接口/数据迁移的计划
4. 在 `materials-index.md` 中记录关联点(含所在版本)

### 步骤 13A — 与用户对齐计划(可选但推荐)
编码计划是开发协作的基线,建议在落地前与用户对齐:
- 任务粒度是否合适
- 顺序是否符合团队节奏
- 里程碑划分是否合理
- 责任人分配(若有)

### 步骤 14A — 撰写 RESULT.md
按 `templates/plan.md` 的章节结构生成,覆盖:
1. 概述(本详细设计的目标、范围、与概要设计的关系)
2. 上游引用(需求/设计/规范)
3. 模块详细化(每个模块的关键类/函数/调用顺序)
4. 算法与逻辑(伪代码 / 流程图)
5. 数据结构完整变更(字段、类型、约束、索引、迁移)
6. 接口细节(完整入参/出参 schema、错误码、示例、版本策略)
7. 异常处理
8. 安全要求
9. 状态机/流程
10. 性能与资源
11. 测试要点
12. 规范遵循
13. 待澄清/未决项
14. 变更记录

### 步骤 15A — 撰写 PLAN.md
按 `templates/task-plan.md` 的结构生成,覆盖:
1. 任务总览表格
2. 任务详情(每条:目标、涉及文件、关键变更、边界与异常、验证手段、回退方式)
3. 任务依赖图
4. 里程碑
5. 变更记录

**首版所有任务的状态统一为"待开始"**。

### 步骤 16A — 同步版本看板(强制)
1. `Read "./assistants/<版本号>/RESULT.md"`,定位"详细设计与任务计划汇总"区段
2. **追加本计划**:
   - 需求编码 / 标题 / 状态=`已完成` / 创建时间 / 任务总数(从 PLAN.md 数出) / 链接
3. **追加"任务清单"区段**(本计划的所有任务):
   - 每条任务一行,字段对齐模板(任务编号 / 需求 / 类型 / 触发/来源 / 标题 / 开发状态 / 测试状态 / 涉及文件)
   - 开发状态统一为 `待开始`,测试状态为 `未编写`(或文档任务 `不适用`)
   - 涉及文件留空(由 `code-it` 完成时填入)
4. **追加"里程碑"区段**(从 PLAN.md 复制)
5. 在"变更记录"追加:
   ```
   YYYY-MM-DD HH:mm  设计新增  <需求编码> 详细设计与编码计划完成(共 N 个任务)  <需求编码>
   ```
6. 更新看板文档头"最近更新"

### 步骤 17A — 交叉验证
- 校验 `RESULT.md` 中每个详细设计点都能在 `PLAN.md` 找到对应任务
- 校验 `PLAN.md` 中每条任务都能在 `RESULT.md` 找到设计依据
- 不一致的地方要么补 `RESULT.md` 的设计,要么调整 `PLAN.md` 的任务

### 步骤 18A — 完善过程文档与汇报
- 收尾所有过程文档
- 向用户汇报:任务总数、关键里程碑、需关注的规范冲突、新增依赖、**版本看板的同步情况**

### 步骤 7B — 增量更新:识别变更源
四类变更源都要识别:
1. **需求侧变更** —— 重读 `<版本号>/require/.../RESULT.md`,对比"变更记录"
2. **概要设计变更** —— 重读 `<版本号>/design/.../RESULT.md`,对比"变更记录"
3. **规范侧变更** —— 重 Glob + Read `./assistants/rules/`,与 `materials-index.md` 对比
4. **代码侧变更** —— 重跑步骤 5 探索项目代码,与 `materials-index.md` 对比

变更源差异表写入 `materials-index.md` 的"本次变更源"部分。

### 步骤 8B — 增量更新 RESULT.md
对每一项变更:
- **需求新增** → 评估对详细设计的影响;在对应模块章节追加
- **需求修改** → 修订对应章节;被影响的接口/算法标注 `(修订于 YYYY-MM-DD)`
- **需求删除** → 评估相关详细设计点;移除或降级为"待澄清"
- **概要设计变化** → 同步详细设计(模块拆分变了 → 详细化跟着变;接口概要变了 → 接口细节跟着变)
- **规范变化** → 重新自检;新的强制约束要在详细设计中体现
- **代码变化(资产变了)** → 更新"复用既有"引用;若既有被破坏可能回退到 7A

更新 `RESULT.md` 时**不重写稳定章节**,只在必要位置编辑。最后在"变更记录"追加本次更新摘要。

### 步骤 9B — 增量更新 PLAN.md(任务管理核心)
**对未完成任务(状态 = 待开始 / 进行中 / 阻塞)**:可修改调整
- 任务标题、涉及文件、关键变更、依赖、估算、描述都可改
- 开发状态保持不变(除非用户明确要改)
- 测试状态保持不变(除非用户明确要改,如发现前期判断"不适用"有误)
- 不重新分配任务编号

**对已完成任务(状态 = 已完成)**:**开发相关字段不能修改**
- 任务描述、关键变更、依赖、估算等"开发相关字段"不可改
- 只能新增"修改类"任务来覆盖/重做
- 新任务的"关联任务"字段指向被它取代/扩展的已完成任务
- 新任务描述中明确"基于 T-XXX 的成果,因...原因需要..."
- **测试状态是例外**:即使开发已完成,测试状态仍可独立更新
  - 已完成任务的测试状态 `未编写`/`已编写` → `已运行-通过` 是正常推进
  - 已完成任务的测试状态 `已运行-通过` → `已运行-失败` 意味着回归,需记录在变更记录
  - 这些更新**只动测试状态字段**,不重新打开开发描述

**对已取消任务(状态 = 已取消)**:**不能修改**
- 已取消任务作为历史保留,新增任务不应再依赖它们

**任务编号分配**:
- 找到 `PLAN.md` 中当前最大的任务序号 N
- 新任务从 N+1 开始递增
- 不复用旧编号
- 不打乱旧任务顺序

**任务状态更新**:
- 用户在重跑技能时可主动告知"哪些任务开发状态/测试状态有变化"
- 技能把这些状态变更写回 `PLAN.md`
- 状态变更在 `PLAN.md` 的"变更记录"中记录
  - 仅开发状态变化 → 变更类型 = `开发状态更新`,版本号小版本递增
  - 仅测试状态变化 → 变更类型 = `测试状态更新`,版本号小版本递增
  - 两者同时变化 → 记录为两条;版本号小版本递增一次

### 步骤 10B — 重新检查任务依赖
- 新增任务可能引入新的依赖关系
- 被取消或已完成的任务不再作为后续任务的前置
- 任务依赖图需要重画

### 步骤 11B — 重新计算里程碑
- 里程碑可能因任务增删而变化
- 同步更新里程碑列表

### 步骤 12B — 交叉验证
同 17A:RESULT.md 与 PLAN.md 保持一致。

### 步骤 13B — 同步版本看板与补充过程文档
1. **更新"详细设计与任务计划汇总"**:本计划的状态/任务数刷新
2. **同步"任务清单"**:
   - **追加** 新拆分/派生任务(给完整一行)
   - **更新** 已存在任务的状态字段(开发状态 / 测试状态)—— 仅当本步骤用户明确告知/PLAN.md 显式变更
   - **不** 改动"任务清单"中其他需求的任务
3. **同步"里程碑"**:本计划相关的里程碑增删改
4. 在"变更记录"追加:
   ```

   ```
5. 更新 `materials-index.md` / `rule-compliance.md` / `clarifications.md`
6. 汇报:

## 缺陷分支(从步骤 1.2 判定为 BUG-NNN 时走)

> **适用场景**:用户已通过 `code-fix` 登记了一个 `BUG-NNN`,现在要为该缺陷制定修复方案。
> **关键简化**:
> - 缺陷已有明确的"问题描述"(`fix/<BUG-NNN>/RESULT.md`),无需再做需求分析
> - 修复方案的规模通常远小于一个完整需求,故 `RESULT.md` 简化为 `fix-plan.md`(单文件)
> - 不需要拆分为多个 `PLAN.md` 任务;如果修复确实需要多步,可在 `fix-plan.md` 中以"步骤"形式列出
> - 任务编号可省略;若一定要用任务系统跟踪,可注册 1 个 `BUG-NNN-001` 任务到 `plan/BUG-NNN-001/PLAN.md`,但更常见的做法是 `code-it BUG-NNN` 直接读 `fix-plan.md` 实施

### 步骤 19 — 定位 / 创建工作目录
1. 检查 `./assistants/<版本号>/fix/<缺陷编号>/` 是否存在
2. 不存在 → `Bash: mkdir -p` 创建
3. `Glob "./assistants/<版本号>/fix/<缺陷编号>/*"` 列出该缺陷目录下现有文件

### 步骤 20 — 读取缺陷上下文
1. `Read "./assistants/<版本号>/fix/<缺陷编号>/RESULT.md"` —— 提取:
   - 缺陷描述(用户原始报告)
   - 复现步骤、期望/实际行为
   - 根因分析(若有,来自 `investigation.md`)
   - 当前状态(应处于"报告"或"调查中")
2. 若存在 `investigation.md`,一并 `Read` —— 提取已确认/已排除的根因假设
3. 把"已知信息"和"待分析项"摘录到本次会话的内部状态

### 步骤 21 — 读取项目级规范(同 REQ 步骤 3)
- 与 REQ 步骤 3 同构,不再重复
- 摘要写入 `fix-plan.md` 起始区(若有需要,可创建临时 `materials-index.md` 跟踪)

### 步骤 22 — 探索项目代码(同 REQ 步骤 5)
- 重点不是"探索架构",而是"为修复方案提供实现约束"
- 特别关注:
  - 缺陷描述中提到的具体文件/函数
  - 既有相似 bug 的修复历史(`git log` / `git blame` 找类似 commit)
  - 既有测试用例(`tests/`)如何组织
- 写入 `fix-plan.md` 的"项目现状"区

### 步骤 23 — 检查 fix-plan.md 是否存在
- **不存在** → 走"**首次规划**"分支(步骤 24A)
- **存在** → 走"**增量更新**"分支(步骤 24B)
  - 读 `fix-plan.md` 现有内容
  - 了解已规划的方案,识别本轮需要调整的地方

### 步骤 24A — 首次规划:根因定位 + 修复方案
**任务**:
1. **根因定位**:
   - 综合缺陷描述、复现步骤、项目代码
   - 明确"哪一行/哪个函数/哪个数据导致 bug"
   - 必要时主动 `Read` 关键代码
2. **修复方案设计**:
   - 至少 2 个候选方案(若适用)
   - 选定方案 + 理由
   - 涉及文件/接口/数据结构变更清单
   - 关键决策旁标注"依据规范:... §X"(如适用)
3. **测试方案**:
   - 如何验证 bug 已修复(回归用例)
   - 如何确保不引入新 bug
4. **风险与回退**:
   - 修复的潜在副作用
   - 出问题时的回退方式
5. **若修复跨多步**,在 `fix-plan.md` 中以"步骤"形式列出每一步(类似任务拆分的简化版):
   ```
   步骤 1:<一句话>
     涉及文件:<文件路径>
     关键变更:<一句话>
   步骤 2:...
   ```

### 步骤 25A — 与用户对齐(可选但推荐)
修复方案可能涉及权衡(性能 vs 简洁、范围 vs 风险),建议在落地前与用户对齐:
- 根因定位是否准确
- 选定方案是否可接受
- 是否有遗漏的边界条件

### 步骤 26A — 撰写 fix-plan.md
按 `templates/fix-plan.md` 的章节结构生成,覆盖:
1. 缺陷摘要(链接到 `fix/<BUG-NNN>/RESULT.md`,只列关键信息:编号/标题/严重度/当前状态)
2. 根因定位(具体到文件:行号)
3. 修复方案(选定 + 候选 + 理由)
4. 涉及文件与变更
5. 测试方案(回归用例 + 验证步骤)
6. 风险与回退
7. 修复步骤(若跨多步,列出顺序)
8. 规范遵循
9. 待澄清/未决项
10. 变更记录(首条)

**注意**:
- `fix-plan.md` 是缺陷修复的**唯一**详细设计文档
- 长度可远小于 `plan/<req-id>/RESULT.md`(因为 bug 通常聚焦)
- 不重写稳定章节(增量更新时尤其重要)

### 步骤 27A — 同步 fix/<BUG-NNN>/RESULT.md 与 fix/RESULT.md(强制)
1. `Read "./assistants/<版本号>/fix/<缺陷编号>/RESULT.md"`

3. `Read "./assistants/<版本号>/fix/RESULT.md"`(总览)
4. 用 `Edit` 更新:
   - 缺陷清单表中本条:状态 → `修复规划中`

   - 变更记录追加一条

### 步骤 28A — 同步版本看板(强制)
1. `Read "./assistants/<版本号>/RESULT.md"`,定位"缺陷清单"区段
2. 更新本条:

3. **规范侧变更**(若有):

### 步骤 25B — 局部更新 fix-plan.md
对每一项变更:
- **缺陷侧变化** → 修订"缺陷摘要"/"根因定位"对应章节

## 衔接
- **下游**:
  - 需求分支:`code-it` 按 `PLAN.md` 的任务执行,逐任务更新状态
  - 缺陷分支:`code-it <BUG-NNN>` 读 `fix-plan.md` 实施修复,产出 `fix-work-log.md` 等
- **下游(次级)**:`code-unit` 与 `code-review` 在 `code-it` 内对单任务展开
- **上游**:
  - 需求分支:`code-version`(必须有激活版本);`code-require` 的 `RESULT.md`、`code-design` 的 `RESULT.md`、项目级规范
  - 缺陷分支:`code-version`(必须有激活版本);`code-fix` 的 `fix/<BUG-NNN>/RESULT.md`、项目级规范
- **横向**:通过 `materials-index.md` 与 `assistants/<版本号>/plan/` 下其他计划形成引用网(可选:跨版本);缺陷分支与同版本其他 `fix/<BUG-NNN>/` 互不干扰

## 不要做的事
- 不要在没有 `./assistants/.current-version` 的情况下继续执行
- 不读上游 `RESULT.md` / `fix-plan.md` / `./assistants/rules/` 就直接做规划
- 在不读取现有 `RESULT.md` / `PLAN.md` / `fix-plan.md` 的情况下重写整个文件
- 重新分配已有任务的编号(编号一旦分配就稳定)
- 修改已完成任务的描述或状态(只能追加"修改类"任务)
- 让 `RESULT.md` 与 `PLAN.md` 不一致(交叉验证是必需步骤)
- 静默违反规范
- 一次性抛出所有澄清问题
- 修改 `./assistants/<版本号>/require/<编号>/`、`<版本号>/design/<编号>/`、`./assistants/rules/` 下的任何内容(只读)
